# P0 高危问题修复方案

## Context

项目审计报告列出 4 项 P0 高危问题：MySQL 无数据持久化、双 schema 不一致、默认弱口令、接口无后端权限校验。经三路 Explore 代理调研 + Plan 代理设计，事实与原报告有重要修正：

- **P0-1 持久化**：两份 compose 文件其实已挂载 `mysql_data:/var/lib/mysql` named volume。真实风险是文档中 `docker compose down -v` / `docker volume prune` 会清掉卷，且 named volume 不可见易误清。
- **P0-2 双 schema**：`vnerpdacahng_schema.sql`（53 表）与 `schema.ts`（41 表）几乎无表名相同。**Plan 代理发现关键前置**：vnerpdacahng_schema.sql 并非运行时全部表结构，Repository 实际用的表分散在 warehouse_inbound.sql、dcprint_migration.sql、create_*.sql 等多文件，且 `prd_standard_card` 存在冲突定义。TS 影响面实测极小——仅 `DrizzleInboundOrderRepository.ts` 导入 schema.ts，其余 Repository 用裸 SQL。
- **P0-3 弱口令**：`database/login.sql:47-52` admin123 bcrypt；`login/route.ts:180-181` 非生产环境后门 `password === '521223'`；`change-password/route.ts:13` 完全无认证可改任意账号密码（额外越权漏洞）。
- **P0-4 权限**：`middleware.ts:38` matcher 显式排除 `/api`，所有 API 绕过中间件。315 个路由仅 45 个（14%）用 `withPermission/withAuth`，270 个无校验。`withPermission`、`ROUTE_PERMISSIONS`（仅 20 条）、`PUBLIC_ROUTES` 基础设施完备。

用户决策：P0-2 以 SQL 为权威；P0-4 逐路由改造；P0-3 保留 admin + 强制改密。

实施顺序按风险与依赖排序：先堵最紧急越权洞（P0-3a），再移除后门+强制改密（P0-3b），再做持久化加固（P0-1），然后权限补全（P0-4，可分批），最后 schema 统一（P0-2，最复杂避免冲突）。

## P0-3a：修复 change-password 越权漏洞（最紧急）

**问题**：[src/app/api/auth/change-password/route.ts](file:///d:/dcprint/erp-project/src/app/api/auth/change-password/route.ts) 从请求体取 `userId` 改密码，无任何认证，任意用户可改任意账号密码。

**修复**：
1. 改用 `withPermission` 包装 handler，从 `userInfo.id` 取当前用户 ID，忽略请求体中的 `userId`（或仅 admin 可指定他人 ID）。
2. 校验旧密码（bcrypt compare）后才允许改。
3. 改密成功后清除 `first_login` 标志。

**验证**：未带 token 调用应返回 401；带普通用户 token 改他人密码应返回 403。

## P0-3b：移除后门 + first_login 强制改密

**问题**：
- [src/app/api/auth/login/route.ts](file:///d:/dcprint/erp-project/src/app/api/auth/login/route.ts):180-181 非生产环境 `password === '521223' && username === 'admin'` 后门。
- `sys_user.first_login` 字段 + `system.force_change_password` 配置存在，但后端无拦截，未改密用户仍可调用所有 API。

**修复**：
1. 删除 `login/route.ts:180-181` 后门代码块。
2. 在 `withPermission`（[src/lib/api-permissions.ts](file:///d:/dcprint/erp-project/src/lib/api-permissions.ts):144）内增加 first_login 拦截：若 `userInfo.firstLogin === true` 且请求路径不在白名单（`/api/auth/change-password`、`/api/auth/logout`、`/api/auth/user-info`），返回 403 + `passwordExpired: true`。
3. `login/route.ts` 登录成功时确认返回 `first_login` 字段给前端。
4. `database/system_config_migration.sql:137` 的 `sys.default.password = 'admin123'` 保留（新用户默认密码），但文档 [docs/04-部署文档/](file:///d:/dcprint/erp-project/docs/04-部署文档/) 增加强制改密提示。

**验证**：admin 首次登录后调用 `/api/warehouse/...` 应返回 403；调用 `/api/auth/change-password` 应成功；改密后再调用业务 API 应正常。

## P0-1：MySQL 持久化加固

**现状**：`docker-compose.yml:16` 和 `docker-compose.prod.yml:23` 已挂载 named volume。风险在文档警示与可见性。

**修复**：
1. `docker-compose.prod.yml` 改 named volume 为 bind mount：`./data/mysql:/var/lib/mysql`（数据落在项目目录可见、不会被 `volume prune` 清）。
2. `docker-compose.yml`（开发环境）保留 named volume（开发可重建）。
3. [docs/04-部署文档/Docker部署指南.md](file:///d:/dcprint/erp-project/docs/04-部署文档/Docker部署指南.md):40 移除 `docker compose down -v` 示例，改为 `docker compose down`（不带 -v）；:403 移除 `docker volume prune` 或加红色警告。
4. 在文档"数据持久化"章节增加显式提示：bind mount 路径需纳入备份计划，引用 [docs/04-运维指南/备份恢复.md](file:///d:/dcprint/erp-project/docs/04-运维指南/备份恢复.md)。

**验证**：`docker compose -f docker-compose.prod.yml up`，写入数据后 `docker compose down` 再 `up`，数据应保留。

## P0-4：API 路由权限补全（逐路由改造）

**策略**：扩充 `ROUTE_PERMISSIONS` 映射表 + 逐个改造 270 个无校验路由用 `withPermission`。

**分模块批次**（按业务风险排序，每批独立可验证）：

**批次 1 - 认证与系统（最高危）**：`/api/auth/*`、`/api/system/*`、`/api/customers`、`/api/suppliers`。修复 change-password 越权（P0-3a）一并完成。

**批次 2 - 财务与人力（敏感数据）**：`/api/finance/*`（payables/cost/receipt/receivable）、`/api/hr/*`（salary/employees/attendance）、`/api/salary`。补 `FINANCE_*`、`HR_*` 权限码到 `API_PERMISSIONS`。

**批次 3 - 仓储与库存**：`/api/warehouse/*`（batch-inventory、inbound、outbound、transfer）、`/api/inventory/*`、`/api/materials/*`。复用现有 `WAREHOUSE_*`、`INVENTORY_*` 权限码。

**批次 4 - 采购与销售**：`/api/purchase/*`（request、orders、suppliers、receipts）、`/api/orders/*`（sales、bom、delivery）、`/api/sales/*`。复用 `PURCHASE_*`、`ORDER_*`。

**批次 5 - 生产与印前**：`/api/production/*`、`/api/dcprint/*`（ink-opening、process-card、trace）、`/api/workorders/*`。复用 `WORKORDER_*`、`PRODUCTION_*`，新增 `DCPRINT_*`。

**批次 6 - 质量与设备**：`/api/quality/*`、`/api/equipment/*`、`/api/maintenance/*`。复用 `QUALITY_*`，新增 `EQUIPMENT_*`。

**每批操作模式**（重复模式，列出代表路径不穷举）：
1. 在 [src/lib/api-permissions.ts](file:///d:/dcprint/erp-project/src/lib/api-permissions.ts) 的 `ROUTE_PERMISSIONS` 补该模块所有路由前缀 + 方法→权限码映射。
2. 改路由文件：`export async function GET(req) { return withErrorHandler(req, async () => {...}) }` → `export async function GET(req: NextRequest) { return withPermission(async (req, userInfo) => {...}, { logTitle: '...', logType: '...' })(req) }`。
3. handler 内需要 userInfo 的（如取当前用户 ID、数据权限过滤）直接用入参 `userInfo`；不需要的可忽略但保留签名。
4. 数据级权限（如仅看本仓库）：在 handler 内调用 `buildDataScopeSql` / `validateResourceAccess`（[src/lib/auth.ts](file:///d:/dcprint/erp-project/src/lib/auth.ts):222-348）。

**验证**：每批改完跑 `pnpm ts-check`；用无 token / 普通用户 token / admin token 三种身份对每批代表路由发请求，验证 401/403/200。

## P0-2：双 Schema 统一（以 SQL 为权威）
