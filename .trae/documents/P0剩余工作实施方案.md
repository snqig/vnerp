# P0 高危问题剩余工作实施方案

## 概述

本方案覆盖 P0 级高危问题的剩余工作，分为两大块：
- **P0-4：API 路由权限补全** — 将 ~247 个路由改造为 `withPermission`，实现认证 + 授权 + 首次登录拦截
- **P0-2：Schema 双轨统一** — 以 SQL 文件为权威，重写 `schema.ts` 对齐 SQL，重置迁移基线

## 第一部分：P0-4 API 路由权限补全

### 1.1 当前状态

| 包装器类型 | 认证 | 权限校验 | firstLogin 拦截 | 路由数 | 处理方式 |
|-----------|------|---------|----------------|--------|---------|
| `withPermission` | ✅ | ✅ | ✅ | 59 | 无需改动 |
| `withAuthAndErrorHandler` | ✅ | ❌ | ❌ | 45 | 转为 withPermission（~43个，排除 logout/refresh） |
| `withErrorHandler` | ❌ | ❌ | ❌ | 202 | 转为 withPermission |
| MIXED (dict-data/dict-type DELETE) | 部分 | 部分 | 部分 | 2 | DELETE 方法改为 withPermission |
| 无包装器（公开路由） | — | — | — | ~52 | 逐一评估 |

**总改造量：~247 个路由**

### 1.2 公开路由例外（不加 withPermission）

以下路由必须保持公开或使用特殊认证，**不改造**：

| 路由 | 原因 |
|------|------|
| `/api/auth/login` | 登录入口，无 token |
| `/api/auth/refresh` | 使用 refresh token，非 access token |
| `/api/auth/register` | 注册入口（如开放注册） |
| `/api/health` | 健康检查 |
| `/api/linkage/*` | 已在 PUBLIC_ROUTES，公开联动校验 |
| `/api/document-number` | 已在 PUBLIC_ROUTES，单据号生成 |
| `/api/setup/create-tables` | 已由 ALLOW_SETUP_API 环境变量控制 |

### 1.6 每个路由的改造模板

**标准改造模式（withErrorHandler → withPermission）：**

```typescript
// Before
import { withErrorHandler } from '@/lib/api-response';

export const GET = withErrorHandler(async (request) => {
  // 业务逻辑
});

// After
import { withPermission } from '@/lib/api-permissions';

export const GET = withPermission(async (request, userInfo) => {
  // 业务逻辑（不变）
});
```

**写操作需加 logTitle：**

```typescript
export const POST = withPermission(
  async (request, userInfo) => { /* ... */ },
  { logTitle: '创建XXX', logType: 'business' }
);
```

## 假设与决策

1. **权限粒度**：采用模块级权限码（`module:view/create/update/delete`），不做按钮级细分。admin 角色绕过所有权限检查。

7. **cachedApiRoute**：所有包裹 cachedApiRoute 的路由移除缓存层，因缓存命中跳过认证存在安全风险。
