# v0.3.0 实施计划：打通「采购 → 入库 → 库存 → 应付」业务链

## Context

当前 ERP 系统最大的问题不是模块少，而是模块孤岛——采购、仓储、财务各自独立，跨模块没有联动。本计划打通第一条核心业务链路：**采购订单审核后可生成入库单 → 入库单审核后自动增加库存、自动生成应付单**，全程数据不落地、单据自动携带。

代码审查发现：基础设施（DDD 聚合、事件驱动、Outbox、幂等处理）已大量存在，但有 **4 条并行且均破损的入库路径**：

| 路径 | 问题 |
|------|------|
| `purchase.received` 事件路径 | PurchaseReceivedHandler 直接更新库存（违反"采购不碰库存"原则）；PurchasePayableHandler 与 FinanceVoucherHandler 重复创建应付 |
| `inbound.approved` DDD 路径 | 正确路径但 InboundOrder 聚合无 `poId`/`supplierId`，事件中 `supplierId: 0`（占位符）；FinanceVoucherHandler 的 fin_payable INSERT 引用不存在的列 |
| `with-po` 原始 SQL 路由 | INSERT 引用 `inv_inbound_item` 不存在的列（`po_line_id`/`line_no`/`material_code`）；绕过聚合/仓储/事件 |
| `audit` 原始 SQL 路由 | 完全损坏：引用 `purchase_order`（应为 `pur_purchase_order`）、`qty`（应为 `quantity`）等 |

**目标**：以 DDD 路径（`inbound.approved`）为唯一真相源，移除重复路径，修复 Schema 不匹配，实现端到端闭环。

## 实施步骤

### 步骤 1：扩展 InboundOrder 聚合根

**文件**：`src/domain/warehouse/aggregates/InboundOrder.ts`

- `InboundOrderProps` 新增 `poId?: number`、`poNo?: string`、`supplierId?: number`
- 构造函数新增对应 readonly 参数
- `create()` / `reconstitute()` 传递新字段
- `approve()` 中 `InboundOrderApprovedEvent` 的 `supplierId` 从硬编码 `0` 改为 `this.supplierId`，payload 新增 `poId`/`poNo`
- `submit()` 中 `InboundOrderSubmittedEvent` 的 `supplierId` 同理修复
- 新增 getter: `poId`、`poNo`、`supplierId`

**文件**：`src/domain/warehouse/events/InboundOrderEvents.ts`

- `InboundOrderApprovedEvent` payload 新增 `poId?: number`、`poNo?: string`
- `InboundOrderCreatedEvent` payload 新增 `poId?: number`、`poNo?: string`

### 步骤 2：更新仓储层

**文件**：`src/infrastructure/repositories/MysqlInboundOrderRepository.ts`

- `findById()`: props 映射新增 `poId: row.po_id`、`poNo: row.po_no`、`supplierId: row.supplier_id`

- `save()`: INSERT 语句新增 `supplier_id, po_id, po_no` 列和参数

**文件**：`src/infrastructure/repositories/DrizzleInboundOrderRepository.ts`

**文件**：`src/lib/db/schema.ts`

- `invInboundOrders` 定义新增 `poId`、`poNo`、`supplierId` 字段映射

### 步骤 3：实现 DDD 版 PO → 入库单转换

**文件**：`src/application/services/InboundApplicationService.ts`

- 构造函数新增 `purchaseRepo: IPurchaseOrderRepository` 参数
- 新增方法 `createInboundFromPO(params)`:
  1. 通过 `purchaseRepo.findById(poId)` 加载采购单
  2. 验证状态为 approved/partially_received
  3. 构造 `InboundOrderProps`，设置 `poId`/`poNo`/`supplierId`/`supplierName`（从采购单获取）
  4. items 从 params 映射（物料信息 + 批次号 + 数量 + 单价）
  5. 调用 `InboundOrder.create()` + `orderRepo.save()` + `persistAndPublishEvents()`
  6. 返回 `{ id, orderNo }`

**文件**：`src/app/api/warehouse/inbound/route.ts`

- `getInboundService()` 注入 `RepositoryRegistry.getPurchaseOrderRepository()`

**新建文件**：`src/app/api/warehouse/inbound/from-po/route.ts`

- `POST` 端点，接收 `{ poId, warehouseId, items[], inboundDate?, remark? }`
- 调用 `service.createInboundFromPO()`
- 返回入库单 ID 和单号

### 步骤 4：修复 FinanceVoucherHandler

**文件**：`src/application/handlers/FinanceVoucherHandler.ts`

当前 3 个 INSERT 全部有 Schema 不匹配问题。v0.3.0 最小可用方案：

1. **移除 `fin_voucher` INSERT** — `fin_voucher` 要求 NOT NULL 的 `period_code` 和 `voucher_type`，且当前 INSERT 使用的 `debit_account`/`credit_account` 等列属于 `fin_voucher_line` 表。凭证生成属于总账模块，推迟到 v0.4.0。

2. **移除 `inv_inventory_transaction` INSERT** — 使用了不存在的 `account_dr`/`account_cr` 列。库存流水记录应由 InventorySyncHandler 统一生成（后续版本补充）。

3. **修复 `fin_payable` INSERT** — 匹配权威 Schema（`vnerpdacahng_schema.sql`）：
   - `source_type` 从字符串 `'inbound'` 改为 TINYINT `2`（采购入库）
   - 删除 `supplier_name` 列（不存在）
   - 删除 `source_id` 列（不存在），用 `source_no` = `inboundNo` 替代
   - 新增 `balance` = `totalAmount`（未付余额初始化）
   - `supplier_id` 直接从事件 payload 获取（步骤 1 已修复）

修复后核心逻辑：
```typescript
async handle(event: InboundOrderApprovedEvent): Promise<void> {
  const { inboundId, inboundNo, totalAmount, supplierId } = event.payload;
  if (totalAmount > 0 && supplierId) {
    const payableNo = 'AP' + Date.now();
    await execute(
      `INSERT INTO fin_payable
       (payable_no, supplier_id, source_type, source_no, amount, paid_amount, balance, status, due_date, remark, create_time)
       VALUES (?, ?, 2, ?, ?, 0, ?, 1, DATE_ADD(CURDATE(), INTERVAL 30 DAY), ?, NOW())`,
      [payableNo, supplierId, inboundNo, totalAmount, totalAmount, `采购入库单 ${inboundNo} 自动生成`]
    );
  }
}
```

### 步骤 6：新增 PO 状态反向同步 + 清理损坏路由

**新建文件**：`src/application/handlers/PurchaseInboundSyncHandler.ts`

- 检查所有行是否完成，更新 `pur_purchase_order.status`（30→approved, 40→partially_received, 50→completed）
- 用 `IdempotentHandler` 包装防止重复累加

**文件**：`src/application/EventRegistry.ts`

- `inbound.approved` 新增 `PurchaseInboundSyncHandler` 订阅

**文件**：`src/app/api/warehouse/inbound/with-po/route.ts`

**文件**：`src/app/api/warehouse/inbound/audit/route.ts`

- 替换为 410 废弃响应桩（完全损坏，功能已被 DDD `approveOrder` 覆盖）

## 已知限制（v0.3.0 接受）

- `fin_voucher` 会计凭证暂不生成（推迟到总账模块 v0.4.0）
- `inv_inventory_transaction` 库存流水暂不生成（由 InventorySyncHandler 后续补充）
- PO 行匹配按 `materialId`（若同一 PO 有多行相同物料会累加到所有匹配行，80% 场景可接受）
- `with-po` 原始 SQL 路由保留但废弃（前端兼容性）
