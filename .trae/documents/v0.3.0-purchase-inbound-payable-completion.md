# v0.3.0 采购→入库→应付 链路打通（剩余步骤实施计划）

## 概述

继续执行 v0.3.0 「采购→入库→库存→应付」端到端业务链路打通工作。前序会话已完成 Step 1-2（InboundOrder 聚合 + 仓储层 + schema 增加 `supplierId`/`poId`/`poNo` 字段），本计划覆盖剩余 Step 3-7。

**目标**：实现从采购订单审核 → 推式生成入库单 → 入库单审核自动增加库存 + 自动生成应付单的全链路，数据全程不落地、单据自动生成、双向可追溯。

## 当前状态分析

### 已完成（Step 1-2）
- `src/domain/warehouse/aggregates/InboundOrder.ts` — 已增加 `supplierId`/`poId`/`poNo` 字段，`create()`/`approve()` 已将字段传入领域事件
- `src/domain/warehouse/events/InboundOrderEvents.ts` — `InboundOrderApprovedEvent`/`InboundOrderCreatedEvent` 已包含 `poId`/`poNo`
- `src/lib/db/schema.ts` — `invInboundOrders` 表已增加 `supplierId`/`poId`/`poNo` 列
- `src/infrastructure/repositories/MysqlInboundOrderRepository.ts` — `findById`/`findByStatus`/`save` 已支持新字段
- `src/infrastructure/repositories/DrizzleInboundOrderRepository.ts` — 同上

### 待完成（Step 3-7）

**核心问题**：当前存在 4 条并行的入库→应付路径，全部损坏：
1. `purchase.received` 事件路径（PurchaseReceivedHandler + PurchasePayableHandler）— 与入库领域重复
2. `inbound.approved` DDD 路径（InventorySyncHandler + FinanceVoucherHandler）— FinanceVoucherHandler 的 3 个 INSERT 全部 schema 不匹配
3. `with-po/route.ts` 原生 SQL 路径 — 引用不存在的 `po_line_id` 列
4. `audit/route.ts` 原生 SQL 路径 — 引用 `purchase_order`（实际为 `pur_purchase_order`）、`qty`（实际为 `quantity`）

**解决方案**：统一收敛到 DDD 路径（`inbound.approved`），废弃其他 3 条。

   - 入参：`{ poId: number; warehouseId: number; items: Array<{ lineNo: number; materialId; materialCode; materialName; materialSpec; unit; batchNo; quantity; unitPrice; warehouseLocation?; produceDate? }> }`
   - 流程：
     1. `purchaseRepo.findById(poId)` 加载采购单

        - `items` 从 params 映射
     5. 调用 `InboundOrder.create(props)` + `orderRepo.save(order)` + `persistAndPublishEvents(order.id, order)`
     6. 返回 `{ id, orderNo }`
   - 错误处理：采购单不存在 → `NotFoundError`；状态不允许 → `DomainError`；超收 → `DomainError`

### Step 4：修复 FinanceVoucherHandler

**文件**：`src/application/handlers/FinanceVoucherHandler.ts`

当前 3 个 INSERT 全部损坏，修复方案：

**Bug 1：fin_voucher INSERT（行 14-30）— 删除**
- 原因：引用 `debit_account`/`credit_account`/`cost_price`/`quantity`/`batch_no`/`material_id`/`material_name`/`warehouse_id`（均不存在于 `fin_voucher`，属于 `fin_voucher_line`）；缺少 NOT NULL 列 `period_code`/`voucher_type`
- 决策：**删除整个 fin_voucher INSERT 块**。凭证体系需要完整的 `fin_voucher` + `fin_voucher_line` 双表写入 + 会计期间校验，属于 Phase 3 财务模块范畴，v0.3.0 暂不实现。仅保留应付单生成。

**Bug 2：inv_inventory_transaction INSERT（行 32-47）— 删除**
- 原因：引用 `account_dr`/`account_cr`（不存在于 `inv_inventory_transaction` 表）
- 决策：**删除整个 inv_inventory_transaction INSERT 块**。库存事务记录属于库存领域，应由 InventorySyncHandler 负责（后续迭代补充），不应放在 FinanceVoucherHandler。当前 InventorySyncHandler 已正确更新 `inv_inventory` + `inv_inventory_batch`，库存数据一致性有保障。

**Bug 3：fin_payable INSERT（行 58-70）— 修复**
- 原因：
  - `supplier_name` 列不存在（`fin_payable` 只有 `supplier_id`）
  - `source_id` 列不存在（`fin_payable` 只有 `source_no`，无 `source_id`）
  - `source_type='inbound'` 类型错误（`fin_payable.source_type` 是 TINYINT，1=采购订单）
  - 缺少 `balance` 列（虽有 DEFAULT，但应显式设置为 `amount` 保持正确语义）
- 修复为：
  ```sql
  INSERT INTO fin_payable
    (payable_no, supplier_id, source_type, source_no, amount, paid_amount, balance, status, due_date, remark, create_time)
  VALUES (?, ?, 1, ?, ?, 0, ?, 1, DATE_ADD(CURDATE(), INTERVAL 30 DAY), ?, NOW())
  ```
  参数：`[payableNo, supplierId, inboundNo, totalAmount, totalAmount, remark]`
  - `source_type=1`（采购订单来源）
  - `source_no=inboundNo`（入库单号作为来源单号，可追溯）
  - `balance=totalAmount`（初始未付余额=应付金额）
  - 移除 `supplier_name` 和 `source_id`

**同时移除**：行 50-55 的 `pur_supplier` 查询（因为 `supplierId` 已从事件 payload 直接获得，无需按名称反查）

**最终 handler 结构**：
```typescript
export class FinanceVoucherHandler implements EventHandler<InboundOrderApprovedEvent> {
  async handle(event: InboundOrderApprovedEvent): Promise<void> {
    const { inboundId, inboundNo, totalAmount, supplierId, supplierName } = event.payload;

    if (totalAmount <= 0 || !supplierId) return;

    await transaction(async (conn) => {
      const payableNo = 'AP' + Date.now();
      await conn.execute(
        `INSERT INTO fin_payable
         (payable_no, supplier_id, source_type, source_no, amount, paid_amount, balance, status, due_date, remark, create_time)
         VALUES (?, ?, 1, ?, ?, 0, ?, 1, DATE_ADD(CURDATE(), INTERVAL 30 DAY), ?, NOW())`,
        [payableNo, supplierId, inboundNo, totalAmount, totalAmount, `采购入库单 ${inboundNo} 自动生成`]
      );
    });

    secureLog('info', 'Payable created for inbound order', { inboundNo, totalAmount, supplierId });
  }
}
```

## 假设与决策

1. **凭证生成延后**：v0.3.0 不实现 `fin_voucher` + `fin_voucher_line` 凭证生成，因为需要会计期间（`fin_period`）校验和科目体系，属于 Phase 3 财务模块。当前只生成应付单。
2. **库存事务记录延后**：`inv_inventory_transaction` 记录不在本次实现，后续由 InventorySyncHandler 补充（需修正列名）。
3. **source_type 枚举**：`fin_payable.source_type=1` 表示「采购订单」来源，入库单视为采购流程的一部分。
4. **不触发循环事件**：PurchaseInboundSyncHandler 更新采购单状态时，不调用 `purchaseOrder.receive()` 的领域方法（会触发 `purchase.received` 事件导致循环），而是直接 SQL 更新 `received_qty` 和 `status`。
5. **保留废弃路由占位**：with-po 和 audit 路由不删除文件，改为返回 410，避免前端调用直接 404。
6. **前端适配延后**：前端 `with-po/page.tsx` 页面需后续改为调用 `/api/warehouse/inbound/from-po`，本次只做后端链路。
