**改动清单**：

1. **`recordTransaction` 方法（L85-134）**：
   - 表名 `inv_inventory_transaction` → `inv_inventory_log`
   - 列名映射：
     - `trans_no` → `business_no`
     - `trans_type` ('in'/'out'/etc) → `operation_type` (TINYINT)

## 三、假设与决策

1. **Schema 是唯一真相来源**：Phase A 创建的 14 张表 + 列对齐是权威定义，代码向 schema 对齐。
2. **inv_inventory_log 不可修改**：schema 注释明确"不可修改、不可删除"，因此 `reverseTransaction` 改为"插入反向流水"模式，不标记原流水。
3. **Phase B 优先级最高**：修复后入库→库存同步恢复，是当前唯一能工作的业务闭环。
4. **与 P0-4 agent 协调**：4 个后台 agent 仍在运行（仓储/采购销售/生产印前/质量设备等路由权限转换），Phase E-2 的 `withPermission` 库修复需在 agent 完成后统一处理冲突。
5. **Phase C/D 可并行**：领域模型与仓储实现可同时开发，无强依赖。
6. **Phase E-3 路由迁移到 DDD 为可选**：先保证路由可用（E-1/E-2），DDD 迁移可延后。
7. **不新增 schema 表**：Phase A 已补全所有缺失表，后续仅在代码侧对齐。
