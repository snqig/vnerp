## 总体架构

```
App Service (事务内 saveEvents 到 outbox，不再 setImmediate publish)
    ↓
domain_event_outbox 表 [status=pending]
    ↓
OutboxPoller (SELECT FOR UPDATE SKIP LOCKED 原子 claim, 5s)
    ↓
StreamPublisher (XADD erp:events * eventType payload)
    ↓ outbox [status=dispatched]
StreamConsumer (消费组 erp-handlers, XREADGROUP COUNT 50 BLOCK 2000)
    ↓
IdempotencyGuard (INSERT sys_event_processed 失败则跳过)
    ↓
本地 InMemoryEventBus.publish → Handler 执行
    ↓
XACK + outbox.markAsProcessed
失败 → 留 PEL → XAUTOCLAIM(60s) 重试 → MAX_RETRY 后 XADD 死信流
```

**多实例语义**：outbox 表为事务真源，Redis Stream 为分发通道；单消费组 `erp-handlers`，每事件由组内一个消费者处理（at-least-once）；`sys_event_processed` 表保证 handler 幂等；Redis 缓存为共享层，单次 `delete` 跨实例生效。

**保留 `InMemoryEventBus`**：仅作为 `StreamConsumer` 内部本地 handler 分发器，不再作为发布入口。

### 阶段 2：缓存击穿/雪崩防护（CacheGuard）

**新增文件**：
- [src/infrastructure/cache/CacheGuard.ts](file:///d:/dcprint/erp-project/src/infrastructure/cache/CacheGuard.ts) — singleflight + SETNX 互斥锁

接口签名：
```typescript
export class CacheGuard {
  constructor(private cache: CacheManager, private redis?: Redis);
  async getOrLoad<T>(key: string, ttl: number, loader: () => Promise<T>): Promise<T>;
}
```
逻辑：① `cache.get` 命中返回；② `SET key:lock token EX 3 NX` 抢锁者执行 loader，其余 `sleep(50ms)` 轮询 `cache.get`；③ loader 完成后 `cache.set(key, val, ttl + random(0..30))` 防雪崩，`del key:lock`；④ 锁超时后竞争者可抢锁回源；⑤ 空值缓存 60s 防穿透。Redis 不可用时降级为本地 `Map` 锁。

```typescript
export class IdempotencyGuard {
  async tryAcquire(eventId: number, handlerName: string): Promise<boolean> {
    try {
      await execute(
        'INSERT INTO sys_event_processed (event_id, handler_name) VALUES (?, ?)',
        [eventId, handlerName]
      );
      return true;
    } catch (e) {
      if (e.code === 'ER_DUP_ENTRY') return false;
      throw e;
    }
  }
}
```

### 阶段 5：Redis Stream 事件分发

**新增文件**：
- [src/infrastructure/event-bus/StreamPublisher.ts](file:///d:/dcprint/erp-project/src/infrastructure/event-bus/StreamPublisher.ts)：`XADD erp:events * outboxId <id> eventType <type> payload <json>`，复用 RedisCacheManager 的连接或新建专用 client
- [src/infrastructure/event-bus/StreamConsumer.ts](file:///d:/dcprint/erp-project/src/infrastructure/event-bus/StreamConsumer.ts)：

```typescript
export class StreamConsumer {
  private consumerId = `${hostname()}:${process.pid}`;
  async start() {
    await this.ensureGroup(); // XGROUP CREATE erp:events erp-handlers $ MKSTREAM
    this.loop();
    this.reclaimLoop(); // 每 30s XAUTOCLAIM erp:events erp-handlers <consumerId> 60000 0
  }
  private async loop() {
    while (this.running) {
      const entries = await this.redis.xreadgroup(
        'GROUP', 'erp-handlers', this.consumerId,
        'COUNT', 50, 'BLOCK', 2000, 'STREAMS', 'erp:events', '>'
      );
      for (const [id, fields] of entries || []) {
        try {
          await this.dispatch(fields); // 本地 eventBus.publish + IdempotencyGuard
          await this.redis.xack('erp:events', 'erp-handlers', id);
        } catch (e) {
          // 不 ACK，留 PEL 等待 XAUTOCLAIM 重试
        }
      }
    }
  }
}
```

**修改文件**：
- [OutboxPoller.ts](file:///d:/dcprint/erp-project/src/infrastructure/event-bus/OutboxPoller.ts) claim 后调用 `StreamPublisher.publish(event)` 而非 `eventBus.publish`，成功后 `markAsDispatched`
- [AppInitializer.ts](file:///d:/dcprint/erp-project/src/application/AppInitializer.ts) db 模式启动 `StreamConsumer`；Redis 不可用时降级为 `OutboxPoller` 直接 `eventBus.publish`（fallback flag）

**死信处理**：超过 `MAX_RETRY_COUNT` 的 PEL entry → `XADD erp:events:deadletter` + `XACK` 原 entry + `outbox.markAsDeadLetter`

**验证**：单事件端到端延迟 < 2s；杀掉消费者进程后重启，PEL 中未 ACK 事件在 60s 内被回收重处理；死信流可查。

**回滚**：`EVENT_BUS_TYPE=db` + 禁用 StreamConsumer，退回 OutboxPoller 直接 publish 模式（阶段 3 已实现的原子 claim 仍保留）。

## 关键决策（已与用户确认）

| 决策点 | 选定方案 |
|---|---|
| 幂等性改造范围 | 同步改造，本期覆盖 8 个非幂等 handler + 新增 sys_event_processed 表 |
| 消费时机统一 | 删除应用服务 `publishOutboxEventsAsync`，统一走 OutboxPoller（接受 ~5s 延迟） |
| 消费组命名 | `erp-handlers`，消费者 ID 用 `hostname:pid` |
| InMemoryEventBus 去留 | 保留，作为 StreamConsumer 内部本地分发器，不再作为发布入口 |
| 原子 claim 方案 | `SELECT FOR UPDATE SKIP LOCKED`（MySQL 8.0 已确认部署） |
| bom-expansion 事件失效 | 接入 CacheInvalidationHandler，`standard-card.confirmed/obsoleted` → `deletePattern('bom:expansion:*')` |

## 不做的事（本期范围外）

- **Saga 补偿事务**：跨聚合的编排式补偿不在本期，级联 publish（如 SalesToWorkOrderHandler）仅保证事件不丢，不保证跨聚合原子性
- **外部系统集成适配层**：短信/邮件/支付等外部对接不在本期
- **Outbox 表纳入 Drizzle schema**：本期保持 raw SQL migration 管理，避免与 drizzle-kit 迁移工具冲突，后续单独评估
- **事件版本化与 schema 演进**：payload JSON 无版本字段，本期不引入
- **事件处理器并发竞争的 FOR UPDATE 改造**：阶段 0 用 upsert 解决 InkCost/ScreenPlateCost，其他 handler 的并发问题另立专项
- **CI/CD 流水线补充**：评估文档提到的 GitHub Actions 缺失属工程基建层，不在本期基础设施升级范围
