# 质量管理模块升级实施方案

## Context

质量管理模块当前完成度 4/10，主要问题：
1. **领域层缺失**：`src/domain/quality/` 不存在，无聚合根/值对象/事件，违反项目 DDD 硬约束
2. **表名不一致**：`schema.sql` 定义 `qc_unqualified`，但 API 路由使用运行时创建的 `qc_unqualified_handle`（由 `init/supplement-tables/route.ts` 创建），`spc-analysis.ts` 的 `handleInspectionFailure` 又写入 `qc_unqualified` —— 两个表数据不互通
3. **无状态机**：`unqualified/route.ts` PUT 直接更新 `handle_status` 字段，无流转校验（待处理→已完成这种跳跃也能成功）
4. **SPC 算法已完整**：`src/lib/spc-analysis.ts` 已实现 XbarR/Pareto/PChart + Cp/Cpk/Pp/Ppk + 失败自动建单 + 批次隔离 + 通知，**无需重写**

目标：补齐领域层、统一表名、引入状态机校验、补单元测试，使质量模块对齐采购/销售模块的 DDD 规范。

## 关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 权威表名 | `qc_unqualified`（schema.sql 定义） | 已在权威 schema 中，符合"SQL 与 ORM 统一"硬约束 |
| 字段吸收 | 从 `qc_unqualified_handle` 吸收 7 个字段 | `handle_no`/`material_code`/`material_name`/`handle_type`/`responsible_dept`/`responsible_person`/`cost_amount` |
| 状态码 | 沿用 DB TINYINT 1/2/3 + 领域字符串 `pending`/`handling`/`completed` | 对齐 PurchaseOrderStatus 模式 |
| 处置方式 | `handle_type` 1-返工/2-报废/3-让步接收/4-退货 | 已是 `qc_unqualified_handle` 实际用法 |
| API DTO | 保留 `handle_status`/`handle_type`/`handle_no` 字段名 | 前端 `unqualified/page.tsx` 已用这些字段，避免破坏 |
| 事件命名 | `quality.unqualified.created`/`.handling_started`/`.completed` | 对齐 `purchase.created` 命名风格 |

## 实施阶段

### Phase 1: Schema 对齐（P0）
**目标**：以 `qc_unqualified` 为唯一权威表，吸收 `qc_unqualified_handle` 字段。

**新增/修改文件**：
- 新建 `database/migrations/020_align_qc_unqualified_schema.sql`（幂等守卫 + ALTER + 数据合并 + DROP 旧表）
- 修改 `database/vnerpdacahng_schema.sql` 同步 `qc_unqualified` 表定义
- 修改 `src/lib/spc-analysis.ts` 的 `handleInspectionFailure` INSERT 列集（增加 `handle_no`/`material_code`/`material_name`/`handle_type`/`status=1`）
- 清理 `database/seed_part4_dcprint_qc_eqp.sql`、`database/seed_business_data.sql`、`src/app/api/init/supplement-tables/route.ts`、`src/app/api/init/seed-data/route.ts`、`src/app/api/init/full-business-seed/route.ts`、`src/app/api/init/settings-seed/route.ts`、`src/app/api/warehouse/production-inbound/route.ts` 中 `qc_unqualified_handle` → `qc_unqualified` 引用

**关键实现点**：
- ALTER 幂等模式：`INFORMATION_SCHEMA.COLUMNS` 守卫 + PREPARE/EXECUTE 动态 SQL（参照 Migration 017/018 模式）
- 若 `qc_unqualified_handle` 存在数据：`INSERT INTO qc_unqualified SELECT ... FROM qc_unqualified_handle` 合并后 `DROP TABLE qc_unqualified_handle`
- 废弃 `handle_method` 字段（旧 schema 定义），统一用 `handle_type`

**验证**：`pnpx tsx scripts/migrate.ts up`；`SELECT COUNT(*) FROM qc_unqualified_handle` 应报错；`tsc --noEmit` 0 错误。

### Phase 2: 领域层（P0）
**目标**：按 `PurchaseOrder` 范式建立 `src/domain/quality/`。

**新增文件**：
- `src/domain/quality/value-objects/UnqualifiedStatus.ts`
  - 状态：`pending(1)`/`handling(2)`/`completed(3)`
  - `transitions`: `pending→handling`, `handling→completed`（单向）
  - `canStartHandle()`/`canComplete()`/`canEdit()`/`canDelete()` 操作权限
  - `fromDbCode/toDbCode/label/equals`
- `src/domain/quality/value-objects/HandleMethod.ts`
  - 枚举：`rework(1)`/`scrap(2)`/`concession(3)`/`return(4)`
  - `fromDbCode/toDbCode/label`
- `src/domain/quality/aggregates/UnqualifiedProduct.ts`
  - `create(props)`：校验 `unqualifiedQty>0`、`inspectionId>0`，初始 `pending`，push `UnqualifiedCreated` 事件
  - `reconstitute(props)`：仓储还原，不触发事件
  - `startHandle()`：`transitionTo('handling')` + push `HandlingStarted`
  - `completeHandle(handlerId, handleResult, costAmount)`：要求 `responsibleDept/Person` 已赋值，`transitionTo('completed')` + push `UnqualifiedCompleted`
  - `assignResponsible(dept, person)`：独立方法
  - `canEdit()=pending`、`canDelete()=pending`
- `src/domain/quality/events/UnqualifiedEvents.ts`
  - `UnqualifiedCreatedEvent` (`quality.unqualified.created`)
  - `HandlingStartedEvent` (`quality.unqualified.handling_started`)
  - `UnqualifiedCompletedEvent` (`quality.unqualified.completed`)
- `src/domain/quality/repositories/IUnqualifiedRepository.ts`
  - `findById`/`save`/`updateStatus`/`softDelete`
- `src/domain/quality/entities/InspectionRecord.ts`（简单实体，仅 `result: 1|2|3`，无状态机）

**参照文件**：
- `src/domain/purchase/aggregates/PurchaseOrder.ts`（聚合根范式）
- `src/domain/purchase/value-objects/PurchaseOrderStatus.ts`（值对象范式）
- `src/domain/purchase/events/PurchaseOrderEvents.ts`（事件范式）

**验证**：`tsc --noEmit`；Phase 4 测试就绪后 `pnpm vitest run tests/unit/domain/quality`。

### Phase 3: 状态机集成（P1）
**目标**：`unqualified/route.ts` 改走应用服务，PUT 强制状态流转校验。

**新增/修改文件**：
- 新建 `src/infrastructure/repositories/MysqlUnqualifiedRepository.ts`
- 新建 `src/application/services/UnqualifiedApplicationService.ts`
- 重写 `src/app/api/quality/unqualified/route.ts`
- 修改 `src/application/EventRegistry.ts` 注册 `quality.unqualified.*` 事件

**关键实现点**：
- 应用服务仿 `PurchaseApplicationService`：`create()`/`startHandle()`/`completeHandle()` 内 `repo.updateStatus(id, newStatus, prevStatus)` + `transaction(conn => saveEvents(conn,'UnqualifiedProduct',id,events))`；`affectedRows===0` 抛 `VersionConflictError`
- PUT 入参按 `action: start|complete` 分派，不再直接写 `handle_status`
- DELETE 走 `canDelete()` 守卫（仅 pending 可删）
- API DTO 保留 `handle_status`/`handle_type`/`handle_no` 字段名兼容前端
- 非法状态流转返回 422 + i18n 错误消息

**验证**：`tsc --noEmit`；手测 PUT `pending→completed` 应返回 422；`pnpm vitest run` 全量回归。

## 依赖关系与执行顺序

```
Phase 1 (Schema) → Phase 2 (Domain) → Phase 4 (Tests，边写边测)
                                   ↘ Phase 3 (API 集成)
```

建议执行顺序：1 → 2 → 4 → 3（边写领域层边补测试，最后接入路由）

## 风险点

1. **迁移风险**：`qc_unqualified_handle` 若已有生产数据，DROP 前必须先合并到 `qc_unqualified`；幂等守卫用 `INFORMATION_SCHEMA.COLUMNS` 检查列存在性
2. **向后兼容**：前端 `unqualified/page.tsx` 使用 `handle_status`/`handle_type`/`handle_no`/`handle_result` 字段名 —— API DTO 保留这些字段名，领域层内部用 `status`/`handleMethod`
3. **`handleInspectionFailure` 事务边界**：当前在调用方事务内执行，改字段后需保证仍可回滚
4. **事件处理器**：Phase 3 注册事件后需实现至少一个处理器（如通知库存解冻），否则事件堆积在 outbox
