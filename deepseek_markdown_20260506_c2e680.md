# VNERP 项目问题报表

> 项目地址：https://github.com/snqig/vnerp  
> 分析时间：2026-05-06  
> 项目定位：丝网印刷行业开源 ERP 系统（Next.js + MySQL）

## 一、总体评估

该项目是一个 **AI 快速生成的原型项目**，功能规划宏大，但工程成熟度较低，多数模块处于 CRUD 占位阶段，**不能直接用于真实印刷厂的生产环境**。

| 评估维度     | 成熟度     | 核心问题                                       |
| ------------ | ---------- | ---------------------------------------------- |
| 业务深度     | 极低       | 印刷特有逻辑（网版/油墨/色卡/工序追溯）缺失    |
| 数据一致性   | 低         | 无事务、无锁、多表联动脆弱                     |
| 安全性       | 低         | API 权限缺失、JWT 隐患、SQL 注入风险           |
| 可维护性     | 低         | AI 生成痕迹严重、代码重复、文档与实现脱节      |
| 测试覆盖     | 极低       | 仅有示例测试，无业务单元测试                   |
| 生产就绪度   | 不可用     | 核心流程断裂，无法支撑真实车间作业             |

---

## 二、核心架构与功能问题

### 2.1 业务流程断裂（高风险）

**症状**：  
- 质检合格后无法自动触发入库  
- 销售订单确认后无法自动生成生产工单  
- 缺少状态驱动的联动逻辑  

**根本原因**：没有引入事件驱动架构或工作流引擎。当前各模块 API 彼此独立，依赖人工操作衔接。

**影响**：ERP 核心闭环无法实现，系统只能作为数据记录工具，不能驱动业务。

### 2.2 API 数据校验缺失（高风险）

**症状**：  
- 绝大多数 API 不校验请求体字段类型、格式、范围  
- 无统一的错误返回格式  

**影响**：无效、错误数据可直接写入数据库，导致业务逻辑异常甚至系统崩溃。

### 2.3 操作日志无记录（中风险）

**用户反馈**：`http://localhost:5000/settings/oper-log` 没有记录。  
**原因**：操作日志模块未实现或未正确接入业务操作。  
**影响**：无法审计、无法追溯异常操作，违反合规需求。

---

## 三、仓库模块问题

### 3.1 FIFO 推荐与执行脱节（高风险）

**优点**：提供了增强版 FIFO 推荐接口（考虑效期、兼容性、风险评分）。  

**严重缺陷**：  
- 推荐接口只返回建议，实际出库操作 **并未强制使用推荐结果**  
- 出库扣减 `available_qty` 时 **没有数据库事务**，并发场景下库存不准确  
- 缺少批次流水表，无法追溯每次出库的成本与批次变动  

**影响**：账实不符、过期物料仍可能被发出、成本核算混乱。

### 3.2 二维码与标签功能严重不足（极高风险）

**现状**：  
- 只存在基础的 QR 生成，未与业务实体（批次、库位、工单）绑定  
- 无标签模板管理、无批量打印、无热敏打印机适配（ZPL/TSPL）  
- FIFO 推荐结果不能一键打印“优先消耗”标签  
- 无扫码入库、扫码出库、扫码盘点接口  

**影响**：**印刷车间几乎无法落地**——工人无法通过扫码作业，追溯成为空谈。标签是印刷厂仓库的核心入口，缺失此功能等于系统不可用。

## 四、生产模块问题

### 4.1 仅是 CRUD 框架（高风险）

| 子模块       | 问题描述                                                         |
| ------------ | ---------------------------------------------------------------- |
| 工单/工艺卡  | 字段映射混乱，无 BOM 支持，无版本控制                            |
| 排产         | 无自动排程算法（甘特图、资源冲突检测、产能负载），仅手动插入计划 |
| 领料/退料    | 无批次精细控制（油墨余量、网版寿命）                             |
| 追溯         | 虽有 `trace/` 路由，但仅能查询简单关系，无全链路追溯              |
| 质量集成     | 生产完工与质量检验无联动，质检结果不能自动影响工单状态           |

**影响**：无法满足印刷行业多工序、多机台、批次追溯的核心需求。

### 4.2 数据库与事务缺陷

- 使用 raw SQL 拼接，虽有一定参数化，但缺乏 ORM 类型安全  
- 复杂业务操作（如领料 + 扣库存 + 更新工单状态）**没有数据库事务**  
- 无乐观锁或悲观锁，高并发下超卖、状态错乱风险高  

---

## 五、安全与权限问题

### 5.1 API 权限控制缺失（高危）

- 文档列出了大量 API 端点，但未说明每个 API 所需角色  
- `register` 端点默认开放，可能被恶意注册  
- 未发现对 `/api/*` 的通用权限中间件  

### 5.2 JWT 实现隐患

- 未检查是否使用强密钥、合理有效期、安全的签名算法  
- 缺少 refresh token 机制  

### 5.3 依赖漏洞风险

- 项目使用 Next.js 14+ 但未注明具体版本，**CVE-2025-66478** (RCE) 可能影响未及时升级的版本  

### 5.4 敏感信息泄露

- 响应头可能暴露技术栈版本  
- `.env.example` 存在，但生产环境配置管理不明  

---

## 六、代码质量与可维护性问题

### 6.1 AI 生成痕迹严重

- 仓库中包含 `.agentic-tools-mcp`、`.claude`、`.continue`、`.windsurf` 等大量 AI 工具配置  
- 提交历史几乎全是 `feat: 自动提交...` 批量提交，无法进行代码审查  
- 存在大量临时文件：`temp_api_response.json`、`temp_employees.json`、`test-results.json` 等未清理  

### 6.2 文档与实现严重脱节

- README 列出的 API 和页面有三分之一实际不存在或只是占位  
- 数据库 schema 分散在多个 SQL 文件中，缺少完整、权威的定义  

### 6.3 测试覆盖几乎为零

- 虽有 Playwright、Vitest 配置文件，但实际业务测试用例极少  
- 无单元测试覆盖核心逻辑（如 FIFO 推荐、库存扣减）

## 九、结论

**VNERP 目前不适合直接用于生产**。它更像一个由 AI 生成的**高概念原型**，展示了功能蓝图，但在数据一致性、安全、行业适配、可维护性四个关键维度存在重大缺陷。若要在真实丝网印刷企业落地，必须按照上述优先级进行**系统级重构**，尤其是补全二维码/标签功能与事务控制。

**最终建议**：  
- 若为学习/演示用途 → 可继续完善文档和简单功能  
- 若为实际生产部署 → 建议基于成熟 ERP 开源框架（如 Odoo）二次开发，或投入 3-6 个月对 VNERP 进行重写核心模块
