# 盘点与调拨设计

## 1. 概述

盘点与调拨是仓库库存准确性的保障机制。盘点用于核对系统账面库存与实物库存差异，调拨用于跨仓库或跨库位的库存转移。本文档基于 `src/app/api/warehouse/stocktaking/route.ts`、`src/app/api/warehouse/stocktaking/[id]/scan/route.ts` 现状编写。

## 3. 盘点流程

```
创建盘点单 (status=1)
   ↓ 锁定仓库库存 (inv_inventory.stocktaking_flag = 1)
   ↓ 自动生成明细（从 inv_inventory 拉取 system_qty）
扫码盘点 (POST /scan)
   ↓ 录入 actual_qty，计算 diff_qty
提交盘点 (action=submit, status=1→2)
   ↓ 校验所有明细已盘点（actual_qty > 0）
审批通过 (action=approve, status=2→3)
   ↓ 事务内更新 inv_inventory.quantity
   ↓ 记录 inv_inventory_log 流水
   ↓ 解锁库存 (stocktaking_flag = 0)
审批驳回 (action=reject, status=2→1)
取消盘点 (action=cancel, status=1→4)
   ↓ 解锁库存
```

## 4. 库存锁定机制

盘点创建时立即锁定整个仓库的库存：

```sql
UPDATE inv_inventory SET stocktaking_flag = 1 WHERE warehouse_id = ?;
```

锁定期间：
- 不允许该仓库的入库/出库操作
- 不允许该仓库的再次盘点
- 解锁发生在审批完成、驳回（保持锁定继续盘点）、取消时

## 5. 扫码盘点

文件：`src/app/api/warehouse/stocktaking/[id]/scan/route.ts`

### 5.1 扫码流程

```
1. 扫描二维码 qr_code
2. 查询 wh_inventory 匹配 qr_code + warehouse_id
3. 查询 inventory_check_items 校验是否在盘点范围内
4. 录入 actual_quantity
5. 计算 difference = actual - book
6. 检查 material_splits 表获取 split_flag 与 parent_qr_code
7. 更新明细状态为已盘点
8. 返回盘点进度（已盘/总数）
```

### 5.2 小料识别

通过 `material_splits` 表识别小料：

```sql
SELECT * FROM material_splits WHERE child_qr_code = ?;
```

若查询有结果，则 `split_flag = 1`（小料），并返回 `parent_qr_code`；否则 `split_flag = 0`（整料）。

## 6. 审批与库存调整

审批通过时使用数据库事务（`transaction`）保证一致性：

```typescript
await transaction(async (conn) => {
  const diffItems = await conn.execute(
    `SELECT * FROM inv_stocktaking_item WHERE taking_id = ? AND diff_qty != 0`, [id]
  );

  for (const item of diffItems) {
    // 负库存防护
    if (currentStock.quantity + item.diff_qty < 0) {
      throw new Error('盘点后将产生负库存');
    }
    // 更新库存
    await conn.execute(
      `UPDATE inv_inventory SET quantity = quantity + ? WHERE material_id = ? AND warehouse_id = ?`,
      [item.diff_qty, item.material_id, check.warehouse_id]
    );
    // 记录流水
    await conn.execute(
      `INSERT INTO inv_inventory_log (..., change_type='STOCKTAKING', ...) VALUES (...)`,
      [...]
    );
  }
  // 更新盘点状态并解锁
  await conn.execute(`UPDATE inv_stocktaking SET status = 3, approver_id = ? WHERE id = ?`, [approver_id, id]);
  await conn.execute(`UPDATE inv_inventory SET stocktaking_flag = 0 WHERE warehouse_id = ?`, [warehouse_id]);
});
```

## 7. 调拨设计

### 7.1 调拨流程

调拨通过 `inv_inventory_transaction` 流水表记录，状态机使用 `WarehouseStateMachine`（见 [26-仓库管理核心原则专项说明](./26-仓库管理核心原则专项说明.md)）：

```
创建调拨单 (status=draft)
   ↓ submit (draft→pending)
确认调拨 (pending→completed)
   ↓ 事务内执行：
   ↓ 源仓库 inv_inventory.quantity -= 调拨数量
   ↓ 目标仓库 inv_inventory.quantity += 调拨数量
   ↓ 写入两条 inv_inventory_transaction 流水（transfer_out / transfer_in）
```

### 7.2 库存交易类型

| 类型 | 说明 |
|---|---|
| `inbound` | 入库 |
| `inbound_cancel` | 入库撤销 |
| `outbound` | 出库 |
| `outbound_cancel` | 出库撤销 |
| `transfer_in` | 调拨入库 |
| `transfer_out` | 调拨出库 |
| `adjust_add` | 盘盈 |
| `adjust_sub` | 盘亏 |

## 9. 工程约定

1. **强制锁定**：盘点期间必须锁定仓库，避免并发出入库造成数据不一致
2. **负库存防护**：审批时校验 `quantity + diff_qty >= 0`，否则拒绝
3. **事务一致性**：库存调整与流水记录必须在同一事务内完成
4. **软删除**：盘点单使用 `deleted` 字段，仅已取消状态可删除
5. **审批阈值**：`stocktaking_diff_threshold` 配置项控制差异审批阈值
