# 仓库管理核心原则专项说明

## 1. 概述

仓库管理是 ERP 系统的基础设施，承担库存准确性、批次追溯、并发安全的核心职责。本文档专项说明仓库模块的核心原则，基于 `src/lib/warehouse-core.ts`、`src/lib/fifo-allocation.ts`、`src/lib/warehouse-state-machine.ts` 现状编写。

## 2. FIFO 批次分配原则

### 2.1 分配优先级

文件：`src/lib/warehouse-core.ts` 的 `allocateFIFO` 方法

```
1. 余料优先：split_flag = 2（生产剩余）最先消耗
2. 近效期优先：expire_date 升序
3. 先入先出：inbound_date 升序
4. 跳过锁定：stocktaking_flag = 1 不参与分配
5. 跳过隔离：quarantine_status = 1 不参与分配
6. 跳过检验中：inspection_status = 'pending' 不参与分配
```

### 2.2 FIFO 覆盖日志

当仓管员手动选择非 FIFO 推荐批次时，必须记录覆盖原因：

```typescript
await logFIFOOverride({
  material_id,
  recommended_batch: 'B001',
  actual_batch: 'B005',
  operator_id,
  reason: 'B001 位于高层货架，取货不便',
  work_order_no,
});
```

写入 `fifo_override_log` 表，用于审计与流程优化。

### 2.3 批次状态字段

`inv_inventory_batch` 表关键字段：

| 字段 | 说明 |
|---|---|
| `available_qty` | 可用数量（实际可分配） |
| `reserved_qty` | 预占数量（已审批未发料） |
| `total_qty` | 总数量 = available + reserved |
| `version` | 乐观锁版本号 |
| `inspection_status` | 检验状态：`pending`/`passed`/`failed` |
| `quarantine_status` | 隔离状态：0=正常 / 1=隔离 |
| `alert_level` | 预警级别：`normal`/`warning`/`expired` |

## 3. 乐观锁与并发控制

### 3.1 乐观锁实现

文件：`src/lib/fifo-allocation.ts`

```typescript
// 分配时使用乐观锁
const result = await execute(
  `UPDATE inv_inventory_batch
   SET available_qty = available_qty - ?,
       reserved_qty = reserved_qty + ?,
       version = version + 1,
       update_time = NOW()
   WHERE id = ? AND version = ? AND available_qty >= ?`,
  [allocateQty, allocateQty, batchId, currentVersion, allocateQty]
);

if (result.affectedRows === 0) {
  throw new Error('批次版本已变更，请重试');
}
```

### 3.2 重试机制

乐观锁冲突时自动重试（最多 3 次），每次重新读取批次版本：

```
1. 读取批次（含 version）
2. 计算分配数量
3. 尝试 UPDATE（WHERE version = ?）
4. 若 affectedRows = 0，回退重试
5. 3 次失败后抛出并发冲突错误
```

## 4. 精度计算原则

所有库存数量与金额计算使用 `Decimal.js`：

```typescript
import Decimal from 'decimal.js';

// 正确：使用 Decimal
const totalCost = new Decimal(unitPrice).mul(quantity).toNumber();

// 错误：浮点运算
const totalCost = unitPrice * quantity;  // 禁止
```

精度规则：
- 数量：保留 4 位小数（支持克/毫米等小数单位）
- 金额：保留 2 位小数
- 单价：保留 6 位小数

## 5. 小料拆分原则

### 5.1 拆分场景

文件：`src/lib/warehouse-core.ts` 的 `splitMaterial` 方法

当整料数量大于需求时，需拆分出小料：

```
原整料：qr_code = MA-001, quantity = 100
需求量：30
拆分后：
  - MA-001（父）：quantity = 70, split_flag = 0
  - MA-001-S1（子）：quantity = 30, split_flag = 1, parent_qr_code = MA-001
```

### 5.2 拆分记录

写入 `material_splits` 表：

| 字段 | 说明 |
|---|---|
| `parent_qr_code` | 父二维码 |
| `child_qr_code` | 子二维码（新建） |
| `split_quantity` | 拆分数量 |
| `operator_id` | 操作员 |
| `split_time` | 拆分时间 |
| `work_order_no` | 关联工单 |

### 5.3 余料回库

生产剩余物料回库时，`split_flag = 2`（余料），优先消耗：

```sql
UPDATE inv_inventory_batch
SET split_flag = 2, available_qty = available_qty + ?
WHERE qr_code = ?;
```

## 6. 库存状态机

文件：`src/lib/warehouse-state-machine.ts`

### 6.1 入库单状态

```
draft → pending → completed
                ↓
              cancelled
```

| 状态 | 允许操作 |
|---|---|
| `draft` | edit, delete, submit |
| `pending` | audit, cancel |
| `completed` | unaudit（撤销） |
| `cancelled` | view |

### 6.2 出库单状态

```
draft → pending → completed
                ↓
              cancelled
```

| 状态 | 允许操作 |
|---|---|
| `draft` | edit, delete, submit |
| `pending` | confirm, cancel |
| `completed` | unconfirm（撤销） |
| `cancelled` | view |

### 6.3 库存交易类型

`InventoryTransType`：`inbound`/`inbound_cancel`/`outbound`/`outbound_cancel`/`transfer_in`/`transfer_out`/`adjust_add`/`adjust_sub`

## 7. 库存流水原则

### 7.1 流水表 `inv_inventory_transaction`

| 字段 | 说明 |
|---|---|
| `trans_no` | 流水号 |
| `trans_type` | 交易类型 |
| `batch_no` | 批次号 |
| `material_id` | 物料 |
| `warehouse_id` | 仓库 |
| `quantity` | 变动数量（正数入库/负数出库） |
| `source_type` | 来源类型（inbound/outbound/stocktaking/transfer/requisition） |
| `source_no` | 来源单号 |
| `operated_by` | 操作员 |
| `operated_at` | 操作时间 |

### 7.2 流水记录原则

1. **每笔变动必记录**：库存增减必须伴随流水记录
2. **事务一致**：库存更新与流水记录在同一事务内
3. **不可修改**：流水一旦生成不可修改，仅可冲销（反向流水）
4. **来源可溯**：`source_type` + `source_no` 指向原始业务单据

## 8. 预警机制

### 8.1 临期预警

文件：`src/lib/warehouse-core.ts` 的 `queryExpiringMaterials`

```sql
SELECT * FROM inv_inventory_batch
WHERE expire_date <= DATE_ADD(NOW(), INTERVAL ? DAY)
  AND available_qty > 0
  AND inspection_status = 'passed'
ORDER BY expire_date ASC;
```

预警天数通过 `expiry_warning_days` 配置项控制（默认 30 天）。

### 8.2 呆滞预警

`queryObsoleteMaterials`：查询长期未变动的批次（默认 90 天无流水）。

```sql
SELECT b.* FROM inv_inventory_batch b
LEFT JOIN inv_inventory_transaction t
  ON b.batch_no = t.batch_no
  AND t.operated_at >= DATE_SUB(NOW(), INTERVAL 90 DAY)
WHERE t.id IS NULL
  AND b.available_qty > 0;
```

## 9. 工程约定汇总

1. **批次维度**：所有库存操作以批次为单位，禁止跨批次合并
2. **乐观锁**：批次更新必须带 `version` 条件，冲突重试
3. **精度**：`Decimal.js` 计算，禁止浮点
4. **流水完整**：库存变动必有流水，事务保证一致
5. **FIFO 强约束**：默认按算法推荐，覆盖必记日志
6. **状态隔离**：锁定/隔离/检验中的批次不参与分配
7. **小料追溯**：拆分必记录 `parent_qr_code`，余料 `split_flag=2`
8. **负库存禁止**：所有扣减前校验 `available_qty >= quantity`
