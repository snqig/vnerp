# 仓库管理模块分析

## 1. 概述

本文档对仓库管理模块进行综合分析，覆盖数据模型、核心能力、业务流程与已知问题。详细原则参见 [26-仓库管理核心原则专项说明](./26-仓库管理核心原则专项说明.md)。

## 2. 数据模型

仓库模块涉及的核心表：

| 表 | 用途 |
|---|---|
| `inv_warehouse` | 仓库主数据 |
| `inv_warehouse_log` | 仓库操作日志 |
| `inv_inventory` | 库存汇总（按物料+仓库） |
| `inv_inventory_batch` | 批次库存（FIFO 单位） |
| `inv_inventory_transaction` | 库存流水 |
| `inv_inventory_log` | 库存变更日志 |
| `inv_inbound_order` / `inv_inbound_item` | 入库单/明细 |
| `inv_stocktaking` / `inv_stocktaking_item` | 盘点单/明细 |
| `material_splits` | 小料拆分记录 |
| `fifo_override_log` | FIFO 覆盖日志 |

### 2.1 `inv_inventory` vs `inv_inventory_batch`

- `inv_inventory`：物料+仓库维度的汇总库存，字段 `quantity`、`stocktaking_flag`
- `inv_inventory_batch`：批次维度库存，字段 `available_qty`、`reserved_qty`、`version`、`expire_date`、`split_flag`

两者关系：`inv_inventory.quantity = SUM(inv_inventory_batch.available_qty + reserved_qty)`。

## 3. 核心能力

### 3.1 批次管理

- 每批物料独立二维码（`qr_code`），记录生产日期、过期日期、供应商
- 批次状态：检验中（`inspection_status=pending`）、隔离（`quarantine_status=1`）、锁定（`stocktaking_flag=1`）
- 批次成本记录在 `material_batch_costs`，用于 FIFO 发料成本核算

### 3.2 FIFO 分配

调用 `allocateFIFO(materialId, warehouseId, quantity)` 返回推荐批次列表：

```
余料(split_flag=2) → 近效期(expire_date ASC) → 先入(inbound_date ASC)
```

跳过锁定/隔离/检验中的批次。冲突时使用 `version` 乐观锁重试。

### 3.3 小料拆分

`splitMaterial(parentQrCode, splitQuantity)`：
1. 创建子二维码（`MA-001-S1`）
2. 父批次 `quantity -= splitQuantity`
3. 新建子批次 `quantity = splitQuantity, split_flag = 1`
4. 写入 `material_splits` 关联记录

### 3.4 预警

- 临期预警：`queryExpiringMaterials(days)` 查询将过期批次
- 呆滞预警：`queryObsoleteMaterials(days)` 查询长期无流水批次

## 4. 业务流程

### 4.1 入库流程

```
采购订单 → 到货验收 → 创建入库单(inv_inbound_order)
   ↓ submit (draft→pending)
   ↓ audit (pending→completed)
   ↓ 事务内：
   ↓   - 写入 inv_inventory_batch（新批次）
   ↓   - 累加 inv_inventory.quantity
   ↓   - 写入 inv_inventory_transaction（type=inbound）
   ↓   - 生成 qrcode_record（物料二维码）
   ↓   - 生成 fin_payable（应付单）
   ↓   - 记录 material_batch_costs（批次成本）
```

### 4.2 出库流程

```
领用/发货 → 创建出库单
   ↓ submit → confirm
   ↓ 事务内：
   ↓   - 按 FIFO 分配批次
   ↓   - 扣减 inv_inventory_batch.available_qty
   ↓   - 扣减 inv_inventory.quantity
   ↓   - 写入 inv_inventory_transaction（type=outbound）
   ↓   - 更新 qrcode_record.status=2
   ↓   - 归集成本（work_order_costs 或 fin_receivable）
```

### 4.3 调拨流程

```
创建调拨单 → submit → confirm
   ↓ 事务内：
   ↓   - 源仓库 inv_inventory.quantity -= 数量
   ↓   - 源仓库 inv_inventory_batch.available_qty -= 数量
   ↓   - 目标仓库 inv_inventory.quantity += 数量
   ↓   - 目标仓库新建/更新批次
   ↓   - 写入两条流水（transfer_out / transfer_in）
```

## 5. 与其他模块联动

| 模块 | 联动点 |
|---|---|
| 采购管理 | 入库触发应付生成 |
| 销售管理 | 发货扣减库存生成应收 |
| 生产管理 | 物料领用扣减库存归集成本 |
| 财务管理 | 批次成本驱动工单成本 |
| 二维码追溯 | 批次绑定二维码，全链路追溯 |
| 质量管理 | 检验状态控制批次可用性 |

## 6. 已知问题与改进方向

### 6.1 双表结构问题

项目存在 `inv_inventory`（业务表）与 `inventory_batches`（Drizzle Schema）两套定义，API 实际操作 `inv_*` 表。建议：
- 统一为 Drizzle ORM 管理，或明确两套表的使用边界
- 迁移脚本保证两套表数据一致

### 6.2 性能优化方向

- `inv_inventory_transaction` 流水表数据量大，建议按月分区或归档历史数据
- FIFO 查询涉及多表 JOIN，可考虑物料维度的库存缓存表
- 临期/呆滞预警可改为定时任务异步计算

### 6.3 业务规则完善

- 余料回库的质检流程需明确（是否需要重新检验）
- 跨仓库调拨的运输成本归集规则待补充
- 负库存场景的处理策略（当前为拒绝，可考虑配置化）
