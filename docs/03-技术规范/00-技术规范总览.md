# 一、全局编码规则（所有单据统一格式）

## 1.1 编码设计原则

### 核心原则

```
✅ 唯一性：全系统范围内唯一，不重复
✅ 可读性：包含业务含义，便于人工识别
✅ 可扩展性：预留足够位数的流水号
✅ 可配置性：前缀和格式可通过系统配置修改
✅ 时间有序：按时间自然排序，便于查询
```

## 1.2 统一编码公式

```
基础数据编码：前缀 + 固定位数流水号
示例：CUST000001

业务单据编码：前缀 + 日期(YYYYMMDD) + 固定位数流水号
示例：WO202605110001
```

**关键参数**（可配置）：

| 参数 | 配置键 | 默认值 | 说明 |
|------|-------|-------|------|
| 日期格式 | `doc_date_format` | YYYYMMDD | 单据中的日期部分 |
| 流水号长度 | `serial_number_length` | 4 | 末尾流水号位数 |

---

## 1.3 基础数据编码规范

### 客户编号
```
格式：CUST + 6位流水号
示例：CUST000001, CUST000002, ... CUST999999
```

### 供应商编号
```
格式：SUPP + 6位流水号
示例：SUPP000001, SUPP000002, ...
```

### 物料编号
```
格式：MAT + 6位流水号
示例：MAT000001, MAT000002, ...
物料类型后缀约定：
├── 原材料(整料) - MAT000001
├── 原材料(小料) - MAT000002
├── 半成品 - MAT005001
├── 成品 - MAT010001
└── 辅料 - MAT020001
```

### 其他基础数据
- 产品编号：PROD + 6位流水号
- 工序编号：PROCESS + 3位流水号
- 设备编号：EQ + 6位流水号
- 仓库编号：WH + 2位流水号
- 库位编号：WH{仓库号}-{区域代码}-{货架号}-{层号}-{位号}
- 员工编号：EMP + 6位流水号
- 部门编号：DEPT + 3位流水号

## 1.5 二维码编码规范（全局唯一）

### 编码结构
```
总长度：19位字符
格式：VNR + 类型(1) + 时间戳(8) + 序号(7) + 拆分标识(1)

字段说明：
├── VNR : 固定前缀（VNERP缩写）
├── T   : 物料类型标识
├── YYYYMMDD : 生成日期
├── NNNNNNN : 当日序号（7位）
└── S   : 拆分标识（0=整料, 1=小料, 2=余料）
```

### 拆分标识详解

| 标识值 | 含义 | 生成场景 | 使用限制 |
|--------|------|---------|---------|
| **0** | 整料 | 采购入库时生成 | 禁止直接领用，必须先拆分 |
| **1** | 小料 | 从整料拆分后生成 | 正常领用的最小单位 |
| **2** | 余料 | 拆分后的剩余部分 | 优先于小料领用 |

### 示例
```
整料二维码：VNR2026051100000010
小料二维码：VNR2026051100000011（从整料拆分）
余料二维码：VNR2026051100000012（拆分剩余）
```

## 2.2 刀模寿命与更换规则

| 参数名称 | 配置键 | 默认值 | 单位 | 取值范围 |
|---------|-------|-------|------|---------|
| 刀模有效天数 | `mould_life_days` | 90 | 天 | 30-365 |
| 刀模最大使用次数 | `mould_max_times` | 5000 | 次 | 100-50000 |
| 刀模预警天数 | `mould_warn_days` | 15 | 天 | 1-30 |
| 刀模报废判定规则 | `mould_scrap_rule` | both | - | both/expire/times |

**报废判定逻辑**：
- `both`: 到期 或 超次 都报废
- `expire`: 仅到期报废
- `times`: 仅超次报废

## 2.3 网版寿命与更换规则

| 参数名称 | 配置键 | 默认值 | 单位 | 取值范围 |
|---------|-------|-------|------|---------|
| 网版有效天数 | `screen_life_days` | 60 | 天 | 15-180 |
| 网版最大使用次数 | `screen_max_times` | 3000 | 次 | 50-20000 |
| 网版预警天数 | `screen_warn_days` | 10 | 天 | 1-20 |

**报废标准**：
- ❌ 拉丝：印刷时出现线条拉丝现象
- ❌ 破损：网布有破洞或裂缝
- ❌ 变形：网框变形导致套印不准
- ❌ 感光胶脱落：感光胶层脱落导致图案不完整

## 2.4 油墨有效期与使用规则

| 参数名称 | 配置键 | 默认值 | 单位 | 说明 |
|---------|-------|-------|------|------|
| 未开盖油墨保质期 | `ink_unopened_shelf_life` | 168 | 小时 | 约7天 |
| 已开盖油墨保质期 | `ink_opened_shelf_life` | 96 | 小时 | 约4天 |
| 混合油墨必须用完时长 | `mixed_ink_expiry_hours` | 24 | 小时 | 调配后有效期 |

**特殊规则**：
⚠️ 调配后的混合油墨必须在24小时内用完
⚠️ 超过24小时自动冻结，禁止继续使用
⚠️ 冻结的油墨只能报废处理，不可解冻

## 2.5 原材料保质期规则

| 材料类别 | 配置键 | 默认值 | 单位 | 预警期 |
|---------|-------|-------|------|-------|
| PET/PVC薄膜 | `pet_film_shelf_life` | 360 | 天 | 30天 |
| 溶剂 | `solvent_shelf_life` | 180 | 天 | 30天 |
| 胶水/辅料 | `glue_shelf_life` | 90 | 天 | 30天 |

| 原料类型 | 整料规格 | 拆分单位 | 小料规格 | 配置键 | 每整料可拆 |

## 2.7 仓库管理强制规则

### FIFO控制
| 规则名称 | 配置键 | 默认值 | 说明 |
|---------|-------|-------|------|
| 是否强制先进先出 | `fifo_enabled` | true | 开启后严格按入库时间推荐批次 |

### 发料控制
| 规则名称 | 配置键 | 默认值 | 说明 |
|---------|-------|-------|------|
| 允许直接发整料 | `allow_whole_material_issue` | false | 整料是否可不拆分直接领用 |

### 领料优先级
```
1️⃣ 余料（remainder）- 最优先，减少浪费
2️⃣ 小料（small）- 其次，正常领用
3️⃣ 整料（whole）- 最后，需审批
```

### 循环盘点周期

|------|------|---------|
| A类 | 高价值、快周转 | 每日 |
| B类 | 中等价值 | 每周 |
| C类 | 低价值、慢周转 | 每月 |

# 三、数据库设计规范

> 📖 详细代码规范请参阅：[代码规范](./代码规范.md)

## 3.1 表结构规范

每张表必须包含的标准字段：

```sql
id INT PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
deleted TINYINT NOT NULL DEFAULT 0 COMMENT '删除标志(0正常 1已删除)',
create_by VARCHAR(50) COMMENT '创建人',
update_by VARCHAR(50) COMMENT '更新人',
remark TEXT COMMENT '备注'
```

## 3.2 命名规范

### 表名规范
- 使用小写字母和下划线：`inv_inventory_batch`
- 模块前缀：`crm_`(客户)、`inv_`(库存)、`prod_`(生产)、`qc_`(品质)

### 字段名规范
- 使用小写字母和下划线：`create_time`, `material_id`
- 外键字段以 `_id` 结尾：`customer_id`, `supplier_id`
- 状态字段统一命名为：`status`
- 类型字段统一命名为：`type`

## 3.3 索引规范

```sql
-- 主键索引
PRIMARY KEY (id)

-- 唯一索引（业务唯一性约束）
UNIQUE INDEX uk_order_no (order_no)

-- 普通索引（高频查询字段）
INDEX idx_material_id (material_id)
INDEX idx_status (status)

-- 复合索引（多条件查询）
INDEX idx_material_status (material_id, status)
INDEX idx_create_time (create_time)
```

# 四、API接口规范

## 4.1 统一响应结构

### 成功响应
```typescript
interface SuccessResponse<T> {
  code: number;        // 200
  message: string;     // "操作成功"
  data: T;             // 业务数据
  timestamp: number;   // 时间戳
}
```

### 错误响应
```typescript
interface ErrorResponse {
  code: number;        // 错误码（非HTTP状态码）
  message: string;     // 错误信息
  errors?: Array<{     // 详细错误（表单验证等）
    field: string;
    message: string;
  }>;
  timestamp: number;
}
```

## 4.2 分页规范

```typescript
interface PaginatedResponse<T> {
  list: T[];
  total: number;       // 总记录数
  page: number;        // 当前页码
  pageSize: number;    // 每页条数
  totalPages: number;  // 总页数
}

// 请求参数
interface PaginationParams {
  page?: number;       // 页码，默认1
  pageSize?: number;   // 每页条数，默认20
}
```

## 4.3 HTTP方法语义

| 方法 | 用途 | 幂等性 |
|------|------|--------|
| GET | 查询资源 | 是 |
| POST | 创建资源 | 否 |
| PUT | 全量更新资源 | 是 |
| PATCH | 部分更新资源 | 是 |
| DELETE | 删除资源 | 是 |

# 六、安全规范

## 6.1 认证授权

- 使用 JWT Token 进行身份认证
- 基于 RBAC（角色权限控制）进行授权
- 敏感操作需要二次验证

## 6.2 数据验证

- 所有输入必须进行服务端验证
- 使用参数化查询防止 SQL 注入
- 对用户输入进行 XSS 过滤

## 6.3 敏感数据处理

- 密码必须加密存储（bcrypt）
- API Key、密钥等不得硬编码在代码中
- 日志中不得记录敏感信息
