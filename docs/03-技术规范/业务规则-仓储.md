# 仓库管理业务规则

> 文档编号：VNERP-BIZ-002 | 版本：V2.0 | 基于：代码现状 2026-06-30
> 关联代码：`src/app/api/inventory/route.ts`、`src/lib/fifo-allocation.ts`、`src/lib/soft-delete.ts`

## 1. 数据模型

库存核心使用 `inv_*` 前缀表，统一通过 `src/app/api/inventory/route.ts` 操作。

| 表名 | 用途 | 关键字段 |
|------|------|---------|
| `inv_inventory_batch` | 库存批次主表 | `batch_no`, `material_id`, `warehouse_id`, `available_qty`, `locked_qty`, `unit_price`, `status`, `version`, `inbound_date`, `expire_date` |
| `inv_material` | 物料主数据 | `material_code`, `safety_stock`, `reorder_point`, `purchase_price` |
| `inv_warehouse` | 仓库主数据 | `warehouse_code`, `warehouse_name` |
| `inv_inventory_transaction` | 库存事务流水 | `trans_no`, `trans_type`(in/out/transfer), `source_type`, `quantity`(出库为负) |
| `inv_inventory_log` | 库存变动日志 | `before_qty`, `after_qty`, `source_no` |
| `inv_outbound_batch_allocation` | 出库批次分配明细 | `fifo_mode`(fifo_auto/specified_batch), `allocated_qty`, `unit_cost`, `total_cost` |

批次状态：`normal`（正常可用）、`frozen`（冻结，如检验不合格）。

## 2. 入库规则

入口：`POST /api/inventory`，`action=inbound`。

- 必填：`warehouseId`、`materialId`、`quantity`；可选：`sourceType`、`sourceNo`。
- 校验：物料与仓库必须存在且 `deleted=0`。
- 事务内执行三步：
  1. `INSERT INTO inv_inventory_batch`（`available_qty=quantity`，`locked_qty=0`，`status='normal'`）；
  2. `INSERT INTO inv_inventory_transaction`（`trans_type='in'`，记录单位成本与总成本）；
  3. `INSERT INTO inv_inventory_log`（记录 `before_qty=0`、`after_qty=quantity`）。
- 单号生成：批次号 `B{时间戳}`，事务号 `IN{时间戳}`（`generateBatchNo`/`generateTransNo`）。
- 操作日志：通过 `logOperation` 写入审计，`oper_type='inventory'`。

## 3. 出库规则

入口：`POST /api/inventory`，`action=outbound`。两种模式：

### 3.1 指定批次出库（`batchNo` 非空）

- `SELECT ... FOR UPDATE` 锁定批次行，校验 `available_qty >= quantity`。
- 乐观锁更新：`WHERE id=? AND version=?`，`affectedRows=0` 时抛 `conflict` 错误并提示重试。
- FIFO 推荐校验：查询同物料同仓库最早可用批次，若与指定批次不一致，返回 `warning: '当前出库批次不是FIFO推荐批次，存在过期物料风险'`。
- 写入 `inv_outbound_batch_allocation`，`fifo_mode='specified_batch'`。

### 3.2 FIFO 自动出库（`batchNo` 为空）

- 必填：`materialId`、`warehouseId`、`quantity`。
- 调用 `allocateFIFO(conn, materialId, warehouseId, quantity)`：
  - 排序：`opened_at` 优先（已开封），其次 `inbound_date ASC`、`expire_date ASC`、`id ASC`；
  - 默认排除过期批次（`expire_date >= CURDATE()`），`allowExpired` 可覆盖；
  - 金额计算使用 `decimal.js` 避免浮点误差。
- 调用 `executeFIFODeductionWithRetry`（最多 3 次，指数退避 100ms×attempts）：
  - 乐观锁：`WHERE id=? AND available_qty>=? AND version=?`；
  - 冲突时重试，超限抛 `FIFO deduction failed after N attempts`。
- `shortage > 0` 时返回 `insufficient` 错误（409）。
- 返回 `batchDetails`（每批次扣减量、单位成本、行成本）与 `totalCost`。

## 4. 调拨规则

入口：`POST /api/inventory`，`action=transfer`。

- 必填：`batchNo`、`warehouseId`（目标仓库）。
- 事务内：锁定源批次与目标仓库行（`FOR UPDATE`），乐观锁更新 `warehouse_id`/`warehouse_name`。
- 不产生数量变动，仅更改归属仓库；记录 `trans_type='transfer'` 事务与日志。
- 注：当前实现直接迁移批次仓库，**不产生**调拨中间态（如"在途"），为简化模型。

## 5. 库存预警

`GET /api/inventory` 列表查询时计算 `alertLevel`：

| 级别 | 条件 |
|------|------|
| `critical` | `available_qty <= 0` |
| `warning` | `available_qty < safety_stock` 或 `available_qty < 10` |
| `normal` | 其他 |

`checkShortageAndWarn`（`fifo-allocation.ts`）在分配不足时生成 `ShortageWarning`，含 `safetyStock`、`reorderPoint`、`reorderSuggestion`。

## 6. 单据作废

通过 `src/lib/soft-delete.ts` 的 `cancelDocument` / `quickCancel` 实现，原则：**数据永久留存，仅作废不物理删除**。

| 单据类型 | 配置 key | 检查条件 |
|---------|---------|---------|
| 入库单 | `inbound_order` | `not_settled` |
| 出库单 | `outbound_order` | `not_settled` |
| 盘点单 | `stocktaking` | 无 |
| 调拨单 | `transfer_order` | `not_completed` |

作废流程：查询原始数据 → 检查 `status`/`deleted` → 业务约束检查 → 事务更新 `status='cancelled'`+`deleted=1` → 写 `document_cancel_log`（含 `snapshot_data`） → `logDocumentCancel` 审计。

物理删除 `physicalDelete` 仅允许 `draft`/`0` 状态，且记录删除前快照。

## 7. 二维码追溯

工单创建时自动生成 `qrcode_record`（`qr_type='workorder'`，见 `src/app/api/workorders/route.ts`），记录 `ref_id`、`work_order_no`、`quantity`、`extra_data`。物料入库的二维码追溯通过 `qrcode_record.batch_no` 与库存批次关联。
