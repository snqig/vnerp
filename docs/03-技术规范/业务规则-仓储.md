# 仓库管理业务规则

> 文档编号：VNERP-BIZ-002 | 版本：V2.1 | 基于：代码现状 2026-07-10
> 关联代码：`src/app/api/warehouse/{inbound,outbound,transfer,stocktaking,inventory}/*`、`src/lib/fifo-allocation.ts`、`src/lib/soft-delete.ts`、`src/lib/inventory-sync.ts`

## 1. 数据模型

库存核心使用 `inv_*` 前缀表（在 `src/lib/db/schema.ts` 中由 Drizzle ORM 消费的 8 张仓库表）。

| 表名 | 用途 | 关键字段 |
|------|------|---------|
| `inv_inbound_order` | 入库单主表 | `order_no`、`order_type`、`warehouse_id`、`supplier_id`、`po_id`、`status`、`qc_status` |
| `inv_inbound_item` | 入库单明细 | `order_id`、`material_id`、`batch_no`、`quantity`、`unit_price` |
| `inv_warehouse` | 仓库主数据 | `warehouse_code`、`warehouse_name`、`warehouse_type`、`manager_id`、`status` |
| `inv_inventory` | 库存台账 | `material_id`、`warehouse_id`、`quantity`、`locked_qty`、`version` |
| `inv_outbound_order` | 出库单主表 | `order_no`、`warehouse_id`、`status` |
| `inv_outbound_item` | 出库单明细 | `order_id`、`material_id`、`batch_no`、`quantity` |
| `inv_transfer_order` | 调拨单 | `transfer_no`、`from_warehouse_id`、`to_warehouse_id`、`status` |
| `inv_stocktaking` | 盘点单 | `stocktaking_no`、`warehouse_id`、`status` |

遗留 SQL 表（非 Drizzle 消费，存在于 `database/vnerpdacahng_schema.sql` 与运行时 SQL）：
- `inv_inventory_batch`：批次主表（`batch_no`、`material_id`、`warehouse_id`、`available_qty`、`locked_qty`、`unit_price`、`status`、`version`、`inbound_date`、`expire_date`、`opened_at`）
- `inv_inventory_transaction`：库存事务流水（`trans_no`、`trans_type`(in/out/transfer/adjust)、`source_type`、`source_id`、`source_no`、`quantity`、`stock_before`、`stock_after`、`unit_price`、`total_amount`）
- `inv_inventory_log`：库存变动明细日志（`before_qty`、`after_qty`、`source_no`）
- `inv_outbound_batch_allocation`：出库批次分配明细（`fifo_mode`(fifo_auto/specified_batch)、`allocated_qty`、`unit_cost`、`total_cost`）

批次状态：`normal`（正常可用）、`frozen`（冻结，如检验不合格）。

## 2. 入库规则

入口：`POST /api/warehouse/inbound`（含 `action=cutting/scan/audit/from-po/with-po`）。

- 必填：`warehouseId`、`materialId`、`quantity`；可选：`sourceType`、`sourceNo`、`batchNo`。
- 校验：物料与仓库必须存在且 `deleted=0`。
- 事务内执行三步：
  1. `INSERT INTO inv_inventory_batch`（`available_qty=quantity`，`locked_qty=0`，`status='normal'`）；
  2. `INSERT INTO inv_inventory_transaction`（`trans_type='in'`，记录单位成本与总成本）；
  3. `INSERT INTO inv_inventory_log`（记录 `before_qty=0`、`after_qty=quantity`）。
- 单号生成：批次号 `B{时间戳}`，事务号 `IN{时间戳}`（`generateBatchNo`/`generateTransNo`）。
- 操作日志：通过 `logOperation` 写入审计，`oper_type='inventory'`。

## 3. 出库规则

入口：`POST /api/warehouse/outbound`（含 `action=fifo/confirm`）。两种模式：

### 3.1 指定批次出库（`batchNo` 非空）

- `SELECT ... FOR UPDATE` 锁定批次行，校验 `available_qty >= quantity`。
- 乐观锁更新：`WHERE id=? AND version=?`，`affectedRows=0` 时抛 `conflict` 错误并提示重试。
- FIFO 推荐校验：查询同物料同仓库最早可用批次，若与指定批次不一致，返回 `warning: '当前出库批次不是FIFO推荐批次，存在过期物料风险'`。
- 写入 `inv_outbound_batch_allocation`，`fifo_mode='specified_batch'`。

### 3.2 FIFO 自动出库（`batchNo` 为空）

- 必填：`materialId`、`warehouseId`、`quantity`。
- 调用 `allocateFIFO(conn, materialId, warehouseId, quantity)`：
  - 排序：`opened_at` 优先（已开封），其次 `inbound_date ASC`、`expire_date ASC`、`id ASC`；
  - 默认排除过期批次（`expire_date >= CURDATE()`），`allowExpired` 可覆盖；
  - 金额计算使用 `decimal.js` 避免浮点误差。
- 调用 `executeFIFODeductionWithRetry`（最多 3 次，指数退避 100ms×attempts）：
  - 乐观锁：`WHERE id=? AND available_qty>=? AND version=?`；
  - 冲突时重试，超限抛 `FIFO deduction failed after N attempts`。
- `shortage > 0` 时返回 `insufficient` 错误（409）。
- 返回 `batchDetails`（每批次扣减量、单位成本、行成本）与 `totalCost`。

## 4. 调拨规则

入口：`POST /api/warehouse/transfer`（含 `[id]/items/outbound/inbound`）。

- 必填：`batchNo`、`warehouseId`（目标仓库）。
- 事务内：锁定源批次与目标仓库行（`FOR UPDATE`），乐观锁更新 `warehouse_id`/`warehouse_name`。
- 不产生数量变动，仅更改归属仓库；记录 `trans_type='transfer'` 事务与日志。
- 注：当前实现直接迁移批次仓库，**不产生**调拨中间态（如"在途"），为简化模型。

## 5. 盘点规则

入口：`POST /api/warehouse/stocktaking`（含 `[id]/scan/items/split-summary/diff-process`）。

- 创建盘点单（`status='pending'`），锁定仓库（`freeze=1`）。
- 扫码盘点，比对系统库存与实盘数量，生成差异明细。
- 差异处理：`diff-process` 调用库存调整接口（`trans_type='adjust'`），写事务与日志。
- 完成盘点：解锁仓库，`status='completed'`。

## 6. 库存预警

`GET /api/warehouse/inventory` 列表查询时计算 `alertLevel`：

| 级别 | 条件 |
|------|------|
| `critical` | `available_qty <= 0` |
| `warning` | `available_qty < safety_stock` 或 `available_qty < 10` |
| `normal` | 其他 |

`checkShortageAndWarn`（`fifo-allocation.ts`）在分配不足时生成 `ShortageWarning`，含 `safetyStock`、`reorderPoint`、`reorderSuggestion`。

`/api/warehouse/inventory/warning` 提供独立的预警查询接口，`/api/warehouse/inventory/logs` 提供变动日志查询。

## 7. 库存同步与并发控制

`src/lib/inventory-sync.ts` 提供基于乐观锁的库存同步：

- `version` 字段 + `transactionWithRetry` 处理并发冲突。
- `inventorySyncHandler` 事件处理器订阅库存变动事件，保证最终一致性。
- `inventoryRollbackHandler` 处理回滚（如出库失败后还原批次数量）。
- 多实例部署下，事件总线使用 Redis Streams（流名 `erp:domain-events`）+ Outbox 两阶段保证幂等。

## 8. 单据作废

通过 `src/lib/soft-delete.ts` 的 `cancelDocument` / `quickCancel` 实现，原则：**数据永久留存，仅作废不物理删除**。

| 单据类型 | 配置 key | 检查条件 |
|---------|---------|---------|
| 入库单 | `inbound_order` | `not_settled` |
| 出库单 | `outbound_order` | `not_settled` |
| 盘点单 | `stocktaking` | 无 |
| 调拨单 | `transfer_order` | `not_completed` |

作废流程：查询原始数据 → 检查 `status`/`deleted` → 业务约束检查 → 事务更新 `status='cancelled'`+`deleted=1` → 写 `document_cancel_log`（含 `snapshot_data`） → `logDocumentCancel` 审计。

物理删除 `physicalDelete` 仅允许 `draft`/`0` 状态，且记录删除前快照。

⚠️ **已知问题**（见《已知问题》INV-001）：`soft-delete.ts` 的 `CANCELLABLE_DOCUMENTS` 配置使用 `warehouse_inbound`/`warehouse_outbound`/`warehouse_stocktaking`/`warehouse_transfer`，与 `inv_*` 真实表名不一致，作废调用可能失败。

## 9. 二维码追溯

工单创建时自动生成 `qrcode_record`（`qr_type='workorder'`，见 `src/app/api/workorders/route.ts`），记录 `ref_id`、`work_order_no`、`quantity`、`extra_data`。物料入库的二维码追溯通过 `qrcode_record.batch_no` 与库存批次关联。

详细开发指南见 [二维码追溯开发](../11-开发指南/二维码追溯开发.md)。

## 10. 切料优化

`src/lib/cutting-optimizer.ts` 提供切料优化算法：

- 输入：原料规格、目标件规格与数量。
- 输出：最优切割方案（最小化损耗）。
- 入库时 `action=cutting` 触发，记录切料明细与余料。
