# 财务核算业务规则

> 文档编号：VNERP-BIZ-005 | 版本：V2.0 | 基于：代码现状 2026-06-30
> 关联代码：`src/app/api/finance/{receivables,payables,receipt,payment,aging,costs,cost-variance,summary,report}/route.ts`、`src/lib/finance-core.ts`

## 1. 应收账款

### 1.1 自动生成（`generateReceivable`）

触发：销售发货确认。`POST /api/finance/receivables` 传 `salesOrderId`/`shipmentId`/`amount`/`dueDate?`。

- 查询 `sal_order`（左联 `crm_customer`）获取客户。
- 单号 `generateDocNo('AR')`，`source_type=1`，`source_no=order.order_no`。
- 默认到期日：当前日期 + 30 天。
- 写入 `fin_receivable`：`amount`、`received_amount=0`、`balance=amount`、`status=1`（未收款）。

### 1.2 状态流转

```
1(未收款) → 2(部分收款) → 3(已结清)
```

### 1.3 收款录入（`recordReceipt`）

事务流程：
1. 锁定应收单，校验 `newReceived = received_amount + amount ≤ amount`（超额抛错）。
2. 写入 `fin_receipt_record`：`receipt_no`（`generateDocNo('RC')`）、`receipt_method`、`bank_account`、`reference_no`、`handler_id`。
3. 更新 `fin_receivable`：`received_amount`、`balance = amount - newReceived`、`status = balance≤0 ? 3 : 2`。

### 1.4 数据表

`fin_receivable` 核心字段：`receivable_no`、`source_type`/`source_no`、`customer_id`、`amount`、`received_amount`、`balance`、`due_date`、`status`(1/2/3)。

`fin_receipt_record`：`receipt_no`、`receivable_id`、`customer_id`、`amount`、`receipt_date`、`receipt_method`、`bank_account`、`handler_id`。

## 2. 应付账款

### 2.1 自动生成（`generatePayable`）

触发：采购入库确认。`generatePayable(purchaseOrderId, inboundId, amount, dueDate?)`。

- 查询 `pur_order`（左联 `pur_supplier`）。
- 单号 `generateDocNo('AP')`，`source_type=1`，`source_no=order.order_no`。
- 默认到期日 +30 天。
- 写入 `fin_payable`：`amount`、`paid_amount=0`、`balance=amount`、`status=1`。

### 2.2 状态流转

```
1(未付款) → 2(部分付款) → 3(已结清)
```

### 2.3 付款录入（`recordPayment`）

事务流程：
1. 锁定应付单，校验 `newPaid = paid_amount + amount ≤ amount`。
2. 写入 `fin_payment_record`：`payment_no`（`generateDocNo('PY')`）、`payment_method`、`bank_account`。
3. 更新 `fin_payable`：`paid_amount`、`balance`、`status = balance≤0 ? 3 : 2`。

## 3. 账龄分析（`/api/finance/aging`）

`GET ?type=receivable|payable&asOfDate=YYYY-MM-DD`，按账龄区间统计未结清款。

账龄区间：未到期、1-30天、31-60天、61-90天、91-180天、181-365天、365天以上。

按往来单位（客户/供应商）汇总 `total_amount`/`remaining_amount`，分配到对应账龄桶。

⚠️ **已知问题**（见《已知问题》DATA-003）：该路由查询 `finance_receivable`/`finance_payable` 表（字段 `remaining_amount`、`status='settled'`），与 `finance-core.ts` 使用的 `fin_receivable`/`fin_payable`（字段 `balance`、`status=1/2/3`）不一致，**账龄分析当前不可用**，需统一表名与字段。

## 4. 成本核算（`calculateWorkOrderCost`）

```
工单总成本 = 直接材料 + 直接人工 + 制造费用
单位成本 = 工单总成本 / 完工数量
```

| 成本项 | 数据源 | 计算 |
|--------|-------|------|
| 直接材料 | `material_requisition_items.total_cost` | `SUM WHERE status=2 AND work_order_id=?` |
| 直接人工 | `prd_work_report` | `SUM(actual_hours × hourly_rate)` |
| 制造费用 | 硬编码简化 | `laborCost × 0.5`（**待优化**，见《已知问题》INV-002） |

结果写入 `work_order_costs`（存在 UPDATE，否则 INSERT），`status=1` 已核算。

## 5. 汇总与报表

- `GET /api/finance/summary`：应收应付汇总。
- `GET /api/finance/stats`：财务统计指标。
- `GET /api/finance/report`：财务报表。
- `queryReceivableSummary(customerId?)` / `queryPayableSummary(supplierId?)`：按 `status` 分组聚合 `COUNT`/`SUM(amount)`/`SUM(received_amount)`/`SUM(balance)`。

## 6. 单据作废

通过 `src/lib/soft-delete.ts` 配置：

| 配置 key | 表名 | 检查条件 |
|---------|------|---------|
| `voucher` | `fin_voucher` | `not_posted`（已过账不可作废） |
| `receivable` | `fin_receivable` | `not_paid`（已收款不可作废） |
| `payable` | `fin_payable` | `not_paid`（已付款不可作废） |

⚠️ `cancelStatus='cancelled'`（字符串）与 `fin_receivable`/`fin_payable` 的数字状态（1/2/3）不兼容，作废前需将状态字段改为字符串枚举或调整配置（见《已知问题》DATA-003）。

## 7. 编号规则汇总

| 单据 | 前缀 | 生成方式 |
|------|------|---------|
| 应收单 | AR | `generateDocNo('AR')` |
| 应付单 | AP | `generateDocNo('AP')` |
| 收款单 | RC | `generateDocNo('RC')` |
| 付款单 | PY | `generateDocNo('PY')` |

## 8. 金额计算规范

- 所有金额使用 `decimal.js`（`decimal.js@10.6`）避免浮点误差。
- 数据库字段类型 `DECIMAL(15,2)`（金额）或 `DECIMAL(15,4)`（数量/单价）。
- 前端展示统一 `toFixed(2)`，汇率字段 `DECIMAL(10,4)`。
