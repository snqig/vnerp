# 采购管理业务规则

> 文档编号：VNERP-BIZ-003 | 版本：V2.0 | 基于：代码现状 2026-06-30
> 关联代码：`src/app/api/purchase/{request,orders,convert-po,return,suppliers}/route.ts`、`src/lib/finance-core.ts`

## 1. 架构分层

采购订单模块采用 DDD 分层（区别于其他模块的"事务脚本"风格）：

| 层 | 文件 | 职责 |
|----|------|------|
| 路由层 | `src/app/api/purchase/orders/route.ts` | HTTP 适配、`withAuthAndErrorHandler` 包装、权限校验 |
| 应用层 | `src/application/services/PurchaseApplicationService` | 用例编排（创建/提交/审核/入库/关闭/删除） |
| 领域层 | `src/domain/purchase/value-objects/PurchaseOrderStatus` | 状态机与 `toDbCode()`/`label()` |
| 基础设施层 | `src/infrastructure/repositories/MysqlPurchaseOrderRepository` | 持久化 |
| 事件 | `src/application/EventRegistry` `registerEventHandlers()` | 事件总线注册 |

权限：`purchase:view`（查询）、`purchase:create`（增改删）。
领域错误类型：`DomainError`、`NotFoundError`、`VersionConflictError`（映射至 400/404/409）。

## 2. 采购申请（`/api/purchase/request`）

### 2.1 数据模型

表 `pur_request` + 明细 `pur_request_item`，单号 `PR{日期}{3位随机}`（旧逻辑）或 `generateDocumentNo('purchase_request')`。

| 字段 | 说明 |
|------|------|
| `request_no` | 申请单号 |
| `request_type` | 申请类型 |
| `request_dept` / `requester_name` | 申请部门/申请人 |
| `reviewer_name` / `approver_name` | 审核人/审批人 |
| `total_amount` / `currency` | 总金额/币种 |
| `status` | 状态（数字） |
| `priority` | 优先级 |
| `expected_date` | 期望到货日 |

### 2.2 状态流转

```
0(草稿) → 提交审核 → 审核通过(4) → 转订单
                            ↓
                      6(?特殊状态，允许修改)
```

- `status >= 4 且 status != 6`：禁止修改。
- `status >= 4`：禁止删除。
- 删除前检查 `pur_order.remark LIKE '%PR-{id}%'`，已被引用则禁止删除。
- 软删除：主表与明细 `deleted=1`（事务）。

## 3. 采购订单（`/api/purchase/orders`）

### 3.1 创建（`POST`）

必填：`supplier_id`、`lines`（至少 1 条）。

订单字段：`po_no`、`order_date`、`delivery_date`、`currency`（默认 `CNY`）、`exchange_rate`（默认 `1.0`）、`tax_rate`（默认 `13`）、`over_receipt_tolerance`、`payment_terms`、`delivery_address`、`remark`、`create_by`（取 `userInfo.userId`）。

明细字段：`line_no`（自增）、`material_id`/`material_code`/`material_name`/`material_spec`、`unit`、`order_qty`、`unit_price`、`require_date`。`received_qty`/`returned_qty` 初始 0，`amount`/`tax_amount`/`line_total` 由服务层计算。

### 3.2 状态动作（`PUT`，`action` 参数）

| action | 行为 | 权限 |
|--------|------|------|
| `submit` | 提交订单 | `purchase:create` |
| `approve` | 审核通过，记录 `audit_by`/`audit_time` | `purchase:create` |
| `close` | 关闭订单 | `purchase:create` |
| `receive` | 采购入库，传 `lineReceives` 数组 | `purchase:create` |

入库逻辑：`service.receiveGoods(id, lineReceives)` 累加 `received_qty`，更新 `remaining_qty`、`is_fully_received`，订单 `total_received_qty` 满足后置 `is_fully_received=true`。

### 3.3 乐观锁与并发

`VersionConflictError` 表示并发冲突（如重复入库），返回 409。`NotFoundError` 返回 404。其他 `DomainError` 返回 400。

## 4. 采购入库与应付生成

入库后由 `src/lib/finance-core.ts` 的 `generatePayable(purchaseOrderId, inboundId, amount, dueDate?)` 自动生成应付：

- 查询 `pur_order`（左联 `pur_supplier`）获取订单与供应商。
- 单号 `generateDocNo('AP')`，`source_type=1`，`source_no=order.order_no`。
- 默认到期日：当前日期 + 30 天。
- 写入 `fin_payable`：`amount`、`paid_amount=0`、`balance=amount`、`status=1`（未付款）。

## 5. 采购退货（`/api/purchase/return`）

独立的退货路由，冲减入库数量与应付金额。退货明细需关联原采购订单与入库批次。

## 6. 供应商管理（`/api/purchase/suppliers`）

表 `pur_supplier`，字段：`supplier_code`、`supplier_name`、`contact`、`phone`、`address`、`status`。支持 CRUD 与分页查询，软删除。

## 7. 单据作废

通过 `src/lib/soft-delete.ts` 配置：

| 配置 key | 表名 | 状态字段 | 作废状态 | 检查条件 |
|---------|------|---------|---------|---------|
| `purchase_order` | `pur_purchase_order` | `status` | `cancelled` | `not_received` |

⚠️ 注意：`soft-delete.ts` 配置的表名为 `pur_purchase_order`，但 `finance-core.ts` 查询的是 `pur_order`，存在命名不一致（见《已知问题》DATA-001）。作废前需确认实际表名。

## 8. 编号规则汇总

| 单据 | 生成方式 |
|------|---------|
| 采购申请 | `generateDocumentNo('purchase_request')` |
| 采购订单 | `PurchaseApplicationService.createOrder` 内部生成 |
| 应付单 | `generateDocNo('AP')` |
