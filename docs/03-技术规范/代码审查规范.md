## 1. SQL 参数化强制规则

### 1.1 【P0-禁止】`IN (?)` 直接传递数组

**问题描述：** `mysql2` 的 `pool.execute()` 走预编译语句协议，**不会**自动展开数组参数。传入 `IN (?)` + `[array]` 时，MySQL 将整个数组视为单个标量值，导致只匹配第一项或完全不匹配。`pool.query()` 虽然客户端展开，但行为隐式且不可靠，统一禁止。

**禁止写法：**

```ts
// ❌ 禁止：数组直接传给 IN (?)，execute() 不展开
await execute(`SELECT * FROM t WHERE id IN (?)`, [materialIds]);

// ❌ 禁止：join(',') 拼成字符串再传给 IN (?)，MySQL 视为单值
const idStr = materialIds.join(',');
await query(`SELECT * FROM t WHERE id IN (?)`, [idStr]);

// ❌ 禁止：参数嵌套数组
await query(`SELECT * FROM t WHERE id IN (?)`, [[1, 2, 3]]);

// ❌ 禁止：query() 依赖隐式展开
await query(`SELECT * FROM t WHERE id IN (?)`, [materialIds]);
```

**强制写法——显式展开占位符 + 扁平化参数：**

```ts
// ✅ 正确：显式生成与数组等长的占位符，参数直接展开（非嵌套）
const placeholders = materialIds.map(() => '?').join(',');
await query(
  `SELECT * FROM t WHERE id IN (${placeholders})`,
  materialIds // 直接传数组，不是 [materialIds]
);

// ✅ execute() 同理
await execute(
  `UPDATE t SET status = 1 WHERE id IN (${placeholders})`,
  materialIds
);
```

**空数组安全降级：**

```ts
// ✅ 空数组时不生成 IN ()（语法错误），改用恒假条件
if (ids.length > 0) {
  const placeholders = ids.map(() => '?').join(',');
  whereClauses.push(`id IN (${placeholders})`);
  params.push(...ids);
} else {
  whereClauses.push('1 = 0'); // 不匹配任何行
}
```

**审查检查点：**
- [ ] SQL 中 `IN` 子句的占位符数量是否与参数数组长度一致
- [ ] 参数是否直接传数组（`ids`），而非嵌套（`[ids]`）或拼接字符串（`ids.join(',')`）
- [ ] 空数组场景是否处理（`1 = 0` 降级）
- [ ] 使用 `execute()` 时尤其注意——它**绝不**展开数组

### 1.2 【P0-禁止】`NOT IN` 数组直接插值（SQL 注入）

**问题描述：** 将数组值直接拼入 SQL 字符串，属于 SQL 注入漏洞。本项目 `src/lib/fifo-allocation.ts` 曾存在此问题，已于 2026-07-10 修复。

**禁止写法：**

```ts
// ❌ 严重：值直接插值，SQL 注入风险
sql += ` AND id NOT IN (${excludeBatchIds.join(',')})`;
```

**强制写法：**

```ts
// ✅ 参数化：占位符展开 + 参数追加
if (excludeBatchIds.length > 0) {
  sql += ` AND id NOT IN (${excludeBatchIds.map(() => '?').join(',')})`;
}
// 参数数组展开追加
const [rows] = await conn.query(sql, [materialId, warehouseId, ...excludeBatchIds]);
```

### 1.3 【P1-禁止】模板字符串插值用户输入

**问题描述：** 任何 `${variable}` 直接出现在 SQL 字符串中，若该变量来源于用户输入或外部数据，均构成 SQL 注入。

**禁止写法：**

```ts
// ❌ 用户输入直接插值
const sql = `SELECT * FROM orders WHERE order_no = '${orderNo}'`;

// ❌ 动态拼接 WHERE 条件含用户输入
let sql = 'SELECT * FROM products';
if (keyword) sql += ` WHERE name LIKE '%${keyword}%'`;
```

**强制写法：**

```ts
// ✅ 参数化占位符
const sql = `SELECT * FROM orders WHERE order_no = ?`;
await query(sql, [orderNo]);

// ✅ 动态 WHERE 使用占位符
let sql = 'SELECT * FROM products WHERE 1=1';
const params: string[] = [];
if (keyword) {
  sql += ` AND name LIKE ?`;
  params.push(`%${keyword}%`);
}
await query(sql, params);
```

### 1.4 【P2-允许】动态表名/列名（需白名单或 escapeId）

动态标识符（表名、列名）**不能**用 `?` 占位符参数化，必须采用以下安全方式之一：

**方式一：白名单校验（推荐）**

```ts
// ✅ 列名白名单
const allowedFields = ['name', 'code', 'status', 'create_time'];
if (!allowedFields.includes(sortField)) {
  throw new Error(`非法排序字段: ${sortField}`);
}
sql += ` ORDER BY ${sortField} ${sortOrder === 'desc' ? 'DESC' : 'ASC'}`;
```

**方式二：`mysql2.escapeId()` 转义**

```ts
import { escapeId } from 'mysql2';

// ✅ 表名转义
sql += ` FROM ${escapeId(tableName)}`;

// ✅ 列名转义
const snakeCol = fieldColumnMap[camelField]; // 显式映射表
sql += ` SET ${escapeId(snakeCol)} = ?`;
```

**禁止：** 使用正则 `validateTableName()` 替代 `escapeId()`——正则无法防御所有注入变体。

## 2. SQL 占位符计数规则

### 2.1 【P0-禁止】占位符与参数数量不匹配

**问题描述：** Bug 1-7 的根因——在含 `NOW()`、`CURDATE()` 等 SQL 函数的 INSERT 语句中，误将函数位置写成 `?`，导致后续参数列错位（参数少传一个，列值整体前移）。

**禁止写法：**

```ts
// ❌ Bug 1：NOW() 位置误写 ?，参数数比列数多一个
await execute(
  `INSERT INTO sys_operation_log (title, create_time, status) VALUES (?, ?, ?, ?)`,
  [title, 1, 1] // 3 个参数，4 个占位符 → create_time 接收 1，status 接收 NULL
);
```

**强制写法：**

```ts
// ✅ SQL 函数直接写，不占占位符位
await execute(
  `INSERT INTO sys_operation_log (title, create_time, status) VALUES (?, NOW(), ?)`,
  [title, 1] // 2 个占位符，2 个参数，一一对应
);
```

**审查检查点：**
- [ ] 逐行核对 SQL 中 `?` 的数量与 `params` 数组长度是否**严格相等**
- [ ] `NOW()`、`CURDATE()`、`COALESCE()`、`IFNULL()` 等内置函数**不占占位符位**
- [ ] `DEFAULT` 关键字不占占位符位
- [ ] `INSERT INTO ... VALUES` 的列数 = 值表达式数 = `?` 数 + SQL 函数数

### 2.2 占位符核对快捷方法

```ts
// 开发/测试时插入断言，快速发现计数错误
const placeholders = sql.match(/\?/g)?.length ?? 0;
const paramCount = params.length;
if (placeholders !== paramCount) {
  throw new Error(`SQL 占位符不匹配: ${placeholders} 个 ? vs ${paramCount} 个参数`);
}
```

## 3. `execute()` vs `query()` 使用规范

| 函数 | 协议 | 数组展开 | 推荐场景 |
|------|------|----------|----------|
| `pool.execute(sql, params)` | 预编译语句 | **不展开** | INSERT/UPDATE/DELETE，重复执行性能优 |
| `pool.query(sql, params)` | 文本协议 | 隐式展开（不可靠） | SELECT，动态 SQL |

**核心规则：** 无论使用哪个函数，`IN` 子句都**必须显式展开占位符**。不依赖 `query()` 的隐式展开行为。

## 4. 动态 SQL 构建规范

### 4.1 字符串拼接 WHERE 子句

```ts
// ✅ 动态 WHERE 使用占位符，参数收集到数组
let whereClause = 'WHERE 1=1';
const params: (string | number)[] = [];

if (status) {
  whereClause += ' AND status = ?';
  params.push(status);
}
if (warehouseId) {
  whereClause += ' AND warehouse_id = ?';
  params.push(warehouseId);
}
if (dateFrom) {
  whereClause += ' AND create_time >= ?';
  params.push(dateFrom);
}

const [rows] = await query(`SELECT * FROM orders ${whereClause}`, params);
```

### 4.2 批量 INSERT

```ts
// ✅ 批量 INSERT 显式展开占位符
const values = items.map(() => '(?, ?, ?, NOW())').join(', ');
const params = items.flatMap(item => [item.code, item.name, item.qty]);
await execute(
  `INSERT INTO t (code, name, qty, create_time) VALUES ${values}`,
  params
);
```

## 5. 代码审查 Checklist

PR 审查时，对每个涉及 SQL 的改动逐条核对：

### SQL 安全
- [ ] 所有 `WHERE`/`SET`/`VALUES` 中的变量值均使用 `?` 占位符，无 `${variable}` 插值
- [ ] `IN`/`NOT IN` 子句使用 `${ids.map(() => '?').join(',')}` 显式展开
- [ ] 动态表名/列名使用白名单或 `escapeId()`，非正则校验
- [ ] 空数组场景已处理（`1 = 0` 降级）

### 占位符计数
- [ ] `?` 数量 === `params` 数组长度（逐行核对）
- [ ] `NOW()`/`CURDATE()`/`COALESCE()` 等函数未误写为 `?`
- [ ] `INSERT` 列数 === 值表达式数

### 参数传递
- [ ] `IN` 子句参数直接传数组（`ids`），非嵌套（`[ids]`）
- [ ] `IN` 子句参数非字符串拼接（`ids.join(',')`）
- [ ] 动态 WHERE 参数数组使用 `push()` 收集，非模板插值

### 测试覆盖
- [ ] 单元测试覆盖多元素数组（≥3 个元素）场景
- [ ] 单元测试覆盖空数组场景
- [ ] 单元测试验证占位符数量与参数数量一致
