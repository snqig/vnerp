# 安全规范

> 基于 `src/lib/auth.ts`、`src/lib/auth-fetch.ts`、`src/lib/api-auth.ts`、`src/lib/rate-limit.ts`、`src/lib/token-blacklist.ts`、`src/lib/csrf.ts`、`src/proxy.ts`、`next.config.ts`、`.env.example` 描述。

## 1. JWT 认证

### 1.1 Token 签发

`src/app/api/auth/login/route.ts` 使用 `jose` 库签发：

```ts
const token = await new SignJWT({ userId, username, realName, roles })
  .setProtectedHeader({ alg: 'HS256' })
  .setIssuedAt()
  .setExpirationTime('24h')
  .sign(new TextEncoder().encode(SECRET_KEY));
```

- 算法：HS256。
- Access Token 有效期：24 小时，写入 `access_token` cookie。
- Refresh Token：`crypto.randomUUID()`，有效期 7 天，存于 `token-blacklist` 模块（Redis / 内存）。
- `JWT_SECRET` 在生产环境必须配置（启动时校验，缺失直接抛错），可用 `crypto.randomBytes(64)` 生成。

### 1.2 Token 校验链路

`withAuth`（`src/lib/api-auth.ts`）按序执行：

1. `extractToken` 提取 `Authorization: Bearer <token>` 或 `access_token` cookie。
2. `verifyToken`（`jose.jwtVerify`）校验签名与有效期，回填 `iat`。
3. `isTokenRevoked(tokenKey)`：单 Token 黑名单（登出 / 主动撤销）。
4. `isUserTokensRevoked(userId, iat)`：用户级撤销——修改密码 / 账号锁定后，所有 `iat < 撤销时间` 的 Token 立即失效。
5. `getUserInfo(userId)`：重新查询用户状态、角色、权限、数据范围。
6. 首次登录强制改密检查（`firstLogin=true` 时仅放行 `/api/auth/change-password`、`/api/auth/logout`、`/api/auth/user-info`、`/api/auth/refresh`）。
7. `hasPermission(user, permission)`：操作权限校验。
8. `validateResourceAccess(user, type, id)`：资源级越权防护（`resourceType` 支持 `'order' | 'workorder' | 'inbound' | 'outbound'`）。

校验失败：401（未认证） / 403（无权限 / 资源越权 / 首次登录未改密）。

### 1.3 密码与登录安全

- 密码使用 `bcryptjs` 哈希，`bcrypt.compare` 验证。
- 失败计数：连续 5 次密码错误锁定账号 15 分钟（`MAX_LOGIN_ATTEMPTS=5`、`LOCKOUT_MINUTES=15`）。
- 异地登录检测：对比 `last_login_ip` 与当前 IP，不一致时写入 `sys_notification`（type=`security`）告警。
- 密码过期：依据 `sys_config.system.password_expire_days` 配置。
- 首次登录强制改密：`first_login=1` 时返回 `passwordExpired=true`，前端引导改密。
- 登录日志写入 `sys_login_log`：用户名、IP、浏览器、OS、状态、消息。

### 1.4 401 无感刷新

`src/lib/auth-fetch.ts` 封装 `authFetch`：

- 收到 401（且非 `/api/auth/refresh` 自身）触发 `refreshAccessToken`。
- 并发 401 复用同一 `refreshPromise`，避免多次刷新互相覆盖。
- 刷新成功 → 写回 localStorage / sessionStorage → 重试原请求一次。
- 刷新失败 / 无 refreshToken → 清除 `token` / `refreshToken` / `userId`，跳转 `/login`。
- Refresh 请求本身使用裸 `fetch`，避免死循环。

## 2. RBAC 权限模型

`用户(User) → 角色(Role) → 权限(Permission) → 菜单/按钮/API`

- `sys_user` ↔ `sys_user_role` ↔ `sys_role`
- `sys_role` ↔ `sys_role_menu` ↔ `sys_menu`（菜单 / 权限点）
- `sys_role.data_scope`：`all` / `dept` / `self` / `dept_and_self` / `custom`。
- `sys_data_scope`：角色级仓库 / 客户 / 供应商白名单。
- `hasPermission`：admin / super_admin 拥有 `*` 通配。
- `buildDataScopeSql`：根据数据范围自动拼接 SQL 条件，避免越权读取。

## 3. Rate Limiting

`src/lib/rate-limit.ts` 提供基于内存 `Map` 的滑动窗口限流：

```ts
checkRateLimit(identifier, { windowMs, maxRequests, keyPrefix })
```

- 登录接口：每 IP 15 分钟 20 次（`/api/auth/login`）。
- 超限返回 429，携带 `Retry-After` / `X-RateLimit-Reset` 头。
- 内存 Map 每 60 秒清理过期项；最大 10000 条，LRU 淘汰。
- `getClientIP`：从 `x-forwarded-for` / `x-real-ip` 提取客户端 IP。

> 当前限流为进程内内存，多实例部署需替换为 Redis 实现（与 token-blacklist 一致）。详见 [已知问题](./已知问题.md) RUN-001。

## 4. CORS 配置

`next.config.ts`：

- `corsAllowOrigin`：从 `CORS_ALLOW_ORIGIN` 读取，默认 `*`（开发）；生产应配置具体域名。
- 所有路由响应携带：
  - `Access-Control-Allow-Origin`
  - `Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS`
  - `Access-Control-Allow-Headers: Content-Type, Authorization`

## 6. CSRF 防护

`src/lib/csrf.ts` 提供 CSRF token 双提交 cookie 模式：

- `ensureCsrfCookie(response)`：在页面响应中种入 CSRF cookie。
- `requiresCsrfValidation(request)`：判断请求是否需要 CSRF 校验（POST/PUT/PATCH/DELETE 等状态变更请求）。
- `validateCsrfToken(request)`：校验请求头 / body 中的 CSRF token 与 cookie 一致。
- `src/proxy.ts` 在 API 路由层调用 `requiresCsrfValidation` + `validateCsrfToken`，校验失败返回 403。

## 7. 中间件 `src/proxy.ts`

`src/proxy.ts`（Next.js 16 自动探测，替代已删 `src/middleware.ts`）：

- **i18n**：`createIntlMiddleware`，`locales=[zh-CN, zh-TW, en, vi]`，`defaultLocale=zh-CN`，`localePrefix='as-needed'`。
- **生产环境封堵调试路由**：`/qrcode`、`/api/init`、`/api/debug`、`/api/diagnose` 返回 404。
- **API 路由**（`/api/`）：
  - 公开 API（`/api/auth/login`、`/api/auth/register`、`/api/health`、`/api/migrations`）放行。
  - 其余须携带 `access_token` cookie（无则 401）。
  - `requiresCsrfValidation` 的请求校验 CSRF token（失败 403）。
- **页面鉴权**：26 个受保护前缀（dashboard/warehouse/sales/purchase/production/quality/hr/finance/equipment/engineering/outsource/orders/reports/settings/dcprint/sample/analysis/base-data/business/srm/prepress/plm/delivery/crm/tools/material-requisitions/modules/qrcode）无 `access_token` → 重定向 `/login?next=原路径`；已登录访问 `/login`/`/register`/`/forgot-password`/`/reset-password` → 重定向 `/dashboard`。
- **config.matcher**：页面路径（排除 `api/_next/_vercel/.*\\..*`）+ `/api/:path*` + `/qrcode/:path*`。
- 注：proxy 只校验 cookie 存在性（Edge runtime 不解析 JWT），JWT 实际有效性由 SSR `layout.tsx` 与 API 路由 `verifyToken` 校验。

## 8. 输入安全

- SQL：所有查询使用参数化占位符 `?`，`IN`/`NOT IN` 必须显式展开占位符（见 [代码审查规范](./代码审查规范.md) 与 [数据库规范](./数据库规范.md)）。
- XSS：响应统一经 `sanitizeObject` HTML 实体转义；React 默认转义。
- CSRF：见第 6 节，状态变更请求强制校验 CSRF token。
- 文件上传：`/api/upload`（含 `/api/upload/sop`、`/api/upload/contract`）校验类型与大小。
- 配置 API：`ALLOW_SETUP_API=false` 时 `/api/setup/*`、`/api/init/*` 在生产被 proxy 层封堵，仅初始化时临时开启。

## 9. 环境变量管理

`.env.example` 关键变量：

| 变量 | 用途 |
|------|------|
| `DB_HOST` / `DB_PORT` / `DB_USER` / `DB_PASSWORD` / `DB_NAME` | MySQL 连接 |
| `JWT_SECRET` | JWT 签名密钥（生产必填，可用 `crypto.randomBytes(64)` 生成） |
| `DEV_ORIGINS` | 开发环境允许的源（逗号分隔） |
| `CORS_ALLOW_ORIGIN` | CORS 允许源（默认 `*`，生产应改具体域名） |
| `EVENT_BUS_TYPE` | `memory` 或 `db`（Outbox 持久化，生产推荐 `db`） |
| `REDIS_URL` | Redis 连接，用于 token 黑名单 / refresh token 共享 / 事件总线 |
| `ALLOW_SETUP_API` | 建表 / 初始化接口开关（生产必须 `false`） |
| `DEBUG_DB` | 数据库查询日志开关 |

- `.env` / `.env.local` / `.env.production.example` 分层管理；密钥不入库、不入日志。
- `secureLog` / `maskSensitiveData` 对敏感字段脱敏后再写入日志。

## 10. 日志与审计

- 操作日志：`sys_operation_log`（见 [系统可审计性设计方案](./系统可审计性设计方案.md)）。
- 登录日志：`sys_login_log`。
- 异常日志：`logger` 模块统一输出，带 `traceId` 串联调用链。
- 保留期：操作日志 180 天、登录日志 90 天、异常日志 30 天；日志不可篡改。

## 11. 事件总线与 Outbox 安全

- 事件总线：Redis Streams（流名 `erp:domain-events`，`STREAM_MAX_LENGTH=10000`，`STREAM_RECLAIM_IDLE_MS=60s`）+ Outbox 两阶段。
- `sys_event_processed`：`checkAndMark` → `status='processing'`，`markAsProcessed` → `status='processed'`，失败 `deleteMark`。
- `XAUTOCLAIM` 回收死消费者；`OutboxPoller` 定时回收过期 `processing`（`IDEMPOTENCY_STALE_THRESHOLD_MINUTES=5min`）+ 清理 30 天前已处理。
- 无 Redis 时降级 direct publish（`EVENT_BUS_TYPE=memory` 不启动 `OutboxPoller`，须 `EVENT_BUS_TYPE=db`）。生产环境强制 `db` 模式。
