# 已知问题与待优化项

> 文档编号：VNERP-TECH-011 | 版本：V2.1 | 基于：代码现状 2026-07-10
> 配合 `.trae/documents/doc-fact-base.md` §4（schema）、§5（API）、§6（事件处理器）核实。

## 一、数据模型层

### DATA-001 Schema 与运行时 SQL 表名不一致【高优先级】

**现象**：`src/lib/db/schema.ts`（Drizzle 定义，42 张消费表）与 API Route 运行时 SQL 使用的表名存在部分不一致（主要在遗留非 Drizzle 消费表上）。

| schema.ts 定义 / Drizzle 消费 | 运行时 SQL 实际使用 | 说明 |
|---------------|-------------------|------|
| `inv_inbound_order` 等仓库 8 表 | 一致 | 已对齐 |
| `sal_order` 等销售 7 表 | 一致 | 已对齐 |
| `pur_purchase_order` + `pur_purchase_order_line` | 部分代码仍用 `pur_order` | 双重命名冲突 |
| `fin_receivable` / `fin_payable` | `finance_receivable` / `finance_payable`（aging 路由） | 见 DATA-003 |
| 遗留表（`crm_customer`、`pur_supplier`、`hr_employee`、`qc_*`、`sys_*` 等） | 一致 | 非 Drizzle 消费，由原始 SQL 管理 |

**影响**：`drizzle-kit generate` 路径已废弃（`drizzle/` 目录已清理），但 ORM 查询构建器活跃使用中。对未对齐的表，`drizzleDb.query.*` 类型化查询可能失效。
**建议**：将 schema.ts 全部对齐为运行时表名；新增表必须同时纳入 schema.ts 与 `database/vnerpdacahng_schema.sql`。

### DATA-002 actual_schema.sql 实际为空【中优先级】

**现象**：`database/actual_schema.sql` 仅含 `DROP TABLE IF EXISTS v_purchase_request_full;`，并非真实表结构。
**影响**：新环境部署无可用 SQL 脚本。
**建议**：权威 schema 来源为 `database/vnerpdacahng_schema.sql`（从目标数据库 `SHOW CREATE TABLE` 导出）；新增表结构变更通过 `pnpm migrate:create` 生成增量迁移（`database/migrations/`，编号 001-060）。

### DATA-003 财务模块三套表名并存【高优先级】

**现象**：财务相关代码同时引用三套表名：

| 来源 | 应收表名 | 应付表名 | 字段 |
|------|---------|---------|------|
| `src/lib/finance-core.ts` | `fin_receivable` | `fin_payable` | `balance`, `received_amount`, `status`(1/2/3) |
| `src/app/api/finance/aging/route.ts` | `finance_receivable` | `finance_payable` | `remaining_amount`, `status='settled'` |
| `src/lib/soft-delete.ts` 配置 | `fin_receivable` | `fin_payable` | `cancelStatus='cancelled'`(字符串) |

**影响**：账龄分析查询必然失败（表不存在）；软删除 `cancelled` 与数字状态不兼容。
**建议**：统一为 `fin_receivable`/`fin_payable`，状态改为枚举；060 迁移已统一部分状态码，但 aging 路由仍需修复。

### DATA-004 schema.ts 文件头注释过时【低优先级】

**现象**：`src/lib/db/schema.ts` 文件头注释写「35 张核心业务表」，实际已有 42 张 Drizzle 消费表（仓库 8 + 销售 7 + 采购 4 + 财务 2 + 生产 12 + 打样 3 + 印前 6）。
**建议**：更新文件头注释为 42 张。

## 二、库存与单据层

### INV-001 soft-delete 表名与库存 API 不匹配【高优先级】

**现象**：`src/lib/soft-delete.ts` 的 `CANCELLABLE_DOCUMENTS` 配置使用 `warehouse_inbound`/`warehouse_outbound`/`warehouse_stocktaking`/`warehouse_transfer`，而 `src/app/api/warehouse/inventory/route.ts` 实际操作 `inv_inventory_batch`、`inv_inbound_order`、`inv_outbound_order`、`inv_stocktaking`、`inv_transfer_order`。
**影响**：调拨/盘点作废调用 `quickCancel('transfer_order', ...)` 会失败。
**建议**：对齐为 `inv_` 前缀表名，并补齐 `inbound_order`/`outbound_order` 的真实表。

### INV-002 制造费用分摊为硬编码【中优先级】

**现象**：`src/lib/finance-core.ts` 中 `manufacturingCost = laborCost * 0.5`，注释标明"简化计算"。
**影响**：成本核算结果不准确，无法满足月末结账要求。
**建议**：接入车间制造费用总额表，按工时比例分摊；可参考 `src/lib/cost-engine.ts` 的扩展实现。

### INV-003 FIFO 分配 SQL 注入已修复【已解决】

**现象**：`src/lib/fifo-allocation.ts` 曾存在 `NOT IN (${excludeBatchIds.join(',')})` 直接插值，构成 SQL 注入。
**修复**：2026-07-10 已改为占位符展开 + 参数追加（见 [代码审查规范](./代码审查规范.md) §1.2）。

## 三、运行时基础设施

### RUN-001 限流为进程内内存【中优先级】

**现象**：`src/lib/rate-limit.ts` 使用内存 `Map` 存储（60s 清理，max 10000 条）。
**影响**：多实例部署时限流失效，单实例上限被绕过。
**建议**：切换为 Redis 计数器（项目已引入 `ioredis`）。

### RUN-002 Token 黑名单多实例不同步【高优先级】

**现象**：`src/lib/token-blacklist.ts` 优先 Redis，但 Redis 不可用时降级为内存 `Set`，且未持久化。
**影响**：实例重启或 Redis 故障期间，已注销 Token 可继续访问。
**建议**：生产环境强制 Redis，禁用降级；Redis 故障应快速失败而非降级。

### RUN-003 事件总线仅内存/DB 两选【低优先级】

**现象**：`EVENT_BUS_TYPE=memory|db`，memory 模式不支持多实例。
**建议**：生产环境强制 `db`（Outbox 模式 + Redis Streams），参考 [系统可审计性设计方案](./系统可审计性设计方案.md) 与 [安全规范](./安全规范.md) §11。

### RUN-004 DevOps 脚本路径已变更【低优先级】

**现象**：`pnpm db:generate` / `db:push` / `db:migrate` 已废弃（运行仅打印警告），应改用 `pnpm setup:db`（全量初始化）与 `pnpm migrate` / `migrate:down` / `migrate:status` / `migrate:create`（增量迁移）。
**建议**：文档与 CI 脚本统一替换命令名。

## 四、测试体系

### TEST-001 部分测试因表缺失失败【中优先级】

**现象**：`tests/unit/` 下部分测试文件失败，主因是 Mock 未覆盖的表或 schema 不一致（如 `sal_order_detail` 缺 `material_name` 字段、`fin_*` 表名不匹配）。
**影响**：CI 流水线不稳定，覆盖率统计存在 flaky。
**建议**：统一 Mock 表结构，对齐真实字段名；按 060 迁移统一状态码后重跑。

### TEST-002 测试环境 window 未定义【低优先级】

**现象**：`tests/unit/setup.ts` 引用 `window` 在 Node 环境报错，已通过 `typeof window !== 'undefined'` 修复。
**建议**：将浏览器全局 Mock 移至 `vitest.setup.ts`。

## 五、DDD 分层债务

### DDD-001 领域层违规导入【中优先级】

**现象**：`eslint.config.mjs` 中 `ddd/layer-dependencies` 规则为 `warn` 级别，存在历史债务：
- `src/domain/production/repositories/IScheduleRepository.ts` 导入 `@/lib/db/schema`
- `src/domain/warehouse/value-objects/WarehouseStateMachine.ts` 导入 `@/lib/logger`

**缓解**：`allowPatterns.domain` 白名单允许 `@/lib/logger`、`@/lib/decimal-utils`、`@/lib/money`，逐步清理后移除白名单。
**建议**：领域层零框架依赖，logger 迁移为领域异常或纯函数。

### DDD-002 equipment 与 hr 模块缺领域层【中优先级】

**现象**：equipment 和 hr 模块仍为"原生 SQL + 胖路由"架构，无聚合根、无状态机、无领域事件。
**建议**：按 [DDD分层迁移方案-equipment-hr](./DDD分层迁移方案-equipment-hr.md) 迁移至 DDD 分层（标杆 warehouse）。
