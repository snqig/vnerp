# 开发规范

> 基于 `CONTRIBUTING.md`、`.husky/`、`commitlint.config.mjs` 描述提交门禁、分支策略与评审流程。

## 1. 环境与工具链

| 工具 | 版本 | 说明 |
|------|------|------|
| Node.js | 18+ | 推荐 LTS |
| pnpm | 9.0+ | `package.json` 中 `preinstall: only-allow pnpm` 强制使用 |
| MySQL | 8.0+ | 字符集 `utf8mb4` |

初始化：

```bash
pnpm install
pnpm setup:db --seed   # 建表 + 种子
pnpm dev               # 监听 :5000
```

## 2. 提交规范（Conventional Commits）

遵循 [Conventional Commits](https://www.conventionalcommits.org/)，格式：

```
<type>(<scope>): <subject>

<body>

<footer>
```

| type | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档 |
| `refactor` | 重构（不改变外部行为） |
| `test` | 测试 |
| `build` | 构建/依赖 |
| `ci` | CI 配置 |
| `chore` | 维护 |

示例：

```
feat(warehouse): 添加入库单审核功能
fix(auth): 修复 401 刷新并发问题
docs(readme): 更新快速开始步骤
```

`commitlint.config.mjs` 继承 `@commitlint/config-conventional`，并放宽 `subject-case` 以允许中英文混合 subject。

## 3. Husky Hooks

`.husky/pre-commit`：

```sh
pnpm exec lint-staged
```

`package.json` 中 `lint-staged` 配置：

```json
{
  "src/**/*.{ts,tsx,js,jsx}": ["eslint --fix", "prettier --write"]
}
```

仅对暂存文件执行 ESLint 自动修复 + Prettier 格式化，避免全量扫描拖慢提交。

`.husky/commit-msg`：

```sh
pnpm exec commitlint --edit $1
```

校验提交信息是否符合 Conventional Commits；不合规将被拒绝。

## 4. 质量检查

提交前请确保：

- `pnpm lint` 通过（含 `i18n/no-chinese-hardcode` 告警处理）。
- `pnpm ts-check` 通过（TypeScript 严格模式无错误）。
- `pnpm test:unit` 通过（vitest 单元测试）。
- 新增功能配套单元测试，覆盖核心业务规则。

测试命令速查：

```bash
pnpm test:unit         # vitest watch
pnpm test:unit:run     # vitest run
pnpm test:coverage     # 覆盖率
pnpm test              # playwright e2e
pnpm test:ui           # playwright 交互模式
```

## 5. 分支策略

| 分支 | 用途 | 合入方式 |
|------|------|----------|
| `main` | 生产分支 | 仅接受 PR，禁止直推 |
| `develop` | 集成分支 | PR 合入 |
| `feat/<scope>-<desc>` | 功能开发 | PR → develop |
| `fix/<scope>-<desc>` | Bug 修复 | PR → develop |
| `release/<version>` | 发布冻结 | PR → main + develop |
| `hotfix/<desc>` | 紧急修复 | PR → main + develop |

- 长期分支（`main` / `develop`）受保护，禁止 force push。
- 功能分支命名建议携带 scope（模块名），如 `feat(auth)-jwt-refresh`。

## 6. 代码评审流程

1. 开发者创建功能分支并提交，确保 CI（`.github/workflows/ci.yml`）通过。
2. 在 GitHub 发起 PR，填写 PULL_REQUEST_TEMPLATE.md 中的变更说明、影响范围、自测项。
3. 至少一名 CODEOWNERS（`.github/CODEOWNERS`）审阅者 approve。
4. 评审重点：
   - 业务规则是否符合 [业务规则-*](./00-技术规范总览.md) 系列文档。
   - 是否引入硬编码中文 / 硬编码颜色（应被 ESLint 拦截）。
   - 数据库操作是否参数化、是否使用事务、是否影响库存流水。
   - 鉴权是否完整（`withAuth` / `withAuthAndErrorHandler`）。
5. 审阅通过后由作者 squash merge；冲突由作者负责 rebase。
6. 合入 `develop` 后触发集成测试；release 分支冻结后合入 `main`。

## 7. CI 流水线

`.github/workflows/ci.yml` 在 PR 与 push 时执行：

- 依赖缓存 + `pnpm install`
- `pnpm lint`、`pnpm ts-check`
- `pnpm test:unit:run`
- Playwright（`pnpm test`，按需触发）

Chromatic（`.github/workflows/chromatic.yml`）用于 Storybook 视觉回归。
