# ERP 模块联动重构方案  
## 目标：消除模块空心化，打通端到端业务链路，实现事件驱动的松耦合联动  

## 一、现状与风险
- 已完成工程底座、目录分层、页面骨架，但各模块独立开发，逻辑零散，缺乏联动。
- 数据库 Schema 存在双轨制（Drizzle + 纯 SQL），未统一。
- 事件总线和状态机已基建，但未形成统一开发规范。
- 核心业务逻辑未落地（MRP、成本核算、SPC 等），仅搭了架子。

## 二、总体策略
- **先通链路，再填深度**：优先跑通「销售订单 → MRP → 生产工单 → 领料入库 → 销售出库 → 财务凭证」主价值链，串联四大模块。
- **先立规范，再写代码**：统一数据库 Schema、事件契约、状态机范式，再开发业务。
- **严格分层**：领域层收敛业务规则，应用层编排流程，模块间仅通过领域事件通信。

---

## 三、第一阶段：夯实底层契约（2 周）
### 1. 统一数据库 Schema（P0）
- 废弃 `vnerpdacahng_schema.sql`，所有表定义迁移到 `src/lib/db/schema.ts`，以 Drizzle 为唯一真值。
- 建立版本化迁移：表结构变更必须通过 `pnpm db:generate` 生成迁移文件，禁止手动改 SQL。
- 所有业务表补充通用字段：`created_at`, `updated_at`, `created_by`, `updated_by`, `is_deleted`。
- 核心关联表添加外键约束，高频查询字段建立索引。

**具体操作**：
- 检查并同步 `schema.ts` 与现有 SQL，确保完全一致。
- 配置 Drizzle 迁移脚本，修改 `setup:db` 命令为执行迁移 + 种子数据。

### 2. 落地领域事件总线开发规范
- **事件命名**：`实体+动作+时态+Event`，如 `InboundOrderApprovedEvent`。
- **事件结构**：继承 `DomainEvent`，仅包含主键、关键状态、操作人，**禁止传递完整实体**。
- **事务性发件箱**：
  - 新建 `sys_event_outbox` 表（字段：id, event_id, event_type, payload, status, retry_count, created_at, processed_at）。
  - 业务操作与事件写入在同一个数据库事务内完成。
  - 事件投递异步进行，支持重试和死信队列。

**具体操作**：
- 在 `src/domain/shared/events/` 定义基类和接口。
- 在 `src/infrastructure/events/` 实现发件箱、投递器、重试逻辑。
- 改造聚合根基类，支持暂存事件（`_domainEvents`）。

### 3. 统一状态机开发范式
- 封装通用单据状态机抽象类，提供 `状态枚举、动作、前置校验、后置事件、审计` 能力。
- 所有单据（入库单、工单、销售单等）复用该抽象类，严禁在页面/API 层直接修改状态字段。

## 四、第二阶段：打通端到端主链路（4~6 周）
### 主链路流程

### 事件联动设计（核心）
| 节点动作 | 发布事件 | 订阅方联动 |
|---------|---------|-----------|
| 销售订单审核 | SalesOrderApprovedEvent | 生产模块触发 MRP；库存模块锁定库存 |
| 生产工单下发 | ProductionOrderReleasedEvent | 仓储生成领料建议；成本模块创建生产成本单 |
| 入库单审核 | InboundOrderApprovedEvent | 库存增加库存；成本更新单位成本；财务生成入库凭证 |
| 出库单审核 | OutboundOrderApprovedEvent | 库存扣减库存；财务结转成本 + 生成应收凭证 |

**各模块本阶段落地边界**：
- **生产管理**：工单全生命周期（创建→下发→领料→完工→结案），单层 BOM 展开 + 无限产能 MRP。
- **仓储管理**：补全出库、库存台账、批次库存，所有库存变动由单据驱动。
- **财务管理**：应收应付台账，出入库凭证自动生成，移动加权平均成本。
- **销售管理**：订单审核、发货出库、退货逆向流程。

**具体实现**：
- 每个模块在 `src/application/handlers/` 下编写事件处理器，实现幂等（按 event_id 去重）。
- 处理器仅调用本模块应用服务，不跨库操作。

## 五、第三阶段：深化核心模块专业能力（6~8 周）
按优先级顺序推进：

### 1. 生产管理
- MRP 升级：多层 BOM 展开，纳入在制工单、在途采购、安全库存、损耗率。
- 有限能力排产（简化版）：按设备/班组产能顺排，输出每日任务。
- 生产执行：报工、工时统计、不良品处理、退料，完工入库自动核算实际工时与物料消耗。

### 2. 财务管理
- 生产成本归集：工单原材料、人工工时、制造费用分摊到入库产品。
- 成本卷积：多层 BOM 成本逐层卷积。
- 财务凭证自动化：业务单据自动生成记账凭证，支持审核、红冲。
- 基础报表：应收/应付账龄、库存资金占用、生产成本明细。

### 3. 质量管理
- 三级检验：IQC（来料）、IPQC（过程）、FQC（成品）完整流程。
- SPC 统计：数据采集 → 控制图（X-R 图）→ 异常判定（八大判异规则）。
- 不合格品处理：评审、返工/报废/特采流程。

### 4. 印前管理（差异化核心）
- 工艺卡标准化（刀模、网版、油墨配方、色序、后工序）。
- BOM 与工艺卡联动，工单自动带出工艺要求。
- 工艺链路追溯：正向/反向追溯原料→工序→设备→人员→检验→成品。
- 刀模/网版资产管理（使用次数、寿命、保养提醒）。

## 六、第四阶段：生产级加固（持续）
- 权限与数据安全：按钮级、数据级权限校验，敏感数据脱敏。
- 异常与兜底：负库存拦截、反写异常回滚、事件消费失败告警。
- 批量操作：批量审核、导入导出、高级筛选。
- 测试覆盖：E2E 测试核心链路，单元测试领域逻辑。

|------|------|---------|---------|

## 八、关键避坑指南（必须遵守）
1. **开发顺序**：领域层 → 应用层 → API → 页面，先保证业务逻辑正确可测。
2. **禁止跨模块直接调用**：模块间通信只能通过领域事件，防止强耦合。
3. **禁止在事件中传递完整实体**：只传主键和核心状态，详情由订阅方自行查询。
4. **必须幂等**：所有事件处理器实现按事件 ID 去重。
5. **不强求一步到位**：MRP、成本、SPC 先简化，迭代深化，避免延期。
6. **每个迭代必须可验证**：有一条完整的业务流程能从头到尾跑通。
