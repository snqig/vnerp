# 系统可审计性设计方案

> 基于 `src/lib/audit-logger.ts`、`src/lib/soft-delete.ts`、`src/lib/data-diff.ts`、`src/app/api/audit/`、`src/lib/api-response.ts#logOperation` 等代码现状梳理。

## 1. 设计目标

> **所有操作留痕，所有修改留底，所有单据不可删，所有变动有据可查。**

ERP 必须满足五大审计要求：操作可追溯、数据可对账、责任可认定、篡改可发现、错误可回查。

## 2. 核心审计表

| 表 | 用途 |
|----|------|
| `sys_operate_log` | 操作日志：模块 / 类型 / 操作人 / 前后数据 / IP / 耗时 |
| `sys_login_log` | 登录日志：用户名 / IP / 浏览器 / OS / 状态 / 消息 |
| `inv_inventory_transaction` | 库存流水：trans_no / trans_type / source / 数量 / 前后值 / 单价 |
| `inv_inventory_log` | 库存变动明细日志 |
| `fin_voucher` / `fin_voucher_line` | 财务凭证与明细 |
| `cancel_record` | 单据作废记录：原数据快照 + 作废原因 |
| `sys_permission_change_log` | 权限变更日志 |

标准审计字段：`module` / `type` / `title` / `username` / `user_id` / `before_data` / `after_data` / `ip` / `user_agent` / `request_url` / `request_method` / `request_param` / `response_result` / `status` / `error_msg` / `duration_ms`。

## 3. 八大手段

### 3.1 全流程操作日志

所有增 / 删 / 改 / 审核 / 反审 / 导入 / 导出 / 打印 必须记录。`src/lib/api-response.ts#logOperation` 写入 `sys_operation_log`，关键字段截断 2000 字。

### 3.2 关键数据变更前后对比

`src/lib/data-diff.ts` 提供 `diffData` / `createSnapshot`：金额、数量、价格、状态、库存变动必须记录 `before_data` 与 `after_data` JSON。

### 3.3 单据不可物理删除

`src/lib/soft-delete.ts` 实现：

- `softDelete`：仅 `deleted=1`，已审核单据抛错（`status >= 20` 禁止删除）。
- `cancelDocument`：`status` 改为作废态，同时写 `cancel_record` 快照原数据。
- `CANCELLABLE_DOCUMENTS` 配置：采购订单 / 销售订单 / 工单等的状态机与作废态映射。

### 3.4 业务必须走审核流程

无审核不生效：采购订单 → 入库、销售订单 → 出库、出入库单 → 库存、生产工单 → 领料、应收应付 → 凭证。审核记录包含 `approver_id` / `approval_action` / `approval_comment` / `approval_time`。

### 3.5 库存流水不可修改、不可删除

`inv_inventory_transaction`：

- 仅 INSERT，无 UPDATE / DELETE 接口。
- 必含 `source_type` / `source_id` / `source_no`，可追溯到原单据。
- 必含 `quantity` / `stock_before` / `stock_after` / `unit_price` / `total_amount`。

### 3.6 权限最小化

角色权限矩阵：销售专员 / 销售主管 / 仓库管理员 / 财务专员 / 系统管理员——开单、审核、修改、删除、作废、查看财务分别授权。`sys_permission_change_log` 记录 grant / revoke。

### 3.7 登录日志与异常监控

`sys_login_log` 记录每次登录尝试。异常规则：密码错误 5 次锁定 15 分钟、异地登录告警、异常时段登录、IP 黑名单。

### 3.8 财务数据自动留痕

`fin_voucher` + `fin_voucher_line`：所有财务操作必须基于业务单据（`source_type` + `source_id`），禁止无单据调整；出入库自动生成应收 / 应付。

## 4. 代码实现要点

- AOP 包装：`withAuthAndErrorHandler` / `withErrorHandler` 自动捕获异常并记录耗时。
- 手动埋点：`AuditLogger.log(entry)` 写 `sys_operate_log`；`logDataChange` 写数据变更日志；`logDocumentCancel` 写作废记录；`logStockFlow` / `logFinanceFlow` 写流水。
- 敏感数据脱敏：`maskSensitiveData` 处理密码 / token 等字段后再落库。
- 事务保护：审计写入与业务写入同事务，避免日志丢失；事务失败时通过 `console.error` 兜底。
- `executeWithAudit`：执行 + 计时 + 自动审计的成功 / 失败两条记录。

## 5. 审计 API

`src/app/api/audit/logs/route.ts` 提供：

- `logType=operate`：操作日志（按 module / type / username / status / 时间区间筛选）。
- `logType=login`：登录日志。
- `logType=stock`：库存流水。
- `logType=finance`：财务流水。

`/api/audit/report` 生成审计汇总报表（按日期 / 模块 / 类型聚合成功 / 失败 / 平均耗时）。

## 6. 上线检查清单

- [ ] 所有写操作调用 `logOperation` 或 `AuditLogger.log`。
- [ ] 关键数据变更记录 `before_data` / `after_data`。
- [ ] 库存变动 100% 写入 `inv_inventory_transaction`。
- [ ] 财务变动 100% 写入 `fin_voucher`。
- [ ] 已审核单据不可删除 / 篡改，仅可作废并写 `cancel_record`。
- [ ] 审核记录可追溯审批人。
- [ ] 日志可导出、可留存，满足财务合规审计要求。
