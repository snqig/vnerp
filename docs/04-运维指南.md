# 一、数据库维护指南

## 1.1 数据库备份策略

### 1.1.1 备份类型与频率

| 备份类型 | 频率 | 保留时间 | 用途 |
|---------|------|---------|------|
| **全量备份** | 每日 02:00 | 30天 | 完整恢复 |
| **增量备份** | 每4小时 | 7天 | 快速恢复到某时间点 |
| **事务日志备份** | 每30分钟 | 3天 | 精确到秒的恢复 |

### 1.1.2 自动备份脚本（MySQL）

```bash
#!/bin/bash
# vnerp_db_backup.sh - VNERP数据库自动备份脚本

# 配置变量
DB_HOST="localhost"
DB_PORT="3306"
DB_USER="vnerp_admin"
DB_PASS="your_secure_password"
DB_NAME="vnerp_db"
BACKUP_DIR="/data/backup/mysql"
DATE=$(date +%Y%m%d_%H%M%S)
RETENTION_DAYS=30

# 创建备份目录
mkdir -p $BACKUP_DIR/$DATE

# 全量备份（mysqldump）
echo "[$(date)] 开始全量备份..."
mysqldump -h$DB_HOST -P$DB_PORT -u$DB_USER -p$DB_PASS \
  --single-transaction \
  --routines \
  --triggers \
  --events \
  --databases $DB_NAME \
  | gzip > $BACKUP_DIR/$DATE/vnerp_full_$DATE.sql.gz

if [ $? -eq 0 ]; then
  echo "[$(date)] 全量备份完成: $BACKUP_DIR/$DATE/vnerp_full_$DATE.sql.gz"
  
  # 记录备份大小
  BACKUP_SIZE=$(du -h $BACKUP_DIR/$DATE/vnerp_full_$DATE.sql.gz | cut -f1)
  echo "[$(date)] 备份文件大小: $BACKUP_SIZE"
else
  echo "[$(date)] ❌ 备份失败！请检查错误日志"
  exit 1
fi

# 清理过期备份
echo "[$(date)] 清理${RETENTION_DAYS}天前的旧备份..."
find $BACKUP_DIR -type f -name "*.sql.gz" -mtime +$RETENTION_DAYS -delete
find $BACKUP_DIR -type d -empty -mtime +$RETENTION_DAYS -delete

echo "[$(date)] ✅ 备份任务完成"

# 发送备份结果通知（可选）
# curl -X POST "https://your-notification-webhook" \
#   -H "Content-Type: application/json" \
#   -d "{\"status\": \"success\", \"size\": \"$BACKUP_SIZE\"}"
```

## 1.2 数据库恢复流程

### 1.2.1 全量恢复（从全量备份）

```bash
#!/bin/bash
# vnerp_db_restore.sh - 数据库恢复脚本

# 配置变量
DB_HOST="localhost"
DB_PORT="3306"
DB_USER="vnerp_admin"
DB_PASS="your_secure_password"
DB_NAME="vnerp_db"
BACKUP_FILE=$1  # 参数：备份文件路径

if [ -z "$BACKUP_FILE" ]; then
  echo "用法: $0 <备份文件路径>"
  echo "示例: $0 /data/backup/mysql/20260511_020000/vnerp_full_20260511_020000.sql.gz"
  exit 1
fi

# 检查备份文件是否存在
if [ ! -f "$BACKUP_FILE" ]; then
  echo "❌ 备份文件不存在: $BACKUP_FILE"
  exit 1
fi

echo "⚠️  警告：此操作将覆盖现有数据库！"
read -p "是否继续？(yes/no): " confirm

if [ "$confirm" != "yes" ]; then
  echo "操作已取消"
  exit 0
fi

# 停止应用服务（防止数据写入冲突）
echo "停止应用服务..."
# systemctl stop vnerp-app  # 根据实际部署方式调整

# 执行恢复
echo "开始恢复数据库，这可能需要几分钟..."
gunzip < $BACKUP_FILE | mysql -h$DB_HOST -P$DB_PORT -u$DB_USER -p$DB_PASS

if [ $? -eq 0 ]; then
  echo "✅ 数据库恢复成功！"
  
  # 重启应用服务
  echo "重启应用服务..."
  # systemctl start vnerp-app
  
  echo "✅ 系统已恢复正常运行"
else
  echo "❌ 数据库恢复失败！请检查错误信息"
  exit 1
fi
```

## 2.2 常用配置场景指南

### 场景一：调整刀模使用寿命

**业务背景**：更换了更高质量的刀模供应商，实际使用寿命从90天延长至120天。

**操作步骤**：

1. 登录系统 → 系统设置 → 全局系统配置
2. 找到"刀模寿命管理"分类
3. 修改"刀模有效天数"：90 → 120
4. 填写修改原因："新供应商刀模质量提升"
5. 提交审批
6. 等待审批通过

**预期效果**：
- ✅ 新激活的刀模有效期变为120天
- ✅ 预警时间自动调整为提前15天（第105天开始预警）
- ✅ 已有未过期刀模的剩余天数重新计算
- ⚠️ 已过期的刀模不会自动复活

### 场景二：启用/禁用FIFO规则

**业务背景**：紧急补料时需要灵活选择批次，暂时关闭FIFO强制执行。

**操作步骤**：

1. 进入配置中心
2. 找到"仓库管理规则"分类
3. 修改"是否强制先进先出FIFO"：true → false
4. 提交审批（说明紧急情况）
5. 审批通过后立即生效

**注意事项**：
- ⚠️ 关闭FIFO仅限特殊情况，日常应保持开启
- ⚠️ 关闭期间每次非FIFO发料都会产生例外记录
- ⚠️ 建议设置定时任务在24小时后自动恢复FIFO开启状态

### 场景三：调整小料拆分规格

**业务背景**：客户要求每卷PET薄膜改为15米包装（原10米）。

**操作步骤**：

1. 进入配置中心
2. 找到"小料拆分标准"分类
3. 修改"PET薄膜标准拆分长度"：10 → 15
4. 提交审批
5. 审批通过后生效

**影响范围**：
- ✅ 之后入库的新物料按15米拆分
- ⚠️ 已拆分的旧小料不受影响（保持10米规格）
- ⚠️ 需通知仓库人员注意新旧规格混存问题

## 3.2 库存相关问题

### 问题3：库存数量不一致

**现象**：总账数量 ≠ 批次明细汇总数量。

**排查SQL**：

```sql
-- 1. 对比总账和批次账
SELECT 
  m.mat_no,
  m.name,
  i.qty as 总账数量,
  IFNULL(SUM(ib.qty), 0) as 批次汇总,
  i.qty - IFNULL(SUM(ib.qty), 0) as 差异
FROM inv_inventory i
LEFT JOIN inv_material m ON i.material_id = m.id
LEFT JOIN inv_inventory_batch ib ON i.material_id = ib.material_id AND ib.status = 'available'
GROUP BY i.id, m.mat_no, m.name, i.qty
HAVING 差异 <> 0;

-- 2. 查找最近的库存变动记录
SELECT * FROM inventory_operation_log
WHERE material_id = {问题物料ID}
ORDER BY operate_time DESC LIMIT 20;

-- 3. 检查是否有未完成的事务
SHOW PROCESSLIST;  -- 查看是否有长时间运行的SQL
```

**修复方案**：

```sql
-- 方案1：以总账为准，调整批次账
UPDATE inv_inventory_batch SET qty = 0 WHERE material_id = {ID};

-- 方案2：以批次为准，更新总账
UPDATE inv_inventory 
SET qty = (SELECT SUM(qty) FROM inv_inventory_batch WHERE material_id = {ID} AND status = 'available')
WHERE material_id = {ID};

-- 方案3：盘点校正（推荐）
-- 创建盘点单，录入实际数量，系统自动调账
```

```bash
fi

# 3. 检查应用服务
if pgrep -f "node.*vnerp" > /dev/null; then
  echo "[应用] 运行正常 ✓" >> $LOG_FILE
else
  echo "❌ 应用服务未运行！" >> $LOG_FILE
fi

# 4. 检查数据库连接
mysqladmin -u root -ping > /dev/null 2>&1
if [ $? -eq 0 ]; then
  echo "[DB连接] 正常 ✓" >> $LOG_FILE
else
  echo "❌ 无法连接数据库！" >> $LOG_FILE
fi

# 5. 检查关键表的数据量
echo "" >> $LOG_FILE
echo "[关键表数据量]" >> $LOG_FILE
mysql -u root -e "
  SELECT 'work_orders' as tbl, COUNT(*) as cnt FROM vnerp_db.prod_work_order
  UNION ALL
  SELECT 'inventory_batches', COUNT(*) FROM vnerp_db.inv_inventory_batch
  UNION ALL
  SELECT 'operation_logs', COUNT(*) FROM vnerp_db.operation_logs;
" >> $LOG_FILE

# 6. 检查最近的错误日志
ERROR_COUNT=$(grep -c "ERROR\|FATAL" /var/log/vnerp/app.log 2>/dev/null || echo 0)
echo "" >> $LOG_FILE
echo "[错误日志] 最近24小时错误数: $ERROR_COUNT" >> $LOG_FILE

echo "" >> $LOG_FILE
echo "========================================" >> $LOG_FILE
echo "检查完成" >> $LOG_FILE
echo "========================================" >> $LOG_FILE

# 如果有严重问题，发送告警邮件
if grep -q "❌" $LOG_FILE; then
  mail -s "VNERP系统告警 - $(date)" $ALERT_EMAIL < $LOG_FILE
fi
```

# 五、安全加固建议

## 5.1 数据库安全

### 5.1.1 用户权限最小化

```sql
-- 创建应用专用账户（不要使用root）
CREATE USER 'vnerp_app'@'%' IDENTIFIED BY 'strong_password_here';

-- 只授予必要的权限
GRANT SELECT, INSERT, UPDATE, DELETE ON vnerp_db.* TO 'vnerp_app'@'%';

-- 禁止危险操作
REVOKE DROP, ALTER, CREATE ON vnerp_db.* FROM 'vnerp_app'@'%';

-- 刷新权限
FLUSH PRIVILEGES;
```

### 5.1.2 敏感数据加密

```sql
-- 1. 密码字段必须加密存储
ALTER TABLE sys_user MODIFY password VARCHAR(255) NOT NULL COMMENT 'bcrypt加密密码';

-- 2. 手机号、身份证等PII数据脱敏显示（应用层处理）
-- SQL层面存储明文，但查询时使用脱敏函数
-- SELECT CONCAT(LEFT(phone, 3), '****', RIGHT(phone, 4)) as phone_masked ...

-- 3. 财务金额等敏感字段访问控制
CREATE VIEW fin_accounts_receivable_safe AS
SELECT ar_no, customer_id, total_amount, status, create_time
  -- 不包含 bank_account, tax_id 等敏感字段
FROM accounts_receivable;
```

## 5.2 应用安全

### 5.2.1 接口安全检查清单

- [ ] 敏感操作（删除、批量修改）需要二次确认
- [ ] 输入参数进行严格的类型校验和XSS过滤
- [ ] SQL查询使用参数化绑定，禁止字符串拼接
- [ ] 文件上传限制类型和大小
- [ ] 配置CORS白名单
- [ ] 启用HTTPS（生产环境强制）
- [ ] 设置安全的HTTP响应头

# 六、灾难恢复预案

## 6.1 故障分级与响应

| 故障等级 | 定义 | 响应时间 | 解决时间 | 通知范围 |
|---------|------|---------|---------|---------|
| **P0-致命** | 系统完全不可用，核心业务停摆 | 5分钟 | 4小时内 | 全体+管理层 |
| **P1-严重** | 主要功能受损，影响大部分用户 | 15分钟 | 8小时内 | 技术团队+产品 |
| **P2-一般** | 部分功能异常，有 workaround | 30分钟 | 24小时内 | 技术团队 |
| **P3-轻微** | 小问题，不影响主要业务 | 2小时 | 3个工作日内 | 相关开发 |

## 6.2 常见灾难场景恢复

### 场景一：数据库服务器宕机

```
检测：监控系统报警 → DB连接失败
      ↓
应急（5分钟内）：
1. 尝试重启MySQL服务
2. 检查磁盘空间、内存
3. 查看错误日志定位原因
      ↓
恢复（如果重启无效）：
1. 切换到备用数据库服务器（如果有主从）
2. 或从最近备份恢复（RTO: 1小时）
3. 验证数据完整性
4. 通知用户恢复时间
      ↓
复盘：
1. 编写故障报告
2. 制定预防措施
3. 更新运维文档
```

### 场景二：数据误删/误更新

```
发现：用户反馈数据丢失或错误
      ↓
止损（立即）：
1. 停止相关应用的写入操作
2. 保护现场（不要尝试自行修复）
      ↓
恢复：
1. 从binlog找回误操作前的数据
   mysqlbinlog --start-position={位置} --stop-position={位置} binlog_file | mysql -u root -p
2. 或从最近的全量备份+增量恢复
3. 验证恢复的数据正确性
      ↓
预防：
1. 审计误操作的SQL语句
2. 加强权限控制（重要表禁止DELETE）
3. 开启操作审计日志
```

### 场景三：勒索病毒攻击

```
发现：文件被加密，出现勒索信息
      ↓
应急（立即）：
1. 断开网络（物理断网）
2. 不要支付赎金
3. 保留样本供警方取证
      ↓
恢复：
1. 格式化受感染服务器
2. 从干净的备份恢复系统和数据
3. 打打所有安全补丁
4. 逐步恢复网络连接
      ↓
预防：
1. 定期离线备份（3-2-1原则）
2. 部署防病毒软件
3. 禁止从互联网直接访问数据库端口
4. 员工安全意识培训
```
