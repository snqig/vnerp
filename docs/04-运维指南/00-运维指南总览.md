# 一、数据库维护指南

## 1.1 数据库备份策略

### 备份类型与频率

| 备份类型 | 频率 | 保留时间 | 用途 |
|---------|------|---------|------|
| **全量备份** | 每日 02:00 | 30天 | 完整恢复 |
| **增量备份** | 每4小时 | 7天 | 快速恢复到某时间点 |
| **事务日志备份** | 每30分钟 | 3天 | 精确到秒的恢复 |

### 自动备份脚本（MySQL）

```bash
#!/bin/bash
# vnerp_db_backup.sh - VNERP数据库自动备份脚本

DB_HOST="localhost"
DB_PORT="3306"
DB_USER="vnerp_admin"
DB_PASS="your_secure_password"
DB_NAME="vnerp_db"
BACKUP_DIR="/data/backup/mysql"
DATE=$(date +%Y%m%d_%H%M%S)
RETENTION_DAYS=30

mkdir -p $BACKUP_DIR/$DATE

echo "[$(date)] 开始全量备份..."
mysqldump -h$DB_HOST -P$DB_PORT -u$DB_USER -p$DB_PASS \
  --single-transaction \
  --routines \
  --triggers \
  --events \
  --databases $DB_NAME \
  | gzip > $BACKUP_DIR/$DATE/vnerp_full_$DATE.sql.gz

if [ $? -eq 0 ]; then
  echo "[$(date)] ✅ 全量备份完成"
  # 清理过期备份
  find $BACKUP_DIR -type f -name "*.sql.gz" -mtime +$RETENTION_DAYS -delete
else
  echo "[$(date)] ❌ 备份失败！"
  exit 1
fi
```

### 设置定时任务（Crontab）

```bash
# 每日凌晨2点执行备份
0 2 * * * /path/to/vnerp_db_backup.sh >> /var/log/vnerp_backup.log 2>&1
```

## 1.2 数据库恢复流程

### 全量恢复（从全量备份）

```bash
#!/bin/bash
# vnerp_db_restore.sh - 数据库恢复脚本

DB_HOST="localhost"
DB_PORT="3306"
DB_USER="vnerp_admin"
DB_PASS="your_secure_password"
DB_NAME="vnerp_db"
BACKUP_FILE=$1

echo "⚠️  警告：此操作将覆盖现有数据库！"
read -p "是否继续？(yes/no): " confirm

if [ "$confirm" = "yes" ]; then
  echo "开始恢复..."
  gunzip < $BACKUP_FILE | mysql -h$DB_HOST -P$DB_PORT -u$DB_USER -p$DB_PASS
  echo "✅ 数据库恢复成功！"
fi
```

# 二、系统配置管理

## 2.1 配置修改流程

### 标准修改流程

```
管理员登录 → 进入系统设置 → 选择配置项 → 修改值 
    ↓
填写修改原因 → 提交审批 → 等待审核
    ↓
[审批通过] → 配置生效 → 清除缓存 → 全局可用
[审批驳回] → 保持原值 → 可重新提交
```

### 常用配置修改示例

#### 示例1：修改刀模寿命参数

```bash
# 通过API修改配置（需管理员权限）
curl -X PUT http://localhost:3000/api/settings/system/config \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "key": "mould_life_days",
    "value": 120,
    "reason": "延长刀模使用寿命以降低成本",
    "require_approval": true
  }'
```

#### 示例2：临时关闭FIFO（紧急情况）

```bash
curl -X PUT http://localhost:3000/api/settings/system/config \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "key": "fifo_enabled",
    "value": false,
    "reason": "紧急生产需要灵活选择批次",
    "require_approval": true
  }'
```

# 三、常见故障排查

## 3.1 数据库连接故障

### 症状：应用无法连接数据库

**检查清单**：

```bash
# 1. 检查MySQL服务状态
systemctl status mysql

# 2. 检查端口是否监听
netstat -tlnp | grep 3306

# 3. 测试数据库连接
mysql -h localhost -u vnerp_admin -p -e "SELECT 1"

# 4. 检查连接数是否达到上限
mysql -u root -p -e "SHOW STATUS LIKE 'Threads_connected'"

# 5. 检查慢查询日志
tail -f /var/log/mysql/slow-query.log
```

**常见解决方案**：

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| Connection refused | MySQL未启动 | `systemctl start mysql` |
| Too many connections | 连接数达上限 | 增大`max_connections`或优化代码 |
| Access denied | 用户名/密码错误 | 重置密码或检查权限 |
| Host blocked | IP被防火墙拦截 | 检查iptables规则 |

## 3.2 应用服务故障

### 症状：页面无法访问或响应缓慢

**诊断步骤**：

```bash
# 1. 检查应用进程
ps aux | grep node

# 2. 查看应用日志
tail -f /var/log/vnerp/app.log

# 3. 检查端口占用
lsof -i :3000

# 4. 检查系统资源
top
free -h
df -h

# 5. 检查PM2状态（如果使用PM2）
pm2 list
pm2 logs vnerp-app
```

**重启应用服务**：

```bash
# 使用PM2重启
pm2 restart vnerp-app

# 或使用systemctl
systemctl restart vnerp-app
```

## 3.3 性能问题排查

### 慢查询分析

```sql
-- 开启慢查询日志（临时）
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 2;  // 超过2秒的查询记录下来

-- 查看当前慢查询设置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';

-- 分析慢查询日志
mysqldumpslow -s t -t 10 /var/log/mysql/slow-query.log
```

### 常见性能优化建议

| 优化项 | 操作 | 预期效果 |
|--------|------|---------|
| 添加索引 | 为高频查询字段创建索引 | 查询速度提升10-100倍 |
| 优化SQL | 避免 SELECT *，使用 EXPLAIN 分析 | 减少IO和内存消耗 |
| 增加缓存 | 启用Redis缓存热点数据 | 响应时间从秒级降到毫秒级 |
| 读写分离 | 主库写，从库读 | 提升读取性能3-5倍 |
| 分库分表 | 大表按业务拆分 | 单表数据量控制在千万级以下 |

# 四、监控与告警

## 4.1 系统健康检查脚本

```bash
#!/bin/bash
# vnerp_health_check.sh - VNERP系统每日健康检查

LOG_FILE="/var/log/vnerp/health_check_$(date +%Y%m%d).log"
ALERT_EMAIL="admin@yourcompany.com"
THRESHOLD_DISK=90    # 磁盘使用率阈值(%)
THRESHOLD_CPU=80     # CPU使用率阈值(%)
THRESHOLD_MEM=85     # 内存使用率阈值(%)

echo "========================================" >> $LOG_FILE
echo "VNERP系统健康检查 - $(date)" >> $LOG_FILE
echo "========================================" >> $LOG_FILE

# 检查磁盘空间
DISK_USAGE=$(df / | tail -1 | awk '{print $5}' | sed 's/%//')
echo "[磁盘] 使用率: ${DISK_USAGE}%" >> $LOG_FILE
if [ $DISK_USAGE -gt $THRESHOLD_DISK ]; then
  echo "⚠️  警告：磁盘使用率超过${THRESHOLD_DISK}%！" >> $LOG_FILE
fi

# 检查MySQL状态
if systemctl is-active --quiet mysql; then
  echo "[MySQL] 服务运行正常 ✓" >> $LOG_FILE
else
  echo "[MySQL] ❌ 服务异常！尝试重启..." >> $LOG_FILE
  systemctl restart mysql
fi

# 检查应用服务
if curl -sf http://localhost:3000/api/health > /dev/null; then
  echo "[应用] 服务响应正常 ✓" >> $LOG_FILE
else
  echo "[应用] ❌ 无响应！尝试重启..." >> $LOG_FILE
  pm2 restart all
fi

echo "检查完成" >> $LOG_FILE
```

## 4.2 关键指标监控

### 必须监控的指标

| 类别 | 指标 | 告警阈值 | 说明 |
|------|------|---------|------|
| **服务器** | CPU使用率 | >80% 持续5分钟 | 可能存在性能瓶颈 |
| **服务器** | 内存使用率 | >85% | 可能导致OOM |
| **服务器** | 磁盘使用率 | >90% | 需要清理或扩容 |
| **数据库** | 连接数 | >max_connections*80% | 需要优化或扩容 |
| **数据库** | 慢查询数量 | >10/分钟 | 需要优化SQL |
| **数据库** | 主从延迟 | >1秒 | 需要检查复制状态 |
| **应用** | 响应时间(P99) | >2s | 用户体验受影响 |
| **应用** | 错误率 | >1% | 存在bug或异常 |
| **业务** | 订单处理成功率 | <99% | 业务流程受阻 |

# 五、安全加固建议

## 5.1 数据库安全

```sql
-- 1. 删除匿名用户
DELETE FROM mysql.user WHERE user='';

-- 2. 禁止root远程登录
UPDATE mysql.user SET host='localhost' WHERE user='root';

-- 3. 删除测试数据库
DROP DATABASE IF EXISTS test;

-- 4. 修改root密码
ALTER USER 'root'@'localhost' IDENTIFIED BY '强密码@2026!';

-- 5. 创建专用应用账号（最小权限原则）
CREATE USER 'vnerp_app'@'localhost' IDENTIFIED BY '应用专用密码';
GRANT SELECT, INSERT, UPDATE, DELETE ON vnerp_db.* TO 'vnerp_app'@'localhost';
FLUSH PRIVILEGES;
```

## 5.2 应用层安全

- ✅ 所有API必须验证JWT Token
- ✅ 敏感操作需要二次确认
- ✅ 输入数据进行严格校验和过滤
- ✅ SQL语句使用参数化查询
- ✅ 定期更新依赖包修复安全漏洞
- ✅ 启用HTTPS加密传输
- ✅ 配置CSP防止XSS攻击
- ✅ 设置合理的Session超时时间

## 5.3 备份安全

- ✅ 备份文件加密存储
- ✅ 定期验证备份可恢复性
- ✅ 异地备份（至少保留一份离线备份）
- ✅ 备份访问权限控制
- ✅ 定期进行灾难恢复演练

| 场景 | 影响范围 | RTO（恢复时间目标） | RPO（恢复点目标） | 应对措施 |
|------|---------|-------------------|-------------------|---------|
| 单服务器宕机 | 部分功能不可用 | <30分钟 | <1小时 | 自动故障转移+快速重启 |
| 数据库崩溃 | 全部功能不可用 | <1小时 | <1小时 | 从最近备份恢复 |

| 数据误删/损坏 | 部分数据丢失 | <2小时 | <1小时 | 从binlog恢复到误操作前 |
| 勒索病毒攻击 | 全部数据加密 | <8小时 | <24小时 | 从离线备份恢复 |
