# 04-运维指南总览

| Docker Compose | v2+ | v2.x | `docker compose` 子命令形式 |

## 脚本命令速查

下列命令摘自 `package.json` scripts，是当前唯一支持的运维入口（已废弃命令见末列）。

| 命令 | 作用 |
|------|------|
| `pnpm dev` | 启动开发服务器（`next dev -p 5000`，端口 5000） |
| `pnpm dev:webpack` | Windows 下用 webpack 模式启动，规避 Turbopack `globals.css` nul 崩溃 |
| `pnpm build` / `pnpm start` | 生产构建 / 启动（`next start -p 5000`；Docker 内用 `node server.js`） |
| `pnpm setup:db [--seed]` | 全量初始化数据库（建库 + 导入 schema + 可选 seed） |
| `pnpm migrate` | 执行增量迁移 `up` |
| `pnpm migrate:status` | 查看迁移状态（已执行 / 待执行） |
| `pnpm migrate:down` | 回滚最后一个迁移 |
| `pnpm migrate:create` | 创建新迁移文件模板（实际调用 `tsx scripts/migrate.ts create <name>`） |
| `pnpm backup` | 全量备份数据库（mysqldump + gzip + SHA256） |
| `pnpm backup:list` | 列出已有备份 |
| `pnpm backup:restore <file>` | 从备份恢复 |
| `pnpm db:studio` | 启动 Drizzle Studio Web UI |
| `pnpm docs:api` | 生成 API 文档 |
| `pnpm cleanup:events` | 清理事件已处理记录（`sys_event_processed`） |
| `pnpm test` / `test:unit` | Playwright E2E / Vitest 单测 |
| ~~`pnpm db:generate` / `db:push` / `db:migrate`~~ | **已废弃**：仅打印警告，请改用 `pnpm setup:db` 或 `pnpm migrate` |

## 阅读建议

1. **新加入的开发者**：按 `本地开发 → 系统初始化配置 → 数据库迁移` 顺序阅读。
2. **部署人员**：先读 `Docker部署指南`，正式上线前对照 `生产部署` 与 `Docker安全扫描报告` 完成安全清单。
3. **DBA**：直接阅读 `数据库迁移` 与 `备份恢复`，注意 SQL 迁移文件位于 `database/migrations/`（编号 001–060），全新环境用 `pnpm setup:db`，增量演进用 `pnpm migrate`。
4. **运维 / SRE**：阅读 `监控告警接入方案` 配置 Prometheus + Grafana + Alertmanager 监控栈，参考 `docker-compose.monitoring.yml`。

## 跨文档约定

- **公司名**：默认「公司名称」（`src/app/[locale]/layout.tsx`），可由 `sys_config.company_name` 覆盖。
- **事件总线**：生产环境必须 `EVENT_BUS_TYPE=db`（启用 Outbox + Redis Streams）；开发无 Redis 时可用 `memory`（不持久化、重启丢失）。
- **认证模式**：`NEXT_PUBLIC_AUTH_COOKIE_MODE` 默认 `header`，切换需 `next build` + 重启（构建时固化）。
- **调试路由**：生产环境 `src/proxy.ts` 封堵 `/qrcode`、`/api/init`、`/api/debug`、`/api/diagnose`（返回 404）。
