# 备份恢复

本文档说明 vnerp（印刷生产经营信息管理系统 Print MIS）数据库的备份与恢复策略，对应 `scripts/backup-database.ts`（`package.json` 中的 `pnpm backup` 系列命令）与项目运维实践。

## 1. 备份策略概览

| 类型 | 频率 | 工具 | 保留期 | 用途 |
|------|------|------|--------|------|
| 全量备份 | 每日 02:00 | `pnpm backup`（mysqldump + gzip） | 30 天（默认） | 日常恢复 |
| 迁移前备份 | 每次迁移前 | `pnpm backup` | 30 天 | 迁移回滚 |
| 手动备份 | 按需 | `pnpm backup` 或 mysqldump | 永久 | 重大变更前快照 |
| Redis 备份 | 每周 | `redis-cli BGSAVE` + `docker cp` | 4 周 | 缓存与 token 黑名单（可重建，非关键） |

## 2. 备份脚本（scripts/backup-database.ts）

项目内置 TypeScript 备份脚本，封装了 `mysqldump` 调用、gzip 压缩、SHA256 校验、自动清理过期备份与恢复验证。

### 2.1 pnpm 命令

| 命令 | 实际执行 | 作用 |
|------|----------|------|
| `pnpm backup` | `tsx scripts/backup-database.ts` | 全量备份（默认） |
| `pnpm backup:list` | `tsx scripts/backup-database.ts list` | 列出已有备份 |
| `pnpm backup:restore` | `tsx scripts/backup-database.ts restore` | 从备份恢复（实际需 `tsx scripts/backup-database.ts restore <file>`） |

`scripts/backup-database.ts` 还支持以下未挂到 pnpm script 的选项（直接 `npx tsx scripts/backup-database.ts [选项]` 调用）：

| 选项 | 作用 |
|------|------|
| `--full` | 全量备份（默认） |
| `--structure` | 仅备份表结构（`--no-data`） |
| `--data-only` | 仅备份数据（`--no-create-info`） |
| `--tables "t1,t2"` | 仅备份指定表 |
| `--retention <天数>` | 覆盖 `BACKUP_RETENTION_DAYS` |
| `--keep <数量>` | 覆盖 `BACKUP_KEEP_COUNT` |
| `--output <目录>` | 覆盖 `BACKUP_DIR` |
| `--no-compress` | 不压缩（默认 gzip） |
| `--verify` | 备份后验证完整性（gzip -t + 校验和） |
| `--dry-run` | 预演，不实际执行 |

### 2.2 配置

脚本从环境变量读取配置（沿用 `.env` 中的 `DB_*` 变量）：

| 变量 | 默认 | 说明 |
|------|------|------|
| `DB_HOST` / `DB_PORT` / `DB_USER` / `DB_PASSWORD` / `DB_NAME` | 127.0.0.1 / 3306 / root / - / vnerpdacahng | MySQL 连接信息 |
| `BACKUP_DIR` | `./backups` | 备份输出目录 |
| `BACKUP_RETENTION_DAYS` | `30` | 备份保留天数 |
| `BACKUP_KEEP_COUNT` | `100` | 保留备份最大数量 |

### 2.3 输出格式

- 文件名：`{database}_{type}_{ISO时间戳}.sql.gz`，例如 `vnerpdacahng_full_2026-07-10T02-00-00.sql.gz`。
- 使用 `gzip` 压缩，并通过 `--single-transaction --routines --triggers --events --quick --lock-tables=false` 保证一致性且不锁表。
- 同时生成 `<filename>.sha256` 校验和文件，内容为 `<sha256>  <filename>`。

### 2.4 自动清理

每次备份完成后自动执行双策略清理：

1. **按天数**：删除修改时间超过 `BACKUP_RETENTION_DAYS`（默认 30 天）的备份。
2. **按数量**：若备份数超过 `BACKUP_KEEP_COUNT`（默认 100），从最旧的开始删除。

清理时同步删除对应的 `.sha256` 文件，并打印清理统计。

### 2.5 失败处理

- 备份失败时删除已生成的部分文件（避免残留半成品），打印错误，进程退出码 1。
- 依赖检查：脚本启动时检查 `mysqldump` 与 `gzip` 是否可用，缺失 `mysqldump` 直接报错退出；缺失 `gzip` 警告并降级为不压缩。

## 3. 手动备份

### 3.1 项目脚本

```bash
# 全量备份（默认压缩）
pnpm backup

# 仅备份结构
npx tsx scripts/backup-database.ts --structure

# 备份指定表
npx tsx scripts/backup-database.ts --tables "sys_user,sys_menu"

# 备份后验证完整性
pnpm backup -- --verify

# 预演（不实际执行）
pnpm backup -- --dry-run

# 自定义保留策略
npx tsx scripts/backup-database.ts --retention 90 --keep 200
```

> `pnpm backup -- <选项>` 中的 `--` 用于将后续参数透传给底层 `tsx` 调用。

### 3.2 直接 mysqldump

```bash
# 不压缩
mysqldump -h 127.0.0.1 -u root -p vnerpdacahng > backup_$(date +%Y%m%d_%H%M%S).sql

# 压缩
mysqldump -h 127.0.0.1 -u root -p vnerpdacahng \
  --single-transaction --routines --triggers --events --quick \
  | gzip > backup_$(date +%Y%m%d).sql.gz

# Docker 容器内
docker exec vnerp-mysql-prod sh -c \
  'exec mysqldump -uvnerp_app -p"$DB_PASSWORD" --single-transaction vnerpdacahng' \
  | gzip > backup_$(date +%Y%m%d).sql.gz
```

### 3.3 迁移前备份

执行 `pnpm migrate`、`pnpm setup:db` 重建、或任何 schema 变更前，必须先备份：

```bash
pnpm backup
# 或
mysqldump -h 127.0.0.1 -u root -p vnerpdacahng > pre_migration_$(date +%Y%m%d).sql
```

## 4. 定时备份

### 4.1 crontab（Linux）

```cron
# 每天凌晨 2 点全量备份，保留 30 天
0 2 * * * cd /opt/vnerp && /usr/local/bin/pnpm backup >> /var/log/vnerp-backup.log 2>&1

# 或纯 mysqldump
0 2 * * * mysqldump -h 127.0.0.1 -u root -pYOUR_PASSWORD vnerpdacahng \
  --single-transaction --routines --triggers --events --quick \
  | gzip > /backup/vnerp_$(date +\%Y\%m\%d).sql.gz
0 3 * * * find /backup -name "vnerp_*.sql.gz" -mtime +30 -delete
```

### 4.2 Windows 任务计划

通过 `taskschd.msc` 创建任务，触发 `pnpm backup`，或将 `scripts/start.bat` 中的备份命令加入计划。

### 4.3 Docker 环境定时备份

```bash
# 在宿主机 crontab 中调用容器内 mysqldump
0 2 * * * docker exec vnerp-mysql-prod sh -c \
  'exec mysqldump -uvnerp_app -p"$DB_PASSWORD" --single-transaction vnerpdacahng' \
  | gzip > /opt/vnerp/backups/mysql/vnerp_$(date +\%Y\%m\%d).sql.gz
0 3 * * * find /opt/vnerp/backups/mysql -name "vnerp_*.sql.gz" -mtime +30 -delete
```

## 5. 恢复流程

### 5.1 项目脚本

```bash
# 列出可用备份
pnpm backup:list

# 恢复（指定文件路径）
npx tsx scripts/backup-database.ts restore backups/vnerpdacahng_full_2026-07-10T02-00-00.sql.gz

# 预演（不实际恢复）
npx tsx scripts/backup-database.ts restore backups/<file>.sql.gz --dry-run
```

恢复逻辑：

- 自动识别 `.gz` 压缩文件，用 `gunzip < <file> | mysql ...` 解压恢复。
- 未压缩文件直接 `mysql ... < <file>`。
- 恢复前打印警告「即将执行数据库恢复，此操作将覆盖现有数据」。
- 失败时打印错误并退出码 1。

> `pnpm backup:restore` 脚本签名未带文件参数，需通过 `npx tsx` 直接调用并显式指定文件。

### 5.2 全量恢复（手动）

```bash
# 解压恢复
gunzip < backup_20260710.sql.gz | mysql -h 127.0.0.1 -u root -p vnerpdacahng

# 或未压缩文件
mysql -h 127.0.0.1 -u root -p vnerpdacahng < backup_20260710.sql
```

### 5.3 Docker 容器内恢复

```bash
# 停止 app 服务避免恢复期间写入
docker compose -f docker-compose.prod.yml stop app

# 恢复
docker exec -i vnerp-mysql-prod sh -c \
  'exec mysql -uvnerp_app -p"$DB_PASSWORD" vnerpdacahng' < backup_20260710.sql

# 或压缩文件
gunzip < backup_20260710.sql.gz | \
  docker exec -i vnerp-mysql-prod sh -c \
  'exec mysql -uvnerp_app -p"$DB_PASSWORD" vnerpdacahng'

# 重启 app
docker compose -f docker-compose.prod.yml start app
```

### 5.4 部分恢复（单表）

```bash
# 从备份中提取单表（需 sed/awk 工具）
sed -n '/-- Table structure for `sys_user`/,/-- Table structure for/p' backup.sql > sys_user.sql
mysql -u root -p vnerpdacahng < sys_user.sql
```

## 6. 备份验证

定期检查备份完整性，避免「备份成功但无法恢复」：

```bash
# 列出备份
pnpm backup:list

# 检查备份文件大小（不应为 0）
ls -lh backups/

# 校验 SHA256
cd backups && sha256sum -c vnerpdacahng_full_2026-07-10T02-00-00.sql.gz.sha256

# 检查 SQL 内容
gunzip -c backups/vnerpdacahng_full_2026-07-10T02-00-00.sql.gz | head -50
gunzip -c backups/vnerpdacahng_full_2026-07-10T02-00-00.sql.gz | grep "CREATE TABLE" | wc -l

# gzip 完整性测试
gzip -t backups/vnerpdacahng_full_2026-07-10T02-00-00.sql.gz
```

`pnpm backup -- --verify` 选项会在备份完成后自动执行上述校验。

## 7. 恢复演练

建议每季度执行一次恢复演练：

1. 在测试环境创建空数据库 `vnerp_test`。
2. 从最近备份恢复：
   ```bash
   gunzip < backups/vnerpdacahng_full_<最近>.sql.gz | mysql -u root -p vnerp_test
   ```
3. 验证关键表行数、最新记录时间戳：
   ```sql
   SELECT COUNT(*) FROM sys_user;
   SELECT COUNT(*) FROM sal_order;
   SELECT MAX(create_time) FROM sys_login_log;
   ```
4. 启动应用连接 `vnerp_test`（修改 `.env` 的 `DB_NAME`），执行登录、查询等核心流程。
5. 记录演练耗时与问题，更新本文档。

## 8. Redis 备份

Redis 数据（token 黑名单、refresh token、缓存）可重建，非关键数据，但建议定期备份：

```bash
# 触发 RDB 快照
docker exec vnerp-redis-prod redis-cli BGSAVE

# 复制 RDB 文件
docker cp vnerp-redis-prod:/data/dump.rdb ./backups/redis/dump-$(date +%Y%m%d).rdb

# 复制 AOF 文件（若启用 appendonly）
docker cp vnerp-redis-prod:/data/appendonly.aof ./backups/redis/appendonly-$(date +%Y%m%d).aof
```

恢复：停止 Redis → 替换 `dump.rdb` → 启动 Redis。

## 9. 注意事项

- `mysqldump` 使用 `--single-transaction` 避免 InnoDB 锁表，但对 MyISAM 表无效（项目表均为 InnoDB）。
- 备份文件含敏感数据（用户密码哈希、JWT 配置等），存储位置应限制访问权限（`chmod 600`）。
- 生产环境备份目录建议挂载到独立磁盘或对象存储，避免与数据库同机故障丢失。
- `BACKUP_RETENTION_DAYS` 默认 30 天，按合规要求调整。
- `BACKUP_KEEP_COUNT` 默认 100 个，避免备份目录无限增长。
- 恢复前务必**停止应用写入**（停止 app 容器或断开连接池），避免恢复期间数据被覆盖。
- 恢复后建议执行 `pnpm migrate:status` 确认迁移记录与 schema 一致。
