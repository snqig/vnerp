# 备份恢复

本文档说明 印刷生产经营信息管理系统 Print MIS 数据库的备份与恢复策略，对应 `scripts/backup-database.ts` 与项目运维实践。

## 1. 备份策略概览

| 类型 | 频率 | 工具 | 保留期 | 用途 |
|------|------|------|--------|------|
| 全量备份 | 每日 02:00 | `mysqldump` 或 `scripts/backup-database.ts` | 7 天 | 日常恢复 |
| 迁移前备份 | 每次迁移前 | `mysqldump` | 30 天 | 迁移回滚 |
| 手动备份 | 按需 | `mysqldump` | 永久 | 重大变更前快照 |
| Docker 卷备份 | 每周 | `docker run --rm -v ...` | 4 周 | 容器级别快照 |

## 2. 备份脚本（scripts/backup-database.ts）

项目内置 TypeScript 备份脚本，封装了 `mysqldump` 调用与过期清理。

### 2.1 运行

```bash
# 通过 tsx 运行
npx tsx scripts/backup-database.ts

# 或配置环境变量
BACKUP_DIR=./backups BACKUP_RETENTION_DAYS=7 npx tsx scripts/backup-database.ts
```

### 2.2 配置

脚本从环境变量读取配置（沿用 `.env` 中的 `DB_*` 变量）：

| 变量 | 默认 | 说明 |
|------|------|------|
| `DB_HOST` / `DB_PORT` / `DB_USER` / `DB_PASSWORD` / `DB_NAME` | localhost / 3306 / root / - / vnerp | MySQL 连接信息 |
| `BACKUP_DIR` | `./backups` | 备份输出目录 |
| `BACKUP_RETENTION_DAYS` | `7` | 备份保留天数 |

### 2.3 输出格式

- 文件名：`{database}_{ISO时间戳}.sql.gz`，例如 `vnerp_2026-06-30T02-00-00.sql.gz`。
- 使用 `gzip` 压缩，并通过 `--single-transaction --routines --triggers --events` 保证一致性。
- 备份完成后自动清理超过 `BACKUP_RETENTION_DAYS` 的旧文件，输出清理数量。

### 2.4 失败处理

脚本失败时返回非零退出码并打印错误，便于 crontab 或 CI 检测。

## 3. 手动备份

### 3.1 命令行

```bash
# 不压缩
mysqldump -h 127.0.0.1 -u root -p vnerp > backup_$(date +%Y%m%d_%H%M%S).sql

# 压缩
mysqldump -h 127.0.0.1 -u root -p vnerp | gzip > backup_$(date +%Y%m%d).sql.gz

# Docker 容器内
docker exec vnerp-mysql-prod sh -c \
  'exec mysqldump -uvnerp_app -p"$DB_PASSWORD" --single-transaction vnerp' \
  > backup_$(date +%Y%m%d).sql
```

### 3.2 迁移前备份

执行 Drizzle 迁移、`setup:db` 重建、或任何 schema 变更前，必须先备份：

```bash
mysqldump -h 127.0.0.1 -u root -p vnerp > pre_migration_$(date +%Y%m%d).sql
```

## 4. 定时备份

### 4.1 crontab（Linux）

```cron
# 每天凌晨 2 点备份，保留 7 天
0 2 * * * cd /opt/vnerp && BACKUP_DIR=/backup npx tsx scripts/backup-database.ts >> /var/log/vnerp-backup.log 2>&1

# 或纯 mysqldump
0 2 * * * mysqldump -h 127.0.0.1 -u root -pYOUR_PASSWORD vnerp | gzip > /backup/vnerp_$(date +\%Y\%m\%d).sql.gz
0 3 * * * find /backup -name "vnerp_*.sql.gz" -mtime +7 -delete
```

### 4.2 Windows 任务计划

通过 `taskschd.msc` 创建任务，触发 `scripts/backup-database.ts`，或将 `scripts/start.bat` 中的备份命令加入计划。

## 5. 恢复流程

### 5.1 全量恢复

```bash
# 解压
gunzip < backup_20260630.sql.gz | mysql -h 127.0.0.1 -u root -p vnerp

# 或未压缩文件
mysql -h 127.0.0.1 -u root -p vnerp < backup_20260630.sql
```

### 5.2 Docker 容器内恢复

```bash
docker exec -i vnerp-mysql-prod mysql -uvnerp_app -p vnerp < backup_20260630.sql
```

### 5.3 部分恢复（单表）

```bash
# 从备份中提取单表（需 sed/awk 工具）
sed -n '/-- Table structure for `sys_user`/,/-- Table structure for/p' backup.sql > sys_user.sql
mysql -u root -p vnerp < sys_user.sql
```

## 6. 恢复演练

建议每季度执行一次恢复演练：

1. 在测试环境创建空数据库 `vnerp_test`。
2. 从最近备份恢复：`mysql -u root -p vnerp_test < backup.sql`。
3. 验证关键表行数、最新记录时间戳。
4. 启动应用连接 `vnerp_test`，执行登录、查询等核心流程。
5. 记录演练耗时与问题，更新本文档。

## 7. 备份验证

定期检查备份完整性，避免"备份成功但无法恢复"：

```bash
# 检查备份文件大小（不应为 0）
ls -lh backups/

# 检查 SQL 内容
gunzip -c backups/vnerp_2026-06-30.sql.gz | head -50
gunzip -c backups/vnerp_2026-06-30.sql.gz | grep "CREATE TABLE" | wc -l
```

## 8. 注意事项

- `mysqldump` 使用 `--single-transaction` 避免 InnoDB 锁表，但对 MyISAM 表无效。
- 备份文件含敏感数据（用户密码哈希、JWT 配置等），存储位置应限制访问权限。
- 生产环境备份目录建议挂载到独立磁盘或对象存储，避免与数据库同机故障丢失。
- `BACKUP_RETENTION_DAYS` 默认 7 天，按合规要求调整。
