# 数据库迁移

本文档说明 vnerp（印刷生产经营信息管理系统 Print MIS）的数据库增量迁移流程，对应 `scripts/migrate.ts`（`package.json` 中的 `pnpm migrate` 系列命令）与 `database/migrations/` 目录。

> 全新环境初始化请用 `pnpm setup:db`，详见 [系统初始化配置.md](./系统初始化配置.md)。本文档仅讨论已有环境的增量演进。

## 1. 迁移体系概览

| 项 | 说明 |
|----|------|
| 迁移目录 | `database/migrations/` |
| 迁移文件格式 | `.sql`（推荐）或 `.ts`（含 `up` / `down` 函数） |
| 迁移编号 | 001–060（当前范围） |
| 记录表 | `sys_migration`（`migration_name` UNIQUE，`batch` 递增，`applied_at`，`execution_time`） |
| 执行脚本 | `scripts/migrate.ts`（`npx tsx scripts/migrate.ts <command>`） |
| pnpm 命令 | `pnpm migrate` / `pnpm migrate:status` / `pnpm migrate:down` / `pnpm migrate:create` |

> **已废弃**：`pnpm db:generate` / `db:push` / `db:migrate` 仅打印警告，请改用本文档命令。drizzle-kit 迁移路径（`drizzle/` 目录）已清理，但 `pnpm db:studio` 仍可用（drizzle-kit studio 仅做可视化查询，不写迁移）。

## 2. 命令一览（package.json scripts）

| 命令 | 实际执行 | 作用 |
|------|----------|------|
| `pnpm migrate` | `tsx scripts/migrate.ts up` | 执行所有待执行的迁移（默认全部） |
| `pnpm migrate:status` | `tsx scripts/migrate.ts status` | 查看迁移状态：总数 / 已执行 / 待执行 |
| `pnpm migrate:down` | `tsx scripts/migrate.ts down` | 回滚最后 1 个迁移（默认） |
| `pnpm migrate:create` | `tsx scripts/migrate.ts create` | 创建新迁移文件模板（实际需 `tsx scripts/migrate.ts create <name>`） |

`scripts/migrate.ts` 还支持以下未挂到 pnpm script 的子命令（直接 `npx tsx scripts/migrate.ts <cmd>` 调用）：

| 子命令 | 作用 |
|--------|------|
| `up --count <n>` | 仅执行前 n 个待执行迁移 |
| `down --count <n>` | 回滚最后 n 个迁移 |
| `down --all` | 回滚所有已执行迁移 |
| `history` | 查看迁移历史（最近 50 条，含 batch / 执行时间） |
| `reset` | 回滚所有 + 重新执行（**危险**） |
| `help` / `--help` / `-h` | 显示帮助 |

## 3. 迁移文件格式

### 3.1 SQL 迁移（推荐）

文件名：`NNN_<description>.sql`，如 `061_add_user_avatar.sql`。内容为纯 SQL，按 `;` 分隔多条语句：

```sql
-- 061_add_user_avatar.sql
ALTER TABLE sys_user ADD COLUMN avatar_url VARCHAR(255) NULL COMMENT '头像 URL' AFTER avatar;
CREATE INDEX idx_sys_user_avatar_url ON sys_user(avatar_url);
```

**注意**：SQL 迁移文件**不支持 `down`**（`pnpm migrate:down` 对 `.sql` 文件会抛错 `"SQL 迁移文件 ... 不支持回滚（down），请手动处理"`）。

### 3.2 TypeScript 迁移

文件名：`YYYYMMDDHHMMSS_<description>.ts`（由 `pnpm migrate:create` 生成）。导出 `up` 与 `down` 函数：

```ts
/**
 * add_user_avatar
 *
 * 正向迁移 (up):   执行变更
 * 反向迁移 (down): 回滚变更
 */

import { Connection } from 'mysql2/promise';

export async function up(conn: Connection): Promise<void> {
  await conn.execute('ALTER TABLE sys_user ADD COLUMN avatar_url VARCHAR(255) NULL COMMENT ? AFTER avatar', ['头像 URL']);
  await conn.execute('CREATE INDEX idx_sys_user_avatar_url ON sys_user(avatar_url)');
}

export async function down(conn: Connection): Promise<void> {
  await conn.execute('ALTER TABLE sys_user DROP INDEX idx_sys_user_avatar_url');
  await conn.execute('ALTER TABLE sys_user DROP COLUMN avatar_url');
}
```

## 4. migrate.ts 执行行为（重要）

### 4.1 SQL 拆分与执行

`scripts/migrate.ts` 的 `executeMigration` 函数处理 `.sql` 文件：

1. 读取文件内容。
2. 移除以 `--` 开头的注释行。
3. 按 `;` 分割为多条语句，过滤空语句。
4. 在同一连接上逐条 `await conn.query(stmt)` 执行（**无事务包裹**）。
5. 全部成功后写入 `sys_migration`（batch = max(batch) + 1，记录 `execution_time`）。

> **关键风险**：因为语句逐条执行且无事务包裹，**某条语句失败时，已执行语句的效果不会回滚**，但整个迁移会被标记为失败（不写入 `sys_migration`）。修复后重跑 `pnpm migrate` 会从头再次执行所有语句。

### 4.2 幂等性要求

由于上述行为，**迁移 SQL 必须幂等**，确保重复执行不报错、不破坏数据。推荐模式：

```sql
-- 1. 用 INFORMATION_SCHEMA 守卫（最稳妥）
SET @col_exists = (SELECT COUNT(*) FROM information_schema.columns
  WHERE table_schema = DATABASE() AND table_name = 'sys_user' AND column_name = 'avatar_url');
SET @sql = IF(@col_exists = 0,
  'ALTER TABLE sys_user ADD COLUMN avatar_url VARCHAR(255) NULL COMMENT ''头像 URL'' AFTER avatar',
  'SELECT ''column already exists'' AS info');
PREPARE stmt FROM @sql; EXECUTE stmt; DEALLOCATE PREPARE stmt;

-- 2. 或用 CREATE TABLE IF NOT EXISTS / CREATE INDEX IF NOT EXISTS（MySQL 8.0+ 支持）
CREATE INDEX IF NOT EXISTS idx_sys_user_avatar_url ON sys_user(avatar_url);

-- 3. 数据修补用 INSERT IGNORE / ON DUPLICATE KEY UPDATE
INSERT IGNORE INTO sys_config (config_key, config_value) VALUES ('system.password_expire_days', '90');
```

### 4.3 TS 迁移执行

`.ts` 文件通过 `await import(filePath)` 动态加载，调用 `migration.up(conn)` 或 `migration.down(conn)`。`up` 成功后写入 `sys_migration`；`down` 成功后从 `sys_migration` 删除该记录。

### 4.4 失败处理

- `up` 阶段：单条迁移失败立即 `break`，输出 `❌ <文件名>: <错误>`，进程退出码 1。
- `down` 阶段：单条迁移失败立即 `break`，输出 `❌ 回滚失败 <文件名>: <错误>`，进程退出码 1。
- 进程退出码非 0 时 CI 应中断流水线。

## 5. 标准迁移流程

### 5.1 创建迁移文件

```bash
# 推荐：用 SQL 文件，命名 NNN_<description>.sql
# 查看当前最大编号
pnpm migrate:status

# 手动创建（如 061_add_user_avatar.sql）
# 或用脚本生成 TS 模板（不挂 pnpm script，直接调用）
npx tsx scripts/migrate.ts create add_user_avatar
# → 生成 database/migrations/YYYYMMDDHHMMSS_add_user_avatar.ts
```

### 5.2 编写迁移 SQL

确保幂等（参考第 4.2 节），编辑 `database/migrations/061_add_user_avatar.sql`。

### 5.3 备份并执行

```bash
# 迁移前必做备份（详见 备份恢复.md）
pnpm backup

# 查看待执行
pnpm migrate:status

# 执行
pnpm migrate
```

### 5.4 验证

```bash
# 查看状态（应显示已执行）
pnpm migrate:status

# 查看历史
npx tsx scripts/migrate.ts history

# 启动 Drizzle Studio 可视化检查
pnpm db:studio

# 或直接查询
mysql -u root -p vnerp -e "DESC sys_user;"
```

### 5.5 失败后重试

```bash
# 1. 查看失败原因
pnpm migrate:status   # 失败的迁移仍标记为「待执行」
npx tsx scripts/migrate.ts history

# 2. 修复迁移 SQL（确保幂等）
vim database/migrations/061_add_user_avatar.sql

# 3. 重跑（已执行语句需幂等不报错）
pnpm migrate

# 4. 若数据库已部分变更，手动清理或恢复备份
pnpm backup:restore backups/vnerp_full_2026-07-10T...sql.gz
```

## 6. 回滚

### 6.1 TS 迁移回滚

```bash
# 回滚最后 1 个
pnpm migrate:down

# 回滚最后 3 个
npx tsx scripts/migrate.ts down --count 3

# 回滚所有（危险）
npx tsx scripts/migrate.ts down --all
```

回滚成功后从 `sys_migration` 删除该记录，下次 `pnpm migrate` 会重新执行。

### 6.2 SQL 迁移回滚

SQL 迁移文件不支持 `down`。回滚策略：

1. **优先**：恢复迁移前备份
   ```bash
   pnpm backup:restore backups/vnerp_full_2026-07-10T...sql.gz
   ```
2. 手写反向 SQL（如 `DROP COLUMN` / `DROP INDEX` / `DROP TABLE`），通过 `mysql` 客户端执行。
3. 手动从 `sys_migration` 删除该记录：
   ```sql
   DELETE FROM sys_migration WHERE migration_name = '061_add_user_avatar.sql';
   ```

### 6.3 重置（危险）

```bash
npx tsx scripts/migrate.ts reset
# 等价于 down --all && up
```

**仅限开发环境**，生产环境禁止使用。

## 7. 与 setup-db.mjs 的关系

| 场景 | 命令 | 说明 |
|------|------|------|
| 全新环境 | `pnpm setup:db --seed` | 建库 + 导入 schema.sql + 可选 seed + 标记所有迁移为已执行（batch 0） |
| 已有环境增量演进 | `pnpm migrate` | 执行 `database/migrations/` 下未执行的迁移 |
| 既有环境误跑 setup:db | - | schema.sql 用 `IF NOT EXISTS` + `INSERT IGNORE`，幂等不会破坏数据；但会重置 `sys_migration` 的 batch 为 0（`INSERT IGNORE` 不覆盖已存在记录） |

> `setup:db` 把所有迁移文件标记为 batch 0，`pnpm migrate:status` 会显示「已执行」。**不要**在已有环境跑 `pnpm setup:db` 期望「重新执行迁移」——它不会跑迁移 SQL，只会跑 schema.sql。

## 9. CI/CD 中的迁移

生产部署前应在 CI 流水线或预发布环境执行迁移：

```bash
# 1. 备份（必做）
pnpm backup

# 2. 执行迁移
pnpm migrate

# 3. 验证状态
pnpm migrate:status

# 4. 若失败，回滚到备份
pnpm backup:restore backups/<最新备份>.sql.gz
```

迁移失败会导致 CI 退出码 1，应中断部署流水线并人工介入。

## 10. 常见问题

### 10.1 `SQL 迁移文件 ... 不支持回滚（down）`

`.sql` 文件无法 `down`。改用 TS 迁移（导出 `down` 函数），或手动写反向 SQL + 删除 `sys_migration` 记录。

### 10.2 重跑后报「Duplicate column name」

迁移 SQL 未做幂等守卫。改用 `INFORMATION_SCHEMA` 检查或 `IF NOT EXISTS` 语法（参考第 4.2 节）。

### 10.3 `pnpm migrate:status` 显示待执行，但表已存在

可能历史上手动执行过 SQL，未走 migrate.ts。手动插入 `sys_migration` 记录：

```sql
INSERT IGNORE INTO sys_migration (migration_name, batch, execution_time)
VALUES ('061_add_user_avatar.sql', 0, 0);
```

### 10.4 迁移文件命名冲突

`sys_migration.migration_name` 是 UNIQUE。若两个迁移文件同名（不应发生），后者无法记录。规范命名 `NNN_<description>.sql`，NNN 递增不重复。

### 10.5 `pnpm migrate:create` 生成的文件名带时间戳

`migrate.ts create` 用 `YYYYMMDDHHMMSS_<name>.ts` 格式，与现有 `NNN_*.sql` 不一致。建议手动创建 `NNN_<name>.sql` 文件以保持一致；或修改模板格式后用 `create`。
