# 数据库迁移

本文档说明 印刷生产经营信息管理系统 Print MIS 基于 Drizzle ORM 的数据库迁移流程，对应 `drizzle.config.ts` 与 `package.json` 的 `db:*` 脚本。

## 1. Drizzle 配置

`drizzle.config.ts` 关键配置：

```ts
export default defineConfig({
  schema: './src/lib/db/schema.ts',  // Schema 源文件
  out: './drizzle',                  // 迁移文件输出目录
  dialect: 'mysql',
  dbCredentials: {
    host: process.env.DB_HOST || '127.0.0.1',
    port: Number(process.env.DB_PORT || 3306),
    user: process.env.DB_USER || 'root',
    password: process.env.DB_PASSWORD || '',
    database: process.env.DB_NAME || 'vnerpdacahng',
  },
  verbose: true,
  strict: true,
});
```

- Schema 单一来源：`src/lib/db/schema.ts`。
- 数据库连接信息从 `.env` 读取，未配置时回退到 `root@127.0.0.1:3306/vnerpdacahng`。
- `verbose: true` 输出执行详情，`strict: true` 严格模式（未识别变更会报错）。

## 2. 迁移脚本（package.json）

| 脚本 | 命令 | 作用 |
|------|------|------|
| `pnpm db:generate` | `drizzle-kit generate` | 对比 `schema.ts` 与上一快照，生成新迁移文件到 `drizzle/` |
| `pnpm db:push` | `drizzle-kit push` | 直接将 schema 推送到数据库（**不生成迁移文件**，仅用于开发） |
| `pnpm db:migrate` | `drizzle-kit migrate` | 应用 `drizzle/` 中未执行的迁移文件 |
| `pnpm db:studio` | `drizzle-kit studio` | 启动 Drizzle Studio Web UI（默认 3000 端口） |
| `pnpm setup:db [--seed]` | `node scripts/setup-db.mjs` | 直接执行 SQL schema 文件，跳过 Drizzle 迁移（用于初始化） |

## 3. 迁移文件结构

`drizzle/` 目录存放生成的迁移：

```
drizzle/
├── 0000_chilly_retro_girl.sql      # 初始迁移
├── 0001_ordinary_valkyrie.sql      # 后续迁移
└── meta/
    ├── _journal.json               # 迁移记录索引
    ├── 0000_snapshot.json          # schema 快照
    └── 0001_snapshot.json
```

`_journal.json` 记录每次迁移的 `idx`、`version`、`when`、`tag`，Drizzle 通过该文件确定迁移顺序与已应用状态。

> 注意：项目主 schema 为 `database/vnerpdacahng_schema.sql`，Drizzle schema（`src/lib/db/schema.ts`）目前覆盖部分表（如 `boms`、`bom_items`、`customers`）。新增表应同步两处，避免漂移。

## 4. 标准迁移流程

### 4.1 修改 Schema

编辑 `src/lib/db/schema.ts`，新增或修改表定义：

```ts
export const newTable = mysqlTable('new_table', {
  id: serial('id').primaryKey(),
  name: varchar('name', { length: 100 }).notNull(),
  createdAt: datetime('created_at').default(sql`CURRENT_TIMESTAMP`),
});
```

### 4.2 生成迁移

```bash
pnpm db:generate
```

Drizzle 输出 `drizzle/000X_<random_name>.sql` 与对应快照。检查 SQL 内容确认无误。

### 4.3 备份并应用

```bash
# 备份（迁移前必做）
mysqldump -h 127.0.0.1 -u root -p vnerp > backup_$(date +%Y%m%d).sql

# 应用迁移
pnpm db:migrate
```

`db:migrate` 通过 `drizzle/` 下的 journal 跟踪执行进度，幂等执行。

### 4.4 验证

```bash
# 启动 Studio 可视化检查
pnpm db:studio

# 或直接查询表结构
mysql -u root -p vnerp -e "DESC new_table;"
```

## 5. 开发期快速迭代

开发阶段如需快速试验 schema，可使用 `db:push` 直接推送：

```bash
pnpm db:push
```

⚠️ `db:push` **不**生成迁移文件，无法在生产追溯。生产环境必须使用 `db:generate + db:migrate`。

## 6. 回滚

Drizzle 不内置自动回滚。回滚策略：

1. **优先**：恢复迁移前备份
   ```bash
   mysql -u root -p vnerp < backup_YYYYMMDD.sql
   ```
2. 手写反向 SQL（如 `DROP TABLE` / `ALTER TABLE DROP COLUMN`），通过 `mysql` 客户端执行。
3. 不建议直接删除 `drizzle/` 下的迁移文件，会破坏 journal 一致性。

## 7. 与 setup-db.mjs 的关系

`scripts/setup-db.mjs` 直接读取 `database/vnerpdacahng_schema.sql` 执行，**不**走 Drizzle 流程，用于全新环境初始化。Drizzle 迁移用于在已有 schema 上增量演进。两者并存：

- 全新部署 → `pnpm setup:db --seed`
- 现有环境增量演进 → `pnpm db:generate && pnpm db:migrate`
