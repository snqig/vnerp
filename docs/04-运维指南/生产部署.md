# 生产部署

本文档说明 vnerp（印刷生产经营信息管理系统 Print MIS）生产环境部署的配置项、安全要求与更新流程，对应 `docker-compose.prod.yml`、`.env.production.example` 与 `nginx/nginx.conf`。完整部署步骤见 [Docker部署指南.md](./Docker部署指南.md)。

## 1. 系统要求

| 组件 | 最低版本 | 推荐版本 | 备注 |
|------|---------|---------|------|
| Node.js | 18.x | 20.x LTS | Docker 镜像使用 `node:20-alpine` |
| pnpm | 9.0+ | 9.0.0 | `corepack` 自动启用 |
| MySQL | 8.0+ | 8.0 | 字符集 `utf8mb4`，collation `utf8mb4_unicode_ci`（schema 推荐用 `utf8mb4_0900_ai_ci`） |
| Redis | 7.0+ | 7-alpine | 用于 token 黑名单 / refresh token / 事件总线（Redis Streams） |
| Docker | 20.10+ | 24.x+ | 含 BuildKit |
| Docker Compose | v2+ | v2.x | `docker compose` 子命令形式 |
| 服务器内存 | 2GB | 4GB+ | MySQL buffer pool 默认 1G（生产） |
| 磁盘 | 10GB | 50GB+ | 含数据卷与备份空间 |

## 2. 生产环境变量（.env.production）

复制 `.env.production.example` 为 `.env.production`，填入真实值（切勿提交到 git）：

| 变量 | 示例值 | 说明 |
|------|--------|------|
| `DB_HOST` | `mysql` | Docker 网络内的服务名 |
| `DB_PORT` | `3306` | MySQL 端口 |
| `DB_USER` | `vnerp_app` | 应用专用账号（不要使用 root） |
| `DB_PASSWORD` | 强密码 | 应用账号密码 |
| `DB_NAME` | `vnerp` | 数据库名 |
| `MYSQL_ROOT_PASSWORD` | 强密码 | 仅 docker-compose 初始化使用 |
| `JWT_SECRET` | 64 字节 hex 串 | `node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"` 生成 |
| `NODE_ENV` | `production` | 启用安全头与调试路由屏蔽 |
| `DEBUG_DB` | `false` | 关闭 SQL 日志 |
| `CORS_ALLOW_ORIGIN` | `https://erp.your-domain.com` | 生产禁止 `*` |
| `DEV_ORIGINS` | （空） | 生产无需配置 |
| `REDIS_URL` | `redis://redis:6379` | Docker 内部地址 |
| `PORT` | `5000` | 应用监听端口（容器内） |
| `HOSTNAME` | `0.0.0.0` | 容器内监听地址 |
| `JWT_EXPIRES_IN` | `7d` | docker-compose.prod.yml 声明，**注意**：当前代码未读取此变量，属已知断连 |
| `APP_VERSION` | `1.0.0` | 应用版本号（健康检查展示） |
| `LOG_LEVEL` | `info` | 日志级别（预留，当前 logger.ts 基于 `NODE_ENV` 判断） |
| `NEXT_PUBLIC_AUTH_COOKIE_MODE` | `header` | 认证模式（header / cookie），切换需 `next build` + 重启 |

> **安全配置现状**：`JWT_ACCESS_TOKEN_TTL`、`JWT_REFRESH_COOKIE_MAX_AGE`、`LOGIN_LOCKOUT_MINUTES`、`LOGIN_RATE_LIMIT_MAX`、`CSRF_COOKIE_MAX_AGE` 等参数在生产代码中部分硬编码（详见 `.env.production.example` 末尾注释），生产加固时建议抽取为环境变量。

## 3. 启动生产环境

```bash
# 构建
docker compose -f docker-compose.prod.yml --env-file .env.production build

# 启动（含 mysql / redis / app / nginx）
docker compose -f docker-compose.prod.yml --env-file .env.production up -d

# 查看应用日志
docker compose -f docker-compose.prod.yml logs -f app

# 查看服务状态
docker compose -f docker-compose.prod.yml ps
```

也可用一键部署脚本：

```bash
chmod +x scripts/deploy-docker.sh
./scripts/deploy-docker.sh up        # 构建并启动
./scripts/deploy-docker.sh health    # 健康检查
./scripts/deploy-docker.sh logs      # 实时日志
./scripts/deploy-docker.sh rebuild   # 重新构建并滚动更新 app
```

## 4. Nginx 反向代理

`docker-compose.prod.yml` 已内嵌 `nginx` 服务（`nginx:1.27-alpine`），配置位于 `nginx/nginx.conf`。职责：

1. **SSL 终结**：HTTP → HTTPS 重定向，TLS 1.2 / 1.3，强密码套件（ECDHE + GCM），`ssl_session_tickets off`。
2. **静态资源缓存**：`/_next/static/` 长缓存 + immutable（1 年）。
3. **gzip 压缩**：text/css/js/json/xml/svg/woff 等。
4. **安全头**：HSTS / CSP / X-Frame-Options / X-Content-Type-Options / Referrer-Policy / Permissions-Policy。
5. **限流**：`/api/auth/login` 限流 10 req/s/IP，burst 20。
6. **反代**：所有其他请求反代到 `http://app:5000`（upstream `nextjs_app`，keepalive 32）。
7. **上传限制**：`client_max_body_size 50m`。

SSL 证书放置于 `nginx/ssl/`：

```
nginx/ssl/
├── cert.pem   # 证书文件（含中间证书链）
└── key.pem    # 私钥文件
```

详细获取方式（Let's Encrypt / 自签名）见 [Docker部署指南.md](./Docker部署指南.md) 的 SSL 章节。

> `next.config.ts` 在生产环境已自动添加 HSTS、X-Frame-Options 等安全响应头，Nginx 层的 `add_header` 是补充强化。

## 5. 更新部署

### 5.1 常规更新（无 schema 变更）

```bash
git pull origin main
docker compose -f docker-compose.prod.yml --env-file .env.production up -d --build app
```

或用脚本：

```bash
./scripts/deploy-docker.sh rebuild
```

### 5.2 含数据库迁移

```bash
# 1. 备份数据库（必做）
pnpm backup

# 2. 执行增量迁移（详见 04-运维指南/数据库迁移.md）
pnpm migrate

# 3. 验证迁移状态
pnpm migrate:status

# 4. 重建并重启 app
docker compose -f docker-compose.prod.yml --env-file .env.production up -d --build app

# 5. 健康检查
curl https://localhost/api/health
```

> 迁移前务必备份；迁移失败时回滚到备份（`pnpm backup:restore backups/<最新>.sql.gz`），详见 [数据库迁移.md](./数据库迁移.md) 与 [备份恢复.md](./备份恢复.md)。

### 5.3 仅重启（不改镜像）

```bash
docker compose -f docker-compose.prod.yml --env-file .env.production restart app
```

## 6. 健康检查与监控

```bash
# 应用健康（通过 Nginx）
curl -f https://localhost/api/health

# 应用健康（容器内部）
docker compose -f docker-compose.prod.yml exec app \
  node -e "require('http').get('http://127.0.0.1:5000/api/health',r=>{process.exit(r.statusCode===200?0:1)})"

# 容器状态
docker compose -f docker-compose.prod.yml ps

# MySQL 连接
docker exec vnerp-mysql-prod mysqladmin -uvnerp_app -p ping

# Redis 连接
docker exec vnerp-redis-prod redis-cli ping

# Nginx 状态
docker exec vnerp-nginx-prod wget --spider -q http://127.0.0.1/ && echo "HEALTHY" || echo "UNHEALTHY"
```

`/api/health` 返回 JSON：`{ status, timestamp, version, environment, latency, checks: { database, memory, uptime } }`。状态码 200 = 健康，503 = 降级。

监控栈部署详见 [监控告警接入方案.md](./监控告警接入方案.md)。

## 7. 安全清单（上线前必查）

- [ ] `.env.production` 未提交到 git，密码为强随机串
- [ ] `JWT_SECRET` 为 64 字节随机 hex，非默认值
- [ ] `ALLOW_SETUP_API=false`（建表接口禁用）
- [ ] `CORS_ALLOW_ORIGIN` 配置为具体域名，非 `*`
- [ ] `EVENT_BUS_TYPE=db`（生产使用持久化事件总线）
- [ ] `DEBUG_DB=false`（关闭 SQL 日志）
- [ ] `NODE_ENV=production`
- [ ] `NEXT_PUBLIC_AUTH_COOKIE_MODE` 经 Staging 验证后再切换（默认 `header`）
- [ ] MySQL root 仅本地访问，应用使用 `vnerp_app` 等受限账号
- [ ] Redis 不对外暴露（仅 `expose`，不 `ports`）
- [ ] 启用 HTTPS（Nginx SSL，证书已放置 `nginx/ssl/`）
- [ ] 防火墙仅开放 80 / 443，不直接暴露 3306 / 6379 / 5000
- [ ] 已配置数据库定时备份（`pnpm backup` cron）
- [ ] 日志轮转已启用（`json-file` max-size 10m / max-file 5）
- [ ] 默认 admin 密码已修改
- [ ] 容器以非 root 用户运行（`nextjs` UID 1001）
- [ ] `read_only: true` + tmpfs + 显式可写卷
- [ ] `cap_drop: ALL` + `cap_add: NET_BIND_SERVICE`
- [ ] 资源限制（CPU + 内存 limits）已配置
- [ ] 监控告警已配置（Prometheus + Alertmanager，详见 [监控告警接入方案.md](./监控告警接入方案.md)）

## 8. 回滚

### 8.1 应用回滚（无 schema 变更）

```bash
# 拉取指定版本镜像
docker pull ghcr.io/snqig/vnerp:<tag>

# 修改 docker-compose.prod.yml 中 image 标签
sed -i 's|ghcr.io/snqig/vnerp:latest|ghcr.io/snqig/vnerp:<tag>|' docker-compose.prod.yml

# 重启 app
docker compose -f docker-compose.prod.yml --env-file .env.production up -d --no-deps --force-recreate app

# 验证健康
docker inspect --format='{{.State.Health.Status}}' vnerp-app-prod
```

### 8.2 含数据库回滚

```bash
# 1. 停止 app
docker compose -f docker-compose.prod.yml --env-file .env.production stop app

# 2. 恢复迁移前备份
pnpm backup:restore backups/<迁移前备份>.sql.gz
# 或
gunzip < backups/<迁移前备份>.sql.gz | \
  docker exec -i vnerp-mysql-prod sh -c \
  'exec mysql -uvnerp_app -p"$DB_PASSWORD" vnerpdacahng'

# 3. 回滚镜像到迁移前版本（参考 8.1）

# 4. 启动 app
docker compose -f docker-compose.prod.yml --env-file .env.production start app
```

## 9. 性能基线

生产模式下（`pnpm build` + `node server.js`）的页面 TTFB 基线（参考值，详见 [Docker部署指南.md](./Docker部署指南.md) 性能测速章节）：

| 页面 | TTFB | Full Load |
|------|------|-----------|
| 仪表盘 | ~33ms | ~83ms |
| 用户管理 | ~34ms | ~85ms |
| 角色管理 | ~36ms | ~87ms |
| 菜单管理 | ~32ms | ~83ms |
| 系统设置 | ~30ms | ~78ms |

> 对比 Dev 模式首次访问 2–9 秒（webpack 编译），生产模式提升约 60–280 倍。
