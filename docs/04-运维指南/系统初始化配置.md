# 系统初始化配置

本文档说明 VNERP 的 CLI 数据库初始化流程，对应 `scripts/setup-db.mjs`。该脚本作为标准初始化方式，替代历史 HTTP `/api/setup/create-tables` 接口。

## 1. 设计目标

| 维度 | 设计 |
|------|------|
| 安全性 | 所有配置从 `.env` 读取，无硬编码密码；仅本地/运维执行，不暴露为 HTTP |
| 独立性 | 不依赖 Next.js 服务运行，纯 Node.js 脚本 |
| 容错 | 单条 SQL 失败不中断整体执行，输出成功/失败统计 |
| 兼容 | 自动预处理 SQL 语句，适配 MySQL 8.0 collation 与 IF NOT EXISTS 语法 |

## 2. 前置条件

1. 已配置 `.env`，包含 `DB_HOST` / `DB_PORT` / `DB_USER` / `DB_PASSWORD` / `DB_NAME`。
2. MySQL 服务已启动且账号可访问。
3. `database/vnerpdacahng_schema.sql` 存在（如需种子，需同时存在 `database/seeds/vnerp-seed-data.sql`）。
4. Node.js 18+，无需额外依赖（脚本仅使用 `mysql2/promise` 与 Node 内置 `fs`/`path`）。

## 3. 用法

```bash
# 仅建表（不导入种子数据）
pnpm setup:db

# 建表 + 导入种子数据
pnpm setup:db --seed

# 直接调用脚本
node scripts/setup-db.mjs --seed
```

## 4. 执行流程

脚本（`scripts/setup-db.mjs`）执行步骤：

### 4.1 加载环境变量

`loadEnv()` 手动解析 `.env`，不引入 `dotenv` 依赖：

- 跳过注释行（`#`）与空行。
- 已存在的 `process.env` 不被覆盖（优先使用真实环境变量）。
- 移除值两端的引号。

### 4.2 连接 MySQL

```js
conn = await mysql.createConnection({
  host: DB_HOST, port: DB_PORT,
  user: DB_USER, password: DB_PASSWORD,
  charset: 'utf8mb4',
  multipleStatements: true,
});
```

未指定数据库，先执行 `CREATE DATABASE IF NOT EXISTS`。

### 4.3 创建数据库

```sql
CREATE DATABASE IF NOT EXISTS `<DB_NAME>`
  DEFAULT CHARACTER SET utf8mb4
  DEFAULT COLLATE utf8mb4_0900_ai_ci;
```

然后执行 `USE <DB_NAME>`。

### 4.4 执行 schema

读取 `database/vnerpdacahng_schema.sql`，做以下预处理：

| 替换规则 | 原因 |
|----------|------|
| 移除 `CREATE DATABASE ...;` | 已在 4.3 完成 |
| 移除 `USE ...;` | 已在 4.3 完成 |
| `CREATE TABLE` → `CREATE TABLE IF NOT EXISTS` | 容错，避免重复执行报错 |
| `utf8mb4_ai_ci` → `utf8mb4_0900_ai_ci` | MySQL 8.0 推荐的 collation |
| `INSERT INTO` → `INSERT IGNORE INTO` | 容错，避免重复主键冲突 |

预处理后按分号分割语句，逐条执行（`executeStatements`）。

### 4.5 导入种子（可选）

`--seed` 时读取 `database/seeds/vnerp-seed-data.sql`，同样将 `INSERT INTO` 替换为 `INSERT IGNORE INTO` 后逐条执行。

### 4.6 输出统计

```
[setup-db] 目标: root@127.0.0.1:3306/vnerp
[setup-db] ✓ MySQL 已连接
[setup-db] ✓ 数据库 vnerp 已就绪
[setup-db] ✓ schema: 成功 N 条，失败 M 条
[setup-db] ✓ 当前表数量: K
[setup-db] 完成（含种子数据）
```

失败明细仅打印前 5 条，避免日志爆炸。

## 5. 错误处理与提示

| 错误码 | 含义 | 提示 |
|--------|------|------|
| `ECONNREFUSED` | MySQL 未启动 | 检查 DB_HOST/DB_PORT，启动 MySQL 服务 |
| `ER_ACCESS_DENIED_ERROR` | 账号或密码错误 | 检查 `.env` 中 DB_USER/DB_PASSWORD |
| `ENOENT` | SQL 文件路径错误 | 确认 `database/vnerpdacahng_schema.sql` 存在 |

脚本失败时 `process.exit(1)`，便于 CI 检测。

## 6. 与其他初始化方式的关系

| 方式 | 命令 | 适用场景 |
|------|------|----------|
| CLI（推荐） | `pnpm setup:db --seed` | 全新环境初始化 |
| Drizzle 迁移 | `pnpm db:migrate` | 已有环境增量演进 |
| Docker 自动初始化 | `docker compose up -d mysql` | 容器首次启动挂载 `01-schema.sql` |
| HTTP 接口 | `POST /api/setup/create-tables` | 已废弃，受 `ALLOW_SETUP_API` 控制，生产必须为 `false` |

## 7. 常见问题

### 7.1 已有数据库重复执行

脚本使用 `CREATE TABLE IF NOT EXISTS` 与 `INSERT IGNORE INTO`，重复执行幂等，不会破坏现有数据。

### 7.2 部分语句失败

`executeStatements` 单条失败不中断整体，会在末尾输出失败表名与错误信息。常见原因：

- 表已存在但结构不同（如 collation 不一致）。
- 外键依赖未按顺序创建。
- 字段类型不支持（如旧版 MySQL）。

排查后可手动 `mysql` 客户端执行失败的 SQL 语句。

### 7.3 字符集问题

脚本统一使用 `utf8mb4_0900_ai_ci`。如已有表使用 `utf8mb4_unicode_ci`，需手动 `ALTER TABLE` 转换，或新建数据库。
