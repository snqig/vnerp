# 系统初始化配置

本文档说明 vnerp（印刷生产经营信息管理系统 Print MIS）的 CLI 数据库全量初始化流程，对应 `scripts/setup-db.mjs`（`package.json` 中的 `pnpm setup:db`）。该脚本作为标准初始化方式，替代历史 HTTP `/api/setup/create-tables` 接口。

## 1. 设计目标

| 维度 | 设计 |
|------|------|
| 安全性 | 所有配置从 `.env` 读取，无硬编码密码；仅本地 / 运维执行，不暴露为 HTTP |
| 独立性 | 不依赖 Next.js 服务运行，纯 Node.js 脚本（仅使用 `mysql2/promise` 与 Node 内置 `fs` / `path`） |
| 容错 | 单条 SQL 失败不中断整体执行，输出成功 / 失败统计 |
| 兼容 | 自动预处理 SQL 语句，适配 MySQL 8.0 collation 与 `IF NOT EXISTS` 语法 |
| 迁移同步 | 自动标记 `database/migrations/` 下所有迁移为已执行，避免重复跑增量迁移 |

## 2. 前置条件

1. 已配置 `.env`，包含 `DB_HOST` / `DB_PORT` / `DB_USER` / `DB_PASSWORD` / `DB_NAME`。
2. MySQL 服务已启动且账号可访问。
3. `database/vnerpdacahng_schema.sql` 存在（如需种子，需同时存在 `database/seeds/vnerp-seed-data.sql`）。
4. Node.js 18+（推荐 20.x LTS）。
5. pnpm 9.0.0+（`preinstall` 钩子强制）。

## 3. 用法

```bash
# 仅建表（不导入种子数据）
pnpm setup:db

# 建表 + 导入种子数据
pnpm setup:db --seed

# 直接调用脚本（不经 pnpm script）
node scripts/setup-db.mjs --seed
```

## 4. 执行流程

脚本（`scripts/setup-db.mjs`）执行步骤：

### 4.1 加载环境变量

`loadEnv()` 手动解析 `.env`，不引入 `dotenv` 依赖：

- 跳过注释行（`#`）与空行。
- 已存在的 `process.env` 不被覆盖（优先使用真实环境变量）。
- 移除值两端的引号（单引号或双引号）。

### 4.2 连接 MySQL

```js
conn = await mysql.createConnection({
  host: DB_HOST, port: DB_PORT,
  user: DB_USER, password: DB_PASSWORD,
  charset: 'utf8mb4',
  multipleStatements: true,
});
```

未指定数据库，先执行 `CREATE DATABASE IF NOT EXISTS`。

### 4.3 创建数据库

```sql
CREATE DATABASE IF NOT EXISTS `<DB_NAME>`
  DEFAULT CHARACTER SET utf8mb4
  DEFAULT COLLATE utf8mb4_0900_ai_ci;
```

然后执行 `USE <DB_NAME>`。

### 4.4 执行 schema

读取 `database/vnerpdacahng_schema.sql`，做以下预处理：

| 替换规则 | 原因 |
|----------|------|
| 移除 `CREATE DATABASE ...;` | 已在 4.3 完成 |
| 移除 `USE ...;` | 已在 4.3 完成 |
| `CREATE TABLE` → `CREATE TABLE IF NOT EXISTS` | 容错，避免重复执行报错 |
| `utf8mb4_ai_ci` → `utf8mb4_0900_ai_ci` | MySQL 8.0 推荐的 collation |
| `INSERT INTO` → `INSERT IGNORE INTO` | 容错，避免重复主键冲突 |

预处理后按分号 `;` 分割语句（移除 `--` 注释行），逐条执行（`executeStatements`）。单条失败不中断，记录到 `failures` 列表。

### 4.5 导入种子（可选）

`--seed` 时读取 `database/seeds/vnerp-seed-data.sql`，同样将 `INSERT INTO` 替换为 `INSERT IGNORE INTO` 后逐条执行。文件不存在则警告并跳过。

### 4.6 标记迁移为已执行

读取 `database/migrations/` 目录（过滤 `.sql` / `.ts` / `.js` 文件，按文件名排序），创建 `sys_migration` 表（如不存在），将所有迁移文件以 batch=0 写入：

```sql
CREATE TABLE IF NOT EXISTS `sys_migration` (
  id INT AUTO_INCREMENT PRIMARY KEY,
  migration_name VARCHAR(255) NOT NULL UNIQUE,
  batch INT NOT NULL DEFAULT 0,
  applied_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  execution_time INT DEFAULT 0,
  INDEX idx_migration_name (migration_name),
  INDEX idx_applied_at (applied_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci;

INSERT IGNORE INTO `sys_migration` (migration_name, batch, execution_time)
VALUES ('001_xxx.sql', 0, 0);
```

> 因为 schema.sql 已包含全部结构，增量迁移无需再跑一遍。setup:db 把它们标记为 batch 0，`pnpm migrate:status` 会显示「已执行」。

### 4.7 输出统计

```
[setup-db] 目标: root@127.0.0.1:3306/vnerp
[setup-db] ✓ MySQL 已连接
[setup-db] ✓ 数据库 vnerp 已就绪
[setup-db] ✓ schema: 成功 N 条，失败 M 条
[setup-db] ✓ 已标记 K/L 个迁移文件为已执行（batch 0）
[setup-db] ✓ 当前表数量: K
[setup-db] 完成（含种子数据）   # 或 "完成（如需种子数据请追加 --seed 参数）"
```

失败明细仅打印前 5 条，避免日志爆炸。

## 5. 错误处理与提示

| 错误码 | 含义 | 提示 |
|--------|------|------|
| `ECONNREFUSED` | MySQL 未启动 | 检查 `DB_HOST` / `DB_PORT`，启动 MySQL 服务 |
| `ER_ACCESS_DENIED_ERROR` | 账号或密码错误 | 检查 `.env` 中 `DB_USER` / `DB_PASSWORD` |
| `ENOENT` | SQL 文件路径错误 | 确认 `database/vnerpdacahng_schema.sql` 存在 |

脚本失败时 `process.exit(1)`，便于 CI 检测。

## 6. 与其他初始化方式的关系

| 方式 | 命令 | 适用场景 |
|------|------|----------|
| **CLI（推荐）** | `pnpm setup:db --seed` | 全新环境初始化（建库 + schema + seed + 标记迁移） |
| 增量迁移 | `pnpm migrate` | 已有环境增量演进（详见 [数据库迁移.md](./数据库迁移.md)） |
| Docker 自动初始化 | `docker compose up -d mysql` | 容器首次启动挂载 `01-schema.sql` 与 `02-seed.sql` |

## 7. 常见问题

### 7.1 已有数据库重复执行

脚本使用 `CREATE TABLE IF NOT EXISTS` 与 `INSERT IGNORE INTO`，重复执行幂等，不会破坏现有数据。同时 `sys_migration` 表用 `INSERT IGNORE` 写入，已存在的迁移记录不会重复。

### 7.2 部分语句失败

`executeStatements` 单条失败不中断整体，会在末尾输出失败表名与错误信息。常见原因：

- 表已存在但结构不同（如 collation 不一致）。
- 外键依赖未按顺序创建。
- 字段类型不支持（如旧版 MySQL）。

排查后可手动 `mysql` 客户端执行失败的 SQL 语句。

### 7.3 字符集问题

脚本统一使用 `utf8mb4_0900_ai_ci`。如已有表使用 `utf8mb4_unicode_ci`，需手动 `ALTER TABLE` 转换，或新建数据库：

```sql
ALTER TABLE <表名> CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_0900_ai_ci;
```

### 7.4 与 `pnpm migrate` 的执行顺序

- 全新环境：先 `pnpm setup:db --seed`，再视需要 `pnpm migrate`（应无待执行迁移）。
- 已有环境：**不要**再跑 `pnpm setup:db`（会重复建表 / 插入种子），改用 `pnpm migrate` 增量演进。
- 若误跑 `pnpm setup:db` 后想恢复，参考 [数据库迁移.md](./数据库迁移.md) 的回滚章节。

### 7.5 Docker 容器内的初始化

`docker-compose.yml` 与 `docker-compose.prod.yml` 中 mysql 服务挂载：

```yaml
volumes:
  - ./database/vnerpdacahng_schema.sql:/docker-entrypoint-initdb.d/01-schema.sql
  - ./database/seeds/vnerp-seed-data.sql:/docker-entrypoint-initdb.d/02-seed.sql
```

容器首次启动时 MySQL 会按文件名顺序自动执行 `docker-entrypoint-initdb.d/` 下的 SQL。**注意**：此机制仅在数据卷为空（首次启动）时触发，已存在的卷不会重复执行。如需重新初始化，需先清空 `./data/mysql`（生产）或 `mysql_data` 卷（开发）。

## 8. 安全注意事项

- `.env` 文件含明文密码，**不要**提交到 git（`.gitignore` 已排除）。
- `ALLOW_SETUP_API` 必须为 `false`（默认），生产环境禁止通过 HTTP 调用建表接口。
- 种子数据含默认管理员账号（如 `admin`），生产环境首次登录后**必须立即修改密码**。
- 脚本执行完检查 `sys_user` 表，确认无默认弱密码账号。
