# 部署指南

本文档描述 vnerp（印刷生产经营信息管理系统 Print MIS）的通用部署流程概览，对应 `Dockerfile`、`docker-compose.yml`、`docker-compose.prod.yml` 与 `docker-compose.monitoring.yml`。详细 Docker 部署见 [Docker部署指南.md](./Docker部署指南.md)，生产环境配置见 [生产部署.md](./生产部署.md)。

## 1. 镜像构建（Dockerfile）

`Dockerfile` 采用三阶段构建，依赖 `next.config.ts` 中 `output: 'standalone'`。

| 阶段 | 基础镜像 | 作用 |
|------|---------|------|
| `deps` | `node:20-alpine` | 安装依赖（`pnpm install --frozen-lockfile`），用 BuildKit 缓存挂载加速 |
| `builder` | `node:20-alpine` | 复制源码并执行 `pnpm build` |
| `runner` | `node:20-alpine` | 仅复制 standalone 产物、`/.next/static`、`/public`、`/messages` |

关键设计：

- 使用 `corepack` 启用 `pnpm@9.0.0`，与本地环境一致。
- `tini` 作为 PID 1，正确传递 SIGTERM / SIGINT 信号，避免僵尸进程。
- 运行阶段使用非 root 用户 `nextjs`（UID / GID 1001）。
- 环境变量 `NODE_ENV=production`、`NEXT_TELEMETRY_DISABLED=1`、`PORT=5000`、`HOSTNAME=0.0.0.0`。
- **启动命令 `node server.js`**（standalone 模式正确方式，非 `next start`）。
- `HEALTHCHECK` 命中 `/api/health` 端点（间隔 30s，超时 5s，启动宽限 60s，重试 3 次）。
- 对外暴露 5000 端口。

> **关键**：`next.config.ts` 中 `output: 'standalone'` 时，必须用 `node server.js` 启动，而非 `next start`。后者会触发警告 `"next start" does not work with "output: standalone" configuration.`，且 standalone 产物已内嵌最小化 `node_modules`，最终镜像约 ~150MB。

手动构建：

```bash
docker build -t vnerp:latest .
```

## 2. 开发环境编排（docker-compose.yml）

`docker-compose.yml` 包含 MySQL、Redis 与 app 三个服务，供本地或开发环境使用（app 的 `NODE_ENV=development`）。

| 服务 | 镜像 | 容器名 | 端口 | 说明 |
|------|------|--------|------|------|
| `mysql` | `mysql:8.0` | `vnerp-mysql` | `${DB_PORT:-3306}:3306` | 初始化时挂载 `database/vnerpdacahng_schema.sql` 与 `vnerp-seed-data.sql`；`innodb-buffer-pool-size=256M`，`max-connections=200`，慢查询日志开启（`long-query-time=2`） |
| `redis` | `redis:7-alpine` | `vnerp-redis` | 6379（仅 expose） | `appendonly yes`，`maxmemory 128mb`，`allkeys-lru` |
| `app` | 自构建（Dockerfile） | `vnerp-app` | `5000:5000` | `NODE_ENV=development`，`DEBUG_DB=true`，依赖 mysql + redis healthcheck |

启动：

```bash
docker compose up -d
docker compose logs -f app
```

## 3. 生产编排（docker-compose.prod.yml）

`docker-compose.prod.yml` 在开发编排基础上增强安全与资源限制，并新增 `nginx` 服务，使用 `--env-file .env.production` 加载生产配置：

```bash
docker compose -f docker-compose.prod.yml --env-file .env.production build
docker compose -f docker-compose.prod.yml --env-file .env.production up -d
```

容器列表：

| 容器 | 名称 | 端口 | 说明 |
|------|------|------|------|
| Nginx | `vnerp-nginx-prod` | `${HTTP_PORT:-80}:80`、`${HTTPS_PORT:-443}:443`（对外） | 反向代理 + SSL 终结 + gzip + 限流 + 安全头 |
| App | `vnerp-app-prod` | 5000（仅 expose，由 Nginx 反代） | `NODE_ENV=production`，`DEBUG_DB=false`，`read_only: true` + tmpfs + 显式可写卷 |
| MySQL | `vnerp-mysql-prod` | 3306（仅 expose） | bind mount `./data/mysql`，`innodb-buffer-pool-size=1G`，`max-connections=500` |
| Redis | `vnerp-redis-prod` | 6379（仅 expose） | `appendonly yes`，`appendfsync everysec`，`maxmemory 512mb` |

关键生产配置：

- 所有服务 `restart: always` + `security_opt: no-new-privileges:true`。
- App / Nginx `cap_drop: ALL` + `cap_add: NET_BIND_SERVICE`。
- 资源限制（CPU + 内存 limits / reservations）全覆盖。
- App `read_only: true` + `tmpfs: /tmp` + 显式可写卷（`app_logs`、`app_uploads`、`app_cache`）。
- 日志驱动 `json-file`，`max-size: 10m`，`max-file: 5`。
- 镜像地址 `ghcr.io/snqig/vnerp:latest`（CI/CD 推送 GHCR）。
- MySQL 数据用 bind mount（`./data/mysql`），`docker compose down -v` 不会删除业务数据。

> App 端口不对外暴露，所有外部请求经 Nginx 反向代理转发。Nginx 配置 SSL 终结、HTTP→HTTPS 重定向、gzip 压缩、登录限流（10 req/s/IP）和安全头（HSTS / CSP / X-Frame-Options / X-Content-Type-Options / Referrer-Policy / Permissions-Policy）。

## 4. 监控编排（docker-compose.monitoring.yml）

`docker-compose.monitoring.yml` 部署 Prometheus + Grafana + Alertmanager + Exporters 监控栈，与生产服务共用 `vnerp-network` 网络：

```bash
docker compose -f docker-compose.monitoring.yml --env-file .env.production up -d
```

详见 [监控告警接入方案.md](./监控告警接入方案.md)。

## 5. 部署流程清单

1. 准备生产服务器（推荐 4GB+ 内存、50GB+ 磁盘）。
2. 克隆仓库并 `cp .env.production.example .env.production`，填入真实密码与 `JWT_SECRET`。
3. 生成强随机 JWT 密钥：
   ```bash
   node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"
   ```
4. 放置 SSL 证书到 `nginx/ssl/`（`cert.pem` + `key.pem`）。
5. 构建并启动：`docker compose -f docker-compose.prod.yml --env-file .env.production up -d --build`。
6. 查看日志：`docker compose -f docker-compose.prod.yml logs -f app`。
7. 健康检查：`curl https://localhost/api/health` 或 `docker compose -f docker-compose.prod.yml exec app node -e "require('http').get('http://127.0.0.1:5000/api/health',r=>{process.exit(r.statusCode===200?0:1)})"`。
8. 停止：`docker compose -f docker-compose.prod.yml down`（数据卷保留）。

> 也可用一键部署脚本 `scripts/deploy-docker.sh`，详见 [Docker部署指南.md](./Docker部署指南.md)。

## 6. 健康检查

应用提供 `GET /api/health` 接口，返回 JSON 含 `status`、`timestamp`、`version`、`environment`、`latency`、`checks.database`、`checks.memory`、`checks.uptime` 字段。状态码 200 表示健康，503 表示降级。Docker `HEALTHCHECK` 与负载均衡探针均可使用。

## 7. 调试路由封堵

`src/proxy.ts`（Next.js 16 自动探测，**`src/middleware.ts` 已删除**）在 `NODE_ENV=production` 下对以下路径返回 404，避免调试代码泄露：

- `/qrcode`
- `/api/init`
- `/api/debug`
- `/api/diagnose`

> 路径前会剥离 locale 前缀（`/zh-CN`、`/zh-TW`、`/en`、`/vi`），故 `/zh-CN/qrcode` 同样被封堵。

## 8. 数据持久化

| 数据卷 | 挂载路径 | 内容 | 备份策略 |
|--------|----------|------|----------|
| `./data/mysql`（bind mount） | `/var/lib/mysql` | MySQL 数据文件 | 每日全量备份（`pnpm backup`） |
| `redis_data`（named volume） | `/data` | Redis RDB / AOF 文件 | 每周 `redis-cli BGSAVE` + `docker cp` |
| `app_logs` | `/app/logs` | 应用日志 | 按天轮转，保留 30 天 |
| `app_uploads` | `/app/uploads` | 用户上传文件 | 每日增量备份 |
| `app_cache` | `/app/.next/cache` | Next.js ISR 缓存 | 无需备份 |
| `nginx_cache` | `/var/cache/nginx` | Nginx 缓存 | 无需备份 |
| `nginx_logs` | `/var/log/nginx` | Nginx 访问 / 错误日志 | 按天轮转 |

> MySQL 使用 bind mount（`./data/mysql`），`docker compose down -v` 不会删除业务数据，但仍建议谨慎使用 `-v` 标志。
