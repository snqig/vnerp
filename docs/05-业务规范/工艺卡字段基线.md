## 二、校验规则

### 2.1 必填校验

| 字段 | 校验条件 | 错误提示 | 来源 |
|------|---------|---------|------|
| customer | 非空字符串 | "请选择客户" | InputCardForm.tsx:193 |
| customerCode | 非空字符串 | "请输入客户料号" | InputCardForm.tsx:197 |
| productName | 非空字符串 | "请输入品名" | InputCardForm.tsx:201 |

### 2.2 数值与格式

| 字段 | 规则 |
|------|------|
| date | date 类型，自动格式 yyyy-MM-dd |
| dashedKnife | boolean，API 序列化为 0/1 |
| coreType | 逗号分隔的多选值：3#, 2#, 1# |
| printType | 逗号分隔的多选值：胶印, 卷料丝印, 片料丝印, 轮转印 |
| processMethod | 逗号分隔的多选值：模切, 冲压 |
| sequences | JSON 字符串，7 行固定，每行 8 字段 |

### 2.3 无前端校验字段

> 以下字段在 card 版录入时不做任何前端校验（空字符串/NULL 均允许提交）：
> - 全部尺寸类字段（finishedSize, tolerance, sheetSpecs 等）
> - 全部编号类字段（moldCode, releasePaperCode 等）
> - 全部关联信息字段（adhesiveType, mylarSpecs 等）
> - 全部签名字段（creator, reviewer 等）
> - sequences 明细行字段（color, inkCode 等）

### 2.4 API 侧校验

| 校验 | 条件 | 来源 |
|------|------|------|
| card_no 唯一性 | POST/PUT 检查 `SELECT id FROM prd_standard_card WHERE card_no = ? AND deleted = 0` | standard-cards/route.ts:222 |
| PUT 时数据保护 | 空值不覆盖已有值 | route.ts:516-524 |
| 状态拦截 | approve 路由检查当前状态 | approve/route.ts:67 |
| 重复审核拦截 | 检查审核人字段是否已填写 | approve/route.ts:72 |

## 三、提交与状态流转

### 3.1 接口

| 操作 | Method | URL | Body |
|------|--------|-----|------|
| 创建 | POST | /api/standard-cards | 全部 78 个平铺字段 + status=1 |
| 更新 | PUT | /api/standard-cards | id + 需更新的字段 |
| 软删除 | DELETE | /api/standard-cards?id=N | - |
| 提交审核 | POST | /api/standard-card/action?action=submit | { id } |
| 审核通过 | POST | /api/standard-card/action?action=approve | { id } |
| 客户确认 | POST | /api/standard-card/action?action=confirm | { id } |
| 作废 | POST | /api/standard-card/action?action=obsolete | { id, reason } |
| 新建版本 | POST | /api/standard-card/action?action=newVersion | { id } |
| 查看审核 | GET | /api/standard-cards/approve?id=N | - |
| 批量删除 | DELETE | /api/standard-cards?id=1,2,3 | - |

### 3.2 请求参数结构

创建/更新请求体与 `mapCardDataToApiPayload()` 输出一致：
```json
{
  "card_no": "SC1740000000000",
  "customer_name": "xxx",
  "customer_code": "xxx",
  "product_name": "xxx",
  "version": "V1.0",
  "status": 1,
  // ... 全部 78 个 snake_case 字段
}
```

### 3.3 状态流转

```
数据库 status 值：
  1 = 草稿 (draft)
  2 = 待审核 (auditing)  — 审核流程进行中
  3 = 已启用 (enabled)    — 批准后生效
  4 = 已归档 (archived)   — 废弃归档

流转规则：
  草稿(1)  ──submit──▶  待审核(2)
  待审核(2) ──reject──▶ 草稿(1)
  待审核(2) ──approve──▶ 已启用(3)
  已启用(3)  (暂无非活跃操作)
  全部状态  ──DELETE──▶  软删除(deleted=1)

备注：前端列表展示 label 映射：
  1 → draft(草稿)
  2 → reviewed(待审核)
  3 → confirmed(已启用)
  4 → invalid(已归档)
```

### 3.4 域模型状态（StandardCard aggregate）

StandardCard 聚合根使用独立状态枚举，与 `prd_standard_card.status` 字段不直接对应：

```
StandardCardStatus 枚举：
  DRAFT = 'draft'         → 草稿
  AUDITING = 'auditing'   → 待审核
  APPROVED = 'approved'   → 已批准
  CONFIRMED = 'confirmed' → 已确认
  OBSOLETE = 'obsolete'   → 已作废

流转规则：
  draft ──submit()──▶   auditing
  auditing ──approve()──▶ approved
  auditing ──reject()──▶  draft
  approved ──confirm()──▶ confirmed (isCurrent=true, isLocked=true)
  approved/confirmed ──obsolete()──▶ obsolete

版本号生成规则：
  StandardCard.generateVersion(currentVersion?) → 无版本时返回 "1.0"
  解析 major.minor → minor+1

编码生成规则：
  StandardCard.generateCode(type) → 前缀 + yyyyMMdd + 4位随机数
```

### 3.6 版本复用逻辑

- `StandardCard.createNewVersion(newVersion, userId)`：仅限已确认状态
- 复制全部字段，id 置空
- 设置新版本号、DRAFT 状态
- 关联 `parentVersionId` 指向原卡
- 设置 `isCurrent=false, isLocked=false`
- 清空审核信息（auditUser, approveUser, confirmUser）
