# 全链路测试方案

> 编号：VNERP-TEST-MASTER | 版本：V2.1
>
> 本方案覆盖 vnerp（印刷生产经营信息管理系统 Print MIS）从框架、页面、计算到流程的全链路测试矩阵，作为测试用例编写与上线验收的依据。

## 一、测试总原则

| 原则 | 说明 |
|------|------|
| 全覆盖 | 覆盖每个框架、录入框、逻辑、计算、环节 |
| 自动化优先 | 可自动化的测试必须编写自动化用例（Vitest 单测 + Playwright E2E） |
| 数据驱动 | 用正常、边界、异常多组数据验证同一功能 |
| 可重复 | 每次执行得到一致结果（Vitest `beforeEach` 重置 Mock，E2E 由 `tests/global-setup.ts` 准备） |

## 二、框架级测试

| 测试项 | 验证方法 | 预期结果 |
|--------|----------|----------|
| 页面路由注册 | Playwright 访问每个页面 | 返回 200，无 404 |
| API 路由注册 | 自动化 GET 请求 | 所有 API 返回合法响应（`src/lib/api-response.ts` 统一格式） |
| 路由守卫 | E2E 未登录访问受保护前缀 | 由 `src/proxy.ts` 重定向到 `/login?next=原路径` |
| 数据库连接池 | 单元测试 | `src/lib/db/index.ts` 单例连接池正常释放 |
| 事务管理 | 集成测试 | commit 后持久化，rollback 后不变 |
| 表单验证 | 手动 + E2E | 必填、类型、长度、格式、范围均拦截非法输入（zod schema + react-hook-form） |
| 权限框架 | E2E 多角色登录 | 菜单、按钮、API 按角色生效（`withPermission` 装饰器 + RBAC）；Token 过期重登录 |
| 日志框架 | 手动 + 看日志 | `sys_oper_log` / `sys_login_log` 齐全，`audit-logger` 落库 |
| 事件总线 | Vitest 集成测试 | Redis Streams + Outbox 两阶段、XAUTOCLAIM 回收、OutboxPoller 死信回收 |
| i18n | E2E + 手动 | 4 语言（zh-CN/zh-TW/en/vi）切换正确，`localePrefix: 'as-needed'` |

## 三、页面与录入框级测试

通用：页面加载无 JS 错误；按钮可点击；输入类型校验；最大长度；必填提示；下拉/日期/分页/搜索/响应式。

| 模块 | 关键测试点 |
|------|-----------|
| 生产 | 工单创建/编辑/状态流转；报工-印刷/模切（含工具列写入）；报工数量上限；标准卡创建与关联；生产调度（含 enhanced） |
| 仓库 | 采购入库、生产入库、切料、领料出库、销售出库、物料拆分、盘点创建/执行/审核、调拨、二维码追溯、库存预警、成本 |
| 采购 | 申请、审批、订单、确认、入库、采购退货、采购对账 |
| 销售 | 销售订单、发货（含部分发货/再发货）、退货、对账核销 |
| 品质 | IQC/IPQC/FQC 检验单与判定、SPC 统计、SGS 报告与到期预警、不合格品处理、客户投诉、供应商审核 |
| 印前/打样 | 工序卡（样品卡/量产卡）、油墨配方版本、刀模/网版寿命、标签打印、扫码追溯 |
| 财务 | 应收/应付/凭证/费用/成本差异/账龄/报表 |
| 系统 | 用户、角色、密码、禁用、菜单管理、字典、配置、公告、操作日志、登录日志、调度器、工作流 |

> 完整页面清单见 `src/app/[locale]/` 下 26 个业务域，140+ `page.tsx`。

| 编号 | 库存状态 | 需求 | 预期分配 |
|------|----------|------|----------|
| FIFO-001 | 批次1: 100kg(1月), 批次2: 100kg(2月) | 80kg | 批次1: 80kg |
| FIFO-002 | 批次1: 50kg(1月), 批次2: 100kg(2月) | 80kg | 批次1: 50kg + 批次2: 30kg |
| FIFO-003 | 批次1: 50kg | 80kg | 失败，提示库存不足 |

| FIFO-005 | 批次1 status=2(已用), 批次2 status=1 | 30kg | 批次2: 30kg |

## 六、二维码追溯链路

> 实现位于 `src/lib/qrcode-service.ts`；表为 `qrcode_record`（参见 ADR-002）。

| 编号 | 场景 | 预期结果 |
|------|------|----------|
| QR-001 | 入库生成 | `qrcode_record` 生成，status=1, split_flag=0 |
| QR-002 | 拆分生成 | 小料 split_flag=1，余料 split_flag=2，parent_qr_id 正确 |
| QR-003 | 出库更新 | 被消耗记录 status=2 |
| QR-004 | 向上追溯 | 通过 parent_qr_id 找到整料 |
| QR-005 | 向下追溯 | 找到所有子记录 |

## 七、异常与边界

| 编号 | 场景 | 预期 |
|------|------|------|
| ERR-001 | 并发创建同编号 | 一个成功，一个 409（乐观锁 version 自增） |
| ERR-002 | 网络中断 | 错误提示，数据不丢失 |
| ERR-003 | 数据库连接失败 | 返回 500，不崩溃 |
| ERR-004 | 数量 999999999 | 正常处理或提示超范围 |
| ERR-005 | 数量 -1 | 拦截，必须为正数 |
| ERR-006 | 数量 0 | 拦截，必须大于 0 |
| ERR-007 | SQL 注入文本 | 参数化查询防护（`?` 占位符 + `mysql2.escapeId()`） |
| ERR-008 | `<script>` 文本 | `src/lib/sanitize.ts` HTML 转义，不执行 |
| ERR-009 | `IN (?)` + 数组 | `execute()` 走预编译不展开数组，须 `${ids.map(() => '?').join(',')}` 显式展开（见 commit `0f8c0d4`） |
| ERR-010 | CSRF token 缺失 | `src/lib/csrf.ts` 校验失败返回 403 |

## 九、并发冲突测试

`tests/concurrency/` 提供四类并发场景，每类测试 N 个客户端同时操作同一资源：

| 测试文件 | 场景 | 关键校验 |
|----------|------|----------|
| `inventory-outbound.test.ts` | 多客户端同时出库 | 库存不超扣，乐观锁 version 自增，失败方收到 409 |
| `material-issue.test.ts` | 多客户端同时领料 | 领用量不超工单需求，库存与批次账一致 |
| `purchase-inbound.test.ts` | 多客户端同时入库同物料 | 批次号不重复，库存累加正确 |
| `stocktaking-approve.test.ts` | 多客户端同时审核盘点单 | 仅一方成功，其余收到 409；库存调整只生效一次 |

报告输出到 `tests/concurrency/reports/`（JSON 格式）。

## 十、测试报告

每次执行后自动生成：用例总数 / 通过数 / 失败数、覆盖率、失败用例详情与截图、执行时间统计。

- Playwright 输出 `playwright-report/`（HTML）+ `test-results.json`（JSON）+ `test-results/`（trace/screenshot/video，失败重试时）
- Vitest 单测输出到控制台；如需覆盖率 HTML/LCOV，运行 `pnpm test:coverage` 输出到 `coverage/`
