# 测试指南

> 面向开发者：如何在本项目中编写、运行与维护测试。
> 所有配置以 `vitest.config.ts` 与 `playwright.config.ts` 实际内容为准。

## 一、测试分层

| 层级 | 工具 | 位置 | 用途 |
|------|------|------|------|
| 单元测试 | Vitest | `tests/unit/**/*.test.ts` | 函数 / 算法 / 状态机 / 值对象 / 聚合根 |
| 集成测试 | Vitest + Mock / 真实 DB | `tests/integration/**/*.test.ts` | API 路由、DB 事务、跨模块协作、事件总线 |
| E2E 测试 | Playwright | `tests/*.spec.ts` | 浏览器端真实业务流程 |
| 并发测试 | Vitest + 模拟 | `tests/concurrency/*.test.ts` | 乐观锁 / 并发冲突 |

> Vitest `include: ['tests/**/*.test.ts']` 与 Playwright `testMatch: '**/*.spec.ts'` 通过扩展名天然隔离，不会互相误抓。

## 二、运行测试

### 单元 / 集成测试（Vitest）

```bash
pnpm test:unit          # 监听模式（开发时使用）
pnpm test:unit:run      # 单次运行（CI / 提交前）
pnpm test:coverage      # 生成覆盖率报告，输出到 coverage/（需在 vitest.config.ts 启用 coverage 段）
```

运行特定文件或用例：

```bash
# 指定测试文件
pnpm vitest run tests/unit/fifo-allocation.test.ts

# 按用例名过滤
pnpm vitest run -t "应该按FIFO顺序分配库存"

# 仅跑集成测试目录
pnpm vitest run tests/integration
```

### E2E 测试（Playwright）

```bash
pnpm test               # 全量 E2E（首次会自动启动 dev server）
pnpm test:headed        # 有头模式，便于观察
pnpm test:ui            # Playwright UI 模式
pnpm test:debug         # 调试模式
pnpm test:report        # 查看 HTML 报告
pnpm test:install       # 安装浏览器二进制
```

E2E 默认 `baseURL: http://localhost:5000`，`webServer` 会自动启动 `pnpm run dev:webpack`（webpack 模式规避 Windows Turbopack `nul` 崩溃）；若已存在同地址服务则复用（`reuseExistingServer: !CI`）。

### 类型与代码质量

```bash
pnpm ts-check           # TypeScript 类型检查
pnpm lint               # ESLint
pnpm format:check       # Prettier 检查
```

## 三、编写单元测试

### 模板

```typescript
import { describe, it, expect, beforeEach, vi } from 'vitest';
import { expandBom } from '@/lib/bom-expansion';

describe('BOM 展开计算', () => {
  beforeEach(() => {
    vi.clearAllMocks();
  });

  it('应该正确展开单层 BOM 并加上损耗', async () => {
    const result = await expandBom(1, 10);
    expect(result.items).toHaveLength(2);
    expect(result.items[0].actualQuantity).toBeCloseTo(21); // 20 * (1 + 5/100)
  });
});
```

### Mock 策略

数据库、配置、日志等通过 `vi.mock` 替换，避免触达真实环境：

```typescript
vi.mock('@/lib/db', () => ({
  query: vi.fn(),
  execute: vi.fn(),
  transaction: vi.fn((fn) => fn(mockConn)),
}));

vi.mock('@/lib/global-config', () => ({
  getConfig: vi.fn((key: string) => configMap[key]),
}));

vi.mock('@/lib/logger', () => ({ secureLog: vi.fn() }));
```

> 集成测试如需真实 DB，由 `tests/setup-env.ts` 加载 `.env`（`process.loadEnvFile()`）注入 `DB_PASSWORD` / `REDIS_URL`；缺失时会报 `Access denied (using password: NO)`。

### 命名与隔离

- 用例名称使用中文描述，清晰表达意图（`it('应该按FIFO顺序分配库存', ...)`）。
- 每个 `it` 独立可重复，`beforeEach` 中重置 Mock 与数据。
- 异步用例必须 `await`，不要忘记 `async`。

## 四、Mock 常见陷阱与防回归规范

> 以下规则源于实际踩坑，写入规范防止重复发生。

### 4.1 箭头函数 mock 不可被 `new` 调用

**问题**：`vi.fn((data) => ({ type: '...', data }))` 创建的是箭头函数。当源代码用 `new SampleCardEvent(data)` 调用时，JavaScript 报 `TypeError: ... is not a constructor`，因为箭头函数没有 `[[Construct]]` 内部方法。

**错误写法**：

```typescript
vi.mock('@/domain/sample-card/events/SampleCardEvents', () => ({
  SampleCardQuoteGeneratedEvent: vi.fn((data) => ({ type: 'SampleCardQuoteGenerated', data })),
}));
```

**正确写法** — 用 `function` 语法或 `class`：

```typescript
vi.mock('@/domain/sample-card/events/SampleCardEvents', () => ({
  SampleCardQuoteGeneratedEvent: vi.fn(function (this: unknown, data: unknown) {
    (this as { type: string; data: unknown }).type = 'SampleCardQuoteGenerated';
    (this as { type: string; data: unknown }).data = data;
  }),
}));
```

**判断标准**：如果被 mock 的模块导出的是 `class`，且源代码用 `new XxxEvent(...)` 调用，则 mock 工厂必须返回 `function`/`class` 语法，不可用箭头函数。

### 4.2 SQL 参数索引与硬编码值的偏移

**问题**：INSERT/UPDATE 语句的 VALUES 子句中，硬编码值（如 `status = 1`、`NOW()`、`'CNY'`）不占用 `?` 占位符位置，因此参数数组的索引 **不等于** 列在 SQL 中的序号。

**示例**：

```sql
-- 列序号:     1          2         3       4        5         6        7(硬编码)
INSERT INTO card (sample_no, sample_name, status, spec, material_cost, labor_cost, create_time)
VALUES         (?,         ?,           1,     ?,   ?,             ?,          NOW())
```

参数数组 `[sampleNo, sampleName, spec, materialCost, laborCost]` 只有 5 个元素：
- `params[0]` = sample_no（列序 1）
- `params[1]` = sample_name（列序 2）
- `params[2]` = spec（列序 4，跳过了 status=1）
- `params[3]` = material_cost（列序 5）
- `params[4]` = labor_cost（列序 6）

**防回归写法** — 每次断言参数索引时，同时验证占位符数量与参数数组长度一致：

```typescript
it('INSERT card: ? 占位符数 === params.length', async () => {
  // ... 触发 service 调用 ...

  const sql = mocks.mockConn.execute.mock.calls[0][0] as string;
  const params = mocks.mockConn.execute.mock.calls[0][1] as unknown[];

  // 核心断言：占位符数 === 参数数，防止新增硬编码值后索引漂移
  const placeholderCount = (sql.match(/\?/g) || []).length;
  expect(placeholderCount).toBe(params.length);

  // 语义映射：每个索引对应的列名
  expect(params[2]).toBe(spec);       // 索引 2 = spec（跳过了硬编码 status=1）
});
```

**规则**：
1. **永远不要**假设参数索引等于列序号。先用 `sql.match(/\?/g).length === params.length` 确认总数，再逐个验证语义。
2. 修改 SQL（新增硬编码值或 `?`）后，必须同步更新测试的参数索引断言。
3. 新增 INSERT/UPDATE 测试时，必须包含占位符数量一致性检查。

### 4.3 `IN (?)` + 数组参数的陷阱

**问题**：`execute()` 走 `pool.execute`（预编译语句），不会展开数组参数。`WHERE id IN (?)` 绑定的整个数组会被当作单个字符串值，导致只匹配第一个 ID 或完全不匹配。

**错误写法**：

```typescript
// ❌ join(',') 把数组变成字符串 "1,2,3"，execute 绑定后等价于 WHERE id IN ('1,2,3')
await execute(`UPDATE sys_notification SET is_read = 1 WHERE id IN (?)`, [
  notification_ids.join(','),
]);
```

**正确写法** — 显式展开占位符：

```typescript
// ✅ 每个元素一个 ?，参数直接传数组
await execute(
  `UPDATE sys_notification SET is_read = 1 WHERE id IN (${notification_ids.map(() => '?').join(',')})`,
  notification_ids
);
```

> **注意**：`query()` 走 `pool.query`（客户端展开），`IN (?)` + `[arr]` 可以工作，但 `execute()` 不展开。项目中统一用 `execute()` 时必须显式展开占位符。该规则已全项目落地（commit `0f8c0d4`）。

### 4.4 logger.error 的 phase 追踪断言

当源代码使用 `let phase = 'init'` + `logger.error(ctx, '... 失败 [phase=xxx]', { error })` 模式时，`phase` 在**消息字符串**中，不在数据对象中。

**错误写法**：

```typescript
expect(mocks.logger.error).toHaveBeenCalledWith(
  ctx,
  'createCard 失败',
  expect.objectContaining({ phase: 'insert_card' }) // ❌ phase 不在数据对象中
);
```

**正确写法**：

```typescript
const errorCall = mocks.logger.error.mock.calls[0];
expect(errorCall[0]).toMatchObject({ module: 'sample-card', action: 'createCard' });
expect(errorCall[1] as string).toContain('createCard 失败');
expect(errorCall[1] as string).toContain('phase=insert_card');
expect(errorCall[2]).toMatchObject({ error: 'DB connection lost' });
```

### 4.5 SQL 占位符排查实战案例（2026-07-10）

> 本节记录了全项目扫描发现并修复的 8 处 SQL 占位符不匹配 Bug，以及 2 个 Handler 新增的日志埋点。对应 commit `e1851eb`、`0f8c0d4`、`c188db7`。

#### 4.5.1 修复的 8 处 SQL 占位符 Bug

| # | 文件 | 场景 | 修复前 | 修复后 |
|---|------|------|--------|--------|
| 1 | `src/app/api/init/menus/route.ts` | sys_menu 降级 INSERT | 10 个 `?` 但只有 9 个参数（缺 `status`） | 补充 `1` 到参数数组末尾 |
| 2 | `src/app/api/warehouse/inbound/cutting/route.ts` | inv_material_label 余料 INSERT | `is_main_material` 硬编码 `0` 但参数数组仍传 `sourceLabel.is_main_material`（18 个 `?` 19 个参数） | 删除多余参数 |
| 3 | `src/application/handlers/PurchaseReceivedHandler.ts` | inv_inventory_batch INSERT | 10 列但 11 个值，`CURDATE()` 前多一个 `?`（8 个 `?` 7 个参数） | 删除多余 `?` |
| 4 | `src/application/handlers/ScreenPlateCostHandler.ts` | screen_plate_history INSERT | 6 列但 7 个值，`NOW()` 前多一个 `?`（5 个 `?` 4 个参数） | 删除多余 `?` |
| 5 | `src/application/services/ProductionApplicationService.ts` | inv_inventory_transaction INSERT | 9 列但 10 个值，`NOW()` 前多一个 `?`（7 个 `?` 6 个参数） | 删除多余 `?` |
| 6 | `src/application/services/SampleProcessCardService.ts` | dcprint_sample_process_card 版本复制 | 26 列但 27 个值，`NOW()` 前多一个 `?`（25 个 `?` 24 个参数） | 删除多余 `?` |
| 7 | `src/application/services/SampleProcessTemplateService.ts` | dcprint_sample_process_template 转模板 | 25 列但 26 个值，`NOW()` 前多一个 `?`（24 个 `?` 23 个参数） | 删除多余 `?` |
| 8 | `src/app/api/warehouse/alert-push/route.ts` | sys_notification 批量已读 | `IN (?)` + `join(',')` 导致只匹配第一个 ID | 改为 `map(() => '?').join(',')` 显式展开 |

**共同模式**：Bug 3-7 都是在 `NOW()` 或 `CURDATE()` 等函数前多写了一个 `?`，导致后续硬编码值和 `?` 全部错位。Bug 1-2 是参数数组与 SQL 占位符不同步。

#### 4.5.2 fifo-allocation SQL 注入修复（commit `c188db7`）

`src/lib/fifo-allocation.ts` 中存在 SQL 注入漏洞，已修复并新增代码审查规范。修复方式：将动态拼接的 SQL 改为参数化查询（`?` 占位符 + `mysql2.escapeId()` 白名单校验）。

#### 4.5.4 验证方法

1. **验证脚本**：`scripts/verify-sql-fixes.mjs` — 10 组模拟数据覆盖 7 处修复的 SQL 路径
   - 运行：`node --env-file=.env scripts/verify-sql-fixes.mjs`
   - 清理：`node --env-file=.env scripts/verify-sql-fixes.mjs --clean`
   - 结果：18/19 PASS（1 项 SKIP 因 `screen_plate_history` 表未创建）

2. **单元测试**：`tests/unit/application/services/ToolManagementService.test.ts` 中的 "SQL 参数索引一致性防回归" 用例验证占位符数 === params.length

3. **全量测试**：53 文件 / 1034 测试全部通过，无回归

#### 4.5.5 防回归规则

1. 新增 INSERT/UPDATE 语句时，**必须**在测试中验证 `(sql.match(/\?/g) || []).length === params.length`
2. `NOW()`、`CURDATE()` 等函数**不**占用 `?`，列数 = 值数时 `?` 数 = 参数数
3. 使用 `execute()`（预编译）时，`IN (?)` **不可**直接传数组，必须 `${ids.map(() => '?').join(',')}` 显式展开
4. 核心业务 Handler 必须包含 phase tracking + 关键分支 logger.info，便于运行时排查

## 五、编写 E2E 测试

### 模板

```typescript
import { test, expect } from '@playwright/test';

test('登录后跳转看板', async ({ page }) => {
  await page.goto('/login');
  await page.fill('input[name="username"]', 'admin');
  await page.fill('input[name="password"]', 'admin123');
  await page.click('button[type="submit"]');
  await expect(page).toHaveURL(/\/dashboard/);
});
```

> 注：默认 locale `zh-CN` 不带前缀（`localePrefix: 'as-needed'`），因此 `goto('/login')` 而非 `goto('/zh-CN/login')`。

### 约定

- 文件统一放 `tests/` 根，命名 `*.spec.ts`。
- 全局准备走 `tests/global-setup.ts`。
- 移动端场景使用 `projects` 中预置的 `Mobile Chrome` / `Mobile Safari`。
- `baseURL` 已在 `playwright.config.ts` 配置为 `http://localhost:5000`，`goto` 用相对路径。

## 六、覆盖率

### 当前状态

`vitest.config.ts` 当前**未配置** `coverage` 段，因此 `pnpm test:coverage` 会使用 v8 provider 的默认设置。如需启用渐进式防退化阈值，需在 `vitest.config.ts` 中补充 `test.coverage`：

```ts
coverage: {
  provider: 'v8',
  reporter: ['text', 'json', 'html', 'lcov'],
  include: [
    'src/lib/{bom-expansion,fifo-allocation,inventory-sync,warehouse-core,material-requisition,state-machine}.ts',
    'src/domain/**',
    'src/infrastructure/event-bus/{EventBus,OutboxPoller,MemoryDomainEventOutbox,DomainEventOutboxFactory}.ts',
  ],
  // thresholds: { lines: 45, functions: 70, branches: 35, statements: 45 },
}
```

### 目标

随补测逐步提升至：lines/functions/statements 80%、branches 70%。查看报告：`coverage/index.html`。

## 七、CI 集成

- PR 创建 / 合并到 main 时自动触发 `.github/workflows/ci.yml`。
- CI 流水线：`pnpm install` → `pnpm ts-check` → `pnpm lint` → `pnpm test:coverage` → `pnpm test`。
- CI 中 Playwright `forbidOnly: true`，`workers: 1`，`retries: 2`。
- pre-push Husky 钩子执行单元测试（commit `edb0906`）。
- 覆盖率不达阈值（如启用）将失败构建。

## 八、故障排查

| 现象 | 排查方向 |
|------|----------|
| Vitest 报 `Access denied (using password: NO)` | `tests/setup-env.ts` 加载 `.env` 失败，检查 `DB_PASSWORD` / `REDIS_URL` 是否配置 |
| Vitest 误抓 `.spec.ts` 报错 | 检查 `include` 是否为 `tests/**/*.test.ts`；`.spec.ts` 由 Playwright 处理 |
| E2E 启动失败 | 确认 5000 端口未被占用，或先 `pnpm dev:webpack` 后利用 `reuseExistingServer` |
| Windows 下 dev server 启动崩溃（`nul`） | 必须用 `pnpm dev:webpack` 而非 `pnpm dev`，规避 Turbopack CSS 处理 bug |
| 覆盖率骤降 | 检查 `include` 列表是否变动，新增文件是否纳入统计 |
| Mock 失效 | `beforeEach` 中遗漏 `vi.clearAllMocks()`；`vi.mock` 路径与 import 路径不一致 |
| 异步断言不生效 | 忘记 `await` 或 `expect().resolves` 用法 |
| `TypeError: ... is not a constructor` | mock 工厂返回了箭头函数，改用 `function` 语法（见 §4.1） |
| 参数索引断言失败 | SQL VALUES 中有硬编码值（`1`、`NOW()`、`'CNY'`），不占 `?` 位，需重新计数（见 §4.2） |
| `Column count doesn't match value count` | VALUES 中 `NOW()`/`CURDATE()` 前多写了 `?`，或列数与值数不一致（见 §4.5） |
| 批量操作只生效一条 | `execute()` 的 `IN (?)` + `join(',')` 导致只匹配第一个 ID，需显式展开占位符（见 §4.3） |
