# NERP 项目全目录精细化评估

按「工程基建层、架构核心层、数据持久层、业务模块层、前端资源层、文档配置层」六大类，对项目所有顶层目录与核心子目录逐一支拆解，包含模块定位、完成度、已落地内容、深度亮点、现存问题、优化建议六个维度。

---

## 一、工程基建层（开发规范 + 质量门禁 + 交付部署）

### 1. .github 目录

**定位**：GitHub 协作模板与 CI/CD 承载目录  
**完成度**：30%  
**已落地内容**：  
- 标准 GitHub 项目模板（Issue、PR 规范模板）

**深度亮点**：  
- 从 0 到 1 搭建开源项目协作规范，降低多人协作的沟通成本

**现存问题**：  
- 仅包含项目模板，未配置 GitHub Actions 流水线，CI/CD 能力完全缺失
- 无自动化测试、构建、镜像发布流程

**优化建议**：  
- 优先补充 PR 自动校验流水线（执行 lint、类型检查、单元测试）
- 后续扩展镜像构建与自动部署流程

## 二、架构核心层（src 目录，DDD 分层核心）

### 1. src/domain 领域层

**定位**：DDD 核心层，封装纯业务逻辑，零外部框架依赖  
**完成度**：全域覆盖度 20%；仓储领域完成度 90%  
**已落地内容**：  
- **shared 共享基础**：领域异常、事件接口、Money 金额值对象
- **warehouse 仓储领域**：InboundOrder 聚合根、InboundItem 实体、入库领域事件、入库仓储接口、OrderStatus 状态值对象

**深度亮点**：  
- 分层严格，领域层零框架依赖，可独立进行单元测试
- 聚合边界清晰，入库单作为聚合根管控明细，事务边界明确
- 值对象封装业务规则（状态流转、金额精度），避免业务逻辑泄露到应用层

**现存问题**：  
- 覆盖范围极窄，仅仓储一个领域落地，其余业务模块均未进入领域层
- 缺少领域服务抽象，FIFO 算法等核心业务逻辑散落在 lib 工具库
- 缺少领域异常的统一定义与分类体系

**优化建议**：  
- 优先扩展印前 / 生产为第二个标杆领域，统一领域层编码规范
- 将 FIFO 分配、状态机等业务属性强的工具迁移为领域服务
- 完善领域异常体系，按业务场景分类定义

---

### 2. src/application 应用层

**定位**：业务流程编排，协调领域对象与基础设施  
**完成度**：30%  
**已落地内容**：  
- **handlers 事件处理器**：审计日志、缓存失效、财务凭证、库存同步、二维码生成
- **services 应用服务**：看板数据服务、入库应用服务

**深度亮点**：  
- 事件驱动解耦，核心业务操作发布事件，副作用由处理器响应
- 应用服务遵循「加载聚合 → 调用领域方法 → 持久化 → 发布事件」的标准 DDD 流程

**现存问题**：  
- 应用服务覆盖不足，仅入库与看板有实现，其余模块业务编排散落在 API 层
- 事件处理器无异常重试与补偿机制，执行失败会导致数据不一致
- 缺少事务编排，跨聚合业务链路无一致性保障

**优化建议**：  
- 补全核心模块应用服务，收敛 API 层的业务逻辑
- 为事件处理器增加重试、死信队列与补偿机制
- 引入应用服务级别的事务管控，保障单业务操作的数据一致性

---

### 3. src/infrastructure 基础设施层

**定位**：技术细节实现，为应用层提供技术支撑  
**完成度**：40%  
**已落地内容**：  
- **cache**：内存缓存管理器
- **event-bus**：内存事件总线
- **config**：事件注册配置
- **repositories**：MySQL 入库单仓储实现

**深度亮点**：  
- 严格实现依赖倒置原则，仓储实现依赖领域层接口，依赖方向正确，后续替换持久化方案成本低

**现存问题**：  
- 全部为单机内存实现，不支持分布式多实例部署
- 缓存仅支持 TTL 过期 + 事件失效，缺少缓存击穿、雪崩防护机制
- 缺少外部系统集成适配层（短信、邮件、支付等对接）

**优化建议**：  
- 替换内存事件总线为 Redis Stream，内存缓存为 Redis
- 增加基础设施层的统一异常处理与监控埋点
- 预留外部系统集成的适配层接口

---

### 4. src/lib 通用工具库

**定位**：通用技术工具与业务无关的能力封装  
**完成度**：70%  
**已落地内容**：  
- **db**：数据库连接与查询封装
- **api-response**：统一 API 响应格式
- **auth**：JWT 认证工具
- **fifo-allocation**：FIFO 出库分配算法
- **state-machine**：通用状态机工具

**深度亮点**：  
- 基础工具封装完整，避免重复造轮子；FIFO 算法、状态机等能力抽象复用性高

**现存问题**：  
- 职责边界模糊，FIFO 分配、状态机等业务属性强的能力不应放在通用工具库
- 缺少工具函数的单元测试
- 数据库查询封装层缺少慢查询监控埋点

**优化建议**：  
- 将业务属性强的工具迁移至对应领域的领域服务
- 补充工具函数的单元测试
- 增加数据库层的慢查询统计与告警能力

---

### 5. src/components 组件层

**定位**：前端共享组件库  
**完成度**：75%  
**已落地内容**：  
- **ui**：shadcn/ui 基础组件库
- **layout**：布局组件

**深度亮点**：

**现存问题**：  
- 业务组件沉淀不足，各页面重复逻辑未抽离
- 缺少组件文档与使用示例
- 未做大数据量场景的性能优化（如列表虚拟滚动）

**优化建议**：  
- 沉淀通用业务组件（表格、表单、筛选器），减少页面重复代码
- 补充组件使用文档
- 大数据量场景增加虚拟滚动能力

---

### 6. src/hooks + src/contexts

**定位**：前端状态管理与逻辑复用  
**完成度**：60%  
**已落地内容**：  
- 自定义 Hooks、React Context 全局状态

**深度亮点**：  
- React Hooks + Context 方案轻量，适合中后台系统，无额外状态库依赖

**现存问题**：  
- 状态粒度不清晰，缺少全局状态规范
- 复杂状态无优化，容易导致不必要的重渲染
- 服务端状态与客户端状态边界模糊

**优化建议**：  
- 引入 SWR/React Query 管理服务端状态，统一数据请求与缓存
- 规范 Context 使用范围，仅存放真正全局的状态

### 7. src/app 表现层

**定位**：Next.js App Router 页面与 API 路由  
**完成度**：页面骨架 80%；业务逻辑深度 30%  
**已落地内容**：  
- 全模块页面路由与对应 API 路由
- 采用 Thin Controller 模式，API 层负责协议转换

**深度亮点**：  
- 模块划分清晰，页面路径与业务模块一一对应
- API 路由与页面同构，开发效率高

**现存问题**：  
- 多数 API 仅实现基础 CRUD，业务逻辑深度不足
- 部分 API 路径命名不一致（如网版 API 为 /api/screen-plates，无模块前缀）
- 缺少接口版本管理与向下兼容机制

**优化建议**：  
- 统一 API 命名规范，按模块前缀划分
- 逐步将 API 层业务逻辑收敛到应用服务
- 预留接口版本号设计

## 四、业务模块层（逐模块精细化评估）

### 1. 仓储管理（warehouse）

**对应路径**：src/app/warehouse、src/domain/warehouse、src/application/services/InboundApplicationService  
**完成度**：80%（核心可用级）  
**已落地功能**：  
- 入库管理、分切入库、物料标签、出库管理、库存管理
- 生产入库、销售出库、调拨管理、库存调整、盘点管理、仓库设置

**深度亮点**：  
- 全链路 DDD 落地，是项目的标杆模块
- 针对印刷行业的分切入库场景做了专属设计
- 核心数据一致性有乐观锁保障，并发场景有专项测试

**现存问题**：  
- 出库、调拨、盘点等子模块未同步落地 DDD 分层，仅入库做了完整领域设计
- 库存成本核算逻辑缺失，无法支持移动加权平均、先进先出等成本计价方式
- 盘点流程无审批流，缺少盘盈盘亏的自动账务处理
- 库位管理粒度不足，不支持货架、库位级精细化管理

**优化建议**：  
- 补全出库、盘点、调拨的领域层设计，统一仓储领域的 DDD 规范
- 增加库存成本核算能力，支持多种计价方式
- 完善盘点全流程，增加审批与自动账务处理
- 增加库位管理能力，支持精细化仓储
