# 技术决策记录：完工入库事件驱动的幂等保护策略

> 编号：VNERP-DEC-005 | 日期：2026-07-14 | 状态：已决定（需补强）

## 背景

`warehouse/production-inbound/route.ts` 在同步路径中已执行库存更新；新增的 `FinishOrderInventoryHandler` 作为 `prod.finish.approved` 事件消费者也会更新库存。两条路径并存有双计数风险；同时 StreamConsumer 的 `XAUTOCLAIM` 回收机制可能导致事件重复投递。

## 选项

### 选项 A：应用层 SELECT 检查

- 优势：实现简单、快速跳过已处理事件、减少不必要的库存写操作
- 劣势：TOCTOU 竞态——并发投递时两个消费者可能同时通过 SELECT 检查后各自 INSERT；防线放在应用层而非数据库层；缺少事务包裹

### 选项 B：数据库层唯一约束

- 优势：幂等性下沉到数据库约束层，彻底消除竞态；`INSERT IGNORE` + `affectedRows` 原子性判定
- 劣势：需要添加唯一索引；需调整代码逻辑

## 决策

先选择 **选项 A** 作为临时方案，后续升级到 **选项 B**。

当前实现：在 Handler 入口处先 `SELECT inv_inventory_transaction WHERE source_type='prod_finish' AND source_id=?`，若已存在则跳过。

## 权衡

| 得到 | 放弃 |
|------|------|
| 应用层快速跳过已处理事件，减少不必要的库存写操作 | TOCTOU 竞态——并发投递时可能双计数 |
| 实现简单 | 防线放在应用层而非数据库层 |
| 初步解决双计数问题 | 缺少事务包裹，SELECT → UPDATE → INSERT 三步非原子 |

## 需补强项

1. 在 `inv_inventory_transaction (source_type, source_id)` 上加唯一索引
2. 将幂等判定改为 `INSERT IGNORE` + 检查 `affectedRows`
3. 将整个 Handler 逻辑包在 `transaction()` 内
4. 补充并发用例测试

## 影响

- 当前实现是"条件性可用"状态，需完成补强后才能用于生产
- 并发场景下存在双计数风险，建议尽快完成补强
