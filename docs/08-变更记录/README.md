# 变更记录目录

> 核心原则：可追溯、可回滚、透明化。

## 目录用途

本目录用于沉淀 VNERP 项目的全部变更记录，涵盖版本变更、发布记录与重要 Bug 修复。

## 文档索引

| 文档 | 用途 | 格式 |
|------|------|------|
| [CHANGELOG.md](./CHANGELOG.md) | 版本变更总记录（按版本号倒序） | Keep a Changelog |
| [发布记录-v0.1.0.md](./发布记录-v0.1.0.md) | v0.1.0 版本发布说明 | Markdown |
| [Bug修复-001-架构整合.md](./Bug修复-001-架构整合.md) | 架构冗余表合并修复记录 | Bug 修复单 |

## 变更流程

```
提出变更 → 评估影响 → 开发实现 → 测试验证 → 记录变更 → 发布
```

- **提出变更**：在 issue 或 PR 中说明变更原因与范围。
- **评估影响**：明确影响的模块、API、数据库表与下游依赖。
- **开发实现**：遵循 [03-技术规范/开发规范.md](../03-技术规范/开发规范.md) 与 [11-开发指南/](../11-开发指南/)。
- **测试验证**：单元测试 + E2E 测试 + 手工验收，参考 [05-测试文档/](../05-测试文档/)。
- **记录变更**：在 `CHANGELOG.md` 对应版本段落追加条目；重要 Bug 修复另起 `Bug修复-NNN-描述.md`；发布版本另起 `发布记录-vX.Y.Z.md`。
- **发布**：合并到 main，打 tag，触发 CI。

## 版本号规则

遵循语义化版本（SemVer）：

```
MAJOR.MINOR.PATCH

MAJOR: 不兼容的 API 变更
MINOR: 向后兼容的功能新增
PATCH: 向后兼容的问题修复
```

## CHANGELOG 条目约定

每个版本段下使用三类小节：

| 小节 | 含义 |
|------|------|
| `### 新增` | 新功能、新模块、新文档 |
| `### 修复` | Bug 修复、缺陷处理 |
| `### 变更` | 重构、配置调整、不兼容但非 Bug 的变更 |

每条记录建议包含：简短描述 + 关联 PR 或 commit hash（如 `(876f911)`）。
