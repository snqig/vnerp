# 数据库优化指南

> 基于 `src/lib/db/schema.ts` 索引定义、`src/lib/db/index.ts` 连接池配置、`database/migrations/059` 缺失索引补齐、项目硬约束（高频字段须索引）整理。

## 一、现状概览

- 数据库 `vnerpdacahng`（MySQL 8.0+），42 张 Drizzle 消费表 + 遗留表（经原始 SQL 管理）。
- Drizzle schema 中通过索引名（`fk_*` / `idx_*` / `uk_*`）声明查询索引与外键语义，生产数据库不强制外键约束。
- 迁移 057 统一主键为 `BIGINT UNSIGNED`，058 统一排序规则为 `utf8mb4_0900_ai_ci`，059 补齐缺失索引，060 统一状态码。
- 连接池单例（`connectionLimit: 20`），所有查询参数化预编译，防 SQL 注入。

## 二、索引策略

### 2.1 索引命名规范

| 前缀 | 含义 | 示例 |
|------|------|------|
| `uk_` | 唯一索引（业务编码或复合唯一） | `uk_order_no`、`uk_material_warehouse` |
| `fk_` | 外键语义索引（声明引用关系） | `fk_sal_order_customer`、`fk_sal_order_detail_order` |
| `idx_` | 普通查询索引 | `idx_status`、`idx_warehouse`、`idx_material` |

### 2.2 索引设计原则

1. **高频查询字段必须有索引**：外键字段（`*_id`）、状态字段（`status`）、日期字段（`*_date`）、业务编码字段（`*_no`，用唯一索引）。
2. **复合索引顺序**：等值查询字段在前，范围查询字段在后。如 `inv_inventory` 的 `uk_material_warehouse`（`material_id` + `warehouse_id`）同时满足按物料查库存与唯一约束。
3. **覆盖索引**：高频列表查询考虑包含返回字段，减少回表。如 `sal_reconciliation` 的 `idx_period`（`period_start` + `period_end`）覆盖对账周期查询。
4. **软删除过滤**：状态索引建议包含 `deleted` 字段。如 `inv_warehouse` 的 `idx_warehouse_status`（`status` + `deleted`）。

### 2.4 迁移 059 补齐的缺失索引

迁移 059 为高基数列补齐索引，主要包括：

- 仓库：`inv_inbound_order`（`supplier_id`、`po_id`）
- 销售：`sal_order`（`status`、`order_date`）、`sal_delivery`（`status`、`delivery_date`）、`sal_return_order`（`delivery_id`、`warehouse_id`）、`sal_quote`（`status`、`quote_date`）
- 采购：`pur_purchase_order`（`create_by`）
- 其他模块的外键、状态、日期字段

> 新增表或字段时，务必同步在 `schema.ts` 与迁移脚本中添加对应索引，避免遗漏导致慢查询。

## 三、查询优化

### 3.1 分页查询

使用 `src/lib/db/index.ts` 的 `queryPaginated<T>()` 辅助函数：

```ts
// 旧格式：自动生成 COUNT
const result = await queryPaginated<InvInventory>(
  'SELECT * FROM inv_inventory WHERE warehouse_id = ? AND deleted = 0',
  [warehouseId], 1, 20
);

// 新格式：手动提供 COUNT SQL（推荐，避免子查询包装）
const result = await queryPaginated<InvInventory>(
  'SELECT * FROM inv_inventory WHERE warehouse_id = ? AND deleted = 0',
  'SELECT COUNT(*) as total FROM inv_inventory WHERE warehouse_id = ? AND deleted = 0',
  [warehouseId], { page: 1, pageSize: 20 }
);
```

优化要点：
- 优先使用新格式提供独立 COUNT SQL，避免旧格式 `SELECT COUNT(*) FROM (subquery)` 的派生表开销。
- 深分页（offset 过大）时考虑改用游标分页（`WHERE id > ? ORDER BY id LIMIT ?`）。

### 3.2 避免 N+1 查询

- 列表查询关联明细时，先批量查询主表 ID，再用 `IN (...)` 一次性查询明细，避免循环内单条查询。
- 使用 Drizzle 查询构建器的 `with` 子查询或 JOIN 替代循环查询。

### 3.3 JOIN 优化

- JOIN 字段必须有索引（外键语义索引 `fk_*` 已覆盖大部分场景）。
- 避免 JOIN 后再过滤大表，优先在子查询中缩小数据集。
- 主从表查询（如订单 + 明细）使用 `order_id` 索引，已在所有明细表建索引。

### 3.4 参数化查询（SQL 注入防护）

所有查询必须使用 `?` 占位符参数化预编译语句，禁止字符串拼接用户输入。`src/lib/db/index.ts` 的 `query` / `execute` / `queryOne` 均基于 mysql2 参数化实现。`src/lib/fifo-allocation` 等历史模块已修复 SQL 注入缺陷。

## 四、乐观锁（version 字段）

### 4.1 适用表

以下表包含 `version` 字段用于乐观锁并发控制：

| 表 | 字段 | 默认值 |
|----|------|--------|
| `inv_inventory` | `version` | `1` |
| `inv_outbound_order` | `version` | `0` |
| `inv_transfer_order` | `version` | `0` |
| `sal_delivery` | `version` | `0` |

### 4.2 使用模式

```sql
UPDATE inv_inventory
SET quantity = quantity + ?, version = version + 1, update_time = NOW()
WHERE id = ? AND version = ? AND deleted = 0
```

- 更新时携带 `WHERE version = ?`，若 `affectedRows = 0` 表示并发冲突。
- 使用 `transactionWithRetry()` 自动重试（指数退避，默认 3 次，检测「已被其他操作修改」「affectedRows」「version」关键字）。
- `src/lib/inventory-sync` 模块已实现乐观锁库存同步。

### 4.3 注意事项

- 新增高并发表时务必添加 `version` 字段。
- 乐观锁冲突日志应包含 `version` 与 `id` 便于排查。

## 五、软删除（deleted 字段）

### 5.1 约定

- 所有业务表包含 `deleted` 字段（`boolean default false` 或 `tinyint default 0`）。
- 查询时必须携带 `WHERE deleted = 0`（或 `deleted = false`）。
- 不物理删除业务数据，仅标记 `deleted = true`/`1`。

### 5.2 兼容性

部分表（`prod_work_order`、`prd_schedule`、`prd_schedule_detail`、`prd_pick_order`、`prd_return_order`、`prd_work_report`、`prd_finish_order`、`dcprint_ink_color`、`dcprint_ink_formula_version`、`dcprint_tool` 等）的 `deleted` 字段为 `tinyint default 0`，应用层需兼容 `0`/`1` 与 `false`/`true` 两种形态。

`src/lib/soft-delete` 模块提供统一的软删除查询辅助。

## 六、连接池配置

`src/lib/db/index.ts` 的连接池配置：

| 参数 | 值 | 说明 |
|------|-----|------|
| `connectionLimit` | 20 | 最大连接数 |
| `maxIdle` | 10 | 最大空闲连接数 |
| `idleTimeout` | 30000 | 空闲连接超时（ms） |
| `queueLimit` | 100 | 等待队列上限 |
| `connectTimeout` | 8000 | 连接超时（ms） |
| `enableKeepAlive` | true | 启用 TCP 保活 |
| `keepAliveInitialDelay` | 10000 | 保活初始延迟（ms） |
| `charset` | `utf8mb4` | 字符集 |

- 使用全局单例模式（`globalForDb`）防止 Next.js 热重载时重复创建连接池。
- `query()` 内置重试：遇到 `ER_CON_COUNT_ERROR` 或 `PROTOCOL_CONNECTION_LOST` 时等待 1s 后重试，最多 2 次。

### 调优建议

- 生产环境根据并发量调整 `connectionLimit`（MySQL `max_connections` 的 10%-20%）。
- 长事务避免占用连接过久，及时释放。
- 监控连接池等待队列长度，超阈值告警。

## 七、慢查询监控

### 7.1 开启慢查询日志

```sql
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;  -- 超过 1 秒记录
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';
```

### 7.2 EXPLAIN 分析

对慢查询执行 `EXPLAIN`，关注：
- `type` 字段：`ALL`（全表扫描）需优化为 `ref` / `range` / `eq_ref`。
- `key` 字段：确认是否命中预期索引。
- `rows` 字段：扫描行数过大时考虑加索引或改写查询。
- `Extra` 字段：`Using filesort` / `Using temporary` 需优化排序与分组。

### 7.3 调试日志

`DEBUG_DB=true` 或 `NODE_ENV=development` 时，`query()` / `execute()` 会输出 SQL 调试日志。

### 7.4 死锁监控

`/api/system/monitor/deadlock` 端点提供死锁监控。`transactionWithRetry()` 在乐观锁冲突时自动重试，降低死锁影响。

## 八、分区建议

### 8.1 适用场景

以下表数据量增长较快，可考虑按时间分区：

| 表 | 分区键 | 分区策略 |
|----|--------|----------|
| `inv_inbound_order` | `inbound_date` | 按月 RANGE 分区 |
| `inv_outbound_order` | `order_date` | 按月 RANGE 分区 |
| `sal_order` | `order_date` | 按月 RANGE 分区 |
| `pur_purchase_order` | `order_date` | 按月 RANGE 分区 |
| `prd_work_report` | `report_date` | 按月 RANGE 分区 |
| `sys_oper_log` | `create_time` | 按月 RANGE 分区（遗留表） |
| `sys_login_log` | `login_time` | 按月 RANGE 分区（遗留表） |
| `domain_event_outbox` | `created_at` | 按月 RANGE 分区 + 30 天后清理 |

### 8.2 注意事项

- 分区键必须是主键的一部分（或表无主键），需先调整主键设计。
- 跨分区查询性能可能下降，查询条件尽量包含分区键。
- 分区维护（DROP PARTITION）比 DELETE 更高效，适合日志类表。
- 当前项目尚未实施分区，建议在单表数据量超过 500 万行时评估。

## 九、表注释与字段注释

补充表与字段注释提升可维护性：

```sql
ALTER TABLE inv_inbound_order COMMENT '入库单主表（采购/生产/退货入库）';
ALTER TABLE inv_inventory COMMENT '库存表（含可用量、锁定量、安全库存、乐观锁）';
ALTER TABLE prd_work_report COMMENT '工序报工单（含刀模/网版工装关联）';
```

字段注释通过 `ALTER TABLE xxx MODIFY column_name ... COMMENT '字段含义';`。Drizzle schema 中已通过内联注释标注关键字段含义（如 `dcprint_tool.tool_type`：`// 1=刀模 2=网版`）。

## 十、执行步骤

1. **备份**：`pnpm backup`（`npx tsx scripts/backup-database.ts`）。
2. **检查索引覆盖**：对照本文档索引清单，确认 `schema.ts` 与数据库一致。
3. **分步执行优化**：先补注释（无风险）→ 补索引（低峰期 `ALTER TABLE ... ADD INDEX`）→ 分区（测试环境验证）。
4. **验证**：查询 `information_schema.STATISTICS` 确认索引，`EXPLAIN` 确认查询计划。
