# 数据库优化指南

> 四个维度的优化建议：外键约束、命名规范、索引、表注释。详细 SQL 见 `scripts/database-optimization.sql` 与 `scripts/check-data-integrity.sql`。

## 一、现状概览

- 数据库 `vnerpdacahng`，约 124 张表，9 大模块（system/order/product/partner/inventory/production/finance/sample/other）。
- Drizzle schema 中通过 `.references()` 声明类型化关系，但生产数据库未强制外键约束。
- 部分表缺少索引与注释，影响 JOIN 性能与新成员上手。

## 二、外键约束建议

### 2.1 优先级模块

| 模块 | 建议外键数 | 关键关系 |
|------|------------|----------|
| 系统管理 | 10 | `sys_user → sys_department`、`sys_user_role → sys_user/sys_role`、`sys_role_menu → sys_role/sys_menu`、`sys_dict_data → sys_dict_type`、`sys_employee → sys_role`、`sys_login_log/sys_operation_log → sys_user`、`sys_salary → sys_employee` |
| 人事 | 3 | `hr_attendance → sys_employee`、`hr_training_participant → sys_employee/hr_training` |
| 客户管理 | 3 | `crm_customer_contact/analysis` 与 `crm_follow_record → crm_customer` |
| 库存 | 6 | `inv_inventory/inv_inventory_batch → inv_material/inv_warehouse`、`inv_inbound_item → inv_inbound_order`、`inv_outbound_item → inv_outbound_order`、`inv_material_category → self` |
| 委外 | 4 | `outsource_issue/receive/settlement → outsource_order`、`outsource_issue_item → outsource_issue` |
| 油墨 | 3 | `ink_mixed_record → base_ink/sys_company`、`ink_mixed_batch_detail → ink_mixed_batch` |
| 采购 | 3 | `pur_request_detail/pur_request_item → pur_request`、`pur_purchase_order → pur_supplier` |
| BOM | 1 | `bom_line → bom_header` |

### 2.2 级联策略

| 策略 | 适用场景 |
|------|----------|
| `ON DELETE CASCADE` | 强关联数据（明细单、关联表） |
| `ON DELETE SET NULL` | 可选关联（部门、角色） |
| `ON DELETE RESTRICT` | 保护性关联（库存对物料/仓库） |
| `ON UPDATE CASCADE` | 所有场景 |

### 2.3 执行前检查

```bash
mysql -u root -p vnerpdacahng < scripts/check-data-integrity.sql
```

发现孤儿数据时先清理（`UPDATE ... SET xxx_id = NULL WHERE NOT EXISTS ...` 或 `DELETE FROM ...`），再添加外键。

## 三、命名规范

### 3.1 待修正字段

| 表 | 字段 | 问题 | 建议 |
|----|------|------|------|
| `pur_purchase_order_line` | `po_id` | 缩写不清晰 | `purchase_order_id` |
| `bom_line` | `header_id` | 未指明哪个头 | `bom_header_id` |
| `inv_inbound_item` | `order_id` | 未指明哪个订单 | `inbound_order_id` |
| `inv_outbound_item` | `order_id` | 未指明哪个订单 | `outbound_order_id` |

### 3.2 命名规则

```
{被关联表名去掉前缀}_id
```

例：`sys_department → department_id`；`pur_purchase_order → purchase_order_id`（不是 `po_id`）。

### 3.3 修改影响

需同步：数据库字段、所有 SQL 查询、应用代码字段引用、API 接口、前端字段。建议新增表严格执行，存量表在重构时统一修改。

## 四、索引建议

### 4.1 索引命名规范

```
idx_{表名}_{字段名}
```

例：`idx_sys_user_department`、`idx_inv_inbound_item_order`。

### 4.2 关键索引清单

| 模块 | 索引示例 |
|------|----------|
| 系统 | `idx_sys_user_department`、`idx_sys_user_role_user/role`、`idx_sys_role_menu_role/menu`、`idx_sys_dict_data_type`、`idx_sys_login_log_user`、`idx_sys_operation_log_user` |
| 客户 | `idx_crm_customer_contact_customer`、`idx_crm_follow_record_customer` |
| 库存 | `idx_inv_inventory_material/warehouse/location`、`idx_inv_inventory_batch_material`、`idx_inv_inbound_item_order/material`、`idx_inv_outbound_item_order/material` |
| 生产 | `idx_prd_work_order_material/product/customer`、`idx_prd_bom_detail_bom/material`、`idx_prd_work_report_work_order` |
| 设备 | `idx_eqp_maintenance_plan/record_equipment`、`idx_eqp_calibration/repair/scrap_equipment` |
| 质量 | `idx_qc_inspection_work_order/material`、`idx_qc_unqualified_inspection` |

| BOM | `idx_bom_line_header/material` |

## 五、表注释建议

补充 `ALTER TABLE xxx COMMENT '业务含义表';`，例：

```sql
ALTER TABLE sys_user COMMENT '用户表';
ALTER TABLE inv_inbound_order COMMENT '入库订单主表';
ALTER TABLE prd_work_order COMMENT '生产工单表';
```

字段注释通过 `ALTER TABLE xxx MODIFY column_name ... COMMENT '字段含义';`。

## 六、执行步骤

1. **备份**：`mysqldump -u root -p vnerpdacahng > backup_$(date +%Y%m%d).sql`
2. **检查数据完整性**：执行 `scripts/check-data-integrity.sql`，清理孤儿数据。
3. **分步执行**：先注释（无风险）→ 索引（低峰期）→ 外键（测试环境验证后）→ 命名（重构时）。
4. **验证**：查询 `information_schema.TABLE_CONSTRAINTS` / `STATISTICS` / `TABLES` 确认结果。

## 七、风险评估

| 操作 | 风险 | 建议 |
|------|------|------|
| 补充表注释 | 低 | 立即执行 |
| 添加索引 | 低 | 低峰期执行 |
| 增加外键约束 | 中 | 测试环境验证 |
| 修改字段命名 | 高 | 重构时统一修改 |
