# 数据库架构

> 基于 `src/lib/db/schema.ts`（42 张 Drizzle 消费表）、`src/lib/db/index.ts`（连接池）、`scripts/migrate.ts`（迁移机制）与 `database/migrations/` 现状整理。
> 权威 SQL 来源：`database/vnerpdacahng_schema.sql`。

## 一、技术栈

| 维度 | 选型 |
|------|------|
| 数据库 | MySQL 8.0+（`utf8mb4`，统一排序规则 `utf8mb4_0900_ai_ci`） |
| ORM | Drizzle ORM 0.45.1（`drizzle-orm/mysql2`） |
| 驱动 | mysql2 3.20.0（Promise API + 连接池） |
| 迁移工具 | 自研 `scripts/migrate.ts`（按 `;` 拆分 + `sys_migration` 记录表）；drizzle-kit 迁移路径已废弃 |
| 默认数据库 | `vnerpdacahng` |
| 配置入口 | `src/lib/db/index.ts`、环境变量 `DB_HOST` / `DB_PORT` / `DB_USER` / `DB_PASSWORD` / `DB_NAME` |

> 注意：`drizzle-kit`（0.31.10）仍作为 devDep 保留用于 Drizzle Studio（`pnpm db:studio`），但其迁移生成路径（`pnpm db:generate` / `db:push` / `db:migrate`）已废弃并打印警告，`drizzle/` 目录已清理。增量迁移统一走 `scripts/migrate.ts`。

## 二、分层概览

数据库表分为两类管理路径：

### 1. Drizzle 消费表（42 张）

定义在 `src/lib/db/schema.ts`，被 Drizzle ORM 查询构建器（`drizzleDb`）实际消费，提供 TypeScript 类型推导。新增 ORM 消费表时，从 `database/vnerpdacahng_schema.sql` 对应 DDL 翻译并在此追加。

> `schema.ts` 文件头注释写「35 张核心业务表」已过时，实际为 42 张。

### 2. 遗留表（经原始 SQL 管理）

系统、人事、品质、设备、基础主数据等模块的表未纳入 Drizzle schema，通过原始 SQL（`database/vnerpdacahng_schema.sql` + `database/migrations/`）管理，应用层使用 `src/lib/db/index.ts` 的 `query` / `execute` 参数化查询访问。包括但不限于：`sys_user`、`sys_role`、`sys_permission`、`sys_role_permission`、`sys_menu`、`sys_config`、`sys_dict_type`、`sys_dict_data`、`sys_oper_log`、`sys_login_log`、`sys_notice`、`sys_announcement`、`sys_event_processed`、`domain_event_outbox`、`hr_employee`、`hr_department`、`hr_attendance`、`hr_salary`、`hr_training`、`qc_inspection`（来料/过程/成品）、`qc_unqualified_product`、`qc_sgs`、`qc_complaint`、`qc_supplier_audit`、`qrcode_record`、`material`、`material_category`、`supplier`、`customer`、`bom`、`bom_item`、`equipment`、`equipment_maintenance`、`equipment_calibration`、`equipment_scrap`、`sop`、`vehicle` 等。具体以 `database/vnerpdacahng_schema.sql` 为准。

## 四、公共字段约定

Drizzle 消费表遵循以下公共字段约定（部分表因历史原因可能缺少个别字段）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `serial` 或 `bigint unsigned autoincrement` | 主键，自增 |
| `create_by` | `bigint unsigned` | 创建人 ID（关联 `sys_user.id`） |
| `update_by` | `bigint unsigned` | 更新人 ID |
| `create_time` | `datetime` | 创建时间，默认 `CURRENT_TIMESTAMP` |
| `update_time` | `datetime` | 更新时间，默认 `CURRENT_TIMESTAMP` |
| `deleted` | `boolean` 或 `tinyint` | 软删除标记（`true`/`1` = 已删除），不物理删除 |
| `version` | `int` | 乐观锁版本号（仅高并发表如 `inv_inventory`、`inv_outbound_order`、`inv_transfer_order`、`sal_delivery`） |

> 注意：部分表（如 `prod_work_order`、`prd_schedule`、`dcprint_tool`）的 `deleted` 字段类型为 `tinyint default 0` 而非 `boolean`，应用层需兼容两种形态。

## 五、字符集与排序规则

- 数据库与表统一使用 `utf8mb4` 字符集。
- 排序规则统一为 `utf8mb4_0900_ai_ci`（MySQL 8.0 默认）。迁移 058 将历史使用 `utf8mb4_unicode_ci` 的列统一转换。
- `sys_migration` 记录表建表语句显式指定 `COLLATE=utf8mb4_0900_ai_ci`。

## 六、设计原则

1. 表命名使用模块前缀（`inv_`、`sal_`、`pur_`、`fin_`、`prd_`/`prod_`、`dcprint_` 等），遗留表使用 `sys_`、`hr_`、`qc_` 等前缀。
2. 字段命名使用 `snake_case`，TypeScript 导出使用 `camelCase`。
3. 状态字段：部分表使用字符串编码（如 `'pending'`、`'completed'`），部分使用 `tinyint` 数值编码（如 `1`=待处理、`2`=已完成）。迁移 060 正在统一为 `tinyint`。
4. 软删除优先（`deleted` 字段），不物理删除。
5. 不强制外键约束（应用层保证一致性），但 Drizzle schema 通过索引名（`fk_*`）声明类型化关系语义，便于类型推导与关系推断。
6. 不使用存储过程和触发器。

## 七、连接与查询

`src/lib/db/index.ts` 提供：

- `getPool()`：全局单例连接池（`connectionLimit: 20`、`maxIdle: 10`、`idleTimeout: 30000`、`queueLimit: 100`、`connectTimeout: 8000`、`enableKeepAlive: true`）。使用全局单例模式防止 Next.js 热重载时重复创建。
- `query<T>()`：SELECT 查询，参数化预编译语句，内置重试（连接丢失/过多时最多 2 次）。
- `execute()`：INSERT/UPDATE/DELETE 写入，参数化预编译语句。
- `queryOne<T>()`：查询单条记录，无结果返回 `null`。
- `transaction()`：手动事务（beginTransaction/commit/rollback）。
- `transactionWithRetry()`：乐观锁冲突自动重试（指数退避，默认 3 次，检测「已被其他操作修改」「affectedRows」「version」关键字）。
- `queryPaginated<T>()`：分页查询辅助，支持旧格式（自动生成 COUNT）与新格式（手动提供 COUNT SQL）。
- `drizzleDb`：Drizzle ORM 实例，加载完整 `schema` 提供类型安全查询构建器。新代码优先使用，旧代码继续用 `query`/`execute`。

配置通过 `DB_HOST` / `DB_PORT` / `DB_USER` / `DB_PASSWORD` / `DB_NAME` 环境变量注入；`DEBUG_DB=true` 或 `NODE_ENV=development` 时输出 SQL 调试日志。

> 注意：`drizzle-orm` 0.45.x 仅识别 `client` 或 `connection` 参数，传入 `pool` 会被忽略。当前实现传 `client: getPool()`。

## 八、迁移机制

### 迁移工具

增量迁移由 `scripts/migrate.ts`（`pnpm migrate`）管理：

- 子命令：`up`（执行未执行迁移）、`down`（回滚）、`status`（查看状态）、`history`（历史）、`create <name>`（创建模板）、`reset`（回滚全部 + 重新执行）。
- 迁移文件位于 `database/migrations/`，命名 `NNN_<description>.sql`（编号 001–060）。
- 按 `;` 拆分逐条执行，记录到 `sys_migration` 表（含 `batch`、`applied_at`、`execution_time`）。
- 单条失败仍记录整体为 applied，故迁移须幂等（推荐用 `INFORMATION_SCHEMA` 守卫 + `CREATE TABLE IF NOT EXISTS` + `ADD INDEX IF NOT EXISTS`）。
- `multipleStatements: true` 连接模式执行。

全量初始化由 `scripts/setup-db.mjs`（`pnpm setup:db`）管理：建库 + 导入 `vnerpdacahng_schema.sql` + 可选 seed。

### 迁移编号 053–060

| 编号 | 文件 | 用途 |
|------|------|------|
| 053 | `053_add_tool_columns_to_work_report.sql` | 报工表新增工装关联列（`tool_id`、`screen_plate_id`），打通报工→工装寿命累计事件联动 |
| 053 | `053_enhance_sample_order.sql` | 打样订单模块增强（工艺卡/工单/销售订单关联、打样费用、版本控制、反馈、样品库存） |
| 054 | `054_add_work_order_cumulative_fields.sql` | `prod_work_order` 新增累计统计字段（`picked_qty`/`finished_qty`/`returned_qty`/`total_material_cost`） |
| 054 | `054_enhance_production_tables.sql` | 生产管理模块增强：工单状态/成本字段、领料/退料/报工/完工入库表 |
| 055 | `055_add_prd_schedule_indexes.sql` | `prd_schedule` 与 `prd_schedule_detail` 补充索引（INFORMATION_SCHEMA 守卫） |
| 056 | `056_expand_tool_unified_fields.sql` | `dcprint_tool` 扩展统一刀模/网版/工装三套体系字段（体系B 刀模 + 体系C 网版） |
| 057 | `057_unify_pk_to_bigint.sql` | 统一主键类型为 `BIGINT UNSIGNED`（25 张使用 INT PK 的表，`SET FOREIGN_KEY_CHECKS = 0`） |
| 058 | `058_unify_collation.sql` | 统一排序规则为 `utf8mb4_0900_ai_ci`（转换 `utf8mb4_unicode_ci` 列） |
| 059 | `059_add_missing_indexes.sql` | 补齐高基数列索引（外键、状态、日期等高查询频率列） |
| 060 | `060_unify_status_codes.sql` | 状态码统一（`varchar` → `tinyint`，含映射表：`inv_inbound_order`/`inv_outbound_order`/`prod_work_order`） |

## 九、命令速查

```bash
pnpm setup:db        # 全量初始化（node scripts/setup-db.mjs，建库 + 导入 schema + 可选 seed）
pnpm migrate         # 执行增量迁移（npx tsx scripts/migrate.ts up）
pnpm migrate:status  # 查看迁移状态
pnpm migrate:down    # 回滚最后一个迁移
pnpm migrate:create  # 创建新迁移文件模板
pnpm db:studio       # Drizzle Studio 可视化（drizzle-kit 仍可用于 Studio）
pnpm backup          # 数据库备份（npx tsx scripts/backup-database.ts）
```

> 已废弃命令（打印警告，勿用）：`pnpm db:generate` / `db:push` / `db:migrate`。
