# i18n 规范

本文档说明 VNERP 国际化（i18n）开发规范，对应 `src/i18n/`、`messages/`、`eslint.config.mjs` 与 `eslint-rules/no-chinese-hardcode.js`。

## 1. 支持语言

项目支持 4 种语言，定义于 `src/i18n/locales.ts`：

| locale | 名称 | 说明 |
|--------|------|------|
| `zh-CN` | 简体中文 | 默认语言 |
| `zh-TW` | 繁體中文 | 港台用户 |
| `en` | English | 国际用户 |
| `vi` | Tiếng Việt | 越南用户 |

```ts
export const locales = ['zh-CN', 'zh-TW', 'en', 'vi'] as const;
export const defaultLocale = 'zh-CN' as const;
```

`localePrefix: 'as-needed'`（`src/i18n/navigation.ts`）表示默认语言不带前缀，其他语言路径为 `/zh-TW/...`、`/en/...`、`/vi/...`。

## 2. 语言包文件

语言包位于 `messages/` 目录：

```
messages/
├── zh-CN.json    # 简体中文（基准）
├── zh-TW.json
├── en.json
└── vi.json
```

每个文件按命名空间组织，当前包含 23 个命名空间（基于 `zh-CN.json`）：

`Common`、`Nav`、`Auth`、`Dashboard`、`Orders`、`Warehouse`、`Production`、`Dcprint`、`QRCode`、`Purchase`、`Sales`、`Finance`、`Quality`、`Equipment`、`Outsource`、`Srm`、`Crm`、`Hr`、`System`、`Engineering`、`Delivery`、`Business`、`Theme`。

新增模块时，应新增对应命名空间，并在 4 个文件中同步添加。

## 3. 配置入口

| 文件 | 作用 |
|------|------|
| `src/i18n/locales.ts` | 语言列表与默认语言 |
| `src/i18n/request.ts` | 服务端 `getRequestConfig`，按请求 locale 加载对应语言包，加载失败回退默认语言 |
| `src/i18n/navigation.ts` | `createNavigation` 创建 `Link` / `useRouter` / `usePathname` / `redirect` |
| `src/components/IntlProvider.tsx` | 客户端 `NextIntlClientProvider` 包装，处理 `MISSING_MESSAGE` 警告 |
| `src/middleware.ts` | `next-intl/middleware` 处理 locale 路由前缀 |

`next.config.ts` 通过 `createNextIntlPlugin('./src/i18n/request.ts')` 集成 next-intl。

## 4. 客户端使用

### 4.1 useTranslations

```tsx
'use client';
import { useTranslations } from 'next-intl';

export function MyComponent() {
  const t = useTranslations('Warehouse');  // 指定命名空间
  return (
    <div>
      <h1>{t('title')}</h1>
      <button>{t('inbound')}</button>
      <span>{t('total', { count: 100 })}</span>
    </div>
  );
}
```

### 4.2 多命名空间

```tsx
const tCommon = useTranslations('Common');
const tWarehouse = useTranslations('Warehouse');
```

### 4.3 带参数文案

语言包中：

```json
"Common": {
  "total": "共 {count} 条",
  "confirmDelete": "确定要删除 {name} 吗？"
}
```

调用：

```tsx
t('total', { count: 100 })
t('confirmDelete', { name: '物料A' })
```

### 4.4 导航

```tsx
import { Link, useRouter, usePathname } from '@/i18n/navigation';

const router = useRouter();
router.push('/warehouse/inbound');
```

不要使用 `next/link` 与 `next/router`，应使用 `@/i18n/navigation` 导出的版本，自动处理 locale 前缀。

## 5. 服务端使用

### 5.1 Server Components

```tsx
import { getTranslations } from 'next-intl/server';

export default async function Page() {
  const t = await getTranslations('Warehouse');
  return <h1>{t('title')}</h1>;
}
```

### 5.2 API 路由

API 路由（`src/app/api/`）通常返回业务错误码而非本地化文案，前端按错误码翻译。如需在服务端返回本地化消息，使用 `src/lib/i18n-server.ts`。

## 6. 硬编码检测

### 6.1 ESLint 自定义规则

`eslint-rules/no-chinese-hardcode.js` 检测以下场景：

- JSX 文本中的中文（如 `<div>中文</div>`）
- 字符串字面量中的中文（如 `const msg = '中文'`）
- 模板字符串中的中文（如 `` `中文${var}` ``）
- 函数调用参数中的中文（如 `errorResponse('错误')`）

`eslint.config.mjs` 中配置：

```js
'i18n/no-chinese-hardcode': ['warn', {
  allowInFunctions: ['tc', 't', 'useTranslations', 'formatMessage'],
  allowInFiles: ['i18n/', 'locales/', 'messages/'],
  ignoreComments: true,
  generateTranslationKey: true,
  ignorePatterns: ['node_modules/', '.next/', 'dist/', 'build/'],
}]
```

规则级别为 `warn`，建议提交前修复所有告警。

### 6.2 允许的例外

- `t`、`tc`、`useTranslations`、`formatMessage` 调用中的中文参数。
- `messages/`、`src/i18n/`、`locales/` 路径下文件。
- 注释中的中文（`ignoreComments: true`）。
- 测试文件（`*.test.ts` / `*.spec.ts` / `__tests__/`）。

### 6.3 检测命令

```bash
pnpm lint
# 或仅检查单个文件
pnpm eslint src/app/[locale]/warehouse/page.tsx
```

## 7. 标准开发流程

1. **新增文案**：先在 `messages/zh-CN.json` 添加 key（基准语言）。
2. **同步翻译**：在 `zh-TW.json`、`en.json`、`vi.json` 中添加对应翻译。
3. **使用 key**：在组件中通过 `useTranslations('Namespace').method('key')` 引用。
4. **本地校验**：`pnpm lint` 确认无硬编码告警。
5. **运行验证**：切换语言（URL 前缀或语言切换器）确认显示正确。

## 8. 翻译辅助脚本

`scripts/` 下提供若干 i18n 辅助脚本：

| 脚本 | 作用 |
|------|------|
| `analyze-i18n.js` / `analyze-i18n-complete.js` | 分析语言包完整度，找出缺失的 key |
| `sync-translations.js` / `sync-nav-zhTW-vi.js` | 同步翻译到其他语言文件 |
| `batch-i18n-migration.js` | 批量替换硬编码中文为翻译函数 |
| `i18n-auto-migrate.js` | 自动迁移 i18n |
| `check-hr-keys.js` | 检查 HR 模块 key 完整性 |

使用方式：

```bash
node scripts/analyze-i18n-complete.js
node scripts/sync-translations.js
```

## 9. 常见问题

### 9.1 翻译键缺失

`IntlProvider` 中 `onError` 捕获 `MISSING_MESSAGE` 并打印 warning，不会崩溃。开发期应关注控制台警告，及时补齐。

### 9.2 语言包加载失败

`src/i18n/request.ts` 在加载失败时回退到 `defaultLocale`，保证服务可用。

### 9.3 服务端与客户端 locale 不一致

可能导致 Hydration 错误。确保 `middleware.ts` 正确解析 locale，`IntlProvider` 接收的 `locale` 与服务端一致。

### 9.4 新增命名空间

如果新增的命名空间未在 4 个语言文件中同步，`useTranslations('NewNs')` 会返回空字符串。必须同步添加。
