# 开发指南

本文档面向 印刷生产经营信息管理系统 Print MIS 项目开发者，说明开发流程、常用命令与目录结构。对应 `CONTRIBUTING.md` 与 `package.json` 脚本。

## 1. 环境准备

详见 `04-运维指南/本地开发.md`。简要命令：

```bash
pnpm install
cp .env.example .env          # 填入本地数据库配置
pnpm setup:db --seed          # 初始化数据库
pnpm dev                      # 启动开发服务器（端口 5000）
```

环境基线：Node.js 18+、pnpm 9.0+、MySQL 8.0+、Redis 7.0（可选）。`package.json` 中 `preinstall` 钩子强制使用 pnpm。

## 2. 目录结构

```
erp-project/
├── src/
│   ├── app/                  # Next.js App Router（含 [locale] i18n 路由 + api/）
│   │   ├── [locale]/         # 多语言页面（qrcode / finance / dcprint / modules 等）
│   │   └── api/              # API 路由（auth / warehouse / orders / qrcode 等）
│   ├── application/           # 应用初始化（AppInitializer / EventRegistry）
│   ├── components/            # UI 组件（ui/ shadcn 组件库、layout/、qr-code/）
│   ├── contexts/             # React Context（AuthContext）
│   ├── domain/               # 领域层（shared/DomainTypes）
│   ├── hooks/                # 自定义 Hooks（useAuth / usePermission 等）
│   ├── i18n/                 # i18n 配置（locales / navigation / request）
│   ├── infrastructure/       # 基础设施层（event-bus / OutboxPoller）
│   ├── lib/                  # 工具库（db / api-response / logger / qrcode-service 等）
│   ├── test/                 # 测试 fixtures 与 setup
│   └── middleware.ts         # i18n + 调试路由屏蔽中间件
├── database/                 # SQL schema 与种子数据
├── drizzle/                  # Drizzle 迁移文件
├── messages/                 # i18n 语言包（zh-CN / zh-TW / en / vi）
├── scripts/                  # 运维 / 数据脚本
├── tests/                    # Playwright E2E 测试
├── Dockerfile / docker-compose*.yml
├── drizzle.config.ts
├── next.config.ts
├── eslint.config.mjs
└── package.json
```

## 3. 常用命令

| 命令 | 作用 |
|------|------|
| `pnpm dev` | 启动开发服务器（`next dev -p 5000`） |
| `pnpm build` | 生产构建（standalone 模式） |
| `pnpm start` | 启动生产服务器 |
| `pnpm lint` / `pnpm lint:fix` | ESLint 检查 / 自动修复 |
| `pnpm format` / `pnpm format:check` | Prettier 格式化 / 检查 |
| `pnpm ts-check` | TypeScript 类型检查 |
| `pnpm test` | 运行 Playwright E2E |
| `pnpm test:ui` / `pnpm test:headed` | Playwright UI / 有头模式 |
| `pnpm test:unit` / `pnpm test:unit:run` | Vitest 单元测试（watch / 单次） |
| `pnpm test:coverage` | 单元测试覆盖率 |
| `pnpm db:generate` / `pnpm db:migrate` / `pnpm db:push` / `pnpm db:studio` | Drizzle 迁移 |
| `pnpm setup:db [--seed]` | 数据库初始化 |

## 4. 提交规范

遵循 [Conventional Commits](https://www.conventionalcommits.org/)：

```
<type>[scope]: <description>
```

| type | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档 |
| `refactor` | 重构 |
| `test` | 测试 |
| `build` | 构建 / 依赖 |
| `ci` | CI 配置 |
| `chore` | 维护 |

示例：

```
feat(warehouse): 添加入库单审核功能
fix(auth): 修复 401 刷新并发问题
docs(readme): 更新快速开始步骤
```

## 5. 提交门禁

项目配置了 husky hooks（`.husky/` 目录）：

- **pre-commit**：`lint-staged` 对暂存文件运行 `eslint --fix` + `prettier --write`。
- **commit-msg**：`commitlint` 校验提交信息格式（`commitlint.config.mjs`）。

`package.json` 中 `lint-staged` 配置：

```json
"lint-staged": {
  "src/**/*.{ts,tsx,js,jsx}": ["eslint --fix", "prettier --write"]
}
```

## 6. 分支策略

- `main`：生产分支，仅接受 PR 合入。
- `develop`：开发集成分支。
- 功能分支：`feat/<scope>-<description>`。
- 修复分支：`fix/<scope>-<description>`。

## 7. 质量检查

提交前请确保：

- `pnpm lint` 通过。
- `pnpm ts-check` 通过。
- `pnpm test:unit` 通过。
- 新增功能有对应测试（单元测试放 `src/lib/*.test.ts`，E2E 放 `tests/*.spec.ts`）。

## 8. 测试约定

- **单元测试**：Vitest，与源码同目录 `*.test.ts`（如 `src/lib/utils.test.ts`）或 `tests/unit/`。
- **E2E 测试**：Playwright，放 `tests/`，配置见 `playwright.config.ts`。
- **测试 fixtures**：`src/test/fixtures/`，提供测试用户与配置。

## 9. 关键约定

- 数据库访问统一通过 `src/lib/db/index.ts` 的 `query` / `execute` / `transaction` / `queryOne` / `queryPaginated` 函数，使用参数化查询，禁止字符串拼接 SQL。
- API 响应统一使用 `src/lib/api-response.ts` 的 `successResponse` / `errorResponse` / `paginatedResponse`，封装 `code` / `success` / `message` / `data`。
- API 路由使用 `withErrorHandler` 或 `withAuthAndErrorHandler` 包装，统一错误处理与认证。
- 日志使用 `src/lib/logger.ts` 的 `logger` / `secureLog`，避免直接 `console.log` 输出敏感信息。
