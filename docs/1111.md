二、数据库问题（P0/P1/P2）
P0 — 最高优先级
1. 硬编码密码暴露（安全）
database/vnerpdacahng_schema.sql 头部注释包含明文密码：Snqig521223

src/lib/db/mysql.ts 注释中也暴露同一密码

种子数据中默认密码 123456 明文写入字典表（未加密）

修复： 立即删除源码/SQL 中所有明文密码，统一用 .env 管理，种子数据密码改用 bcrypt 哈希。

2. 外键约束极度缺失（数据完整性）
87 张表仅 3 个 FK → 已通过 Migration 017/018 补充至 19 个

仍有 80+ 个明确引用关系完全无 FK 保护

已新增覆盖（16 个）：

销售模块（8 个）：sal_order→customer，sal_delivery→order，sal_delivery_detail→delivery/material，sal_return_order→order 等

库存模块（6 个）：inventory→material/warehouse，batch→material/warehouse，outbound→warehouse

生产模块（3 个）：work_order→sales_order，work_order→material，color_seq→work_order

财务模块（2 个）：receivable→customer，payable→supplier

遗留项：

inv_outbound_order 缺少 customer_id、work_order_id 列（Schema 与实际 DB 不一致）

inv_inbound_order.warehouse_id 类型为 INT，与 inv_warehouse.id（BIGINT UNSIGNED）不匹配

P1 — 重要问题
3. 排序规则（Collation）三种混用
排序规则	表数量	问题
utf8mb4_0900_ai_ci	~62	MySQL 8.0 默认
utf8mb4_unicode_ci	~19	与 0900_ai_ci 混用导致 JOIN 隐式转换
utf8mb4_ai_ci	1（inv_inventory_log）	孤表，与所有表 JOIN 都有风险
修复： 统一为 utf8mb4_0900_ai_ci，ALTER TABLE ... CONVERT TO。

4. 状态码体系不统一（4 种方案并存）
方案	表数	示例
TINYINT 0-1	~12	sys_user：0-禁用/1-启用
TINYINT 1-N	~15	sal_sales_order：1-待确认/2-已确认...
TINYINT 10-90	~5	pur_purchase_order：10-草稿/20-待审批...
VARCHAR(20)	~8	inv_inbound_order：'draft'/'pending'/'completed'
问题： 同一数字（如 1）在不同表中含义完全不同；同一模块内也不统一（pur_purchase_request 用 1-5，pur_purchase_order 用 10-90）。

修复： 新表统一 VARCHAR 字符串状态，存量表渐进迁移。

5. 主键类型不一致（INT vs BIGINT UNSIGNED）
主键类型	表数	典型表
BIGINT UNSIGNED	~79	绝大多数
INT UNSIGNED	4	pur_purchase_order, fin_account 等
INT（无 UNSIGNED）	2	domain_event_outbox, sys_event_processed
跨表类型不匹配（无法建 FK）：

inv_inbound_order.warehouse_id（INT）→ inv_warehouse.id（BIGINT UNSIGNED）

pur_purchase_order.supplier_id（INT UNSIGNED）→ pur_supplier.id（BIGINT UNSIGNED）

fin_voucher.created_by（VARCHAR(50)）→ sys_user.id（BIGINT UNSIGNED）

修复： 统一为 BIGINT UNSIGNED，修改子表引用列类型。

P2 — 低优先级
6. DECIMAL 精度不统一（4 种精度混用）
精度	模块	问题
DECIMAL(18,4)	CRM/INV/PRD/SAL（多数）	4 位小数，适合印刷业
DECIMAL(14,2)	PUR（采购）	跨模块汇总时精度丢失
DECIMAL(15,3)	INV（入库单）	3 位小数，与 18,4/14,2 都不一致
DECIMAL(18,2)	FIN（财务凭证）	会计模块仅 2 位小数
修复： 统一为 DECIMAL(18,4)（扩精度安全）。

7. 审计字段命名不统一（3 种风格并存）
风格	创建人	更新人	创建时间	更新时间	表数
下划线式	create_by	update_by	create_time	update_time	~70+
created_by 式	created_by	—	—	—	2
_at 式	—	—	created_at	—	2
特例： prd_standard_card.creator_id（唯一用 creator_id），fin_voucher.created_by 存用户名而非 ID。

修复： 统一为 create_by / update_by / create_time / update_time（BIGINT UNSIGNED + DATETIME）。

8. Drizzle ORM 覆盖率极低（2/87 表）
仅定义了 inv_inbound_order 和 inv_inbound_item 两张表

其余 85 张表全部通过 pool.query()/pool.execute() 执行原生 SQL

无编译期类型检查，列名拼写错误/类型不匹配只能在运行时发现

修复： 新功能优先使用 Drizzle，存量代码按模块逐步迁移。

9. 软删除字段定义不一致
定义	表数	问题
deleted TINYINT DEFAULT 0	~60+	主流方案
deleted TINYINT NOT NULL DEFAULT 0	~5	有 NOT NULL 约束
deleted TINYINT(1) DEFAULT 0	1	MySQL 8.0 已废弃显示宽度
无 deleted 字段	~10（日志/事件表）	设计合理
修复： 统一为 deleted TINYINT NOT NULL DEFAULT 0。

三、前端问题（P0/P1/P2）
P0 — 最高优先级
1. TypeScript any 类型滥用 — 2,772 处
覆盖 436 个文件（53%），TypeScript strict 模式形同虚设

Top 5 重灾区：

api/dashboard/ceo/route.ts — 51 个 any

api/dashboard/kpi/route.ts — 47 个 any

lib/audit-logger.ts — 37 个 any

application/services/StandardCardApplicationService.ts — 31 个 any

lib/standard-card-service.ts — 29 个 any

基础设施文件也在用 any： lib/api-response.ts 的 ApiResponse<T = any>、PaginatedResponse<T = any> 被全项目引用

修复：

: any → : unknown + 类型守卫，或定义具体 interface

ApiResponse<T> 在调用处传入具体类型

ESLint 添加 @typescript-eslint/no-explicit-any: "error"

2. API 路由鉴权缺失 — 315/317 个路由未用 withAuth（覆盖率 0.6%）
Middleware 只做 CSRF 校验，不做 JWT 认证 → 所有 /api/ 路由直接放行

虽然项目有 withAuth 包装器，但几乎无人使用

修复方案（二选一）：

方案 A（推荐）： middleware 统一 JWT 校验，白名单路由（/api/auth/login 等）显式排除

方案 B： 强制所有 API handler 用 withAuth 包装，ESLint 自定义规则检测裸露的 export async function GET/POST/PUT/DELETE

3. JWT Token 存储在 localStorage — XSS 可窃取
AuthContext.tsx 将 token、refreshToken、user 都存入 localStorage/sessionStorage

任何 XSS 漏洞都可直接读取，导致账号被盗

修复：

JWT 改为 httpOnly + Secure + SameSite=Strict Cookie

Refresh Token 同样存入 httpOnly Cookie

前端仅保留用户非敏感信息在内存（Context state）中

P1 — 重要问题
4. 超大组件文件（违反单一职责）
文件	行数	问题
warehouse/inbound/page.tsx	3,000	表单+表格+弹窗+扫描+明细全混合
warehouse/outbound/page.tsx	1,712	同上

settings/user/page.tsx	856	用户管理全功能
purchase/request/page.tsx	832	采购申请全流程
修复： 每个 500+ 行页面拆分为表单组件、表格组件、弹窗组件、业务 Hook。目标：单组件 ≤ 300 行。

5. 错误边界几乎缺失
全项目仅 1 个 error.tsx（warehouse/error.tsx）

无全局 error.tsx，无 not-found.tsx

仅 1 个 loading.tsx，140 个页面几乎无路由级加载态

0 处 ErrorBoundary 组件使用

修复： 添加全局 error.tsx、not-found.tsx，关键路由补 loading.tsx + Skeleton，关键组件包裹 <Suspense>。

6. 330 处行内样式（style={{}}）— 违反样式分离
项目已用 Tailwind v4，但仍有 330 处 style={{}} 分散在 240 个 TSX 文件中。

修复： 全部替换为 Tailwind 等效类名，ESLint 添加 react/forbid-component-props 禁止 style 属性。

7. 调试/临时文件残留
类型	路径	问题

8. 51 处 console.log/error 残留生产代码
项目已有 lib/logger.ts 统一日志模块，但仍有 51 处直接使用 console.log/error。

修复： 全局替换 console.log/error/warn → logger.info/error/warn，ESLint 添加 no-console 规则。

9. 缺少标准目录（constants / types / store / utils）

修复： 创建 src/constants/ 和 src/types/，考虑引入 Zustand 替代复杂 Context。

核心业务逻辑（MRP 引擎、FIFO 分配、财务核心、工作流引擎）几乎无测试覆盖

修复： 优先为核心 lib/ 补单元测试（mrp-engine, fifo-allocation, finance-core），关键 API 补集成测试（auth/login, warehouse/inbound）。目标：核心模块 80%+，API 路由 50%+。
