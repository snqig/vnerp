# 故障排查

本文档说明 印刷生产经营信息管理系统 Print MIS 常见问题的排查思路与解决方案。

## 1. 通用排查流程

1. **确认现象**：错误信息、影响范围、复现条件。
2. **检查健康**：`curl http://localhost:5000/api/health`，查看 `database` / `memory` 状态。
3. **查看日志**：应用日志、数据库日志、浏览器 Console。
4. **隔离原因**：基础设施（MySQL/Redis/网络）vs 应用代码 vs 配置。
5. **临时恢复**：重启服务或回滚版本，优先恢复业务。
6. **根因分析**：修复后补测试用例，更新本文档。

## 2. 应用启动失败

### 2.1 端口被占用

```bash
# Windows
netstat -ano | findstr :5000
# Linux/Mac
lsof -i :5000
```

解决：终止占用进程或修改 `PORT` 环境变量。

### 2.2 `JWT_SECRET must be set in production environment`

生产环境未配置 `JWT_SECRET`。在 `.env.production` 中填入 64 字节随机串：

```bash
node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"
```

### 2.3 `ER_ACCESS_DENIED_ERROR`

数据库账号或密码错误。检查 `.env` 中 `DB_USER` / `DB_PASSWORD`：

```bash
mysql -h 127.0.0.1 -u <DB_USER> -p<DB_PASSWORD> -e "SELECT 1"
```

### 2.4 `ECONNREFUSED`

MySQL 或 Redis 未启动。Docker 部署检查：

```bash
docker compose -f docker-compose.prod.yml ps
docker compose -f docker-compose.prod.yml logs mysql redis
```

### 2.5 构建失败

```bash
pnpm build 2>&1 | tee build.log
```

常见原因：

- TypeScript 类型错误：`pnpm ts-check` 定位。
- ESLint 错误：`pnpm lint`。
- 依赖缺失：`pnpm install`。
- 内存不足：增加 Node 内存 `NODE_OPTIONS=--max-old-space-size=4096`。

## 3. 登录问题

### 3.1 401 未授权

- token 过期（默认 24h）：刷新 token（`POST /api/auth/refresh`）或重新登录。
- token 被撤销：修改密码后旧 token 失效。
- token 格式错误：检查 `Authorization: Bearer <token>` 头。

### 3.2 429 请求过频

- 登录限流：单 IP 15 分钟 20 次，等待或换 IP。
- 账号锁定：失败 5 次锁定 15 分钟，可手动解锁：

  ```sql
  UPDATE sys_user SET login_fail_count = 0, lock_time = NULL WHERE username = 'xxx';
  ```

### 3.3 403 无权访问

- 角色未关联菜单：检查 `sys_role_menu`。
- 权限点不匹配：前端 `usePermission` 与后端 `permission` 参数一致。
- `data_scope` 限制：查询 `src/lib/api-permissions.ts` 的数据范围过滤。

### 3.4 异地登录告警

非 Bug，是安全机制。登录时对比 `last_login_ip`，不一致会插入 `sys_notification` 提醒。

## 4. 数据库问题

### 4.1 连接数耗尽

```sql
SHOW STATUS LIKE 'Threads_connected';
SHOW VARIABLES LIKE 'max_connections';  -- docker-compose 默认 200
```

排查长连接或慢查询：

```sql
SHOW PROCESSLIST;
KILL <连接ID>;
```

`src/lib/db/index.ts` 默认 `connectionLimit: 20`、`queueLimit: 100`，可根据负载调整。

### 4.2 慢查询

启用慢查询日志：

```sql
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 2;
SHOW VARIABLES LIKE 'slow_query_log_file';
```

通过 `EXPLAIN` 分析慢 SQL，添加索引或重写查询。

### 4.3 死锁

```sql
SHOW ENGINE INNODB STATUS\G
```

关注 `LATEST DETECTED DEADLOCK` 段。常见原因：

- 事务中多表更新顺序不一致。
- 长事务持锁。

`src/lib/db/index.ts` 的 `query` 函数自动重试 2 次，缓解偶发死锁。

### 4.4 字符集乱码

```sql
SHOW VARIABLES LIKE 'character_set%';
SHOW CREATE TABLE <表名>;
```

确认表与连接均使用 `utf8mb4`。`setup-db.mjs` 已将 collation 转换为 `utf8mb4_0900_ai_ci`。

### 4.5 数据不一致

- 软删除字段 `deleted` 误判：检查查询是否带 `WHERE deleted = 0`。
- 事务未提交：检查 `transaction()` 是否正确 `await`。
- 并发更新：考虑使用乐观锁（version 字段）或 `SELECT ... FOR UPDATE`。

## 5. Redis 问题

### 5.1 token 撤销不跨实例

未配置 Redis 时降级为进程内 Map，`revokeAllUserTokens` 仅当前实例生效。生产环境必须配置 `REDIS_URL`。

### 5.2 缓存不生效

- `EVENT_BUS_TYPE` 配置错误。
- `invalidateCache` 未在数据变更后调用。
- 客户端缓存（浏览器）干扰，尝试强制刷新。

### 5.3 Redis 内存满

```bash
redis-cli INFO memory
redis-cli CONFIG GET maxmemory
```

清理策略为 `allkeys-lru`，会自动淘汰。如仍不足，调整 `docker-compose.prod.yml` 中 `--maxmemory`。

## 6. 前端问题

### 6.1 Hydration 错误

服务端与客户端渲染结果不一致。常见原因：

- `useTranslations` 在客户端调用但服务端 locale 不一致。
- 使用 `Date.now()` 或 `Math.random()` 等非确定性值。
- 第三方组件依赖 `window` 对象。

排查：检查 `IntlProvider` 的 `locale` 与 `messages` 是否一致，使用 `'use client'` 标记客户端组件。

### 6.2 翻译键缺失

控制台 warning：`[i18n] 翻译键缺失: ...`。在对应语言包 `messages/*.json` 补齐 key。

### 6.3 页面 404（生产环境）

`src/middleware.ts` 在生产环境屏蔽 `/debug`、`/test-api`、`/diagnostic`、`/test`、`/qrcode` 路径，确认访问的不是调试路由。

## 7. 性能问题

### 7.1 接口慢

- 启用 `DEBUG_DB=true` 查看 SQL 执行时间。
- 使用 `EXPLAIN` 分析执行计划。
- 检查 N+1 查询（循环中调用 `query`）。
- 使用 `cachedApiRoute` 缓存读多写少的接口。

### 7.2 内存增长

```bash
docker stats vnerp-app-prod
```

- 检查 `process.memoryUsage()`（`/api/health` 返回 `heapUsed / heapTotal`）。
- 排查内存泄漏：长生命周期对象、未清理的 setInterval、未关闭的连接。
- 调整 `NODE_OPTIONS=--max-old-space-size=1024`。

### 7.3 数据库连接池耗尽

`src/lib/db/index.ts` 默认 `connectionLimit: 20`。如并发高，调整环境变量或直接修改配置。

## 8. 部署问题

### 8.1 Docker 构建失败

```bash
docker compose -f docker-compose.prod.yml build --no-cache app
```

常见原因：

- `pnpm install --frozen-lockfile` 失败：lockfile 与 `package.json` 不一致，本地 `pnpm install` 后提交。
- 网络问题：配置 npm 镜像或代理。

### 8.2 容器启动后立即退出

```bash
docker compose -f docker-compose.prod.yml logs app --tail=100
```

常见原因：

- 依赖服务（mysql/redis）未就绪：`depends_on` healthcheck 已配置，但仍可能因网络问题失败，重启即可。
- 环境变量缺失：检查 `.env.production` 是否完整。

### 8.3 数据卷丢失

`docker compose down` 不会删除数据卷。如使用 `docker compose down -v` 会清空数据，谨慎使用。

## 9. 紧急联系

| 现象 | 紧急程度 | 操作 |
|------|----------|------|
| 生产完全不可用 | P0 | 立即回滚到上一版本，恢复备份 |
| 数据库不可用 | P0 | 检查 MySQL 容器，恢复最新备份 |
| 大量用户登录失败 | P1 | 检查限流与账号锁定状态 |
| 部分接口报错 | P2 | 查看日志定位，必要时降级功能 |
| 性能严重下降 | P2 | 检查慢查询与连接池 |
