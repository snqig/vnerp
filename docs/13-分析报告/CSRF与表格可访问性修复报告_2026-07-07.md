## 二、ISSUE-1: CSRF Token 缺失导致 global-setup 失败

### 2.1 根因分析

**文件**: [tests/global-setup.ts](file:///d:/dcprint/erp-project/tests/global-setup.ts)

原实现直接 POST `/api/auth/reset-lock` 重置 admin 账号的登录锁定状态:

```ts
// 原代码(已删除)
const response = await fetch(`${baseURL}/api/auth/reset-lock`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ username: 'admin' }),
});
```

**中间件校验链路**([src/middleware.ts:111-121](file:///d:/dcprint/erp-project/src/middleware.ts#L111-L121)):

1. `/api/auth/reset-lock` 是 POST 请求 → `requiresCsrfValidation()` 返回 `true`
2. 该路径不在 CSRF 豁免列表中([src/lib/csrf.ts:7](file:///d:/dcprint/erp-project/src/lib/csrf.ts#L7) 仅豁免 `/api/auth/login`、`/api/auth/register`、`/api/auth/refresh`)
3. `validateCsrfToken()` 校验 `X-CSRF-Token` header 与 `csrf_token` cookie 是否一致
4. 原请求两者皆无 → 返回 `403 CSRF token validation failed`
5. admin 锁定状态未重置 → 后续所有测试登录被拒(每次 30s 超时 × 3 次重试 = 90s/用例)

### 2.2 修复方案

采用 **Double Submit Cookie** 模式([src/lib/csrf.ts](file:///d:/dcprint/erp-project/src/lib/csrf.ts) 已实现的基础设施):

```ts
// 修复后的 global-setup.ts
// Step 1: GET /en/login → middleware 第 147-149 行调用 ensureCsrfCookie 种入 csrf_token cookie
const pageResp = await fetch(`${baseURL}/en/login`, {
  headers: { 'Accept': 'text/html' },
});
const setCookie = pageResp.headers.get('set-cookie') || '';
const csrfToken = setCookie.match(/csrf_token=([^;]+)?.[1];

// Step 2: POST /api/auth/reset-lock 携带 CSRF token(header + cookie 双重提交)
const response = await fetch(`${baseURL}/api/auth/reset-lock`, {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'X-CSRF-Token': csrfToken,      // header
    'Cookie': `csrf_token=${csrfToken}`, // cookie
  },
  body: JSON.stringify({ username: 'admin' }),
});
```

**增强**: 添加 3 次重试(间隔 2s),应对 dev server 启动时的瞬态不可用。

### 2.3 验证

测试日志确认修复生效:
```
[GlobalSetup] Admin lock reset: Lock reset for admin
```

## 四、ISSUE-2: 表格列头 aria-sort 属性缺失

### 4.1 根因分析

**文件**: [src/app/[locale]/orders/sales/page.tsx](file:///d:/dcprint/erp-project/src/app/[locale]/orders/sales/page.tsx)

销售订单表格有 6 个可排序列头(`order_no`、`customer_name`、`order_date`、`delivery_date`、`total_amount`、`status`),排序状态由 `sortField` + `sortOrder` 管理([第 168-169 行](file:///d:/dcprint/erp-project/src/app/[locale]/orders/sales/page.tsx#L168-L169))。

**存在的三个缺陷**:

| 缺陷 | 影响 | 测试用例 |
|------|------|---------|
| 列头未设置 `aria-sort` 属性 | 屏幕阅读器无法感知排序状态;`getAttribute('aria-sort')` 返回 `null` | TC-SO-004 |
| 列头未设置 `tabIndex={0}` | 无法通过键盘 Tab 聚焦 | TC-SO-005/006 |
| 列头仅有 `onClick`,无 `onKeyDown` | Enter/Space 键无法触发排序 | TC-SO-005/006 |

### 4.2 修复方案

#### 4.2.1 新增辅助函数([第 258-268 行](file:///d:/dcprint/erp-project/src/app/[locale]/orders/sales/page.tsx#L258-L268))

```ts
// 根据 sortField/sortOrder 返回 ARIA 标准值
const getAriaSort = (field: SortField): 'ascending' | 'descending' | 'none' => {
  if (sortField !== field || !sortOrder) return 'none';
  return sortOrder === 'asc' ? 'ascending' : 'descending';
};

// 键盘事件处理:Enter/Space 触发排序(与 click 行为一致)
const handleSortKeyDown = (e: React.KeyboardEvent, field: SortField) => {
  if (e.key === 'Enter' || e.key === ' ') {
    e.preventDefault();
    handleSort(field);
  }
};
```

#### 4.2.2 为 6 个可排序 TableHead 添加可访问性属性([第 1077-1140 行](file:///d:/dcprint/erp-project/src/app/[locale]/orders/sales/page.tsx#L1077-L1140))

每个可排序 `<TableHead>` 增加:
- `aria-sort={getAriaSort('field')}` — 动态反映排序状态
- `tabIndex={0}` — 使 `<th>` 可通过键盘聚焦
- `onKeyDown={(e) => handleSortKeyDown(e, 'field')}` — 响应 Enter/Space 键

**关键决策**: 不设置 `role="button"`。原因:测试用 `getByRole('columnheader')` 查找列头,若设置 `role="button"` 会覆盖 `<th>` 隐式的 `columnheader` 角色,导致选择器失效。`tabIndex` + `onKeyDown` 已足够支持键盘操作。

示例(以 `order_no` 列为例):
```tsx
<TableHead
  className="cursor-pointer select-none hover:bg-muted/50"
  tabIndex={0}
  aria-sort={getAriaSort('order_no')}
  onClick={() => handleSort('order_no')}
  onKeyDown={(e) => handleSortKeyDown(e, 'order_no')}
>
  <span className="inline-flex items-center">
    {t('orderNo')}{getSortIcon('order_no')}
  </span>
</TableHead>
```

## 八、后续建议

1. **TC-SO-001 flaky**: 排查销售订单页面加载时的 console error 根因(可能与菜单权限 API 响应时序有关),考虑在测试中增加 `waitForResponse` 或延长 `waitForTimeout`。
2. **Next.js middleware 弃用警告**: 测试日志显示 `The "middleware" file convention is deprecated. Please use "proxy" instead.` — 属于 Next.js 版本升级提示,建议在后续升级中迁移至 `proxy.ts`。
3. **aria-sort 最佳实践**: 当前方案通过 `tabIndex` + `onKeyDown` 实现键盘可访问性。若后续需更严格的 WAI-ARIG 规范,可考虑在 `<th>` 内嵌套 `<button>` 元素(需同步调整测试选择器)。
4. **其他表格页面**: 建议排查采购订单等页面的表格是否也存在相同的 `aria-sort` 缺失问题,统一修复。
