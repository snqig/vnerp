## 1. 问题概述

在 ERP 项目的全链路验证中，发现 8 处 SQL 占位符与参数数组不同步的 Bug，以及 1 处 `IN (?)` 数组展开隐患。这些问题会导致数据库写入时列值错位、数据静默丢失或批量操作只生效第一条。

### 1.1 Bug 模式分类

| 模式 | 描述 | 影响 |
|------|------|------|
| **多余 `?` 占位符** (Bug 1-7) | 在 `NOW()` / `CURDATE()` 等 SQL 函数前多写了 `?`，导致参数数组与占位符数量不匹配 | 列值错位，数据写入错误列 |
| **`join(',')` + `IN (?)`** (Bug 8) | 用 `join(',')` 将数组拼成字符串后传给 `IN (?)`，MySQL 视为单个值 | 批量操作只生效第一条 |
| **`buildOptimizedQuery` 隐患** | 动态生成 `IN (?)` + `params.push(value)` 依赖调用方选择 `query` vs `execute` | 潜在运行时错误 |

### 2.2 Bug 8: `join(',')` + `IN (?)` 反模式

**文件**: [alert-push/route.ts](../../src/app/api/warehouse/alert-push/route.ts)

**修复前** (第 127-129 行):
```typescript
await execute(`UPDATE sys_notification SET is_read = 1, read_time = NOW() WHERE id IN (?)`, [
  notification_ids.join(','),
]);
```

**修复后**:
```typescript
await execute(
  `UPDATE sys_notification SET is_read = 1, read_time = NOW() WHERE id IN (${notification_ids.map(() => '?').join(',')})`,
  notification_ids
);
```

**根因**: `execute()` 走 `pool.execute`（预编译语句），不展开数组参数。`join(',')` 将 `[1,2,3]` 拼成 `"1,2,3"`，MySQL 将整个字符串视为单个值，导致只匹配第一个 ID。

### 2.3 db-optimization.ts 隐患修复

**文件**: [db-optimization.ts](../../src/lib/performance/db-optimization.ts)

**修复前** (第 63-65 行):
```typescript
if (Array.isArray(value)) {
  whereClauses.push(`${key} IN (?)`);
  params.push(value);  // 嵌套数组，依赖调用方用 query() 而非 execute()
```

**修复后**:
```typescript
if (Array.isArray(value)) {
  if (value.length > 0) {
    whereClauses.push(`${key} IN (${value.map(() => '?').join(',')})`);
    params.push(...value);  // 展开到 params 顶层
  } else {
    whereClauses.push('1 = 0');  // 空数组安全降级：不匹配任何行
  }
```

## 6. 防回归规则

1. **`IN (?)` + 数组参数**: 必须使用 `query()`（客户端展开）或显式展开 `${ids.map(() => '?').join(',')}`，禁止 `join(',')` 拼接
2. **`execute()` + `IN`**: 必须显式展开占位符，禁止传嵌套数组
3. **`NOW()` / `CURDATE()`**: 前面不应有 `?` 占位符
4. **参数数组长度**: 必须与 SQL 中 `?` 数量严格一致，代码审查时需逐一核对
