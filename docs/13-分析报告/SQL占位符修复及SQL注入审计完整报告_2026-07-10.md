## 3. `IN (?)` 显式展开规范化

### 3.1 问题描述

项目中存在两种 `IN (?)` 使用方式：
- `query()` + `IN (?)` + `[ids]` — mysql2 的 `pool.query` 会客户端展开数组，**技术安全**
- `execute()` + `IN (?)` + `[ids]` — `pool.execute` 走预编译语句，**不展开数组**

虽然使用 `query()` 的场景技术安全，但存在**隐性维护风险**：开发者可能在不了解差异的情况下将 `query()` 改为 `execute()`，导致运行时错误。

### 3.2 修复策略

统一采用显式占位符展开模式，消除对 `query()` vs `execute()` 差异的依赖：

```typescript
// 修复前（隐性依赖 query() 展开数组）
await query(`SELECT ... WHERE id IN (?)`, [ids]);

// 修复后（显式展开，query/execute 均安全）
await query(`SELECT ... WHERE id IN (${ids.map(() => '?').join(',')})`, ids);
```

## 4. SQL 注入风险审计

### 4.1 审计范围

对 `src/` 目录下所有 `.ts` 文件进行了以下模式的扫描：
1. SQL 语句中的模板字符串插值 `${}`
2. SQL 语句的字符串拼接 `+`
3. 动态表名/列名
4. 用户输入直接注入查询

### 4.2 审计结果

#### 4.2.1 无风险（安全模式）

| 模式 | 使用位置 | 安全原因 |
|------|----------|----------|
| **列常量插值** `${COLUMNS}` | MysqlReceivableRepository, MysqlPayableRepository, MysqlDeliveryRepository 等 15+ 文件 | 常量字符串，编译时确定 |
| **动态 WHERE 子句** `${where}` | SampleProcessTemplateService, SampleProcessCardService, MysqlPurchaseReconciliationRepository, contract-review/route.ts 等 10+ 文件 | WHERE 子句由硬编码 SQL 片段 + `?` 占位符构建，用户输入通过 params 数组传递 |
| **动态 UPDATE SET** `${fields.join(',')}` | equipment/route.ts, equipment/plan/route.ts, equipment/maintenance/route.ts, contract-review/route.ts | 使用 `allowedFields` 白名单 + `?` 占位符 |
| **colMap 白名单** `${col} = ?` | MysqlToolRepository.ts | 列名来自硬编码 colMap 映射 |
| **硬编码字段名** `${fields.join(',')}` | MysqlFormulaVersionRepository.ts | 字段名为硬编码常量 |
| **已正确的 IN 展开** `${placeholders}` | MysqlTransferOrderRepository, MysqlStocktakingOrderRepository, MysqlPurchaseOrderRepository, MysqlOutboundOrderRepository, MysqlInboundOrderRepository, lib/auth.ts | 已使用 `map(() => '?').join(',')` 模式 |

#### 4.2.2 低风险（约定偏离，非可利用）

| 文件 | 行号 | 模式 | 风险说明 |
|------|------|------|----------|
| [lib/db/index.ts](../../src/lib/db/index.ts) | 392-436 | `${table}` + `${keys.join(',')}` | 表名通过 `validateTableName()` 正则验证，列名来自 `Object.keys(data)` (应用定义)。**偏离项目约定**：项目规范要求使用 `mysql2.escapeId()` + `assertValidIdentifier()` 白名单验证，当前仅用正则。 |
| [lib/auth.ts](../../src/lib/auth.ts) | 282-309 | `${tableAlias}.${fieldName}` | `tableAlias`/`deptField`/`userField` 等为函数参数，默认值硬编码，调用方使用应用常量。未使用 `escapeId()`。 |

#### 4.2.3 SQL 注入漏洞修复（fifo-allocation.ts）

**文件**: [fifo-allocation.ts](../../src/lib/fifo-allocation.ts) 第 122 行

```typescript
// 修复前（SQL 注入风险）：excludeBatchIds 值直接插值到 SQL 中
```

## 6. 防回归规则

1. **`IN` + 数组参数**: 必须使用 `${ids.map(() => '?').join(',')}` 显式展开，禁止 `IN (?)` + `[array]` 依赖 `query()` 自动展开
2. **`execute()` + `IN`**: 禁止传嵌套数组，必须显式展开占位符
3. **空数组保护**: `IN` 子句前检查 `array.length > 0`，空数组使用 `1 = 0` 降级
4. **`NOW()` / `CURDATE()`**: SQL 函数前不应有 `?` 占位符
5. **参数数组长度**: 必须与 SQL 中 `?` 数量严格一致
6. **动态表名/列名**: 使用 `mysql2.escapeId()` 或白名单验证，禁止直接插值
