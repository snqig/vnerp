### 任务 03：事件总线可靠性改造（事务性发件箱模式）✅ 已完成

* **状态**：✅ 已完成（2026-07-08）
* **任务详情**：基于「数据库发件箱表 + 异步任务队列」实现领域事件的可靠投递，保证「业务数据提交」与「事件发布」的原子性；替换原有内存事件总线的核心链路，保留内存实现作为开发环境降级方案。
* **交付物**
  1. 数据库新增发件箱表的 Drizzle 迁移脚本
  2. `src/infrastructure/event-bus/` 可靠事件总线实现
  3. 入库单审核场景试点改造，验证一致性
  4. 配套死信、重试、监控埋点
* **验收标准**
  1. 业务事务回滚时，事件不会发出
  2. 服务重启后，未投递的事件可继续补发
  3. 入库单审核场景下，库存同步、审计日志等副作用不丢失、不重复
  4. 开发环境无 Redis 时，可降级为内存事件总线
* **实施记录**
  * 迁移脚本：`database/migrations/001_create_domain_event_outbox.sql`、`002_add_outbox_dispatching.sql`。
  * 事件总线实现位于 `src/infrastructure/event-bus/`，共 10 个文件：
    * `DomainEventOutbox.ts`：发件箱仓储接口与 MySQL 实现
    * `OutboxPoller.ts`：轮询发件箱投递到 Redis Stream，含定时回收过期 `processing` 记录、清理 30 天前已处理记录
    * `StreamPublisher.ts`：XADD 投递到 `erp:domain-events` Stream，受 `STREAM_MAX_LENGTH` 自动裁剪
    * `StreamConsumer.ts`：XREADGROUP 消费 + XACK 确认 + XAUTOCLAIM 回收死消费者遗留消息
    * `IdempotentHandler.ts` + `IdempotencyGuard.ts`：基于 `sys_event_processed` 表的两阶段幂等标记（`processing` → `processed`），失败时 `deleteMark` 允许重试
    * `MemoryDomainEventOutbox.ts` + `DomainEventOutboxFactory.ts`：开发环境无 Redis 时降级为内存实现
    * `EventBus.ts`：统一事件总线入口
  * 健康检查：`src/infrastructure/monitoring/InfrastructureHealthCheck.ts` 检查 database/redis/outbox/streamConsumer 四项状态。
  * 环境变量：`EVENT_BUS_TYPE=db`（默认）、`STREAM_MAX_LENGTH`、`STREAM_RECLAIM_IDLE_MS`、`IDEMPOTENCY_STALE_THRESHOLD_MINUTES` 等已抽取到 `.env.example`。
