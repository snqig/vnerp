## 🔴 P0 — 严重问题（6 项）

### 1. 动态表名拼入 SQL — 10+ 处

**风险**：SQL 注入

`soft-delete.ts`(151/180/258)、`diagnose/material/route.ts`(11)、`settings/category-rules/route.ts`(59) 等文件将变量直接拼为表名。虽然是硬编码调用链，但公共库函数应使用 `mysql2.escapeId(tableName)` 或白名单校验。

```sql
-- src/lib/soft-delete.ts:151
SELECT * FROM `{tableName}` WHERE id = ?
```

### 2. 动态字段名拼入 UPDATE SET

**风险**：SQL 注入  
**文件**：`src/app/api/dcprint/process-cards/route.ts:343`

`snakeCase(field)` 转换后的字段名直接拼入 SQL，来自请求 body。应使用字段名白名单校验。

```sql
UPDATE prd_process_card SET {updateFields.join(', ')} WHERE id = ?
```

### 3. `${row.id}` 直接拼入 WHERE

**风险**：SQL 注入  
**文件**：`MaterialLifecycleService.ts:615/620`

虽是数据库查询结果但违反参数化原则，应为 `?` 占位符。

### 4. 主键类型不一致 — 4 张表 INT UNSIGNED vs 79 张 BIGINT

**风险**：类型安全 / 性能

| 表名 | 当前 PK | 应改为 |
|------|--------|--------|
| `pur_purchase_order` | INT UNSIGNED | BIGINT |
| `pur_purchase_order_line` | INT UNSIGNED | BIGINT |
| `fin_account` | INT UNSIGNED | BIGINT |
| `fin_period` | INT UNSIGNED | BIGINT |

与 `supplier(id=BIGINT)`、`voucher_line(account_id→BIGINT)` 的 JOIN 将产生隐式类型转换，**索引失效**。

### 5. JOIN 字段无 FK 约束 — 83 表仅 5 个 FK

**风险**：数据完整性

核心父子关系仅靠应用层保证完整性，存在孤儿记录风险：

- `sal_order_detail.order_id`
- `sal_delivery_detail.delivery_id`
- `pur_request_detail.request_id`
- 等数十处

### 6. UPDATE ... WHERE IN (SELECT ...) — 性能反模式

**风险**：性能  
**文件**：`quality/incoming/route.ts:223/234/301/315`

MySQL 性能最差的反模式之一，应改为 JOIN UPDATE。

```sql
UPDATE inv_inbound_order
SET status = ?
WHERE order_no IN (SELECT ref_no FROM qrcode_record WHERE ...)
```

| 组件 | 文件 | 职责 | 评估 |
|------|------|------|:--:|

## 📝 总体评价

1. **Schema 层面积累了大量技术债务的解决方案**：37 个迁移文件已经编写了 FK 补建、PK 对齐、Collation 统一、DECIMAL 精度的脚本，只差执行和 schema.sql 回写这临门一脚。

2. **SQL 安全存在 2 类高危注入点**：动态表名（10+ 文件）和动态字段名（1 文件）。虽然当前调用链大多使用硬编码表名，但 `soft-delete.ts` 作为公共库函数缺乏防护是安全隐患。

3. **性能反模式集中在 SELECT \***：150+ 处 `SELECT *` 是最大的性能债务。N+1 事件写入和 UPDATE...IN(SELECT) 次之，但影响面较窄。

4. **Outbox / 事务架构是亮点**：DomainEventOutbox + IdempotencyGuard + OutboxPoller 的三件套设计成熟，FOR UPDATE SKIP LOCKED + 三阶段幂等 + 指数退避均为正确的工程实践。

5. **Drizzle ORM 需要明确路线**：当前 2.4% 的覆盖率处于"既用又不用"的尴尬状态，继续维持会增加维护成本。
