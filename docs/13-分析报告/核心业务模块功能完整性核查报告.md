## 2. 已实现功能详解（纠正误判）

### 2.1 仓储 FIFO 库存分配 ✅

**原判定**：🔴 未实现 — "UI 有，逻辑未接入"
**实际状态**：✅ 已实现并接入 3 个 API 路由

**证据**：
- 库文件：`src/lib/fifo-allocation.ts`
  - 完整接口定义：`FIFOAllocationItem`、`FIFOAllocationResult`、`ShortageWarning`
  - 使用 `decimal.js` 保证金额精度
  - 内置乐观锁冲突重试机制
  - 支持事务化多物料批量出库 + 指定批次出库
- 接入点（3 处）：
  1. `src/app/api/warehouse/outbound/confirm/route.ts` — 出库确认时调用 `allocateFIFO` + `executeFIFODeductionWithRetry`
  2. `src/app/api/inventory/route.ts` — 库存查询
  3. `src/app/api/production/material-issue/route.ts` — 生产领料
- 出库确认路由包含完整事务逻辑：`SELECT ... FOR UPDATE` 行锁 → 状态机校验 → FIFO 分配 → 扣减 → 记录日志

**结论**：FIFO 分配逻辑已完整接入出库和领料流程，原判定有误。

### 2.2 仓储批次追溯 ✅

**原判定**：🔴 未实现 — "batch_no 字段存在，无追溯链路"
**实际状态**：✅ 已实现，252 行完整追溯 API

**证据**：
- API 路由：`src/app/api/warehouse/batch/trace/route.ts`
- 功能：
  1. 按 `batch_no` 查询 `inv_inventory_batch` 表获取批次快照
  2. 查询 `inv_inventory_log` 获取该批次所有操作记录（入库/出库/调整/调拨/领料/退料/退货）
  3. 关联业务单据：入库单、出库单、调拨单、盘点单、库存调整单
  4. 返回完整追溯链 `trace_chain` + 汇总（总入库/总出库/净变动）
- 支持 7 种操作类型标签映射
- 每条记录含操作时间、操作人、仓库、业务单号、备注

**结论**：批次追溯链路完整，原判定有误。

### 2.3 生产 MRP 计划生成 ✅

**原判定**：🟡 部分 — "两个版本（v1/v2），尚未用于生产"
**实际状态**：✅ 已实现并接入 API 路由

**证据**：
- 库文件：`src/lib/mrp-engine.ts`（v1）、`src/lib/mrp-engine-v2.ts`（v2）
- API 路由：`src/app/api/production/mrp/route.ts`
  - **POST**：`runFullMRP()` — 执行完整 MRP 运算（BOM 展开 → 时间分段 → 净需求 → 计划订单 → 可选自动生成采购申请）
  - **GET**：支持 `bom-explode`（BOM 展开）、`time-buckets`（时间桶）、`net-requirements`（净需求）、`planned-orders`（计划订单）
- 测试文件：`src/lib/__tests__/mrp-engine-v2.test.ts`

**待改进**：MRP 目前通过 API 手动触发，无定时自动运行（需结合任务 02 BullMQ 定时任务实现）。但核心计算引擎已完整可用。

**结论**：MRP 引擎已实现并可通过 API 调用，原判定"尚未用于生产"过于保守。

### 2.4 质量 SPC 统计分析 ✅

**原判定**：🔴 未实现 — "spc-analysis.ts 有框架，无实际使用"
**实际状态**：✅ 已实现，API 路由含 3 种 SPC 图表

**证据**：
- 库文件：`src/lib/spc-analysis.ts`
  - 完整数据结构：`SPCDataPoint`、`ControlLimit`、`XbarRChart`、`ParetoItem`、`PChart`
  - 统计系数表：`A2_TABLE`（子组大小 2-10 的 A2 系数）
  - 函数：`calculateXbarRChart`、`calculatePareto`、`calculatePChart`、`getSPCDataFromDB`
- API 路由：`src/app/api/quality/spc/route.ts`
  - `action=xbar-r`：Xbar-R 控制图（含 UCL/CL/LCL、失控点、过程能力指数 Cp/Cpk/Pp/Ppk）
  - `action=pareto`：帕累托分析（缺陷类型排序 + 累计百分比）
  - `action=p-chart`：P 控制图（不合格率）
- 从数据库读取实际检验数据，非模拟

**结论**：SPC 统计分析已完整实现并接入 API，原判定有误。

### 2.5 财务应付对账 ✅

**原判定**：🔴 未实现 — "仅有应收对账，应付部分缺失"
**实际状态**：✅ 已实现，含应付生成、付款、核销、账龄分析

**证据**：
- 数据库表：`fin_payable`（应付单表）
- API 路由（8 处涉及应付）：
  1. `src/app/api/finance/payables/route.ts` — 应付单列表 + 生成（调用 `generatePayable`）
  2. `src/app/api/finance/payables/[id]/payment/route.ts` — 单笔应付付款
  3. `src/app/api/finance/payable/route.ts` — 应付查询
  4. `src/app/api/finance/payment/route.ts` — 付款记录
  5. `src/app/api/finance/invoice/route.ts` — 发票核销（同时支持 `payableId` 和 `receivableId`）
  6. `src/app/api/finance/aging/route.ts` — 账龄分析（`type=payable` 查 `fin_payable`）
  7. `src/app/api/finance/stats/route.ts` — 应付统计（总额/已付/余额/逾期）
  8. `src/app/api/finance/summary/route.ts` — 应付汇总
- 核心函数：`generatePayable`、`recordPayment`、`queryPayableSummary`（来自 `@/lib/finance-core`）

**结论**：应付对账功能完整，原判定有误。
