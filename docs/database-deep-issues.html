影响评估 无 FK 保护意味着：(1) 删除父记录不会阻止子记录留存 → 产生孤儿数据；(2) 无法在数据库层阻止插入无效引用；(3) 数据迁移/清理时难以自动发现断链；(4) 随数据增长，清理孤儿数据的成本指数级上升。
修复方案

分阶段补齐外键（建议 3 个 Sprint）：Phase 1 — 核心业务链路（sys、sal、inv、fin 模块的高频 JOIN 关系）；Phase 2 — 生产/采购链路；Phase 3 — 辅助表（日志、字典等）。

每个 FK 添加前需先执行 SELECT COUNT(*) FROM child WHERE parent_id NOT IN (SELECT id FROM parent) 清理孤儿数据，再添加 ALTER TABLE ... ADD CONSTRAINT ... FOREIGN KEY。对于无法确定 ON DELETE 行为的，统一使用 ON DELETE RESTRICT。

P1 2. 排序规则（Collation）三种混用 — JOIN 性能与正确性风险

87 张表中存在 3 种不同的 utf8mb4 排序规则，当不同 Collation 的表进行 JOIN 时，MySQL 无法使用索引，导致全表扫描。

排序规则表数量典型表影响
utf8mb4_0900_ai_ci ~62 sys_user, crm_customer, inv_material, sal_sales_order 等大多数 MySQL 8.0 默认值
utf8mb4_unicode_ci ~19 eqp_equipment, prd_schedule, prd_requisition, prd_material_return, prd_bom_std, fin_account, fin_voucher, inv_stock_batch 等 与 0900_ai_ci 混用会导致 JOIN 时的隐式转换
utf8mb4_ai_ci 1 inv_stock_transaction 孤表，与所有其他表 JOIN 都有风险
影响场景示例 inv_stock_transaction（utf8mb4_ai_ci）与 inv_material（utf8mb4_0900_ai_ci）通过 material_id JOIN 时，如果关联字段含字符串比较（如 material_code），MySQL 会因 Collation 不匹配而放弃索引，产生 Illegal mix of collations 错误或全表扫描。
修复方案

统一为 utf8mb4_0900_ai_ci（MySQL 8.0 默认值）：

-- 批量修改表级 Collation
ALTER TABLE inv_stock_transaction CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_0900_ai_ci;
ALTER TABLE eqp_equipment CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_0900_ai_ci;
ALTER TABLE prd_schedule CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_0900_ai_ci;
-- ... 对其余 17 张表执行同样操作
-- 建议生成迁移脚本批量执行
P1 3. 状态码体系不统一 — 4 种方案并存

status 字段在 87 张表中出现 40+ 次，但使用了 4 种完全不同的编码方案，给前后端对接和状态机维护带来巨大认知负担。

-- 用于: sys_user, sys_department, sys_role, sys_menu, sys_dict_type, 
--       sys_dict_data, inv_material, inv_warehouse, crm_customer, 
--       pur_supplier_material, inv_material_category, prd_bom_std

修复方案

统一策略：新表统一使用 VARCHAR 字符串状态（可读性最佳，适合 ERP 领域）。对存量表采用渐进式迁移：先添加新 VARCHAR 列，同步写入，最后切换列名。不建议直接修改 TINYINT 列值，因为前后端、报表、历史数据都依赖现有编码。

P2 5. DECIMAL 精度不统一 — 财务计算精度风险

数据库中有 4 种不同的 DECIMAL 精度方案，跨模块金额计算时存在精度丢失风险。

精度方案使用模块表数量问题
DECIMAL(18,4) CRM/INV/PRD/SAL (大多数) ~40+ 4 位小数，适合印刷行业精确计量
DECIMAL(14,2) PUR (采购金额) ~6 仅 2 位小数，与 18,4 混算时精度丢失
DECIMAL(15,3) INV (入库单) ~3 3 位小数，与 18,4 和 14,2 都不一致
DECIMAL(18,2) FIN (会计凭证/余额) ~12 会计模块仅保留 2 位小数
影响场景 采购明细表 pur_purchase_order_line 的 amount DECIMAL(14,2) 与销售明细表 sal_sales_order_detail 的 amount DECIMAL(18,4) 在跨模块汇总（如利润分析）时，14,2 会截断后 2 位小数。而财务模块 fin_voucher_line.debit_amount DECIMAL(18,2) 在与库存成本 DECIMAL(18,4) 对账时，也存在精度差异。
修复方案

统一为 DECIMAL(18,4)（印刷行业需要 4 位小数精度，如公斤/平方米换算）：

ALTER TABLE pur_purchase_order 
  MODIFY total_amount DECIMAL(18,4) DEFAULT 0 COMMENT '订单总金额';
ALTER TABLE pur_purchase_order_line 
  MODIFY amount DECIMAL(18,4) DEFAULT 0 COMMENT '金额';
-- 注意: 扩精度是安全的，截断才需要谨慎

P0 7. 硬编码密码暴露 — 安全漏洞

在源码中直接暴露了数据库明文密码，这是最高优先级的安全问题。

7.1 src/lib/db/mysql.ts（已废弃文件）

/**
 * @deprecated 此文件已废弃，请使用 ./index.ts 中的数据库配置
 *
 * 统一数据库配置已移至 index.ts：
 * - 统一环境变量: MYSQL_HOST, MYSQL_USER, MYSQL_PASSWORD, MYSQL_DATABASE
 * - 默认密码: Snqig521223          ← 明文密码暴露！
 * - 默认数据库: vnerpdacahng
 */
export * from './index';
风险 该文件虽然标记为 @deprecated，但仍存在于代码库中。任何有代码访问权限的人（包括公开仓库的访问者）都能看到数据库密码。如果该密码在生产环境中使用，将构成严重安全漏洞。

7.2 database/vnerpdacahng_schema.sql（种子数据）

INSERT INTO sys_dict_data (dict_type_id, dict_label, dict_value, sort_order, status) VALUES
('默认密码', 'sys.default.password', '123456', 1, '用户默认密码');  ← 明文密码写入字典表
风险 默认密码 123456 存储在数据库字典表中且未被加密。新用户首次登录使用此密码，如果未强制修改，则构成安全漏洞。
修复方案
  1. 立即删除 mysql.ts 中的密码注释，或更好：直接删除该废弃文件
  2. 确保 index.ts 中 password: process.env.DB_PASSWORD || '' 不含硬编码默认值
  3. 将种子数据中的默认密码改为 bcrypt 哈希值
  4. 确保 sys_user.first_login = true 强制首次登录修改密码
  5. 轮换数据库密码（已暴露的密码应视为已泄露）
  6. 将密码相关配置移入 .env.local 并确保 .gitignore 覆盖
P2 8. Drizzle ORM 覆盖率极低 — 85/87 表使用原生 SQL

项目引入了 Drizzle ORM，但仅定义了 2 张表的 schema（inv_inbound_order 和 inv_inbound_item），其余 85 张表全部通过 pool.query() 或 pool.execute() 执行原生 SQL。

// src/lib/db/schema.ts — 仅 2 张表
export const invInboundOrders = mysqlTable('inv_inbound_order', { ... });
export const invInboundItems = mysqlTable('inv_inbound_item', { ... });
// ... 其余 85 张表呢？
影响 原生 SQL 查询无编译期类型检查，列名拼写错误、类型不匹配只能在运行时发现。Drizzle 的类型推导能力未被利用，开发体验和安全性均打折扣。但考虑到 87 张表的迁移成本，建议新功能优先使用 Drizzle，存量代码按模块逐步迁移。
P2 9. 软删除字段定义不一致

几乎所有业务表都有 deleted 字段，但定义方式有细微差异：

定义表数问题

deleted TINYINT NOT NULL DEFAULT 0~5NOT NULL 约束，与其他不一致

影响 轻微差异不影响功能，但建议统一为 deleted TINYINT NOT NULL DEFAULT 0（NOT NULL 更安全，避免 NULL 歧义）。
