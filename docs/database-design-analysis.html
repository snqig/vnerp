三、设计模式与约定

3.1 模块前缀命名

所有表名使用 模块前缀_业务名 格式（如 inv_inventory_batch），便于按域识别归属。少数历史遗留表未遵循（如 material_requisitions、work_order_costs）。

3.2 软删除

绝大多数业务表包含 deleted TINYINT DEFAULT 0 字段，采用应用层软删除而非物理删除。inv_inventory_log（库存流水）明确标注"不可修改、不可删除"，满足审计要求。

3.3 审计字段

标准审计四件套：create_time、update_time（含 ON UPDATE CURRENT_TIMESTAMP）、create_by、update_by。部分日志表（如 sys_operation_log、sys_login_log）仅保留 create_time。

3.4 乐观锁

高并发写入表使用 version INT DEFAULT 0 乐观锁版本号：出库单、调拨单、盘点单、退货单、库存调整单、库存批次表等。代码层配合 transactionWithRetry() 实现自动重试（指数退避）。

3.5 冗余字段

大量明细表存储主表冗余字段以优化查询性能：

inv_inventory_batch: material_code, material_name（冗余自 inv_material） sal_delivery_detail: material_code, material_name, material_spec（冗余自 inv_material） sal_return: order_no, customer_name, delivery_no（冗余自关联主表） inv_outbound_order: warehouse_code, warehouse_name, customer_name（冗余自关联主表）

3.6 事件驱动架构（Outbox 模式）

domain_event_outbox + sys_event_processed 实现 Outbox 模式：

  • 事件状态机：pending → dispatching → processed（失败时 failed）
  • 指数退避重试：next_execute_at 字段控制（1s/3s/9s，最大 3 次）
  • 幂等保障：uk_event_handler (event_id, handler_name) 联合唯一键
  • 聚合根溯源：idx_aggregate (aggregate_type, aggregate_id)

3.7 金额精度

财务相关字段统一使用 DECIMAL(18,4) 或 DECIMAL(14,2)，采购模块使用 DECIMAL(14,3) 数量精度。汇率字段 DECIMAL(10,4)。总账模块使用 DECIMAL(18,2)。

3.8 权限模型

经典 RBAC 模型：sys_user → sys_user_role → sys_role → sys_role_menu → sys_menu。菜单支持三级类型（目录/菜单/按钮），角色支持数据范围控制（全部/本部门/本部门及下级/仅本人）。

五、问题诊断与改进建议

P0 · 已修复 外键约束补充完成（3 → 19）

通过 Migration 017 + 018 已将外键约束从 3 个提升至 19 个，覆盖销售、库存、生产、财务、采购 5 大模块的核心父子关系。

新增外键清单（16 个）：

  • 销售模块（8 个）：sal_order→crm_customer, sal_delivery→sal_order, sal_delivery_detail→sal_delivery, sal_delivery_detail→inv_material, sal_return_order→sal_order, sal_return_order_item→sal_return_order（表名按实际 DB）
  • 库存模块（6 个）：inv_inventory→inv_material, inv_inventory→inv_warehouse, inv_inventory_batch→inv_material, inv_inventory_batch→inv_warehouse, inv_outbound_order→inv_warehouse（外加 Migration 009 的出入库单→明细）
  • 生产模块（3 个）：prd_work_order→sal_order, prd_work_order→inv_material, prd_work_order_color_seq→prd_work_order
  • 财务模块（2 个）：fin_receivable→crm_customer, fin_payable→pur_supplier

遗留项： inv_outbound_order 缺少 customer_id 和 work_order_id 列（schema.sql 与实际 DB 不一致，需 P0-2 基线重置时补列）；inv_inbound_order.warehouse_id 类型为 INT（与 inv_warehouse.id BIGINT UNSIGNED 不匹配，需后续迁移对齐）。

P0 · 安全 Schema 文件包含明文密码

vnerpdacahng_schema.sql 头部注释包含明文数据库密码：Snqig521223。src/lib/db/mysql.ts 同样在注释中暴露密码。

建议： 立即从 SQL 文件和源码注释中移除密码，使用 .env 文件管理敏感配置，确保 .gitignore 已排除。

P1 · 一致性 排序规则（Collation）不统一

三套排序规则混用：

  • utf8mb4_0900_ai_ci — 系统管理、客户、采购、销售、库存出入库等（多数）
  • utf8mb4_unicode_ci — 设备、排程、领退料、标准主档、总账
  • utf8mb4_ai_ci — 库存流水表 inv_inventory_log

影响： 跨表 JOIN 时可能出现排序规则冲突错误（Illegal mix of collations），需要额外 COLLATE 子句。

建议： 统一为 utf8mb4_0900_ai_ci（MySQL 8.0 默认推荐），通过迁移脚本逐步 ALTER TABLE ... CONVERT TO。

P1 · 一致性 状态码体系不统一

不同模块使用不同的状态码表示方式：

模块状态字段类型示例
采购单TINYINT10/20/30/40/50/90（两位码）
销售订单TINYINT1/2/3/4/5（一位码）
入库单VARCHAR'pending'/'approved'/'completed'
出库单VARCHAR'pending'/'approved'/'completed'
设备VARCHAR'available'/'maintenance'/'offline'
排程TINYINT1/2/3/4/5

建议： 制定统一状态码规范（建议 TINYINT + 状态枚举类），后续迭代逐步对齐。

P1 · 一致性 主键数据类型不统一

同一逻辑外键在不同表中使用不同类型：

  • inv_inbound_order.warehouse_id 为 INT，但 inv_warehouse.id 为 BIGINT UNSIGNED
  • pur_purchase_order.id 为 INT UNSIGNED，而 crm_customer.id 为 BIGINT UNSIGNED
  • fin_account.id 为 INT UNSIGNED，而 fin_voucher_line.voucher_id 为 BIGINT UNSIGNED

建议： 统一主键为 BIGINT UNSIGNED，新表遵循此标准，存量表在重大版本迁移时对齐。

P1 · 架构 ORM 覆盖率极低

Drizzle ORM 的 schema.ts 仅定义 2 张表（inv_inbound_order、inv_inbound_item），87 张表中覆盖率不足 3%。绝大多数查询走原生 SQL（db.query() / db.execute()）。

影响： 丧失类型安全查询构建能力；ORM 与原生 SQL 混用增加维护复杂度。

建议： 明确技术路线——要么逐步补全 Drizzle schema（推荐），要么移除 Drizzle 依赖统一走原生 SQL + 类型生成。

P2 · 维护 命名规范不一致

少数表未遵循模块前缀命名：

  • material_requisitions / material_returns — 应为 inv_requisition / inv_return 或 mat_requisition
  • work_order_costs — 应为 prd_work_order_cost
  • material_batch_costs — 应为 inv_batch_cost

P2 · 性能 部分明细表缺少合适索引

尽管 Migration 009 大幅补充了索引，仍有部分场景可优化：

  • sal_reconciliation_line 的 source_type + source_id 组合索引已有，但缺少 reconciliation_id + source_type 复合索引
  • inv_inventory_batch 的 idx_material_warehouse_status 复合索引设计良好，可作为其他表参考
  • 日志表（sys_operation_log、sys_login_log）可考虑按时间分区

P2 · 设计 标准卡表字段过于宽泛

prd_standard_card 表包含 60+ 列，大量字段为 VARCHAR(20)~VARCHAR(200)，包含印刷行业特有工艺参数（跳距、膜编号、背胶、麦拉等）。许多数值型参数以字符串存储（如 jump_distance VARCHAR(20)），不利于范围查询和计算。

建议： 考虑将工艺参数拆分为子表或使用 JSON 列，减少主表宽度。

七、改进路线建议

短期（1-2 周）

  1. 清除密码泄露 — 从 SQL 文件和源码注释中移除明文密码
  2. 统一排序规则 — 编写迁移脚本将所有表 CONVERT TO utf8mb4_0900_ai_ci
  3. 清理废弃目录 — 确认 migrations/ 和 drizzle/ 不再使用后归档或删除

中期（1-2 月）

  1. 补充核心外键 — 优先处理父子关系最强的表对（订单→明细、工单→BOM、凭证→明细），每批 5-8 个，测试验证后上线
  2. 统一状态码体系 — 制定 TINYINT 状态码规范文档，新表强制遵守，存量表按模块逐步迁移
  3. 补全 ORM schema — 将活跃查询的表补入 Drizzle schema.ts，逐步替换原生 SQL
  4. 统一主键类型 — 新表统一 BIGINT UNSIGNED，存量表在重大版本时对齐

长期（3-6 月）

  1. 标准卡表重构 — 拆分 60+ 列的 prd_standard_card，将工艺参数移入子表或 JSON 列
  2. 日志表分区 — 对 sys_operation_log、sys_login_log、inv_inventory_log 按月分区
  3. 命名规范统一 — 将 material_*、work_order_* 表重命名至模块前缀规范
  4. 数据归档策略 — 建立历史数据归档机制，控制核心表数据量
