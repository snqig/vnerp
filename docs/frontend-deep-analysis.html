4. P0 级问题（立即修复）

P0 1. TypeScript `any` 类型滥用 — 2,772 处，覆盖 436 个文件（53%）

违反规范: 「禁止 any，能用 unknown 替代」「函数参数与返回值必须标注类型」

影响: TypeScript strict 模式形同虚设，编译期类型保护被绕过，运行时错误风险极高。

Top 10 重灾区文件:

文件any 数量说明
app/api/dashboard/ceo/route.ts51CEO 仪表盘 API
app/api/dashboard/kpi/route.ts47KPI 仪表盘 API
app/api/init/core-flow-seed/route.ts39种子数据初始化
app/api/init/full-business-seed/route.ts39业务种子数据
lib/standard-card-service.ts29标准卡核心服务
application/workflow/WorkflowEngine.ts27工作流引擎
lib/mrp-engine.ts24MRP 物料需求计划
lib/fifo-allocation.ts20FIFO 分配算法
lib/audit-logger.ts37审计日志
application/services/StandardCardApplicationService.ts31标准卡应用服务

典型反模式:

// src/lib/api-response.ts — 核心基础设施文件也在用 any export interface ApiResponse<T = any> { ... } export interface PaginatedResponse<T = any> extends ApiResponse { ... } export function withErrorHandler<T extends (...args: any[]) => Promise<NextResponse>>(...) export function validateRequestBody<T extends Record<string, any>>(...) // src/hooks/usePermission.ts — Hook 返回值无类型保护 function safeGetUser(): any | null { ... } const role = roleResult.data.find((r: any) => r.id === user.role_id);

修复建议:

  • 将所有 : any 替换为 : unknown + 类型守卫，或定义具体的 interface
  • API 响应统一泛型化：ApiResponse<T> 在调用处传入具体类型
  • 对 api-response.ts 等核心文件立即修复 — 这些被全项目引用
  • ESLint 添加 "@typescript-eslint/no-explicit-any": "error" 规则阻止新增
P0 2. API 路由鉴权覆盖率极低 — 317 个路由仅 2 个使用 withAuth（0.6%）

违反规范: 「接口必须做权限控制」「敏感信息不放在 localStorage」

影响: 315 个 API 路由可能未做认证检查。虽然 middleware 做了 CSRF 校验，但 JWT 验证不强制，存在未授权访问风险。

// src/middleware.ts — 中间件只做 CSRF，不做 JWT 认证 if (pathname.startsWith('/api/')) { if (requiresCsrfValidation(request)) { if (!validateCsrfToken(request)) { return NextResponse.json({ success: false, ... }, { status: 403 }); } } return NextResponse.next(); // ← 直接放行，未验证 JWT！ } // src/lib/api-auth.ts — withAuth 包装器存在，但几乎没人用 export function withAuth(handler, options?) { ... } // 全项目 317 个路由中仅 2 个文件 import 了 withAuth

修复建议:

  • 方案 A（推荐）: 在 middleware 中统一做 JWT 校验 — 所有 /api/ 路由（白名单除外）必须携带有效 Token
  • 方案 B: 强制所有 API route handler 使用 withAuth 包装器，ESLint 自定义规则检测裸露的 export async function GET/POST/PUT/DELETE
  • 白名单路由（/api/auth/login, /api/auth/register, /api/health）显式排除
P0 3. JWT Token 存储在 localStorage/sessionStorage — XSS 可窃取

违反规范: 「敏感信息不放在 localStorage」「禁止前端明文存储密钥/Token」

影响: 任何 XSS 漏洞都可直接读取 localStorage 中的 token 和 refreshToken，导致账号被盗。

// src/contexts/AuthContext.tsx const storage = rememberMe ? localStorage : sessionStorage; storage.setItem('token', result.data.token); storage.setItem('user', JSON.stringify(result.data.user)); storage.setItem('refreshToken', result.data.refreshToken); // ← refreshToken 也存了！ storage.setItem('userId', String(result.data.user.id));

修复建议:

  • 将 JWT 改为 httpOnly + Secure + SameSite=Strict Cookie 存储，前端 JS 无法读取
  • Refresh Token 同样存入 httpOnly Cookie
  • 前端只需要用户基本信息（非敏感），可保留在内存（Context state）中

违反规范: 「组件应小而专注，单一职责」「禁止嵌套过深（<3 层）」

P1 5. 错误边界几乎缺失 — 仅 1 个 error.tsx，无全局 not-found

违反规范: 应有完善的错误处理和用户引导

  • 全项目仅 src/app/[locale]/warehouse/error.tsx 1 个错误边界文件
  • 没有全局 error.tsx（src/app/[locale]/error.tsx）
  • 没有 not-found.tsx（404 页面）
  • 只有 1 个 loading.tsx — 140 个页面几乎无路由级加载态
  • 0 处 ErrorBoundary 组件使用

修复建议:

  • 添加 src/app/[locale]/error.tsx 全局错误边界
  • 添加 src/app/[locale]/not-found.tsx 404 页面
  • 为关键路由添加 loading.tsx + Skeleton 骨架屏
  • 关键组件包裹 <Suspense fallback={...}>
P1 6. 330 处行内样式（style={{}}） — 违反样式分离原则

违反规范: 「禁止行内样式」「颜色使用变量，禁止硬编码」

330 处 style={{}} 分散在 240 个 TSX 文件中。项目使用 Tailwind CSS v4，行内样式应完全用 Tailwind 类名替代。

// 反模式 — 行内硬编码样式 <div style={{ display: 'flex', gap: '8px', padding: '16px' }}> // 正确 — 使用 Tailwind 类名 <div className="flex gap-2 p-4">

修复建议:

  • 将所有 style={{}} 逐批替换为 Tailwind 等效类名
  • 动态样式使用 cn() + 条件类名（项目已有 tailwind-merge）
  • ESLint 添加 react/forbid-component-props 规则禁止 style 属性

P1 8. 51 处 console.log/error 残留生产代码

违反规范: 「禁止 console（生产环境）」

项目已有 src/lib/logger.ts 统一日志模块，但仍有 51 处直接使用 console.log/error。

// 反模式 — 直接 console console.error('登录失败:', error); // login/route.ts:384 console.error('[AuthContext] 菜单API返回401'); // AuthContext.tsx:99 // 正确 — 使用统一 logger logger.error(ctx, '登录失败', error);

P1 9. 缺少标准目录 — constants / types / utils / store 均不存在

违反规范: 「项目目录应包含 constants/ utils/ 等标准目录」

规范要求目录是否存在现状
constants/❌ 缺失常量散落在各组件内，如 usePermission.ts 中硬编码 permissionModules
types/❌ 缺失类型定义散落在各文件，无全局类型
utils/⚠️ 用 lib/ 替代功能等价但命名不统一
store/❌ 缺失无状态管理方案，全靠 Context

修复建议:

  • 创建 src/constants/ 目录，提取所有硬编码常量（状态码、权限模块、分页大小等）
  • 创建 src/types/ 目录，集中管理跨模块共享的 interface/type
  • 对于复杂状态管理，考虑引入 Zustand 或 Jotai 替代纯 Context

6. P2 级问题（低优先级改进）

P2 10. 测试覆盖率严重不足 — 36 测试文件 / 819 源文件（4.4%）

项目配置了 Playwright（E2E）和 Vitest（单元测试），但测试文件极少。核心业务逻辑（MRP 引擎、FIFO 分配、财务核心、工作流引擎）几乎没有测试覆盖。

修复建议:

  • 优先为核心 lib/ 模块补单元测试：mrp-engine, fifo-allocation, finance-core, warehouse-core
  • 为关键 API 路由补集成测试：auth/login, warehouse/inbound, orders
  • 目标：核心模块 80%+ 覆盖率，API 路由 50%+ 覆盖率
P2 11. React StrictMode 已关闭 — 错失潜在 bug 检测
// next.config.ts reactStrictMode: false, // ← 应为 true

关闭 StrictMode 意味着不会检测：不安全的副作用、过时的 API 使用、意外的 state 变化等问题。

修复建议: 设为 true，修复暴露出的潜在问题。

P2 12. 开发环境回退密钥安全性不足
// src/app/api/auth/login/route.ts const SECRET_KEY = JWT_SECRET || 'dev-only-secret-key'; // src/lib/env.ts JWT_SECRET: process.env.JWT_SECRET || 'dev-only-insecure-secret-32chars',

虽然生产环境会 fail-fast，但开发环境的回退密钥仍然可被利用。建议开发环境也强制要求 JWT_SECRET。

P2 13. 18 处硬编码 URL/IP 地址

源代码中有 18 处硬编码的 http://, https://, localhost, 127.0.0.1。应通过环境变量管理。

P2 14. 5 处 dangerouslySetInnerHTML 使用

虽然有 sanitizeInput 函数做 XSS 防护，但 dangerouslySetInnerHTML 仍是高风险点。需逐一确认是否对内容做了充分的 HTML 转义。

9. 总结

这个 ERP 项目在架构设计上做得很好 — DDD 分层清晰、安全 Headers 完整、i18n 国际化到位、工具链（ESLint + Prettier + Husky + Commitlint）齐全。这些是很多项目缺失的基础设施。

但在执行细节上存在显著问题。最核心的三个问题是：

  1. any 类型泛滥（2,772 处）— TypeScript strict 模式形同虚设
  2. API 鉴权几乎缺失（0.6% 覆盖率）— 315 个路由可能未授权访问
  3. 超大组件文件（最大 3,000 行）— 可维护性极差

建议按优先级从安全加固入手，逐步解决类型安全和组件拆分问题。项目基础好，修复后可以达到 8+ 的水平。
