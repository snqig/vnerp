一、严重问题 P0 · 立即修复

以下问题可被直接利用造成数据泄露或系统破坏，或严重阻碍项目可维护性，必须在上线前修复。

1. 未鉴权的远程命令执行 (RCE) + 命令注入 安全

文件: src/app/api/project-files/upload/route.ts
影响: 攻击者无需登录即可通过 commitMsg 注入 "; rm -rf / # 实现任意命令执行，同时 git add . 会将含密码的 .env 提交进仓库。
execSync(`git commit -m "${commitMsg || 'Initial commit'}"`, { cwd: PROJECT_ROOT });
execSync('git add .', { cwd: PROJECT_ROOT });
修复建议: 立即删除该路由（或迁移至 scripts/ 脚本），严禁在生产环境暴露 git 操作 API。如必须保留，需添加鉴权 + 使用参数化方式调用 git。

2. 未鉴权的文件系统信息泄露 安全

文件: src/app/api/project-files/route.ts
影响: GET 递归遍历 PROJECT_ROOT 全部文件并返回路径/大小，无任何鉴权。攻击者可枚举源码结构、定位配置文件。
修复建议: 删除该路由或添加严格鉴权，生产环境不应暴露文件系统遍历接口。

3. SQL 注入 (字符串拼接) 安全

文件: src/application/handlers/StandardCardHandlers.ts:200-211
影响: 直接模板字符串插值拼接 SQL，未参数化。materialId 若上游校验不严即可注入。
SELECT id FROM prd_work_order WHERE material_id = ${materialId} ...
UPDATE prd_work_order SET standard_card_id = ${standardCardId} ... WHERE id = ${wo.id}
修复建议: 改为参数化查询 query(sql, [materialId, standardCardId, wo.id])，并全局审计所有 SQL 拼接。

4. Loose = any 类型黑洞 (1911 处) 类型安全

文件: src/types/loose.d.ts · 影响范围: 300+ 文件
影响: type Loose = any 被 1911 处使用，等价于全局禁用类型检查。配合 ESLint 关闭 no-unsafe-* 规则，整个项目的类型安全形同虚设。
type Loose = any;      // src/types/loose.d.ts
type LooseArray = any[];
// 1911 处: const rows: Loose = await query(...)  ← 返回 any[]
修复建议: 分步替换——先为 query() 调用补全泛型 query<MyRow>(sql)，再逐步删除 Loose 定义，最后重新启用 ESLint no-unsafe 规则。

5. 218 个空 catch 块静默吞掉错误 代码质量

集中区域: src/app/api/dashboard/ · src/app/api/dcprint/ink-consumption/
影响: 数据库查询失败时返回空数据但 HTTP 仍 200 成功，前端无法感知数据缺失原因，生产环境故障将完全不可见。
try {
  const rows = await query(`SELECT COUNT(*) ...`);
  overview.totalItems = Number(rows[0].total || 0);
} catch {}  // ← 错误被完全忽略，overview 保持全 0
修复建议: 统一错误处理策略——至少 catch (e) { logger.error('...', e) }，关键查询应向上抛出或返回错误状态码。

二、中等问题 P1 · 尽快处理

影响代码质量、性能与可维护性，建议在迭代中逐步解决。

6. 中间件不校验 API 身份，鉴权依赖各路由自觉 安全

文件: src/middleware.ts:111-121
影响: 中间件对 /api/ 仅做 CSRF 校验后放行，不在边缘层验证 JWT。开发者遗漏 withPermission 即导致裸奔路由（问题 1/2 即为实例）。BLOCKED_ROUTES 仅生产环境拦截，开发/预发全部暴露。
修复建议: 在中间件对 /api/ 统一校验 access_token，危险路由（init/diagnose）改为构建时排除。

7. Drizzle ORM 名存实亡，253 个路由用原始 SQL 架构

覆盖: 0 个路由使用 Drizzle 查询构建器 · 253 个路由调用 query()/execute() 拼接 SQL
影响: schema.ts 仅定义 36 张表（实际 164 张），ORM 类型安全与迁移可追溯能力完全浪费。74 个路由手动拼接 WHERE 子句。
修复建议: 统一收敛到 Drizzle 查询构建器，或正式放弃 Drizzle 改为声明式数据访问层（DAL）。

8. DDD 与传统架构并行，迁移未完成 架构

并存目录: src/domain/ + src/application/ + src/infrastructure/ ‖ src/lib/services/ + src/services/
影响: 仅 52/349 路由使用 @/application，服务类散落 4 处，边界混乱。半截子 DDD 重构增加认知负担。
修复建议: 明确一种范式并完成迁移，删除孤儿目录。如选 DDD 则将 lib/services 迁入 application/services。

9. N+1 查询普遍存在 (80 个文件) 性能

典型: src/infrastructure/repositories/MysqlVoucherRepository.ts · src/application/services/DashboardDataService.ts
影响: 循环内逐条查子表，N 条主记录触发 N 次查询，数据量增长后响应时间线性恶化。
修复建议: 改用 WHERE id IN (...) 批量查询后内存 join。

10. 数据库主键类型不统一 (BIGINT vs INT) 数据库

147 表 BIGINT PK · 16 表 INT PK（含 inv_inbound_order, pur_purchase_order, prd_die 等核心表）
影响: 跨类型 JOIN 触发隐式转换导致索引失效；数据量超 21 亿时 INT 溢出。如 inv_inbound_item.order_id 是 bigint 但主表 inv_inbound_order.id 是 int。
修复建议: 统一为 BIGINT UNSIGNED，分批 ALTER TABLE 对齐。

11. 状态码三体系并存 数据库

tinyint 75 个 (1-5 体系) · int 5 个 (10-90 体系) · varchar 7 个 ('pending' 等)
影响: 同一系统内数字与字符串状态码混用，前端需为每个模块维护独立映射，跨模块状态筛选无法统一。
修复建议: 全量统一为 tinyint + 字典表 sys_dict_data。

12. xlsx 依赖存在已知 CVE 漏洞 工程化

package.json: "xlsx": "^0.18.5" · CVE-2023-30533 (原型污染) + CVE-2024-22363 (ReDoS)
影响: npm 仓库已撤包，存在可被利用的安全漏洞。
修复建议: 迁移至 @e965/xlsx 或官方修复版。

13. 测试覆盖率不足 18% 工程化

349 个 API 路由 · 仅 61 个测试文件 · hr/crm/reports/audit/dashboard 等 10+ 模块零测试
影响: 核心业务域缺乏回归保障，重构风险高。vitest.config 未匹配 .tsx、未启用 jsdom、无 coverage 阈值。
修复建议: include 改为 ['tests/**/*.test.{ts,tsx}']；按需设置 jsdom 环境；添加 coverage 阈值。

14. 'use client' 滥用，丧失 SSR 优势 性能

250+ 文件标注 'use client'，几乎所有列表/详情页均为客户端组件
影响: 数据在客户端 fetch，完全丧失 Next.js 16 服务端组件的流式 SSR 与数据预取优势，首屏加载慢。
修复建议: 列表/详情页改为服务端组件 + fetch + cache，仅交互部分下沉为客户端子组件。

15. 监控配置不完整 工程化

alertmanager.yml 使用 ${SMTP_USER} 变量插值（Alertmanager 原生不支持）· Grafana 无仪表盘 · 无 /metrics 端点
影响: 邮件告警将失败；Grafana 启动后是空白实例；无法观测请求延迟、错误率、事件积压等业务指标。
修复建议: 用 envsubst 生成 alertmanager 配置；补充 Grafana 仪表盘 JSON；集成 prom-client 暴露 /metrics。

16. 46 个 init/diagnose 路由暴露在生产 API 中 架构

如 src/app/api/init/full-seed/route.ts (3616 行) · src/app/api/diagnose/all-tables/route.ts
影响: 应为 CLI 脚本的生产逻辑暴露为 API，init/full-seed 单文件 3616 行。40% 路由无 try/catch。
修复建议: 迁移到 scripts/ 目录，加路由级错误兜底中间件。
