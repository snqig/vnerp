## 1. 模块结构分析

模块采用 **领域驱动设计（DDD）** 架构，目标是实现与印刷行业业务高度匹配的仓库管理能力。目前结构如下：

```
src/
├── domain/warehouse/                  # 领域层（核心业务规则）
│   ├── aggregates/
│   │   └── InboundOrder.ts            # 入库单聚合根
│   ├── entities/
│   │   └── InboundItem.ts
│   ├── events/
│   ├── repositories/                  # 仓储接口
│   └── value-objects/
├── application/                       # 应用服务
│   └── services/
│       └── InboundApplicationService.ts
├── infrastructure/                    # 基础设施（仓储实现、事件总线等）
├── app/api/warehouse/                 # API 路由（表现层）
└── lib/
    ├── fifo-allocation.ts              # FIFO 核心算法
    ├── state-machine.ts                # 状态机逻辑
    ├── db/                             # 数据库操作
    └── ...                             # 其他工具模块
```

### 1.1 各层级职责简述
| 层 | 功能 | 体现 |
|----|------|------|
| **Domain层** | 实现业务规则和模型（聚合根、实体、值对象） | `InboundOrder` 为完整聚合根，其他单据模型尚未完善 |
| **Application层** | 业务流程编排 | 仅 `InboundApplicationService` 有应用服务实现 |
| **Infrastructure层** | 数据库、事件总线、文件存储等基础设施 | 基础能力，缺少稳定抽象层 |
| **lib层（通用库）** | 核心算法和工具 | `fifo-allocation.ts` 为核心逻辑，但尚未充分领域化 |

## 2. 功能模块现状

### 2.1 入库管理
- **支持**：采购入库、分切入库、生产入库
- **架构体现**：`InboundOrder` 作为聚合根，支持状态机和事件驱动
- **风险**：接口层和事件处理应更紧密配合，Domain 层未完全覆盖全部流程

### 2.2 出库管理
- **支持**：销售出库、生产领料
- **现状**：基础CRUD实现，但缺乏完整的领域模型和领域事件
- **风险**：业务流程缺乏闭环性，如状态机、审计、权限、审批等能力缺失

### 2.3 库存管理
- **支持**：汇总库存（`inv_inventory`）、批次库存（`inv_inventory_batch`）
- **当前问题**：
 - 无库存流水台账，无法进行审计、报表
 - 现存量与可用量未区分，易超卖
 - 无审计与操作日志

### 2.4 调拨与盘点
- **现状**：仅有入口页，流程不完整
- **风险**：无事务一致性保障，无在途管理，无法循环盘点

### 2.5 库存调整
- **支持**：直接修改库存
- **问题**：缺少审批流和审计能力，可能造成数据混乱

### 2.6 FIFO 成本分配
- **实施**：核心算法实现（`lib/fifo-allocation.ts`）
- **优点**：基于 `Decimal.js` 精准计算，具备幂等与并发校验
- **风险**：
 - 累加 Bug（未重新赋值）
 - 未分页支持，可能导致性能问题
 - 并发操作未充分处理，部分场景可能冲突

### 2.7 标签与批次追踪
- **实现**：支持 QR 码生成、批次查询入口
- **待完善**：未能打通全链路批次追踪系统（与生产、销售打通）

## 3. 深度评估与问题识别

### 3.1 P0 级核心高风险问题（直接影响库存数据准确性）

| 问题 | 描述 | 影响 |
|------|------|------|
| 🔴 **无库存流水台账** | 没有记录每个增减操作，包括来源单据、操作人、方向（入库/出库）、数量 | 无法追责、无法生成进销存报表、审计失败 |
| 🔴 **负库存无强制管控** | 未进行出库前可用量校验，无配置开关允许负库存 | 导致数据失真、生产、发货中断 |
| 🔴 **库存更新无幂等与并发保护** | SQL无乐观锁、事件驱动未处理重试 | 重复操作导致超卖、并发冲突 |
| 🔴 **现存量与可用量不分** | 库存表仅有数量字段 | 多单据同时消耗同一库存，实际执行时才发现不足 |

### 3.2 P1 级业务流程缺陷

| 问题 | 影响 |
|------|------|
| 🟡 **出入库业务类型不全** | 缺少销售退货、采购退货、其他入库（盘盈/赠品）、其他出库（报废/样品/损耗）等完整业务类型 |
| 🟡 **盘点流程缺失** | 无完整的“创建→录入→差异→审核→调整”流程，不支持循环盘点 |
| 🟡 **调拨无事务一致性** | 调出扣减与调入增加未绑定事务，无在途管理 |
| 🟡 **批次属性缺失** | 缺少生产日期、有效期、质检状态等关键属性，FIFO 与出库未强绑定 | 效期预警无法落地 |
| 🟡 **期初库存初始化机制缺失** | 上线初期仅靠调整录入，无规范流程 | 基线数据易错 |

### 3.3 印刷行业适配性不足（行业差异化短板）

| 痛点 | 现状 | 风险 |
|------|------|------|
| ⚠️ **分切业务入口不完整** | 仅存在“分切入库”页面，缺少原卷出库→分切损耗→成品拆分→余料回库完整流程 | 仍需手工统计分切损耗和余料，失去行业竞争力 |
| ⚠️ **油墨仓储未实现垂直化** | 缺少开盖有效期、色号、配方属性 | 无法精细化管理损耗、成本核算 |
| ⚠️ **余料无正规回库流程** | 未建立纸张、油墨余料的入库和领用机制 | 成为账外循环，库存管理失效 |
| ⚠️ **全链路批次追踪未打通** | 仓储端未与生产、销售端批次关联 | 无法实现质量追溯，不满足行业规范 |

### 3.4 管控与审计能力缺口（P2 级）

| 问题 | 影响 |
|------|------|
| 🟠 **单据审批机制不完善** | 入库单有状态，其他单据如出库、调拨、盘点等无“制单-审核”分离 |
| 🟠 **无仓库级数据权限隔离** | 所有管理员可操作所有仓库，无部门级权限 |
| 🟠 **库存调整机制不规范** | 缺少审批流程、原因分类、操作审计 |
| 🟠 **无库存预警机制** | 未实现上下限、效期、呆滞库存预警 | 导致异常库存管理缺失 |

### 3.5 架构与性能隐患

| 问题 | 描述 |
|------|------|
| ⚠️ **库存聚合根缺失** | 入库单是唯一完整聚合根，其他单据未实现领域模型 | 地域内聚性差，不利于扩展 |
| ⚠️ **出库/调拨/盘点无领域模型** | 依赖原始数据表操作，计算逻辑分散 | 维护成本高，缺乏统一业务规则 |
| ⚠️ **库位数据模型缺失** | 仅管理仓库，未能支持库区/货架/货位 | 多仓库管理困难，精细化存取不支持 |
| ⚠️ **无库存结存快照** | 历史记录需要逐笔累加 | 数据量增大时查询性能下滑 |
| ⚠️ **算法与基础设施耦合** | 依赖直接的 SQL 查询，未抽离为通用算法 | 难以进行跨业务复用与替换 |

## 9. 项目进阶建议

### 9.1 架构层重构建议

- 将 `FIFO` 等算法封装为 **`domain/warehouse/services`** 下的领域服务，如 `FIFOAllocator.ts`
- 引入统一的持久化抽象层（如 `IInventoryRepository`）
- 对库存业务，设计 `Inventory` 作为聚合根
- 对 `OutboundOrder`、`TransferOrder` 等数据，引入 `IOutboundRepository` 等嗅觉层接口

### 9.2 业务增强建议

- **分切业务**：对外部系统（如生产排程）做对接，建立 `CuttingOrder` 或 `RemainingStock` 单据
- **油墨管理**：建立 `InkStock` 实体，封装 `opened_at`、`remaining_weight`、`color_code`、`color_formula_id`、`shelf_life` 等属性
- **库位系统**：建立分层存储模型：`warehouse`, `zone`, `rack`, `position`
- **库存快照** | 通过定时任务生成每日/月度库存快照，支持 `inv_inventory_snapshot` 表 | 提高历史查询性能 |
- **预警系统** | 引入预警规则（如 `if available_qty < safety_stock`），通过 `checkShortageAndWarn()` 实现 | 提高风控能力 |

### 9.3 测试与 CI 建议

- 确保 `Vitest` 覆盖率 ≥ 80%（`lib/` 和 `domain/` 层）
- 引入 **Testcontainers** 实现测试数据库（或 **Local DB**），覆盖跨层调用测试
- 增加行为测试（如异常出库、并发扣减、审批回退等）
- 使用 Playwright 增强 E2E 测试，确保表现层功能正常调用
- 建立 localhost 环境下的 **测试数据库**，保障 MongoDB/MySQL 数据隔离

## 11. 总结

### ✅ **系统优势**：
- 实现了核心业务（入库）的完整 DDD 逻辑
- FIFO 成本分配逻辑精准
- 已有事件驱动设计基础
- 部分字段具备行业特性（如库存事务、QR 生成）

### ❌ **主要问题**：
- **库存数据完整性缺失**：缺少流水记录，无法支持审计与报表
- **并发与事务完整性低**：未实现真正的 Power-of-2 重试 + 乐观锁
- **批次与库存关联薄弱**：未打通生产、销售、调拨批号追溯
- **业务流程控制不足**：未建立审批、权限、警告等体系
- **扩展性差**：DB 耦合、算法未抽象、未封装逻辑

### 🚀 **新一代仓库系统架构方向**
- 领域驱动架构 + DDD 聚合根模式
- **库位模型**（Warehouse → Zone → Rack → Position）
- **库存快照系统**（提升历史查询性能）
- **领域服务封装算法**（如 FIFO、LIFO、按效期分配）
- **多仓库支持**（确保权限隔离）
- **业务流程完整化**（进货→库存→出货→调拨→盘点→预警）
