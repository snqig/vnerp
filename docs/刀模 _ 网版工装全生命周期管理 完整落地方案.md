## 一、核心业务规则

### 1.1 工装分类与基础定义

采用 **“单表继承 + 类型区分”** 设计，统一管理不同工装。

| 工装类型 | 类型编码     | 寿命计量单位 | 典型使用场景     |
| -------- | ------------ | ------------ | ---------------- |
| 刀模     | die          | 模切次数     | 后工序模切工单   |
| 网版     | screen_plate | 印刷次数     | 丝网印刷工序工单 |

### 1.2 全生命周期状态流转

统一数字枚举，流转规则封装在领域层：

| 枚举值 | 状态   | 说明                        | 可执行操作               |
| ------ | ------ | --------------------------- | ------------------------ |
| 1      | 待用   | 新制作/采购入库，未投入生产 | 领用、编辑、删除         |
| 2      | 在用   | 正常投入生产使用            | 报工累计次数、维修、报废 |
| 3      | 维修中 | 送修/保养中                 | 维修完成、报废           |
| 4      | 预警   | 使用次数达到预警阈值        | 正常使用、维修、报废     |
| 5      | 已报废 | 寿命耗尽/损坏报废，不可再用 | 查看、归档               |

**强制流转规则：**

- 只有「在用 / 预警」状态的工装可被生产工单引用，维修中、报废状态自动过滤。
- 使用次数达到预警阈值（默认额定寿命的 80%）自动进入预警状态；达到额定寿命自动锁定，需走维修或报废。
- 维修完成后可手动调整剩余寿命，调整记录全程留痕。
- 报废必须填写原因，已报废工装永久保留档案与历史记录，不可物理删除。

### 1.3 使用寿命追踪规则

- **自动累计**：生产报工审核通过后自动累加使用次数，一次报工对应一次使用累计，报工数量为 N 则累计 N 次。
- **两级预警机制**：
  - 黄色预警：剩余寿命 ≤ 额定寿命的 20%，系统推送提醒，列表标记黄色。
  - 红色预警：剩余寿命 ≤ 额定寿命的 5%，强制提醒，工单引用二次确认。
- **寿命调整留痕**：必须记录调整人、时间、原因、前后次数，全程可追溯。
- **使用记录全量归档**：每次使用关联对应工单、工序、操作人员，支持双向追溯。

### 1.4 工装成本分摊规则

**基础版（第一阶段）**：按额定寿命均摊法
单次分摊成本 = 工装采购/制作总成本 ÷ 额定总使用次数
工单工装成本 = 单次分摊成本 × 本次报工使用次数

**进阶版（第二阶段）**：纳入维修成本动态调整剩余寿命单位成本
剩余单位分摊成本 = (当前账面净值 + 本次维修费用) ÷ 维修后剩余寿命
账面净值 = 原值 - 累计已分摊成本

**分摊规则约束：**

- 成本在报工审核时自动计算并写入工单成本明细，无需人工核算。
- 历史工单分摊成本快照固化，后续调整不影响已生效数据。
- 提前报废的工装，剩余未分摊净值一次性计入当月制造费用。

### 1.5 维修与报废流程规则

- **维修**：发起申请 → 状态置为维修中 → 录入费用、内容、调整后剩余寿命 → 完成 → 状态恢复为在用/预警。
- **报废**：发起申请 → 校验无未完成工单关联 → 填写原因、残值 → 审核通过 → 状态置为已报废 → 剩余成本结转 → 从可用列表移除。

## 二、数据模型设计

> **Drizzle ORM 实现**，表名前缀 `dcprint_`，全量外键约束与审计字段。

### 2.1 工装主档案表 `dcprint_tool`

```ts
export const dcprintTool = mysqlTable('dcprint_tool', {
  id: bigint('id', { mode: 'number' }).primaryKey().autoincrement(),
  toolType: tinyint('tool_type').notNull(),          // 1-刀模 2-网版
  toolCode: varchar('tool_code', { length: 50 }).notNull().unique(),
  toolName: varchar('tool_name', { length: 100 }).notNull(),
  spec: varchar('spec', { length: 255 }),
  materialId: bigint('material_id', { mode: 'number' }),
  totalLife: int('total_life').notNull(),
  warningThreshold: int('warning_threshold').notNull(),
  usedCount: int('used_count').notNull().default(0),
  remainLife: int('remain_life').notNull(),
  originalCost: decimal('original_cost', { precision: 10, scale: 2 }).notNull(),
  accumulatedCost: decimal('accumulated_cost', { precision: 10, scale: 2 }).notNull().default(0),
  netValue: decimal('net_value', { precision: 10, scale: 2 }).notNull(),
  unitCost: decimal('unit_cost', { precision: 10, scale: 4 }).notNull(),
  status: tinyint('status').notNull().default(1),    // 1-待用 2-在用 3-维修中 4-预警 5-已报废
  manufactureDate: date('manufacture_date'),
  warehouseLocation: varchar('warehouse_location', { length: 100 }),
  remark: text('remark'),
  scrapReason: text('scrap_reason'),
  scrapTime: datetime('scrap_time'),
  scrapBy: bigint('scrap_by', { mode: 'number' }),
  createBy: bigint('create_by', { mode: 'number' }),
  createTime: datetime('create_time').notNull().defaultNow(),
  updateBy: bigint('update_by', { mode: 'number' }),
  updateTime: datetime('update_time').onUpdateNow(),
  isDeleted: tinyint('is_deleted').notNull().default(0),
}, (table) => ({
  typeStatusIdx: index('type_status_idx').on(table.toolType, table.status),
}));
```

### 2.2 工装使用记录表 `dcprint_tool_usage`

每次生产报工审核后自动写入一条使用记录，关联工单、工序、操作人员，支撑双向追溯与成本归集。

```ts
export const dcprintToolUsage = mysqlTable('dcprint_tool_usage', {
  id: bigint('id', { mode: 'number' }).primaryKey().autoincrement(),
  toolId: bigint('tool_id', { mode: 'number' }).notNull().references(() => dcprintTool.id),
```

## 三、DDD 分层落地实现

完全复用项目现有 DDD 分层架构，业务规则封装在领域层，应用层仅编排，基础设施层对接 Drizzle/MySQL。

### 3.1 领域层（`src/domain/dcprint/`）

**聚合根：`Tool`**，封装状态流转、寿命累计、成本分摊核心规则：

```ts
static create(props: ToolProps): Tool              // 新建工装（待用状态）
activate(operatorId: number): void                  // 投入使用（待用→在用）
recordUsage(useCount: number, unitCost: number): {   // 记录一次使用
  usedCountDelta: number;
  amortizedCost: number;
  newStatus: ToolStatus;                            // 可能触发预警状态切换
}
startMaintenance(): void                            // 在用/预警→维修中
completeMaintenance(cost: number, lifeAfter: number): void  // 维修完成，重算剩余寿命与单位成本
scrap(operatorId: number, reason: string, residualValue: number): void  // 报废
```

**值对象**：
- `ToolStatus`：状态枚举与流转校验（封装 1.2 节强制流转规则）
- `ToolType`：工装类型（die / screen_plate）
- `LifeMetric`：寿命计量（usedCount / remainLife / warningThreshold 一致性校验）

**领域事件**：
- `ToolWarningTriggeredEvent`：使用累计触发黄色/红色预警时发布
- `ToolScrappedEvent`：工装报废时发布，成本模块订阅结转剩余净值

**仓储接口**：

```ts
export interface IToolRepository {
  findById(id: bigint): Promise<Tool | null>;
  findByIdWithUsage(id: bigint): Promise<Tool | null>;
  findByType(toolType: ToolType, status?: ToolStatus): Promise<Tool[]>;
  save(tool: Tool): Promise<void>;
  saveUsageRecord(record: ToolUsageRecord): Promise<void>;
  saveMaintenanceRecord(record: ToolMaintenanceRecord): Promise<void>;
}
```

### 3.2 应用层（`src/application/services/ToolManagementService.ts`）

`ToolManagementService` 负责业务流程编排，不含业务规则：

- `listTools(filter)` / `getToolDetail(id)`：查询
- `createTool(dto)`：创建工装档案
- `activateTool(id, operatorId)`：投入使用
- `recordUsage(workOrderId, toolId, useCount)`：记录使用（由报工事件触发，计算分摊成本并写入使用记录，更新主表累计字段）
- `startMaintenance(id, dto)` / `completeMaintenance(id, dto)`：维修流程
- `scrapTool(id, operatorId, reason, residualValue)`：报废流程（校验无未完成工单关联）
- `getToolDashboard()`：预警/在用/报废统计看板

### 3.3 基础设施层

- **仓储实现**：`src/infrastructure/repositories/ToolRepository.ts`，基于 Drizzle/mysql2，处理事务（寿命累计与使用记录写入同一事务）。
- **事件总线**：接入现有 `domain_event_outbox` + Stream 机制，保证「寿命累计成功 = 事件必发」。

### 3.4 表现层（API 路由）

路径前缀 `/api/dcprint/tool`，薄控制器，仅参数校验与协议转换：

| 方法 | 路径 | 功能 |
| ---- | ---- | ---- |
| GET | `/` | 工装列表（按类型/状态/预警筛选分页） |
| POST | `/` | 新建工装档案 |
| GET | `/[id]` | 工装详情（含累计统计） |
| PUT | `/[id]` | 编辑工装基础信息 |
| POST | `/[id]/activate` | 投入使用 |
| POST | `/[id]/scrap` | 报废（填原因/残值） |
| GET | `/[id]/usage` | 使用记录列表 |
| POST | `/[id]/maintenance` | 发起/完成维修 |
| GET | `/dashboard` | 工装看板（在用/预警/报废数量、近效预警清单） |

## 四、跨模块联动逻辑

全部通过领域事件解耦，工装模块不直接感知生产/库存/成本模块。

### 4.1 寿命累计联动（报工 → 工装）

生产报工审核通过后发布 `WorkReportedEvent`，工装模块订阅处理：

1. 按报工记录的 `toolIds` 找到关联刀模/网版
2. 累加 `usedCount`（增量 = 本次报工数量），同步递减 `remainLife`
3. 计算本次分摊成本 `= unitCost × useCount`，写入 `dcprint_tool_usage` 快照
4. 累加 `accumulatedCost`、重算 `netValue = originalCost - accumulatedCost`
5. 触发两级预警：
   - 黄色：`remainLife ≤ warningThreshold`（默认额定寿命 80%）→ 状态切「预警」，列表黄色标记
   - 红色：`remainLife ≤ totalLife × 5%` → 强制提醒，工单引用二次确认

> **幂等**：以 `workOrderId + toolId` 为幂等键，重复消费不重复累计。

### 4.2 成本分摊联动（工装 → 工单成本）

报工审核时同步归集工装分摊成本到工单成本明细：

- 基础版：`单次分摊成本 = originalCost ÷ totalLife`，`本次分摊 = 单次分摊 × useCount`
- 进阶版：维修后动态重算 `剩余单位分摊成本 = (netValue + 本次维修费用) ÷ lifeAfter`
- 分摊成本快照固化在使用记录 `amortizedCost`，历史数据不受后续调整影响

### 4.3 报废结转联动（工装 → 成本）

工装报废时发布 `ToolScrappedEvent`，成本模块订阅：

- 剩余未分摊净值 `netValue - 残值` 一次性计入当月制造费用
- 工装从可用列表移除，档案与历史记录永久保留

### 4.4 工单引用校验

- 生产工单/打样工艺卡引用工装时，仅「在用 / 预警」状态可选，维修中/报废自动过滤
- 预警状态二次确认：引用红色预警工单时弹窗确认，留痕

## 五、前端页面设计

### 5.1 工装管理列表页（`/dcprint/tool`）

- 顶部筛选：工装类型（刀模/网版）、状态、预警等级、关键字
- 表格列：编码、名称、类型、规格、状态标签、已用/额定寿命进度条、剩余寿命、净值、操作
- 预警工装行黄色/红色高亮，置顶展示
- 操作：编辑、投入使用、发起维修、报废、查看详情

### 5.2 工装详情页

- **基础信息区**：编码、名称、类型、规格、制作日期、存放位置、原值/累计分摊/净值
- **寿命进度区**：可视化进度条（已用/预警阈值/额定），剩余寿命大字显示
- **使用记录 Tab**：`dcprint_tool_usage` 明细，按时间倒序，可跳转关联工单
- **维修记录 Tab**：`dcprint_tool_maintenance` 明细，展示维修前后寿命、费用、调整量
- **成本曲线**：累计分摊成本随时间趋势

### 5.3 工装看板（`/dcprint/tool` 内嵌或独立）

- 在用/预警/维修中/报废数量统计卡
- 近效预警清单（剩余寿命 Top N）
- 月度维修费用、报废净值结转统计

### 5.4 交互约束

- 报废前校验：若有未完成工单关联，弹窗拦截并提示关联工单号
- 维修完成调整寿命时，必须填写调整原因，前后值留痕
- 已报废工装所有写操作禁用，仅查看归档

## 六、库存模块联动细节补充

工装模块与库存模块的联动较弱，主要是工装存放与物料档案关联：

1. **物料档案关联**：`dcprint_tool.material_id` 关联 `inv_material.id`，工装制作入库时可关联物料扣减（采购外协制作的刀模/网版）。
2. **存放位置**：`warehouse_location` 字段记录工装物理存放位，与仓库库位档案保持编码一致，便于盘点。
3. **盘点联动**：库存盘点时同步展示工装在库情况，工装状态为「待用」的可纳入盘点校验。
4. **不参与批次/成本核算**：工装作为固定资产类管理，不走库存移动加权平均成本，独立按额定寿命均摊，避免与原材料成本混淆。
