## 第二部分：技术架构评估

### 2.1 架构合规性评估

#### DDD 分层合规性：基本到位，但有耦合裂缝

通过对 26 个聚合根的扫描验证，`_domainEvents` 私有字段模式已在所有聚合根（`InkFormulaVersion`、`SampleOrder`、`WorkOrder`、`Tool` 等）中一致落地，说明"聚合根持有领域事件、由应用服务提取并投递 Outbox"的范式已被团队内化为约定。`InkFormulaVersion` 采用"私有字段 + 显式 Props 接口 + 值对象状态（`FormulaStatus`）+ `canTransition` 函数式状态机"的写法，聚合根封装性合格。

但 `BaseRepository`（位于 `src/infrastructure/repositories/BaseRepository.ts`）的实现存在一处**架构层级穿透**：基类直接 `import { query, execute, queryOne } from '@/lib/db'`，使所有继承它的仓储实现都被钉死在 MySQL/Drizzle 上。这与项目硬约束"可通过 `REPOSITORY_IMPL` 环境变量切换 MySQL/Drizzle"形成张力——切换底层实现时必须替换 `BaseRepository` 本体或重写全部子类。**正确的分层应是 `BaseRepository` 依赖一个抽象的 `DbExecutor` 接口**，由具体工厂注入实现。

#### 领域事件统一治理：可落地但语义有瑕疵

`src/domain/shared/events/index.ts` 作为统一 barrel 已聚合采购/销售/仓储/印前/生产/打样六大领域共 40+ 事件，并按模块分块注释。新增的 `ToolUsedEvent`、`ProcessCardConfirmedEvent` 已就位，分别支撑"报工联动工装使用次数累计"与"工艺卡确认触发打样工单生成"两条联动链路。

但事件 barrel 中对 `ReturnOrderCreatedEvent` 的重名冲突处理方式值得关注：

```typescript
export { ReturnOrderCreatedEvent as ProdReturnOrderCreatedEvent, ... }
  from '../../production/events/ReturnOrderEvents';
```

别名只解决了**编译期符号冲突**，运行时事件总线上的 `event.type` 仍是 `return_order.created`，销售退货（退货单）与生产退料（退料单）在事件订阅、日志检索、Outbox 重放时无法区分。**根因是两个限界上下文共用了一个事件类名**，正确做法是生产模块的事件应更名为 `MaterialReturnCreatedEvent`（语义：退料单创建），而非同名 + 别名遮蔽。

#### 仓储模式：通用模板有效，但抽象不彻底

`BaseRepository` 提供 10 个通用方法（findById/findByCode/findList/findAll/insertRow/updateFields/updateStatus/softDelete/existsByCode/countByStatus）+ 2 个 protected 辅助方法（queryPaginatedRaw/buildWhereClause）+ 1 个 abstract mapRow，模板方法模式运用得当。

**但 `buildWhereClause` 的默认实现硬编码了三个假设**：① 所有表都有 `name` 列用于 keyword 模糊匹配；② 所有表都有 `create_time` 字段用于日期范围；③ 所有表都用 `status` 数字字段过滤。这三个假设在工艺卡（`sample_process_card`）、打样反馈（`sal_sample_feedback`）等表上是否成立需要在子类中逐一验证。更稳妥的设计是 `buildWhereClause` 也设为 `abstract`，强制子类显式声明其过滤语义。

### 2.2 技术实现亮点

#### BaseRepository 模板方法：DRY 与扩展性的合理平衡

`insertRow`/`updateFields` 被设计为 `protected` 而非 `public`，强制外部调用必须通过子类语义化方法（如 `save()`、`update()`）发起，避免了上层直接操作通用 SQL 导致的"贫血仓储"陷阱。`queryPaginatedRaw` 留给子类构造复杂联表分页查询的逃生通道，是务实的折中。

#### 事件驱动解耦的闭环验证

`FinishOrderInventoryHandler` 作为 `prod.finish.approved` 事件的消费者，将"完工审核 → 成品入库 → 工单回写"从同步调用解耦为事件订阅，使生产模块无需感知仓储领域。`warehouse/production-inbound/route.ts` 的事件发布与 Handler 的消费形成完整闭环。

#### 幂等保护意识：方向正确

`FinishOrderInventoryHandler` 通过查询 `inv_inventory_transaction` 表中是否已存在 `source_type='prod_finish' AND source_id=?` 的记录来防止双计数，体现了"事件可能被重投递"的分布式系统意识。

#### 状态机的一致建模

打样单 7 态、工单 7 态、油墨配方 3 态、工装 5 态，均统一采用"`canTransition(from, to)` 函数 + 聚合根方法内校验"模式，状态流转规则被收敛在值对象中而非散落在应用服务，聚合根的不变量保护到位。

### 2.3 架构风险与改进建议

#### 风险 1（高）：幂等保护存在 TOCTOU 竞态，且无事务包裹

`FinishOrderInventoryHandler.handle()` 的幂等检查是经典的"先 SELECT 后 INSERT"模式，存在两处缺陷：
- **TOCTOU 竞态**：当 StreamConsumer 的 `XAUTOCLAIM` 回收机制将同一事件重新投递给另一个消费者时，两个消费者可能同时通过 SELECT 检查（均读到 0 行），随后各自执行 INSERT，造成**双计数**。
- **无事务包裹**：SELECT → UPDATE/INSERT inv_inventory → INSERT inv_inventory_transaction 三步不在同一事务内。

**改进建议**：
1. 在 `inv_inventory_transaction` 表上添加 `UNIQUE INDEX (source_type, source_id)`，将幂等性下沉到数据库约束层；
2. 使用 `INSERT ... ON DUPLICATE KEY UPDATE` 或 `INSERT IGNORE` + 检查 `affectedRows` 来原子性地判定"是否首次处理"；
3. 将整个 Handler 逻辑包在 `transaction()` 内；
4. 补充并发用例测试。

#### 风险 2（高）：聚合写入与事件 Outbox 写入未在统一事务内

`SampleOrderApplicationService.createOrder()` 的实现顺序为：

```typescript
const id = await this.orderRepo.save(order);   // 步骤 A：写聚合
if (id) {
  await this.persistEvents(id, order, ctx);    // 步骤 B：写 Outbox
}
```

A、B 是两次独立的数据库操作，未观察到 `transaction()` 包裹。这违反了 Transactional Outbox 模式的**核心契约**：聚合状态变更与事件写入必须在同一事务内，否则会出现：
- A 成功、B 失败 → 聚合已变更但事件丢失，下游永远收不到事件（静默数据不一致）；
- A 失败、B 成功 → 事件已发出但聚合未变更，下游处理时会引用不存在的聚合 ID。

**改进建议**：在 `createOrder`、`submitOrder`、`convertToSalesOrder` 等所有变更聚合的方法中，使用项目已导入的 `transaction()` 将 `repo.save()` 与 `persistEvents()` 包在同一事务内。**当前实现是"伪 Outbox"**。

#### 风险 3（中）：跨聚合事务边界与状态一致性

`SampleOrderApplicationService.createSalesOrderFromSample()` 在单个应用服务中同步完成了两件跨聚合操作：
1. `SampleOrder.convertToSalesOrder()` 状态流转（打样单 → 已转大货）；
2. 在 `sal_order` 表插入新的销售订单记录。

这跨越了"打样"与"销售"两个限界上下文。若不在事务内，则打样单已转大货但销售订单创建失败时，业务上会出现"打样单显示已转单但销售系统查无此单"的故障。

**改进建议**：至少用 `transaction()` 包裹；更优解是改为事件驱动（打样聚合发布 `SampleOrderConvertedEvent`，由销售模块 Handler 异步创建），与项目已建立的事件总线范式保持一致。

#### 风险 4（中）：销售订单号生成存在并发冲突

`createSalesOrderFromSample()` 的订单号生成逻辑：

```typescript
const seqRows = await query<{ cnt: number }>(
  'SELECT COUNT(*) AS cnt FROM sal_order WHERE order_no LIKE ?',
  [`${prefix}%`]
);
const nextSeq = (seqRows[0]?.cnt || 0) + 1;
const orderNo = `${prefix}${String(nextSeq).padStart(4, '0')}`;
```

两个并发请求可能同时读到相同 `cnt`，生成重复订单号。

**改进建议**：① 在 `sal_order.order_no` 上加 `UNIQUE INDEX`；② 改用独立的序列号表（带 `SELECT ... FOR UPDATE` 行锁）或 Redis INCR 生成序号；③ 应用层捕获唯一约束冲突并重试 2-3 次。

#### 风险 5（低）：事件 barrel 中销售退货与生产退料的语义混淆

详见 2.1 节。建议将生产模块的 `ReturnOrderCreatedEvent` 重命名为 `MaterialReturnCreatedEvent`（event.type 改为 `material_return.created`），与销售退货彻底区分。
