# 生产工单 - 领料 - 库存 - 完工入库 全链路完善方案

严格遵循**生产管业务状态、库存管数量成本、事件驱动解耦**的核心原则，完全复用仓储模块成熟的出入库、批次、成本核算能力，生产模块仅负责工单生命周期与业务单据编排，不直接操作任何库存表，保证模块边界清晰、架构不腐化。

---

## 一、核心业务规则与全链路状态流转

### 1.1 完整端到端链路

plaintext

```
生产工单创建 → 工单审核 → 生成领料单 → 领料单审核
    ↓ （事件驱动）
库存模块：原材料出库扣减 + 成本核算 → 回写关联关系
    ↓
生产执行 → 生产报工（累计工时/产量/工装寿命）
    ↓
生成完工入库单 → 完工入库单审核
    ↓ （事件驱动）
库存模块：成品入库增加 + 成本归集 → 回写关联关系
    ↓
工单完工 → 成本核算 → 工单结案
```

### 1.2 生产工单状态机（数字枚举，全局统一）

表格

| 枚举值 | 状态   | 触发条件                     | 可执行操作                       |
| ------ | ------ | ---------------------------- | -------------------------------- |
| 1      | 草稿   | 新建工单                     | 编辑、删除、提交审核             |
| 2      | 已审核 | 工单审核通过                 | 生成领料单、修改、作废           |
| 3      | 领料中 | 首次领料单审核通过           | 继续领料、退料、报工、生成完工单 |
| 4      | 生产中 | 首次生产报工完成             | 报工、领料、退料、完工入库       |
| 5      | 已完工 | 累计完工数量 ≥ 工单计划数量 | 结案、补录退料                   |
| 6      | 已结案 | 成本核算完成、所有单据闭环   | 仅查看                           |
| 7      | 已作废 | 未领料前作废                 | 仅查看归档                       |

 **强制流转规则** ：

1. 已发生领料的工单不可直接作废，必须先完成全部退料后才可作废
2. 已完工工单不可新增领料，仅允许补录退料
3. 已结案工单不可进行任何业务操作，数据永久冻结
4. 支持 **部分领料、部分完工** ：一个工单可对应多张领料单、多张完工入库单，适配分批生产场景

### 1.3 核心单据关联规则

表格

| 生产侧单据                       | 库存侧单据                         | 关联方式                       | 业务类型           |
| -------------------------------- | ---------------------------------- | ------------------------------ | ------------------ |
| 生产领料单 `prod_pick_order`   | 其他出库单 `inv_outbound_order`  | 领料单 ID → 出库单业务关联 ID | 出库类型：生产领料 |
| 生产退料单 `prod_return_order` | 其他入库单 `inv_inbound_order`   | 退料单 ID → 入库单业务关联 ID | 入库类型：生产退料 |
| 完工入库单 `prod_finish_order` | 产成品入库单 `inv_inbound_order` | 完工单 ID → 入库单业务关联 ID | 入库类型：生产完工 |

> 库存侧完全复用现有出入库能力，包括 FIFO 批次分配、移动加权平均成本、软删除、审计日志，生产模块零重复开发。

## 三、DDD 分层与事件驱动解耦实现

### 3.1 分层职责边界

表格

| 层级 / 模块 | 职责                                                         | 禁止操作                             |
| ----------- | ------------------------------------------------------------ | ------------------------------------ |
| 生产领域层  | 工单状态流转、领料 / 退料 / 完工单业务规则校验、数量累计逻辑 | 直接操作库存表、直接调用库存模块接口 |
| 生产应用层  | 工单全流程编排、单据生成、事件发布                           | 包含业务规则、直接修改库存数据       |
| 库存模块    | 出入库执行、批次分配、成本计算、库存数量变更                 | 感知生产工单业务逻辑                 |
| 事件总线    | 跨模块数据同步、状态通知                                     | 承载业务逻辑                         |

### 3.2 核心领域事件定义

生产模块发布 3 类核心事件，库存模块订阅处理，双向解耦：

ts

```
// 1. 领料单审核通过事件 → 库存扣原材料
export class PickOrderApprovedEvent implements DomainEvent {
  readonly eventType = 'prod.pick.approved';
  pickOrderId: bigint;
  pickOrderNo: string;
  workOrderId: bigint;
  items: Array<{
    materialId: bigint;
    quantity: number;
    batchNo?: string; // 可选指定批次，不填则库存自动FIFO分配
  }>;
}

// 2. 退料单审核通过事件 → 库存加原材料
export class ReturnOrderApprovedEvent implements DomainEvent {
  readonly eventType = 'prod.return.approved';
  returnOrderId: bigint;
  workOrderId: bigint;
  sourcePickId: bigint;
  items: Array<{
    materialId: bigint;
    quantity: number;
    batchNo: string; // 退料必须指定原批次
    unitCost: number;
  }>;
}

// 3. 完工入库单审核通过事件 → 库存加成品
export class FinishOrderApprovedEvent implements DomainEvent {
  readonly eventType = 'prod.finish.approved';
  finishOrderId: bigint;
  workOrderId: bigint;
  productId: bigint;
  quantity: number;
  unitCost: number; // 完工核算的单位成本
  totalCost: number;
}
```

### 3.3 反向回调事件（库存→生产回写）

库存处理完成后发布事件，生产模块更新单据状态与成本数据，保证生产侧数据可独立查询：

ts

```
// 出库完成事件 → 生产更新领料单状态与成本
export class OutboundCompletedEvent {
  bizOrderId: bigint; // 对应领料单ID
  outboundOrderId: bigint;
  items: Array<{ materialId: bigint; actualQty: number; unitCost: number; totalCost: number }>;
}

// 入库完成事件 → 生产更新退料单/完工单状态
export class InboundCompletedEvent {
  bizOrderId: bigint; // 对应退料单/完工单ID
  inboundOrderId: bigint;
}
```

### 3.4 一致性与可靠性保障

1. **事务性发件箱** ：生产单据审核与事件发布在同一数据库事务中，保证「单据生效 = 事件必发」，不会出现单据审核了但库存没动的情况
2. **幂等处理** ：所有事件处理器以「单据 ID」为幂等键，重复消费不重复扣加库存、不重复更新状态
3. **失败重试与死信** ：事件处理失败自动重试 3 次，仍失败进入死信队列告警，人工介入，数据最终一致
4. **反向回滚** ：领料单反审核发布 `PickOrderCancelledEvent`，库存自动回补库存，状态同步回退

## 四、关键业务场景细化

### 4.1 领料场景：BOM 自动带出 + 批次策略

#### 基础版（P0 落地）

* 手动录入领料明细，支持按物料搜索选择
* 提交审核时校验物料库存是否充足，不足则拦截报错（遵循负库存禁止规则）
* 库存自动按 FIFO 规则分配批次，无需手动指定

#### 进阶版（P1 迭代）

* 工单关联产品 BOM，生成领料单时自动展开 BOM 带出原材料清单与应领数量
* 支持按完工比例自动计算领料量，支持损耗率自动加成
* 超额领料管控：超过 BOM 数量 ±5% 需二次审批，超过 10% 需特殊审批
* 支持指定批次领料、优先领余料 / 临期料规则

### 4.2 退料场景：原批次退回 + 成本冲减

1. 退料必须关联原领料单，只能退原领料批次的物料，禁止跨批次退料
2. 退料成本按原领料出库成本原路冲减，保证工单材料成本准确
3. 退料审核通过后，自动冲减工单累计已领数量与累计材料成本
4. 剩余未用完的物料必须在工单结案前全部退回，否则不允许结案

### 4.3 完工入库场景：多维度成本自动归集

完工入库审核时自动核算成品成本，写入完工单并传递给库存：

plaintext

```
成品总成本 = 累计材料成本 + 累计人工成本 + 累计工装分摊成本 + 制造费用分摊
单位成本 = 总成本 ÷ 合格完工数量
```

* **材料成本** ：累计领料成本 - 累计退料成本
* **人工成本** ：按报工工时 × 工时工资率计算
* **工装分摊成本** ：刀模 / 网版使用累计分摊成本（联动工装模块）
* **制造费用** ：按工时分摊月度制造费用（联动财务模块）

### 4.4 双向追溯链路

1. **正向追溯（工单→库存）** ：工单详情可查看所有领料单、退料单、完工入库单，点击可直接跳转对应库存单据，查看批次、成本、操作人员
2. **反向追溯（库存→工单）** ：库存批次详情可查看该批次物料被哪些工单领用，成品批次可追溯对应生产工单、用料、操作人员
3. 全链路通过单据 ID 关联，无需冗余数据，查询性能通过索引保障

## 五、异常与边界场景处理

表格

| 场景               | 处理规则                                                                     |
| ------------------ | ---------------------------------------------------------------------------- |
| 领料审核时库存不足 | 直接拦截报错，提示具体缺料物料与缺口数量，不允许部分出库                     |
| 工单中途作废       | 校验是否有已领物料，必须全部退库后才可作废，作废后所有未执行单据自动失效     |
| 超产完工           | 默认允许超产 ±10%，超过需审批；超产部分成本正常归集                         |
| 部分完工多次入库   | 支持一个工单多次完工入库，成本按比例分摊，累计完工达到计划量自动标记完工状态 |
| 领料后物料批次变更 | 以库存实际出库批次为准，生产侧回写实发批次与成本，不强制匹配申请批次         |
| 跨月退料 / 完工    | 成本按实际发生月份归集，不影响已结账月份数据                                 |

### 8.2 关键架构差异：事件驱动 vs 直接调用

本文档第三章核心设计是「生产模块禁止直接操作库存表，所有库存变更通过事件驱动」。**实际实现未遵循此原则**：

- 领料审核：`material-issue/route.ts` 直接调用 `allocateFIFO` + `executeFIFODeductionWithRetry` 扣减库存，未发布 `PickOrderApprovedEvent` 由库存模块订阅
- 退料/完工：同理直接调用库存入库接口，未走 `ReturnOrderApprovedEvent` / `FinishOrderApprovedEvent`
- 反向回写：无 `OutboundCompletedEvent` / `InboundCompletedEvent` 回调，生产单据状态更新内联在事务中

> 实际实现虽功能可用，但破坏了模块解耦边界。后续重构建议：将库存出入库操作收敛到库存模块的事件处理器，生产模块仅发布事件，对齐本文档设计。
